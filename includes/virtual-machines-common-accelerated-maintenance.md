

## <a name="what-is-happening"></a>Что происходит?

[3 января была обнаружена](https://googleprojectzero.blogspot.com/2018/01/reading-privileged-memory-with-side.html) аппаратная уязвимость системы безопасности, касающаяся всей отрасли. Безопасность клиентов всегда имеет для нас высочайший приоритет, и мы делаем все возможное, чтобы защитить каждого клиента Azure от таких уязвимостей.

После раскрытия уязвимости в системе безопасности мы [ускорили периоды планового обслуживания](https://azure.microsoft.com/blog/securing-azure-customers-from-cpu-vulnerability/) и начали автоматическую перезагрузку виртуальных машин, к которым еще не было применено необходимое обновление.


## <a name="how-can-i-see-which-of-my-vms-are-already-updated"></a>Как узнать, какие из моих виртуальных машин уже обновлены? 

Если виртуальные машины уже перезапущены, вы можете просмотреть сведения об их состоянии в [списке виртуальных машин на портале Azure](https://aka.ms/T08tdc). Для виртуальных машин отображается состояние Already updated (Уже обновлена), если обновление выполнено, или Scheduled (Обновление запланировано), если обновление еще требуется. Чтобы просмотреть только виртуальные машины, для которых запланировано обновление, перейдите к службе [Работоспособность служб Azure](https://portal.azure.com/).

## <a name="can-i-find-out-exactly-when-my-vms-will-be-rebooted"></a>Могу ли я узнать точное время перезагрузки виртуальных машин?

Чтобы заранее получать предупреждение о перезапуске, настройте [запланированные события](https://docs.microsoft.com/azure/virtual-machines/windows/scheduled-events). Тогда вы будете получать уведомления за 15 минут до отключений виртуальной машины, связанных с техническим обслуживанием.

## <a name="can-i-manually-redeploy-now-to-perform-the-required-maintenance"></a>Могу ли я вручную выполнить повторное развертывание, чтобы провести необходимое обслуживание? 

Мы не можем гарантировать, что виртуальная машина будет повторно развернута на узле, к которому уже применено обновление. По мере возможности фабрике Azure старается размещать виртуальные машины на уже обновленных узлах. Но существует возможность, что при повторном развертывании виртуальная машина будет развернута на необновленном узле. Тогда в рамках запланированного обслуживания потребуется еще одна перезагрузка виртуальной машины. Поэтому мы не рекомендуем повторное развертывание для решения этой проблемы.

## <a name="how-long-will-the-reboot-take"></a>Сколько времени занимает перезапуск? 

Большинство перезагрузок выполняются приблизительно за **30 минут**.

## <a name="does-the-guest-os-need-to-be-updated"></a>Нужно ли обновлять гостевую ОС? 

Это обновление инфраструктуры Azure устраняет раскрытую уязвимость на уровне гипервизора и не требует обновлять образы виртуальных машин Windows или Linux. Но вам, как и обычно, необходимо соблюдать все рекомендации по безопасности для образов виртуальной машины. Обновления и инструкции, если они потребуются, можно получить у поставщиков используемых операционных систем. Для клиентов виртуальных машин Windows Server опубликовано руководство, с которым можно ознакомиться [здесь](../articles/virtual-machines/windows/mitigate-se.md).

## <a name="will-there-be-a-performance-impact-as-a-result-of-resolving-this-update"></a>Повлияет ли это обновление на производительность?

Большинство клиентов Azure не обнаружили заметного снижения производительности после этого обновления. Мы оптимизировали пути ЦП и дискового ввода-вывода и не заметили существенного снижения производительности после применения этого исправления. У небольшого количества клиентов может незначительно снизиться производительность сети. Эту проблему можно устранить, применив ускорение работы в сети Azure (для [Windows](https://docs.microsoft.com/azure/virtual-network/create-vm-accelerated-networking-powershell) и [Linux](https://docs.microsoft.com/azure/virtual-network/create-vm-accelerated-networking-cli)). Эта бесплатная возможность доступна для всех клиентов Azure.

## <a name="i-follow-your-recommendations-for-high-availability-will-my-environment-remain-highly-available-during-the-reboot"></a>Я выполняю все ваши рекомендации для обеспечения высокого уровня доступности. Сохранится ли высокая доступность моего окружения во время перезагрузки?

Да. Для виртуальных машин, развернутых в группе доступности или масштабируемом наборе виртуальных машин, применяется концепция доменов обновления. Во время обслуживания Azure учитывает ограничения доменов обновления и не выполняет перезагрузку виртуальных машин из разных доменов обновления (в пределах одной и той же группы доступности). Дополнительные сведения об обеспечении высокого уровня доступности см. в руководствах по управлению доступностью виртуальных машин [Windows](https://docs.microsoft.com/azure/virtual-machines/windows/manage-availability) и [Linux](https://docs.microsoft.com/azure/virtual-machines/linux/manage-availability) в Azure.

## <a name="i-have-architected-my-business-continuitydisaster-recovery-plan-using-region-pairs-will-reboots-to-my-vms-occur-in-region-pairs-at-the-same-time"></a>В моем плане обеспечения непрерывности бизнес-процессов (аварийного восстановления) применяются пары регионов. Может ли перезапуск виртуальных машин в регионах такой пары происходить одновременно?

Как правило, мероприятия по плановому обслуживанию Azure выполняются в парных регионах поочередно, чтобы свести к минимуму риск сбоя в обоих регионах. Но из-за срочности этого обновления системы безопасности мы развертываем его во всех регионах одновременно.

## <a name="what-about-paas-services-on-azure"></a>Как это повлияет на службы PaaS в Azure?  

В службах платформы Azure (службах веб-устройств, мобильных устройств, данных, Интернете вещей, бессерверных приложениях и т. д.) эта уязвимость уже устранена. Для клиентов, использующих эти службы, дополнительные действия не требуются.

## <a name="intel-released-additional-guidance-on-january-22-2018-related-to-the-security-vulnerabilities--will-this-guidance-cause-any-additional-maintenance-activities-by-azure"></a>22 января 2018 года компания Intel выпустила дополнительные рекомендации, связанные с уязвимостями системы безопасности.  Требуют ли эти рекомендации дополнительных действий по обслуживанию в Azure?  

Все меры по устранению рисков, объявленные для Azure 3 января 2018 года, остаются действительными после выпуска [обновленных рекомендаций](https://newsroom.intel.com/news/root-cause-of-reboot-issue-identified-updated-guidance-for-customers-and-partners/) корпорации Intel. Эти обновления не требуют дополнительных действий на виртуальных машинах клиентов.
