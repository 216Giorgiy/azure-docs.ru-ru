# Сценарии развертывания и варианты использования
В этом разделе мы рассмотрим некоторые сценарии и варианты использования, демонстрирующие преимущества использования доменных служб Azure Active Directory (AD).

## Простое и безопасное администрирование виртуальных машин Azure
С истечением срока эксплуатации серверов и прочей инфраструктуры компания Contoso перемещает в облако множество приложений, которые в настоящее время размещены локально. Согласно текущему корпоративному стандарту использования ИТ-ресурсов серверы, на которых размещаются корпоративные приложения, должны быть присоединены к домену, а управляться они должны с помощью групповой политики. ИТ-администратор Contoso предпочитает присоединять к домену виртуальные машины, развернутые в Azure, для упрощения администрирования (т. е. администраторы могут входить в систему с использованием корпоративных учетных данных). В интересах Contoso — отсутствие необходимости выполнять развертывание и мониторинг контроллеров домена, а также управлять этими объектами в Azure с целью защиты виртуальных машин Azure.

При рассмотрении этого сценария необходимо отметить ряд важных моментов.

- Управляемые домены, предоставляемые службами доменов Azure AD, поддерживают только плоскую структуру подразделений (организационных единиц). Все присоединенные к домену компьютеры располагаются в едином плоском подразделении, а иерархические структуры подразделений не поддерживаются.
- Службы доменов Azure AD поддерживают простую групповую политику, реализуемую в виде встроенных объектов групповой политики для всех контейнеров пользователей и компьютеров. Нельзя назначить групповую политику отдельным подразделениям или отделам, выполнить фильтрацию WMI или создать настраиваемые объекты групповой политики.
- Службы доменов Azure AD поддерживают базовую схему объекта компьютера AD. Расширить схему объекта компьютера нельзя.


## Перемещение локального приложения, использующего проверку подлинности с привязкой LDAP, в службы инфраструктуры службы Azure
В компании Contoso используется локальное приложение, которое было приобретено у независимого поставщика ПО много лет назад. Приложение в настоящее время обслуживает этот независимый поставщик, и стоимость запроса на внесение в него изменений будет недопустимо высокой для компании. Это приложение имеет внешний веб-интерфейс, который собирает пользовательские учетные данные с помощью веб-формы, а затем проверяет подлинность пользователей, выполняя привязку LDAP к корпоративной службе Active Directory. Компании Contoso нужно перенести это приложение в службы инфраструктуры Azure. Желательно, чтобы приложение работало как есть, без необходимости внесения изменений. Кроме того, пользователи должны иметь возможность проходить проверку подлинности с помощью существующих корпоративных учетных данных, т. е. без необходимости повторного обучения пользователей другим алгоритмам действий. Иными словами, конечным пользователям должно быть все равно, где выполняется приложение, а сама миграция должна быть для них прозрачной.

При рассмотрении этого сценария необходимо отметить ряд важных моментов.

- Убедитесь, что для приложения не требуется вносить изменения или записи в каталог. Доступ на запись LDAP к управляемым доменам (предоставляется службами доменов Azure AD) не поддерживается.
- Пользователи не могут изменить пароль непосредственно на управляемом домене. Пользователи могут изменить пароль либо с помощью механизма изменения паролей Azure AD в режиме самообслуживания, либо в локальном каталоге. Эти изменения автоматически синхронизируются с их последующим применением в управляемом домене.


## Перемещение локального приложения, использующего чтение LDAP для доступа к каталогу, в службы инфраструктуры Azure
В компании Contoso используется локальное бизнес-приложение, которое было разработано около десяти лет назад. Это приложение с поддержкой каталогов было разработано для использования в среде Windows Server AD. Приложение использует протокол LDAP (Lightweight Directory Access Protocol) для чтения атрибутов и информации о пользователях из Active Directory. Приложение не изменяет атрибуты и ничего не записывает в каталог. Компании Contoso необходимо перенести это приложение в службы инфраструктуры Azure и вывести из эксплуатации устаревшее локальное оборудование, на котором в настоящее время это приложение размещено. Приложение нельзя переписать с использованием современных API-интерфейсов каталогов, например Azure AD API Graph на основе REST. Таким образом, требуется перенести приложение для запуска в облаке без необходимости менять код или переписывать приложение.

При рассмотрении этого сценария необходимо отметить ряд важных моментов.

- Убедитесь, что для приложения не требуется вносить изменения или записи в каталог. Доступ на запись LDAP к управляемым доменам (предоставляется службами доменов Azure AD) не поддерживается.
- Убедитесь, что приложению не требуется настраиваемая или расширенная схема Active Directory. Расширения схемы не поддерживаются в службах доменов Azure AD.
- Убедитесь, что приложению не требуется доступ LDAPS. LDAPS не поддерживается службами доменов Azure AD.


## Миграция локальной службы или управляющего приложения в службы инфраструктуры Azure
В компании Contoso используется написанное по заказу приложение — хранилище, состоящее из внешнего веб-интерфейса, SQL-сервера и внутреннего FTP-сервера. При работе с приложением выполняется проверка внешнего веб-интерфейса на FTP-сервере (встроенная проверка подлинности Windows с помощью учетных записей служб). Компании необходимо переместить это приложение в службы инфраструктуры Azure без необходимости развертывать виртуальную машину контроллера домена в облаке для поддержки требуемой приложением идентификации. ИТ-администратор Contoso может развернуть серверы, на которых размещаются внешний веб-интерфейс, SQL-сервер и FTP-сервер, на виртуальных машинах в Azure и присоединить их к домену служб доменов Azure AD. Следовательно, администратор может использовать одну учетную запись службы в каталоге для проверки подлинности приложения.

При рассмотрении этого сценария необходимо отметить ряд важных моментов.

- Убедитесь, что приложение использует имя пользователя и пароль для проверки подлинности. Проверка подлинности на основе сертификатов или смарт-карт не поддерживается службами доменов Azure AD.


## Azure RemoteApp
С помощью Azure RemoteApp администратор Contoso может создавать коллекции, присоединенные к домену. Благодаря этим коллекциям удаленные приложения, обслуживаемые Azure RemoteApp, могут запускаться на компьютерах, присоединенных к домену, а также получать доступ к другим ресурсам с помощью встроенной проверки подлинности Windows. Contoso может использовать доменные службы Azure AD для предоставления управляемого домена, который используется коллекциями Azure RemoteApp, присоединенными к домену.

Дополнительные сведения об этом сценарии развертывания см. в статье блога о службах удаленного рабочего стола [Перемещение рабочих нагрузок с помощью Azure RemoteApp и доменных служб Azure AD](http://blogs.msdn.com/b/rds/archive/2016/01/19/lift-and-shift-your-workloads-with-azure-remoteapp-and-azure-ad-domain-services.aspx).
