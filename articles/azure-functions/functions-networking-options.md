# <a name="azure-functions-networking-options"></a>Функции Azure, вариантах сетевого взаимодействия

Этот документ описывает набор сетевых функций, доступных по функциям Azure, варианты размещения. Все следующие сетевые параметры предоставляют возможность доступа к ресурсам без использования маршрутизируемые адреса Интернета, или ограничить доступ к Интернету к приложению-функции. Все модели размещения имеют различные уровни изоляции сети доступны и выбрав нужную позволит вам в соответствии с требованиями изоляции сети.

Приложения-функции можно размещать несколькими способами.

* Существует ряд параметров плана, которые выполняются на мультитенантную инфраструктуру, с различными уровнями подключение виртуальных СЕТЕЙ и масштабирования.
    1. План потребления, который может масштабироваться динамически в ответ для загрузки и предлагает параметры минимальной сетевой изоляции.
    1. "Премиум" плана, который также масштабируется динамически, предлагая более полный сетевой изоляции.
    1. План службы приложений, который работает на фиксированный масштаб, а также предлагает аналогичные сетевой изоляции в план уровня "премиум".
* Функции также можно запустить на службы приложений среда (ASE) развертывает функции в виртуальную сеть, а также предлагаем полный сетевой контроль и Улучшенная изоляция.

## <a name="inbound-ip-restrictions"></a>Входящий ограничения IP-адресов

Ограничения IP-адресов позволяют определить список IP-адресов, упорядоченных по приоритету, с которых разрешен или запрещен доступ к вашему приложению. Список разрешений может содержать IPv4- и IPv6-адреса. Если он содержит одну или несколько записей, то действует неявный запрет на все адреса в конце этого списка. Ограничения IP-адресов возможность работает начиная с всех вариантах размещения функции.

> [!NOTE]
> Чтобы иметь возможность использовать редактор портала Azure, портал должен иметь возможность получить прямой доступ к выполняющегося приложения-функции, и устройство, которое вы используете для доступа к порталу должен иметь его список разрешенных IP-адрес. Ограничений по сети позволяют по-прежнему обращаться к любой функции в **функции платформы** вкладки.

Дополнительные сведения см. [здесь](https://docs.microsoft.com/azure/app-service/app-service-ip-restrictions).

## <a name="vnet-integration"></a>Интеграция виртуальной сети

Интеграция с виртуальной СЕТЬЮ позволяет приложения-функции для доступа к ресурсам в виртуальной сети. Интеграция VNET доступна как в план "премиум", так и для плана службы приложений. Если приложение находится в среде службы приложений, он уже находится в виртуальной сети и не требует использования функции интеграции с виртуальной сетью для доступа к ресурсам в той же виртуальной сети.

Интеграция с виртуальной сетью предоставляет приложению-функции доступ к ресурсам в виртуальной сети, но не предоставлять [доступ к частным сайтам](#private-site-access) для приложения-функции из виртуальной сети.

Интеграция с виртуальной сетью часто используется для доступа приложений к базам данных и веб-службам, работающим в виртуальной сети. Благодаря интеграции с виртуальной сетью вам не нужно предоставлять общедоступную конечную точку приложениям, запущенным на виртуальной машине. Вместо этого будут использоваться частные адреса, недоступные из Интернета.

Общедоступная версия Интеграция виртуальной сети зависит от VPN-шлюза для подключения приложения-функции к виртуальной сети. Она доступна в функциях, размещенного в плане службы приложений. Узнайте, как настроить эту функцию, см. в разделе [документ службы приложений для той же функции](../app-service/web-sites-integrate-with-vnet.md#enabling-vnet-integration).

### <a name="preview-vnet-integration"></a>Установщик интеграции предварительной версии виртуальной сети

Доступна предварительная версия новой функции интеграции с виртуальной сетью. Она не зависит от сети VPN типа "точка — сеть", а также поддерживает доступ к ресурсам через ExpressRoute и конечные точки служб. Эта функция доступна в план "премиум" и в планах службы приложений масштабируется до категории премиум v2.

Новая версия интеграции с виртуальной сетью, которая в настоящее время в предварительной версии, обеспечивает следующие преимущества:

* Для использования новой функции интеграции виртуальной сети не нужен шлюз.
* Доступ к ресурсам можно получить с помощью подключения ExpressRoute без дополнительной настройки, за исключением интеграции с виртуальной сетью, подключенной при помощи ExpressRoute.
* Вы можете использовать конечную точку службы, защищенным ресурсам из функции. Для этого включите конечные точки служб в подсети, используемой для интеграции с виртуальной сетью.
* Невозможно настроить триггерами для использования конечной точки службы, защищенным ресурсам, благодаря новой возможности интеграции с виртуальной сетью. 
* Приложения-функции и виртуальной сети должен быть в одном регионе.
* Новой функции требуется неиспользуемая подсеть в виртуальной сети Resource Manager.
* Рабочие нагрузки не поддерживаются на новую версию интеграции с виртуальной сетью, находящимся на этапе предварительной версии.
* Таблицы маршрутизации и глобальное пиринговое соединение пока что недоступны для новой функции интеграции с виртуальной сетью.
* Один адрес, используемый для каждой потенциальной экземпляре приложения-функции. Так как размер подсети нельзя изменить после назначения, используйте подсеть, которая может легко поддерживать размер максимальный масштаб. Например, чтобы обеспечить поддержку план "премиум", который можно масштабировать до 80 экземпляров, мы рекомендуем `/25` подсети, которая предоставляет 126 адреса узла.

Дополнительные сведения об использовании интеграции с виртуальной СЕТЬЮ предварительной версии, см. в разделе [интегрировать приложение-функцию с виртуальной сетью Azure](functions-create-vnet.md).

## <a name="hybrid-connections"></a>Гибридные подключения

[Гибридные подключения](../service-bus-relay/relay-hybrid-connections-protocol.md) входит в состав ретранслятора Azure, который может использоваться для доступа к ресурсам приложений в других сетях. Они обеспечивают доступ из вашего приложения к конечной точке приложения. Он не может использоваться для доступа к приложению. Гибридные подключения доступен для функций, выполняемых в [план службы приложений](functions-scale.md#app-service-plan) и [среды службы приложений](../app-service/environment/intro.md).

При использовании в функциях, каждое гибридное подключение сопоставляется с одной комбинацией узла и порта TCP. Это означает, что конечная точка гибридного подключения может быть размещена в любой операционной системе, а любое ваше приложение использует TCP-порт прослушивания. Гибридным подключениям не важен используемый протокол приложения или ресурсы, к которым вы обращаетесь. Они просто обеспечивают доступ к сети.

Дополнительные сведения см. в разделе [документации по службе приложений для гибридных подключений](../app-service/app-service-hybrid-connections.md), который поддерживает функции и веб-приложений.

## <a name="private-site-access"></a>доступ к частным сайтам;

Доступ к частным сайтам подразумевает доступность приложения только из частной сети, например из виртуальной сети Azure. Доступ к частным сайтам возможен, только если для ASE настроен внутренний балансировщик нагрузки (ILB). Дополнительные сведения об использовании ILB ASE см. в разделе [Создание и использование ASE с ILB](../app-service/environment/create-ilb-ase.md).

Существует много способов доступа к ресурсам виртуальной сети в другие варианты размещения, но ASE — это единственный способ Разрешить триггерам для функции по виртуальной сети.
