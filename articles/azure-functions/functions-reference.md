# Справочник разработчика по функциям Azure

Функции Azure имеют ряд общих технических особенностей и компонентов вне зависимости от используемого языка или привязки. Прежде чем приступать к детальному изучению особенностей, характерных для определенного языка или привязки, ознакомьтесь с этими общими сведениями.

В этой статье предполагается, что вы уже прочитали [обзор функций Azure](functions-overview.md) и знакомы с [понятиями, связанными с пакетом SDK веб-заданий, такими как триггеры, привязки и среда выполнения JobHost](../app-service-web/websites-dotnet-webjobs-sdk.md). Функции Azure основаны на пакете SDK веб-заданий.

## function

Ключевым элементом решения "Функции Azure" является *функция*. Вы пишете код функции на любом языке на свой выбор, а затем сохраняете файлы кода и файл конфигурации в одной папке. Файл конфигурации имеет формат JSON и имя `function.json`. Поддерживаются различные языки, и способы работы с каждым из них немного различаются в целях удобства. Пример структуры папок:

```
mynodefunction
| - function.json
| - index.js
| - node_modules
| | - ... packages ...
| - package.json
mycsharpfunction
| - function.json
| - run.csx
```

## function.json и привязки

Файл `function.json` содержит конфигурацию, относящуюся к функции, включая привязки. Среда выполнения считывает этот файл, чтобы определить, какие события следует активировать, какие данные включить при вызове функции и куда отправить данные, переданные из самой функции.

```json
{
    "disabled":false,
    "bindings":[
        // ... bindings here
        {
            "type": "bindingType",
            "direction": "in",
            "name": "myParamName",
            // ... more depending on binding
        }
    ]
}
```

Чтобы запретить среде выполнения запускать функцию, присвойте свойству `disabled` значение `true`.

В свойстве `bindings` указываются триггеры и привязки. Каждая привязка имеет ряд общих параметров и некоторые параметры, характерные для данного типа привязки. Для каждой привязки требуются указанные ниже параметры.

|Свойство|Значения и типы|Комментарии|
|---|-----|------|
|`type`|string|Тип привязки. Пример: `queueTrigger`.
|`direction`|"in", "out"| Указывает, служит ли привязка для получения данных в функции или для отправки их из функции.
| `name` | строка | Имя, которое будет использоваться для связанных данных в функции. Для C# это будет имя аргумента, а для JavaScript — ключ в списке ключей и значений.

## Среда выполнения (средство обработки сценариев и веб-узел)

Среда выполнения, также называемая средством обработки сценариев, является базовым узлом SDK веб-заданий, который прослушивает события, собирает и отправляет данные и, наконец, выполняет ваш код.

Чтобы упростить инициацию триггеров HTTP, также используется веб-узел, который размещается перед средством обработки сценариев в рабочих средах. Это позволяет изолировать средство обработки сценариев от трафика переднего плана, которым управляет веб-узел.

## Структура папок

Средство обработки сценариев указывает на папку, содержащую файл конфигурации и одну или несколько функций.

```
parentFolder (for example, wwwroot)
 | - host.json
 | - mynodefunction
 | | - function.json
 | | - index.js
 | | - node_modules
 | | | - ... packages ...
 | | - package.json
 | - mycsharpfunction
 | | - function.json
 | | - run.csx
```

Файл *host.json* содержит данные конфигурации, относящиеся к средству обработки сценариев, и находится в родительской папке.

У каждой функции есть папка с файлами кода, файлом *function.json* и другими зависимостями.

При настройке проекта для развертывания функций в приложении функции в службе приложений Azure эту структуру папок можно рассматривать как код сайта. Для установки пакетов или транспиляции кода во время развертывания можно использовать существующие средства, такие как непрерывная интеграция и развертывание, или пользовательские сценарии развертывания.

## Параллельное выполнение

Если однопотоковая среда выполнения функций не успевает обрабатывать активируемые события, она может вызывать функцию несколько раз в параллельном режиме. Если для приложения функции используется [динамический план обслуживания](functions-scale.md#dynamic-service-plan), то оно может развертываться в параллельных экземплярах, количество которых.до 10. Каждый экземпляр приложения функции, как при использовании динамического плана обслуживания, так и при использовании обычного [плана обслуживания приложения](../app-service/azure-web-sites-web-hosting-plans-in-depth-overview.md), может обрабатывать несколько параллельных вызовов функции с помощью нескольких потоков. Максимальное число параллельных вызовов функции в каждом экземпляре приложения функции зависит от размера памяти этого приложения.

## Пульс в Функциях Azure  

Пульс — это динамический поток событий, в котором показано, как часто выполняются функции, а также указывается, успешно ли они были выполнены. Вы также можете отслеживать среднее время выполнения. Со временем функциональность этого потока будет расширяться. Получить доступ к странице **Пульс** можно на вкладке **Мониторинг**.
