# Импорт данных в службу поиска Azure
> [AZURE.SELECTOR]
- [Overview](search-what-is-data-import.md)
- [Portal](search-import-data-portal.md)
- [.NET](search-import-data-dotnet.md)
- [REST API](search-import-data-rest-api.md)
- [Indexers](search-howto-connecting-azure-sql-database-to-azure-search-using-indexers-2015-02-28.md)

Поиск Azure — это размещенная облачная служба поиска, включающая поисковую систему, функции поиска и хранилище данных. Служба работает с сохраненными данными (индексом), в которых содержатся документы и сведения, используемые для обработки индекса, выполнения запросов и формулирования результатов поиска. Среди этих высокоуровневых операций есть и определенные низкоуровневые операции: для анализа текста, ранжирования, фильтрации, сортировки, уточнения и других функций, используемых для вычисления и отображения результатов поиска.

Релевантность результатов поиска зависит от качества передаваемых данных и требуемой частоты их обновления. В этой статье описывается передача и извлечение (сканирование) данных для импорта и обновления индекса.

Индекс должен существовать до начала импорта данных, а передаваемые данные должны соответствовать схеме. Дополнительные сведения см. в статье [Индексы в службе поиска Azure Search](search-what-is-an-index.md).

##Рекомендации по наборам данных

Если схема и наборы данных сформулированы в виде структур JSON, ограничения для типа передаваемых данных отсутствуют.

Источником загружаемых данных может быть любая база или источник данных, созданные или используемые вашим пользовательским приложением. Например, если приложение представляет собой каталог интернет-магазина, индекс, созданный для службы поиска Azure, должен получать данные из номенклатуры товаров или баз данных продаж, поддерживающих ваше приложение.

Индекс службы поиска Azure получает данные из одной таблицы, представления, контейнера больших двоичных объектов или аналогичного источника. Возможно, вам потребуется создать в своей базе данных или приложении NoSQL структуру данных, предоставляющую данные для службы поиска Azure. Для некоторых источников данных, например базы данных SQL Azure или DocumentDB, можно создать индексатор, который сканирует внешнюю таблицу, представление или контейнер больших двоичных объектов, на предмет данных, которые следует передать в службу поиска Azure.

##Требования к задержке и синхронизации данных

В следующей таблице приведен перечень общих требований и рекомендации по их соблюдению.

|Требования|Рекомендации|
|------------|---------------|
|Синхронизация данных практически в реальном времени|Код (.NET или REST API) для передачи обновлений в индекс. Прием данных с помощью извлечения является запланированной операцией, скорость которой недостаточна для того, чтобы своевременно отображать изменения в первичном источнике данных.|
|База данных SQL Azure, DocumentDB или SQL Server на виртуальных машинах Azure|Индексаторы привязываются к конкретным типам источников данных. Если первичные источники данных находятся в поддерживаемом источнике данных, вы сможете сканировать источник данных с помощью индексатора и запланировать обновление данных каждый час. Индексатор можно настроить на портале или в коде.|
|Обновление данных по расписанию|Используйте индексатор (см. выше).|
|Программа-обходчик|Используйте индексатор (см. выше).|
|Создание прототипов или редактирование без использования кода|На портале есть мастер импорта данных, предназначенный для настройки индексатора. Иногда мастер создает предварительные схемы, если база данных-источник содержит достаточно соответствующих сведений. Мастер включает параметры для настройки обновления данных по расписанию. При необходимости можно добавить анализаторы языка или параметры CORS. Недостатки: невозможно добавлять профили повышения, а также экспортировать схему, созданную на портале, в JSON-файл для использования в коде.|
