# Импорт данных в службу поиска Azure
> [AZURE.SELECTOR]
- [Overview](search-what-is-data-import.md)
- [Portal](search-import-data-portal.md)
- [.NET](search-import-data-dotnet.md)
- [REST API](search-import-data-rest-api.md)
- [Indexers](search-howto-connecting-azure-sql-database-to-azure-search-using-indexers-2015-02-28.md)

Служба поиска Azure работает с сохраненными данными (индексом). Индекс содержит документы и сведения, которые служба использует для обработки данных, выполнения запросов и подготовки результатов поиска. Для заполнения индекса можно использовать модели передачи или извлечения данных.

Импорт выполняется в уже существующий индекс. Дополнительные сведения см. в статье [Индексы в службе поиска Azure](search-what-is-an-index.md).

##Передача данных в индекс

Так называется процесс передачи в службу поиска уже существующего набора данных, который соответствует схеме индекса. Для приложений с очень высокими требованиями к задержкам обработки (например, если операции поиска должны выполняться синхронно с работой учетной базы данных) модель передачи будет единственным вариантом.

Передать данные в индекс можно с использованием интерфейса API REST или .NET SDK. Инструментов для передачи данных через портал пока не существует.

Передача является более гибким подходом, чем извлечение, поскольку документы можно передавать по одному или пакетами (не более 1000 документов и не более 16 МБ в каждом пакете).

##Извлечение (сканирование) данных 

Модель извлечения данных подразумевает сканирование источника данных для автоматического заполнения индекса. Поиск Azure реализует эту возможность с помощью *индексаторов*, которые в настоящее время поддерживают базы данных Azure SQL, DocumentDB и SQL Server на виртуальных машинах Azure. Раздел, посвященный [индексаторам](search-howto-connecting-azure-sql-database-to-azure-search-using-indexers-2015-02-28.md), даст вам дополнительные сведения о передаче данных в Azure SQL.

Извлечь данные в индекс можно с использованием портала, интерфейсов API REST или .NET SDK.

##Требования к набору данных

Если схема и наборы данных сформулированы в виде структур JSON, ограничения для типа передаваемых данных отсутствуют.

Источником данных может быть любая база данных или другой источник данных, который создает или использует ваше приложение. Например, если приложение представляет собой каталог интернет-магазина, индекс, созданный для службы поиска Azure, должен получать данные из номенклатуры товаров или баз данных продаж, поддерживающих ваше приложение.

Набор данных можно получать из одной таблицы, представления, контейнера больших двоичных объектов или аналогичного источника. Возможно, вам потребуется создать в своей базе данных или приложении NoSQL структуру данных, предоставляющую данные для службы поиска Azure. Для некоторых источников данных, например базы данных SQL Azure или DocumentDB, можно создать индексатор, который сканирует внешнюю таблицу, представление или контейнер больших двоичных объектов, на предмет данных, которые следует передать в службу поиска Azure.

##Выбор подхода к импорту данных

|Критерии|Рекомендуемый подход|
|------------|---------------|
|Синхронизация данных практически в реальном времени|Код (.NET или REST API) для передачи обновлений в индекс. Прием данных с помощью извлечения является запланированной операцией, скорость которой недостаточна для того, чтобы своевременно отображать изменения в первичном источнике данных.|
|База данных SQL Azure, DocumentDB или SQL Server на виртуальных машинах Azure|Индексаторы привязываются к конкретным типам источников данных. Если нужные данные хранятся в источниках поддерживаемого типа, индексатор будет самым простым способом заполнить индекс. Вы можете запланировать регулярное обновление данных, вплоть до ежечасного. Индексатор можно настроить на портале или в коде.|
|Обновление данных по расписанию|Используйте индексатор (см. выше).|
|Создание прототипов или редактирование без использования кода|На портале есть мастер импорта данных, предназначенный для настройки индексатора. Иногда мастер создает предварительные схемы, если база данных-источник содержит достаточно соответствующих сведений. Мастер включает параметры для настройки обновления данных по расписанию. При необходимости можно добавить анализаторы языка или параметры CORS. Недостатки: невозможно добавлять профили повышения, а также экспортировать схему, созданную на портале, в JSON-файл для использования в коде.|
