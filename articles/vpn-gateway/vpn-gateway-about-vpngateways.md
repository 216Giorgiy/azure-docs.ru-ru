# Основные сведения о VPN-шлюзах Azure

VPN-шлюз — это набор параметров, настраиваемых для обмена сетевым трафиком между виртуальными и локальными сетями. В этой статье рассматриваются параметры, связанные с VPN-шлюзом. VPN-шлюз сети используется для подключений типа "сайт — сайт" и "точка — сайт", а также подключений ExpressRoute. Он также используется для обмена трафиком между несколькими виртуальными сетями в Azure (подключения типа "виртуальная сеть — виртуальная сеть").

Чтобы создать подключение, добавьте VPN-шлюз к виртуальной сети. У каждой виртуальной сети может быть только один VPN-шлюз. Кроме того, каждое подключение предусматривает специальную настройку. Схемы подключения см. в статье [Топологии подключения через VPN-шлюз Azure](vpn-gateway-topology.md).

## <a name="gwsku"></a>SKU шлюзов

Во время создания шлюза VPN нужно указать SKU, который вы хотите использовать для шлюза. SKU шлюзов применяются к обоим типам шлюзов: ExpressRoute и VPN. Цены на шлюзы зависят от артикулов. Сведения о ценах см. в разделе [Стоимость VPN-шлюзов ](https://azure.microsoft.com/pricing/details/vpn-gateway/). Дополнительные сведения об ExpressRoute см. в [техническом обзоре ExpressRoute](../expressroute/expressroute-introduction.md).

Существует 3 SKU VPN-шлюза:

- базовая;
- Standard
- HighPerformance.

В примере ниже для `-GatewaySku` задано значение *Standard*.

	New-AzureRmVirtualNetworkGateway -Name vnetgw1 -ResourceGroupName testrg -Location 'West US' -IpConfigurations $gwipconfig -GatewaySku Standard -GatewayType Vpn -VpnType RouteBased

## <a name="gwtype"></a>Типы шлюзов

По типу шлюза определяется, как подключается шлюз. Это обязательный параметр конфигурации для модели развертывания с помощью Resource Manager. Тип шлюза не следует путать с типом VPN, по которому определяется тип маршрутизации в VPN. Для `-GatewayType` доступны такие значения:

- Vpn
- ExpressRoute


В этом примере модели развертывания Resource Manager для параметра -GatewayType задано значение *Vpn*. При создании шлюза необходимо убедиться, что в конфигурации указан правильный тип шлюза.

	New-AzureRmVirtualNetworkGateway -Name vnetgw1 -ResourceGroupName testrg -Location 'West US' -IpConfigurations $gwipconfig -GatewayType Vpn -VpnType RouteBased

## <a name="connectiontype"></a>Типы подключения

Для каждой конфигурации необходим конкретный тип подключения. Для `-ConnectionType` в PowerShell можно указать такие значения (развертывание Resource Manager):

- IPsec
- Vnet2Vnet
- ExpressRoute
- VPNClient

В следующем примере создается подключение "сайт — сайт", которое требует подключения типа IPsec.

	New-AzureRmVirtualNetworkGatewayConnection -Name localtovon -ResourceGroupName testrg -Location 'West US' -VirtualNetworkGateway1 $gateway1 -LocalNetworkGateway2 $local -ConnectionType IPsec -RoutingWeight 10 -SharedKey 'abc123'

## <a name="vpntype"></a>Типы VPN

Чтобы VPN работала, необходимо указать ее тип в конфигурации. Если необходимо совместить две конфигурации, например создать подключения типа "сеть — сеть" и "точка — сеть" в одной виртуальной сети, нужно указать тип VPN, который отвечает требованиям обоих подключений.

При конфигурации, предусматривающей сосуществование подключений типа "точка — сеть" и "сеть — сеть", вам необходимо задать VPN на основе маршрутов, если используете модель развертывания с помощью Azure Resource Manager, или динамический шлюз, если используете классический режим развертывания.

Во время создания конфигурации нужно выбрать тип VPN, требуемый в зависимости от подключения.

Существует два типа VPN:

[AZURE.INCLUDE [vpn-gateway-vpntype](../../includes/vpn-gateway-vpntype-include.md)]

В этом примере модели развертывания Resource Manager для `-VpnType` задан тип *RouteBased*. При создании шлюза необходимо убедиться, что в конфигурации указано правильное значение -VpnType.

	New-AzureRmVirtualNetworkGateway -Name vnetgw1 -ResourceGroupName testrg -Location 'West US' -IpConfigurations $gwipconfig -GatewayType Vpn -VpnType RouteBased

## <a name="gwsub"></a>Подсеть шлюза

Чтобы настроить VPN-шлюз, сначала нужно создать подсеть шлюза для виртуальной сети. Чтобы подсети шлюзов работали правильно, каждую подсеть нужно назвать *GatewaySubnet*. Это имя сообщит Azure, что подсети должны использоваться для шлюза.<BR>Если вы используете классический портал, подсеть шлюза будет автоматически отображаться в интерфейсе портала под именем *Gateway*. Это примечание касается только просмотра подсети шлюзов на классическом портале. Подсеть фактически создается в Azure под именем *GatewaySubnet*, и под этим именем она отображается на портале Azure и в PowerShell.

Минимальный размер подсети шлюза зависит исключительно от того, какую конфигурацию вы хотите создать. Хотя можно создать подсеть шлюза размером /29, мы рекомендуем создавать подсети размером /28 или больше (/28, /27, /26 и т. д.).

Создание шлюза большего размера предотвращает превышение ограничений его размера при работе. Например, если вы создали шлюз с подсетью размером /29 и хотите настроить конфигурацию с сосуществованием VPN типа "сеть — сеть" и ExpressRoute, придется удалить шлюз и подсеть шлюза, создать подсеть размером /28 или больше, а затем снова создать шлюз.

Если вы изначально создадите подсеть шлюза большего размера, то сэкономите время при добавлении новых возможностей конфигурации в сетевую среду.

В примере ниже показана подсеть шлюза с именем GatewaySubnet. Здесь приведена нотация CIDR, указывающая размер /27, при котором можно использовать достаточно IP-адресов для большинства конфигураций, которые уже существуют.

	Add-AzureRmVirtualNetworkSubnetConfig -Name 'GatewaySubnet' -AddressPrefix 10.0.3.0/27

>[AZURE.IMPORTANT] Убедитесь, что у GatewaySubnet нет никаких групп безопасности сети (NSG), так как из-за этого подключение может не работать.



## <a name="lng"></a>Локальные сетевые шлюзы

Обычно термин "шлюз локальной сети" означает локальное расположение. В классической модели развертывания шлюз локальной сети называется "локальным сайтом". Для локального сетевого шлюза нужно задать имя и общий IP-адрес локального VPN-устройства, а также указать префиксы адресов, принадлежащие к локальному расположению. Azure будет проверять наличие сетевого трафика по префиксам адресов назначения, учитывать конфигурацию, указанную для локального сетевого шлюза, и соответствующим образом направлять пакеты. При необходимости эти префиксы адресов можно изменить.


### Изменение префиксов адресов с помощью Resource Manager

Процедура изменения префиксов адресов зависит от того, создан ли уже VPN-шлюз. См. раздел об [изменении префиксов адресов для шлюза локальной сети](vpn-gateway-create-site-to-site-rm-powershell.md#modify).

В следующем примере вы можете увидеть заданный локальный сетевой шлюз MyOnPremiseWest, у которого будет два префикса IP-адресов.

	New-AzureRmLocalNetworkGateway -Name MyOnPremisesWest -ResourceGroupName testrg -Location 'West US' -GatewayIpAddress '23.99.221.164' -AddressPrefix @('10.0.0.0/24','20.0.0.0/24')	

### Изменение префиксов адресов при классическом развертывании

Если при использовании классической модели развертывания вам нужно изменить локальные сайты, это можно сделать на классическом портале на странице конфигурации локальных сетей. Кроме того, это можно сделать в файле конфигурации сети NETCFG.XML.
