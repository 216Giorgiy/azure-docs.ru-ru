# Сценарии приложений структуры служб

Структура служб и Azure предлагают надежную и гибкую облачную инфраструктурную платформу, которая позволяет запускать множество типов бизнес-приложений и служб. Эти приложения и микрослужбы могут регистрировать или не регистрировать состояния, при этом для них выполняется балансировка ресурсов на используемых виртуальных машинах для обеспечения максимальной эффективности. Уникальная архитектура структуры служб позволяет выполнять анализ данных в режиме, близком к режиму реального времени, вычисления в памяти, параллельные транзакции и обработку событий для ваших приложений. Вы можете без труда масштабировать приложения в зависимости от изменяющихся требований к ресурсам.

Платформа структуры служб в Azure является идеальной для следующих категорий приложений и служб.

- **Высокодоступные службы**. Службы структуры служб обеспечивают исключительно быструю отработку отказа. Доступными остаются несколько вторичных реплик службы. После выхода узла из строя из-за сбоя аппаратного обеспечения одна из вторичных реплик немедленно становится первичной при ничтожно малой приостановке обслуживания клиентов. Службы можно без труда масштабировать от нескольких экземпляров до тысяч с последующим обратным масштабированием до нескольких экземпляров в зависимости от потребностей в ресурсах. Вы можете использовать структуру служб для создания жизненного цикла этих масштабируемых облачных служб и управления им.

- **Масштабируемые службы**. Индивидуальные службы можно разделить на разделы для обеспечения возможности масштабирования в кластере. Кроме того, в структуре служб можно создавать и масштабировать множество служб.
 
- **Вычисление по нестатичным данным**. Структура служб позволяет создавать приложения с регистрацией состояний с интенсивной вычислительной нагрузкой и нагрузкой на операции ввода-вывода. Ресурсы обработки и данных в приложениях структуры служб размещаются вместе, поэтому, когда вашему приложению необходимо осуществлять быстрые операции чтения и записи, исключается сетевая задержка, которая обычно возникает при использовании внешнего кэша данных или уровня хранения. Например, предположим, имеется приложение, выполняющее выбор рекламных материалов в режиме, близком к режиму реального времени, с задержкой при взаимном обмене данными < 100 мс, в котором правила вычисления выбора рекламных материалов обеспечивают эффект отклика с точки зрения пользователя.
 
- **Интерактивные приложения, использующие сеансы**. Структура службы полезна, если ваши приложения, такие как онлайн-игры или приложения для обмена мгновенными сообщениями, требуют низких значений задержки при операциях чтения и записи. Структура службы позволяет вам создавать эти интерактивные приложения с регистрацией состояний без необходимости создавать отдельное хранилище или кэш, необходимый для приложений без регистрации состояний.
 
- **Распределенная обработка графики**. Рост социальных сетей обеспечил спрос на параллельный анализ крупных графических объектов. Быстрое масштабирование и параллельная обработка нагрузки делают структуру службы естественной платформой для обработки крупных графических объектов. Структура службы позволяет строить высокомасштабируемые службы для таких групп, как взаимодействие в социальных сетях, бизнес-аналитика и научные исследования.
 
- **Анализ данных и рабочие процессы**. Быстрота операций чтения и записи, которыми характеризуется структура службы, позволяет приложениям, которым это необходимо, надежно обрабатывать события или потоки данных. Структура службы также позволяет обеспечивать работу приложений, описывающих конвейер обработки, где результаты должны быть надежными для последующей передачи без потерь на следующую ступень обработки, в том числе в транзакционных и финансовых системах, в которых исключительной важностью обладает целостность данных и гарантии правильности вычислений.

## Разработка приложений, состоящих из микрослужб, регистрирующих и не регистрирующих состояния ##
Создание приложений с ролями Worker в облачной службе Azure является примером служб, не регистрирующих состояния. И наоборот, микрослужбы с отслеживанием состояния сохраняют утвержденное состояние за пределами запроса и ответа на него, обеспечивая репликацию состояний при помощи простых API. Службы с отслеживанием состояния представляют собой демократический вариант реализации высокого уровня доступности для всех систем, а не только для баз данных или других хранилищ данных. Такая реализация обеспечивает естественное развитие в проектировании, поскольку приложения отошли от использования чисто реляционных баз данных для обеспечения высокой доступности и перешли к базам данных NoSQL, а теперь и к самим приложениям, состояние которых управляется в них для "горячих" данных, которые необходимо держать в доступности для вычисления.

При создании приложений, состоящих из микрослужб, обычно имеется сочетание веб-приложений без регистрации состояния (ASP.NET, node.js и т. п.), вызывающих службы среднего уровня как с регистрацией состояния, так и без нее, все из которых развернуты в одной и той же структуре служб при помощи команд структуры служб. Независимость масштаба, надежности и использования ресурсов для каждой из этих микрослужб обеспечивает гибкость при разработке жизненного цикла и управлении им.
  
Микрослужбы с регистрацией состояния упрощают разработку приложений, поскольку устраняют необходимость в дополнительных очередях и кэше, которые традиционно требовались для обеспечения доступности соответствия требованиям по задержке для чистых приложений без регистрации состояний. Поскольку службы с регистрацией состояний естественным образом обладают высокой доступностью и низкими значениями задержки, это означает меньшее количество движущихся частей в вашем приложении в целом. На диаграммах ниже представлены различия между проектированием приложения без регистрации состояния и с регистрацией состояния. В случае приложений с отслеживанием состояния, использованием моделей программирования [Reliable Services](../Service-Fabric/service-fabric-reliable-services-introduction.md) и [Reliable Actors](service-fabric-reliable-actors-introduction.md), службы с отслеживанием состояния снижают сложность приложений, одновременно обеспечивая высокую пропускную способность и низкие значения задержки.
