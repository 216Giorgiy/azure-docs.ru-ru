## <a id="WhatIs"> </a>Что такое блок приложения автомасштабирования?

Блок приложения автомасштабирования позволяет автоматически масштабировать приложение Microsoft Azure на основе правил, которые определяются специально для приложения. Эти правила можно использовать, чтобы приложение Azure поддерживало свою пропускную способность в ответ на изменения своей рабочей нагрузки, а также чтобы управлять расходами, связанными с размещением приложения в Azure. Наряду с масштабированием путем увеличения или уменьшения количества экземпляров ролей в приложении этот блок также позволяет использовать другие действия масштабирования, например регулирование определенных функциональных возможностей приложения или использование пользовательских действий.

Можно выбрать размещение блока в роли Azure или в локальном приложении.

Блок приложения автоматического масштабирования является частью [Microsoft Enterprise Library 5.0 Integration Pack для Azure][].

## <a id="Concepts"> </a>Основные понятия

На следующей схеме зеленая линия является проекцией числа работающих экземпляров роли Azure за последние два дня. Число экземпляров меняется автоматически с течением времени в ответ на набор правил автомасштабирования.

![схема примерного автомасштабирования](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling01.png)

Блок использует два типа правил для определения поведения автоматического масштабирования приложения.

-   **Правила ограничений.** Чтобы задать верхнюю и нижнюю границы числа экземпляров, например между 8:00 и 10:00 каждого утра, вам необходимо не менее четырех и не более шести экземпляров, поэтому вы можете использовать **правило ограничения**. На схеме красная и синяя линии представляют правила ограничений. Например, в точке **A** на схеме минимальное число экземпляров ролей поднимается с двух до четырех, чтобы удовлетворить ожидаемое увеличение рабочей нагрузки на приложение в это время. В точке **B** на схеме запрещается увеличивать число экземпляров ролей более пяти, чтобы контролировать эксплуатационные расходы на приложение.

-   **Реактивные правила.** Чтобы количество экземпляров менялось в ответ на непрогнозируемые изменения спроса, следует использовать **реактивные правила**. В точке **C** на схеме блок автоматически уменьшает число экземпляров роли с четырех до трех в ответ на уменьшение рабочей нагрузки. В точке **D** блок обнаруживает увеличение рабочей нагрузки и автоматически повышает количество работающих экземпляров ролей с трех до четырех.

Блок сохраняет свои параметры конфигурации в двух хранилищах:

-   **Хранилище правил.** В хранилище правил находится ваша бизнес-конфигурация — список всех правил автомасштабирования, определенных для вашего приложения Azure. Как правило, это хранилище использует XML-файл, который расположен в месте, доступном для чтения блоку приложения автомасштабирования, например в хранилище больших двоичных объектов Azure или в локальном файле.

-   **Хранилище служебных сведений.** В хранилище служебных сведений размещена ваша рабочая конфигурация, которая является моделью службы вашего приложения Azure. Эта модель службы включает в себя все сведения о приложении Azure (например, имена ролей и сведения об учетной записи хранения), которые необходимы блоку для сбора данных из целевого приложения Azure и выполнения операций масштабирования.

## <a id="PerfCounter"> </a>Сбор данных счетчика производительности из целевого приложения Azure

Реактивные правила могут использовать данные счетчиков производительности ролей в рамках определения правила. Например, реактивное правило может отслеживать использование ЦП ролью Azure для определения, следует ли блоку инициировать операцию по масштабированию. Блок считывает данные счетчиков производительности из таблицы диагностики Azure с именем **WADPerformanceCountersTable**, которая размещена в хранилище Azure.

По умолчанию Azure не записывает данные счетчиков производительности в таблицу диагностики Azure в хранилище Azure. Таким образом, следует изменить роли, из которых можно брать данные счетчиков производительности для сохранения этих данных. Дополнительные сведения о включении счетчиков производительности в приложении см. в разделе [Использование счетчиков производительности в Azure][].

## <a id="CreateHost"> </a>Настройка ведущего приложения для блока приложения автомасштабирования

Можно выбрать размещение блока приложения автомасштабирования в роли Azure или в локальном приложении. Блок приложения автомасштабирования, как правило, размещается в отдельном от целевого приложении, которое необходимо автоматически масштабировать. В этом разделе представлены рекомендации по настройке ведущего приложения.

### Получите пакет NuGet для блока приложения автомасштабирования

Прежде чем использовать блок приложения автомасштабирования в проекте Visual Studio, необходимо получить двоичный код блока приложения автомасштабирования и добавить ссылки на него в свой проект. Расширение NuGet для Visual Studio упрощает установку и обновления библиотек и инструментов в Visual Studio и Visual Web Developer. Пакет NuGet блока приложения автомасштабирования является самым простым способом получения интерфейсов API блока приложения автомасштабирования. Дополнительные сведения о **NuGet**, а также об установке и использовании расширения **NuGet** для Visual Studio см. на веб-сайте [NuGet][].

После установки диспетчера пакетов NuGet для установки пакета NuGet для автомасштабирования следует выполнить следующие действия.

1.  Откройте окно **консоли диспетчера пакетов NuGet**. В меню **Сервис** выберите **Диспетчер пакетов библиотеки**, а затем выберите **Консоль диспетчера пакетов**.

2.  Введите следующую команду в окне консоли диспетчера пакетов NuGet:

        PM> Install-Package EnterpriseLibrary.WindowsAzure.Autoscaling

Установка пакета NuGet обновляет проект, добавляя в него все необходимые сборки и ссылки, необходимые для блока приложения автомасштабирования. Теперь проект содержит файлы XML-схемы для определений правил автомасштабирования и информации о службе автомасштабирования. Теперь проект также включает файл readme, содержащий важную информацию о блоке приложения автомасштабирования:

![файлы, настроенные автомасштабированием пакета NuGet](./media/cloud-services-dotnet-autoscaling-application-block/auotscaling02.png)


### Задайте целевую платформу как .NET Framework 4

1.  В обозревателе решений щелкните правой кнопкой мыши имя проекта и выберите пункт **Свойства**.

2.  На вкладке **Приложение** окна "Свойства" убедитесь, что целевой платформой является **.NET Framework 4**.

	![изображение](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling03.png)

Добавьте следующие объявления пространств имен кода в начало любого файла C#, в котором вы собираетесь получать доступ к блоку приложения автомасштабирования программными средствами:

## <a id="Instantiate"> </a>Практическое руководство. Создание экземпляра и запуск средства автомасштабирования

Создайте экземпляр средства автомасштабирования с помощью метода **IServiceLocator.GetInstance**, а затем вызовите метод **Autoscaler.Start** для запуска **средства автомасштабирования**.

    Autoscaler scaler =
        EnterpriseLibraryContainer.Current.GetInstance<Autoscaler>();
    scaler.Start();

## <a id="DefineServiceModel"> </a>Практическое руководство. Определение модели службы

Как правило, модель службы (описание среды Microsoft Azure, в которое включены сведения о подписках, ведущих службах, ролях и учетных записях хранения) сохраняется в XML-файле. Копию схемы для этого XML-файла можно найти в файле **AutoscalingServiceModel.xsd** вашего проекта. В Visual Studio эта схема предоставляет Intellisense и способ проверки редактируемого XML-файла модели службы.

Создайте новый XML-файл с именем **services.xml** в своем проекте.

В Visual Studio необходимо убедиться, что файл модели службы скопирован в выходную папку. Для этого:

1.  Щелкните правой кнопкой мыши файл и выберите пункт **Свойства**.

2.  На панели «Свойства» задайте для параметра **Копировать в выходной каталог** значение **Всегда копировать**.

    ![Задайте значение параметра "Копировать в выходной каталог"](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling04.png)


	В следующем примере кода показан пример модели службы в файле **services.xml**:

    <?xml version="1.0" encoding="utf-8" ?> <serviceModel xmlns="http://schemas.microsoft.com/practices/2011/entlib/autoscaling/serviceModel"> <subscriptions> <subscription name="[subscriptionname]"
                      certificateThumbprint="[managementcertificatethumbprint]"
                      subscriptionId="[subscriptionid]"
                      certificateStoreLocation="CurrentUser"
                      certificateStoreName="My"> <services> <service dnsPrefix="[hostedservicednsprefix]" slot="Staging"> <roles> <role alias="AutoscalingApplicationRole"
                      roleName="[targetrolename]"
                      wadStorageAccountName="targetstorage"/> </roles> </service> </services> <storageAccounts> <storageAccount alias="targetstorage"
              connectionString="DefaultEndpointsProtocol=https;AccountName=[storageaccountname];AccountKey=[storageaccountkey]"> </storageAccount> </storageAccounts> </subscription> </subscriptions> </serviceModel>

Значения в квадратных скобках необходимо заменить на значения, относящиеся к вашей среде и целевому приложению. Чтобы найти многие из этих значений, придется войти на [портал управления Azure][].

Выполните вход на портал управления.

-   **[subscriptionname]\:** выберите понятное имя для ссылки на подписку Azure, которая содержит приложение, где требуется использовать автоматическое масштабирование.

-   **[subscriptionid]\:** уникальный идентификатор подписки Azure, содержащей приложение, в котором требуется использовать автоматическое масштабирование.

    1.  На портале управления Azure щелкните элемент **Облачные службы**.

    2.  В списке облачных служб щелкните службу, содержащую приложение, в котором вы хотите использовать автоматическое масштабирование. На панели сводки справа будет отображаться **идентификатор подписки**.

        ![изображение](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling05.png)

  
	-   **[hostedservicednsprefix]\:** префикс DNS размещенной службы, в которой требуется использовать автомасштабирование.

    1.  На портале управления Azure щелкните элемент **Облачные службы**.

    2.  В списке облачных служб найдите службу, содержащую приложение, в котором требуется использовать автоматическое масштабирование. Имя облачной службы — **префикс DNS**.

        ![изображение](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling06.png)
 
	-   **[targetrolename]\:** имя роли, которая является целевой для ваших правил автомасштабирования.

    1.  На портале управления Azure щелкните элемент **Облачные службы**.

    2.  В списке облачных служб щелкните службу, содержащую приложение, в котором требуется использовать автоматическое масштабирование, а затем щелкните пункт **Экземпляры**. Столбец **Роль* содержит имя целевой роли.

        ![изображение](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling07.png)


	-   **[storageaccountname]** и **[storageaccountkey]\:** имя учетной записи хранения Azure, которая используется для целевого приложения Azure.

    1.  На портале управления Azure щелкните **Служба хранилища**.

    2.  В списке учетных записей хранения выберите используемую учетную запись хранения. Столбец **Имя** содержит **имя**.

    3.  Нажмите кнопку **Управление ключами** в нижней части экрана, чтобы получить первичный ключ доступа.

        ![изображение](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling08.png)
  
 
	-   **[managementcertificatethumbprint]\:** **отпечаток** сертификата управления, который будет использоваться блоком для обеспечения безопасности запросов масштабирования, адресованных целевому приложению.

    1.  На портале управления Azure щелкните элемент **Параметры**.

    2.  В столбце **Отпечаток** отображается **Отпечаток**.

        ![изображение](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling09.png)
 

Дополнительные сведения о содержимом файла модели службы см. в разделе [Сохранение сведений о службе][].

## <a id="DefineAutoscalingRules"> </a>Практическое руководство. Определение правил автоматического масштабирования

Обычно правила автомасштабирования управляют рядом экземпляров ролей в целевом приложении в XML-файле. Копию схемы для этого XML-файла вы можете найти в файле **AutoscalingRules.xsd** вашего проекта. В Visual Studio эта схема предоставляет Intellisense и способ проверки редактируемого XML-файла.

Создайте новый XML-файл с именем **rules.xml** в своем проекте.

В Visual Studio необходимо убедиться, что файл правил скопирован в папку выхода. Для этого:

1.  Щелкните правой кнопкой мыши файл и выберите пункт **Свойства**.

2.  На панели «Свойства» задайте для параметра **Копировать в выходной каталог** значение **Всегда копировать**.

В следующем примере кода показан пример набора правил в файле **rules.xml**.

    <?xml version="1.0" encoding="utf-8" ?>
    <rules xmlns="http://schemas.microsoft.com/practices/2011/entlib/autoscaling/rules">
      <constraintRules>
        <rule name="default" enabled="true" rank="1" description="The default constraint rule">
          <actions>
            <range min="2" max="6" target="AutoscalingApplicationRole"/>
          </actions>
        </rule>
      </constraintRules>
      <reactiveRules>
        <rule name="ScaleUpOnHighUtilization" rank="10" description="Scale up the web role" enabled="true" >
          <when>
            <any>
               <greaterOrEqual operand="WebRoleA_CPU_Avg_5m" than="60"/>
            </any>
          </when>
          <actions>
            <scale target="AutoscalingApplicationRole" by="1"/>
          </actions>
        </rule>
        <rule name="ScaleDownOnLowUtilization" rank="10" description="Scale up the web role" enabled="true" >
          <when>
            <all>
              <less operand="WebRoleA_CPU_Avg_5m" than="60"/>
            </all>
          </when>
          <actions>
            <scale target="AutoscalingApplicationRole" by="-1"/>
          </actions>
        </rule>
      </reactiveRules>
      <operands>
        <performanceCounter alias="WebRoleA_CPU_Avg_5m"
          performanceCounterName="\Processor(_Total)% Processor Time"
          source ="AutoscalingApplicationRole"
          timespan="00:05:00" aggregate="Average"/>
      </operands>
    </rules>

В данном примере имеется три правила автоматического масштабирования (одно **правило ограничения** и два **реактивных правила**), относящиеся к целевому объекту **AutoscalingApplicationRole**, который является псевдонимом роли, определенной в **модели службы**.

-   Правило ограничения всегда активно и задает минимальное число экземпляров роли, равное двум, и максимальное число экземпляров роли, равное шести.

-   Оба реактивных правила используют **операнд** с именем **WebRoleA_CPU_Avg_5m**, который вычисляет среднее использование ЦП за последние пять минут для роли Azure с именем **AutoscalingApplicationRole**. Эта роль определяется в **модели службы**.

-   Реактивное правило с именем **ScaleUpOnHighUtilization** увеличивает на единицу число экземпляров целевой роли, если среднее использование ЦП за последние пять минут было не менее 60 %.

-   Реактивное правило с именем **ScaleDownOnLowUtilization** уменьшает на единицу число экземпляров целевой роли, если среднее использование ЦП за последние пять минут было менее 60 %.

## <a id="Configure"> </a>Практическое руководство. Настройка блока приложения автомасштабирования

Определив правила модели службы и автомасштабирования, необходимо настроить блок приложения автомасштабирования для их использования. Эти сведения об оперативной конфигурации сохраняются в файле конфигурации приложения.

По умолчанию блок приложения автомасштабирования ожидает сохранения правил автомасштабирования и модели службы в больших двоичных объектах Azure. В этом примере выполняется настройка блока для загрузки правил из локальной файловой системы.

### Настройте блок приложения автомасштабирования в ведущем приложении

1.  Щелкните правой кнопкой мыши файл **App.config** в обозревателе решений и выберите команду **Изменить файл конфигурации**.

2.  В меню **Блоки** щелкните **Добавить параметры автомасштабирования**: ![изображение](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling10.png)
  
3.  Разверните узел **Параметры автомасштабирования** и щелкните многоточие (...) рядом с **учетной записью хранения точек данных**, добавьте **имя учетной записи** и **ключ учетной записи** для учетной записи хранения Azure, в которой блок будет хранить собранные точки данных (см. [Практическое руководство. Определение модели службы][], если вы не уверены, где можно найти эти значения), а затем нажмите кнопку **ОК**.

	![изображение](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling11.png)

4.  Разверните раздел **Параметры автомасштабирования**, чтобы открыть разделы **Хранилище правил** и **Хранилище сведений о службе**. По умолчанию они настроены для использования хранилища больших двоичных объектов Azure: ![изображение](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling12.png)


5.  Щелкните знак «плюс» (+) рядом с пунктом **Хранилище правил**, выберите пункт **Задать хранилище правил**, затем щелкните **Использовать локальное хранилище правил файлов** и нажмите кнопку **Да**.

6.  В поле **имя файла** введите **rules.xml**. Это имя файла, который содержит правила автоматического масштабирования: ![изображение](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling13.png)


7.  Щелкните знак «плюс» (+) рядом с пунктом **Хранилище сведений о службе**, выберите пункт **Задать хранилище сведений о службе**, затем щелкните **Использовать локальное хранилище файлов сведений о службе** и нажмите кнопку **Да**.

8.  В поле **имя файла** введите **services.xml**. Это имя файла, который содержит правила автоматического масштабирования: ![изображение](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling14.png)


9.  В окне «Конфигурация библиотеки предприятия» в меню **Файл** нажмите **Сохранить**, чтобы сохранить изменения конфигурации. Затем в окне конфигурации библиотеки предприятия в меню **Файл** нажмите **Выход**.

Для получения подробной информации о действиях, выполняемых блоком приложения автомасштабирования, необходимо записать сообщения, записываемые в журналы. Например, при размещении блока в приложении консоли можно просматривать сообщение журнала в окне вывода Visual Studio. В следующем разделе показано, как настроить такое поведение.

### Настройте ведение журналов в ведущем приложении блока приложения автомасштабирования

1.  В Visual Studio дважды щелкните файл **App.config** в обозревателе решений, чтобы открыть его в редакторе. Затем добавьте раздел **system.diagnostics**, как показано в следующем примере.

        <?xml version="1.0" encoding="utf-8" ?>
        <configuration>
          ...
          <system.diagnostics>
            <sources>
              <sourcename="Autoscaling General" switchName="SourceSwitch" switchType="System.Diagnostics.SourceSwitch" />
              <sourcename="Autoscaling Updates" switchName="SourceSwitch" switchType="System.Diagnostics.SourceSwitch" />
            </sources>
            <switches>
              <addname="SourceSwitch" value="Verbose, Information, Warning, Error, Critical" />
            </switches>
          </system.diagnostics>
        </configuration>

2.  Сохраните изменения.
