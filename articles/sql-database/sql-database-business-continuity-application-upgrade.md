#Обновление приложений без простоев

В контексте Microsoft Azure термин «приложение» относится к таким компонентам, как внешние интерфейсы, развернутые в облаке службы, а также уровень данных, используемый для хранения данных приложения или метаданных. Облачные приложения часто разрабатываются для непрерывной и круглосуточной работы. Развертывание новой версии приложения, при котором изменения на уровне данных применяются в рабочей среде, может привести к нарушению операций, например к ограничению доступных возможностей или полному прекращению работы.

Планируя обновление приложения, следует в первую очередь позаботиться об устранении или значительном сокращении срока, в течение которого оно работает с ограничениями. Для этого в процесс обычно включают создание временной копии приложения, которая будет использоваться, если при обновлении произойдет сбой. При планировании обновления следует учитывать следующие факторы:

1.	Максимальное приемлемое время, в течение которого приложение будет работать в ограниченном режиме. 
2.	Минимальный набор функций, который будет доступен во время обновления.
3.	Возможность отката в случае возникновения ошибок во время обновления.
4.	Общие затраты. Под общими затратами подразумеваются расходы на любые вспомогательные компоненты приложения, которые потребуются для создания временной копии (например, дополнительные базы данных Premium для активной георепликации) или дополнительные расходы на временное развертывание, используемое при обновлении. 

Если приложение может какое-то время работать в режиме только для чтения, процесс обновления можно выстроить так, чтобы исключить простой полностью. Чтобы узнать, как спланировать обновление с учетом топологии конкретного приложения, читайте статью [Минимальное нарушение работы базы данных SQL Azure при развертывании обновлений для приложений](https://msdn.microsoft.com/library/azure/dn790385.aspx).
