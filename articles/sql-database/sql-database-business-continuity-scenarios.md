В этой статье вы узнаете о нескольких сценариях обеспечения непрерывности бизнес-процессов для базы данных SQL Azure.

## Восстановление после сбоя

В разделе [Восстановление базы данных SQL Azure или переход на базу данных-получатель при отказе](sql-database-disaster-recovery.md) описывается восстановление после сбоя с помощью следующих возможностей.

- [Активная георепликация](sql-database-geo-replication-overview.md)
- [Геовосстановление](sql-database-geo-restore.md)

В этой статье рассматривается, когда следует начать восстановление, как его выполнить с помощью каждой возможности, а также как настроить базу данных и приложения после восстановления.

## Восстановление после ошибки пользователя

В разделе [Восстановление базы данных SQL Azure после ошибки пользователя](sql-database-user-error-recovery.md) описывается восстановление после ошибок пользователей или непреднамеренного изменения данных с помощью следующих возможностей.

- [Восстановление состояния системы в определенной временной точке](sql-database-point-in-time-restore.md) 
- [Восстановление удаленной базы данных](sql-database-restore-deleted-database.md)

## Отработка аварийного восстановления

В разделе [Отработка аварийного восстановления](sql-database-disaster-recovery-drills.md) описывается, как выполнить аварийное восстановление с помощью следующих возможностей.

- [Активная георепликация](sql-database-geo-replication-overview.md)
- [Геовосстановление](sql-database-geo-restore.md)

Рекомендуется периодически проверять готовность приложений к рабочему процессу восстановления. Проверка поведения приложения и возможных последствий потери данных и нарушений работы, которая включает отработку отказа, предусмотрена стандартами разработки. Эта процедура также является обязательной для большинства отраслевых стандартов в рамках сертификации непрерывности бизнес-процессов.

Этапы отработки аварийного восстановления:

- моделирование сбоя уровня данных;
- восстановление; 
- проверка целостности приложения после восстановления.

## Управление безопасностью после аварийного восстановления

В разделе [Управление безопасностью после аварийного восстановления](sql-database-geo-replication-security-config.md) описываются требования к аутентификации для настройки [активной георепликацией](sql-database-geo-replication-overview.md) и управления ею, а также приведены действия, необходимые для настройки доступа пользователей к базе данных-получателю. В нем также описано, как включить доступ к восстановленной базе данных после [геовосстановления](sql-database-geo-restore.md).

## Управление последовательными обновлениями для облачных приложений с помощью активной георепликации

В разделе [Управление последовательными обновлениями для облачных приложений с помощью активной георепликации базы данных SQL](sql-database-manage-application-rolling-upgrade.md) описывается использование [георепликации](sql-database-geo-replication-overview.md) в базе данных SQL для применения последовательных обновлений к облачному приложению. Так как обновление представляет собой нарушающую работу операцию, оно должно учитываться при планировании и проектировании непрерывности бизнес-процессов. В этой статье рассматривается два разных метода управления процессом обновления, а также описываются преимущества и недостатки каждого из них.

## Проектирование приложения для облачного аварийного восстановления с использованием активной георепликации

В разделе [Создание приложения для аварийного восстановления облака с использованием активной георепликации в базе данных SQL](sql-database-designing-cloud-solutions-for-disaster-recovery.md) описывается использование [активной георепликации](sql-database-geo-replication-overview.md) в базе данных SQL для проектирования приложений базы данных, которые устойчивы к региональным сбоям и разрушительным простоям. В этой статье рассматривается топология развертывания приложения, целевое соглашение об уровне обслуживания, задержка трафика и издержки, а затем приводятся распространенные шаблоны приложений, каждый из которых имеет преимущества и компромиссы.
