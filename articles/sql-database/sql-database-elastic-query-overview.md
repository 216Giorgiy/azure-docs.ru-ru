# Запрос к эластичной базе данных (предварительная версия от июня 2015 года) 

Запрос к эластичной базе данных — это новая функция базы данных SQL Azure (предварительная версия появится в конце июня 2015 года). Она позволяет запускать запрос T-SQL, который охватывает несколько баз данных, используя одно подключение к базе данных SQL Azure. Это позволяет сегментированному набору данных, определенному с помощью карты сегментов эластичных баз данных, предстать в виде единого интегрированного хранилища данных. Используется для отчетности и интеграции данных, вся распределенная обработка запросов проходит в фоновом режиме. Соединения поддерживаются любым драйвером, который способен взаимодействовать с базой данных SQL, включая ADO.Net, ODBC, JDBC, Node.js, PHP и т. д.

## Сценарии эластичных запросов к базам данных

Цель эластичного запроса к базам данных — облегчить сценарии отчетности по всему уровню сегментированных данных, в котором несколько баз данных или сегментов предоставляют строки для единственного общего результата запроса T-SQL. Запрос T-SQL может быть создан напрямую пользователем или приложением либо косвенно — с помощью средств, которые подключены к базе данных эластичных запросов для глобального запроса. Эта возможность особенно полезна для коммерческой бизнес-аналитики, отчетности или средств интеграции данных, комплектов программного обеспечения, которые не могут быть расширены непосредственно с помощью эластичных библиотек масштабирования. Она также позволяет легко получить доступ ко всей коллекции сегментированных данных через запросы, выданные SQL Server Management Studio или Visual Studio, и поддерживает прозрачный доступ к многосегментным данным из Entity Framework или других сред ORM. Этот сценарий представлен на рисунке 1. Эластичный запрос к БД глобальных запросов предлагает второй, отличный от первого путь для отправки запросов к уровню сегментированных данных в дополнение к существующему облачному приложению (который может использовать эластичные библиотеки масштабирования).

![Эластичные запросы к базе данных][1]

Сценарии могут характеризоваться следующими топологиями:

-	Уровень сегментированных данных (топология 1). Этот уровень данных был разработан для сегментирования. Сегментирование осуществляется и управляется с помощью (1) клиентской библиотеки эластичных баз данных или (2) самостоятельного сегментирования. Вариант отчетности — вычислить отчеты по большому количеству сегментов, учитывая требования к функциональности или подключению, за пределами которых можно выполнить многосегментный запрос в клиентской библиотеке эластичных баз данных. 

-	Макет с несколькими базами данных (топология 2). В данном случае уровень данных разделен вертикально таким образом, что различные типы данных располагаются в различных базах. Вариант отчетности — вычислить отчеты по данным, которые распределены по нескольким базам данных, но должны быть интегрированы в один общий результат для отчета.

Предварительная версия запроса эластичной базы данных охватит оба варианта. Если для управления сегментами клиенты с топологией 1 используют клиентскую библиотеку эластичных баз данных, то они могут полагаться на свою карту существующих сегментов. Чтобы воспользоваться этой возможностью предварительной версии, самостоятельным сегментировщикам, в свою очередь, потребуется создать карту сегментов с помощью средств эластичных баз данных для их уровня данных. Клиентам, следующим топологии 2, потребуется создать другую карту сегментов для каждой базы данных. Каждая из этих карт сегментов будет указывать только на один сегмент, который потом может предоставить свои таблицы для запросов между сегментами.
