# <a name="what-is-azure-redis-cache"></a>Описание кэша Redis для Azure

Кэш Redis для Azure основан на популярном [кэше с открытым кодом Redis](https://redis.io/). Он обычно используется в качестве кэша для повышения производительности и масштабируемости систем, которые в значительной степени зависят от хранилищ данных. Для повышения производительности часто запрашиваемые данные временно копируются в быстрое хранилище данных, расположенное ближе к приложению. С [кэшем Redis](https://redis.io/) это быстрое хранилище находится в памяти, а не загружается с диска с помощью базы данных.

Кэш Redis для Azure также может использоваться как хранилище данных в памяти, распределенная нереляционная база данных и брокер сообщений. Производительность приложений повышается за счет использования низкой задержки и высокой пропускной способности модуля Redis.

Кэш Redis для Azure предоставляет вам доступ к безопасному выделенному кэшу Redis, размещенному в Azure и управляемому корпорацией Майкрософт и доступному из любых приложений на платформе Azure или вне ее.

## <a name="why-use-azure-redis-cache"></a>Зачем использовать кэш Redis для Azure

Существует много распространенных шаблонов, в которых кэш Redis используется для поддержки архитектуры приложений или для повышения производительности приложений. Ниже перечислены некоторые наиболее распространенные шаблоны.

| Модель      | Описание                                        |
| ------------ | -------------------------------------------------- |
| [Отдельно от кэша](cache-web-app-cache-aside-leaderboard.md) | Так как база данных может быть большой, загрузка всей базы данных в кэш не рекомендуется. Обычно используется шаблон [кэш на стороне](https://docs.microsoft.com/azure/architecture/patterns/cache-aside) для загрузки элементов данных только при необходимости. Когда система вносит изменения в серверные данные, она может в это же время обновлять кэш, который распространяется вместе с другими клиентами. Кроме того, система может установить истечение срока действия элементов данных или использовать политику вытеснения, чтобы вызвать перезагрузку обновленных данных в кэш.|
| [Кэширование содержимого](cache-aspnet-output-cache-provider.md) | Большинство веб-страниц создаются из шаблонов с заголовками, нижними колонтитулами, панелями инструментов, меню и т. д. Они не изменяются часто и не должны создаваться динамически. При использовании кэша в памяти, например кэша Redis для Azure, ваши веб-серверы получают быстрый доступ к этому типу статического содержимого по сравнению с базовыми хранилищами данных. Этот шаблон уменьшает время обработки и нагрузку на сервер, которые бы требовались для динамического создания содержимого. Благодаря этому скорость взаимодействия веб-серверов повышается, а количество серверов, необходимых для обработки нагрузок, может быть сокращено. Кэш Redis для Azure предоставляет поставщик кэша вывода Redis для поддержки этого шаблона с помощью ASP.NET.|
| [Кэширование сеанса пользователя](cache-aspnet-session-state-provider.md) | Этот шаблон обычно используется с корзинами покупок и другой информацией о типе из истории пользователя, которую веб-приложение может связать с файлами пользователя. Хранение большого объема содержимого в файле cookie может отрицательно сказаться на производительности, так как размер этого файла растет и передается и проверяется с каждым запросом. Типичным решением является использование файла cookie в качестве ключа для запроса данных в серверной базе данных. Использование кэша в памяти, например кэша Redis для Azure, для связывания информации с пользователем намного быстрее, чем взаимодействие с полной реляционной базой данных. |
| Очереди задач и сообщений | Когда приложения получают запросы, часто для выполнения операций, связанных с запросом, требуется дополнительное время. Общим сценарием является отсрочка более длительных операций путем добавления их в очередь, которая обрабатывается позже и, возможно, на другом сервере. Этот метод отсрочки работы называется постановкой задач в очередь. Существует много программных компонентов, предназначенных для поддержки очереди задач. Кэш Redis также предназначен для этой цели, как и распределенная очередь.|
| Распределенные транзакции | Общим требованием для приложений является возможность выполнения ряда команд по отношению к хранилищу данных в виде одной операции (атомной). Все команды должны успешно завершиться или все транзакции должны быть возвращены в исходное состояние. Кэш Redis поддерживает выполнение пакета команд в рамках одной операции в форме [транзакций](https://redis.io/topics/transactions). |

## <a name="azure-redis-cache-offerings"></a>Предложения кэша Redis для Azure

Кэш Redis для Azure доступен на следующих уровнях:

| Уровень | Описание |
|---|---|
базовая; | Кэш одного узла. Этот уровень поддерживает несколько объемов памяти (250 МБ–53 ГБ). Этот уровень идеально подходит для разработки или тестирования и некритических рабочих нагрузок. Для уровня "Базовый" соглашение об уровне обслуживания не предусмотрено. |
| Стандартная | Реплицируемый кэш в конфигурации с двумя узлами: основным и дополнительным, управляемый корпорацией Майкрософт, с соглашением об уровне обслуживания с высоким уровнем доступности (99,9 %). |
| Премиум | Уровень "Премиум" предназначен для работы в корпоративной среде. Кэш уровня "Премиум" поддерживает больше возможностей и имеет более высокую пропускную способность с более низкими задержками. Кэши уровня "Премиум" развертываются на более мощном оборудовании и обеспечивают лучшую производительность по сравнению с уровнями "Базовый" и "Стандартный". Это преимущество означает, что пропускная способность для кэша такого же размера будет выше для уровня "Премиум" по сравнению с уровнем "Стандартный". |

> [!TIP]
> Дополнительные сведения о размере, пропускной способности и полосе пропускания для кэшей уровня "Премиум" см. в статье [Кэш Redis для Azure. Вопросы и ответы](cache-faq.md#what-redis-cache-offering-and-size-should-i-use).
>

Созданный кэш можно перевести на более высокий уровень. Переход на более низкий уровень не поддерживается. Пошаговые инструкции см. в разделах [Масштабирование кэша Redis для Azure](cache-how-to-scale.md) и [Автоматизация операции масштабирования](cache-how-to-scale.md#how-to-automate-a-scaling-operation).
