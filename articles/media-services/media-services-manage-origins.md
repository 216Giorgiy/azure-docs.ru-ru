#<a id="managemediaservicesorigins"></a>Управление конечными точками потоковой передачи в учетной записи служб мультимедиа

> [AZURE.SELECTOR]
- [Portal](media-services-manage-origins.md)
- [Java](https://github.com/southworkscom/azure-sdk-for-media-services-java-samples)
- [PHP](https://github.com/Azure/azure-sdk-for-php/tree/master/examples/MediaServices)

В службах мультимедиа Microsoft Azure **конечная точка потоковой передачи** — это служба потоковой передачи, которая может доставить содержимое непосредственно в клиентское приложение проигрывателя или в сеть доставки содержимого (CDN) для дальнейшего распространения. Службы мультимедиа также обеспечивают прозрачную интеграцию с Azure CDN. Исходящий поток из службы StreamingEndpoint может быть потоком трансляции или активом видео по запросу в учетной записи служб мультимедиа.

Кроме того, чтобы справиться с растущими потребностями в пропускной способности, можно контролировать мощности службы конечной точки потоковой передачи, регулируя число единиц масштабирования (которые также называются единицами потоковой передачи). Рекомендуем выделить одну или несколько единиц масштабирования для приложения в рабочей среде. Единицы масштабирования предоставляют как выделенные мощности исходящего трафика, которые можно приобрести с шагом 200 Мбит/с, так и дополнительную функциональность, которая в настоящее время включает: [динамическое упаковывание](media-services-dynamic-packaging-overview.md), интеграцию с CDN и расширенную конфигурацию.

Плата взимается, только когда служба StreamingEndpoint используется.

В этом разделе представлен обзор основных возможностей, предоставляемых конечными точками потоковой передачи. Здесь также показано, как использовать классический портал Azure для управления конечными точками потоковой передачи.


##Добавление и удаление конечных точек потоковой передачи

Добавлять и удалять конечные точки потоковой передачи можно с помощью пакета SDK для .NET, API-интерфейса REST или классического портала Azure.

Чтобы добавить или удалить конечную точку потоковой передачи с помощью классического портала Azure, выполните указанные ниже действия.

1. На [классическом портале Azure](https://manage.windowsazure.com/) щелкните **Службы мультимедиа**. Затем щелкните имя службы мультимедиа.
2. Откройте страницу **КОНЕЧНЫЕ ТОЧКИ ПОТОКОВОЙ ПЕРЕДАЧИ**.
3. В нижней части страницы нажмите кнопку ДОБАВИТЬ или УДАЛИТЬ. Обратите внимание, что конечную точку потоковой передачи по умолчанию удалить нельзя.
4. Нажмите кнопку ЗАПУСК для запуска конечной точки.
5. Щелкните имя конечной точки потоковой передачи, чтобы настроить ее.

![Страница конечной точки потоковой передачи][streaming-endpoint]


По умолчанию можно использовать до двух конечных точек потоковой передачи. Если вам необходимо больше точек, изучите раздел [Квоты и ограничения](media-services-quotas-and-limitations.md).

##<a id="scale_streaming_endpoints"></a>Масштабирование конечной точки потоковой передачи

Единицы потоковой передачи предоставляют как выделенные мощности исходящего трафика, которые можно приобрести с шагом 200 Мбит/с, так и дополнительную функциональность, которая в настоящее время включает [возможности динамического упаковывания](media-services-dynamic-packaging-overview.md). По умолчанию потоковая передача настроена в модели общего экземпляра, для которой серверные ресурсы (например, вычислительные ресурсы, мощности исходящего трафика и т. д.) совместно используются со всеми другими пользователями. Чтобы повысить пропускную способность для потоковой передачи, рекомендуем приобрести единицы потоковой передачи.

Масштабировать конечные точки можно с помощью пакета SDK для .NET, API-интерфейса REST или классического портала Azure.

Чтобы изменить число единиц потоковой передачи с помощью портала, выполните следующие действия.

1. Чтобы указать число единиц потоковой передачи, перейдите на вкладку МАСШТАБ и переместите ползунок **зарезервированная мощность**.

	![Страница "Масштаб"](./media/media-services-manage-origins/media-services-origin-scale.png)

4. Нажмите кнопку СОХРАНИТЬ, чтобы сохранить изменения.

	Выделение новых единиц потоковой передачи занимает около 20 минут.

	 
	>[AZURE.NOTE]Сейчас переход от любого положительного значения единиц потоковой передачи к нулевому может привести к отключению потоковой передачи по требованию на период до одного часа.

	>[AZURE.NOTE]Для расчета затрат используется наибольшее число единиц, указанных для 24-часового периода. Дополнительные сведения о ценах см. в разделе [Сведения о ценах для служб мультимедиа](http://go.microsoft.com/fwlink/?LinkId=275107).
	
##<a id="configure_streaming_endpoints"></a>Настройка конечной точки потоковой передачи

Конечная точка потоковой передачи позволяет настроить следующие свойства при наличии по крайней мере 1 единицы масштабирования:

- управление доступом;
- настраиваемые имена узлов;
- управление кэшем;
- межсайтовые политики доступа.

Дополнительные сведения об этих свойствах см. в разделе [StreamingEndpoint](https://msdn.microsoft.com/library/azure/dn783468.aspx).

Настроить эти свойства можно с помощью пакета SDK для .NET, API-интерфейса REST или классического портала Azure.

Чтобы изменить число единиц потоковой передачи с помощью портала, выполните следующие действия.

1. Выберите конечную точку потоковой передачи, которую хотите настроить.
1. Перейдите на вкладку "НАСТРОЙКА".
  
Далее представлено краткое описание полей.

![Configure origin][configure-origin]
  

1. Задать максимальный период кэширования, который будет указан в заголовке управления кэшем в ответах HTTP. Это значение не будет переопределять максимальное значение кэша, заданное явным образом в контенте BLOB-объекта.

2. Указать IP-адреса, которым будет разрешено подключение к опубликованной конечной точке потоковой передачи. Если IP-адреса не указаны, можно подключаться с любого IP-адреса.

3. Определить настройку для проверки подлинности заголовка подписи Akamai.

4. Можно указать политику междоменного доступа для клиентов Adobe Flash (дополнительные сведения см. в документе [Спецификация файла политики междоменного доступа](http://www.adobe.com/devnet/articles/crossdomain_policy_file_spec.html)), а также политику доступа клиента для клиентов Microsoft Silverlight (дополнительные сведения см. в разделе [Доступ к службам за пределами домена](https://msdn.microsoft.com/library/cc197955(v=vs.95).aspx).

5. Можно также настроить пользовательские имена узлов, нажав кнопку **Настройка**. Дополнительную информацию см. в описании свойства **CustomHostNames** в разделе [Конечная точка передачи данных](https://msdn.microsoft.com/library/dn783468.aspx).


##<a id="enable_cdn"></a>Включение интеграции Azure CDN

Вы можете включить интеграцию Azure CDN для конечной точки потоковой передачи (по умолчанию она отключена).

Включение интеграции Azure CDN

- У конечной точки потоковой передачи должна быть по крайней мере одна единица потоковой передачи (масштабирования). Если позже вы захотите задать число единиц масштабирования равным 0, вначале потребуется отключить интеграцию CDN. По умолчанию при создании новой конечной точки потоковой передачи автоматически устанавливается одна единица потоковой передачи.

- Конечная точка потоковой передачи должна быть остановлена. После включения CDN вы можете запустить конечную точку потоковой передачи.

Для включения интеграции Azure CDN может потребоваться 90 минут. Активизация изменений по всем CDN POP занимает до двух часов.


Интеграция CDN включена во всех центрах обработки данных Azure: Запад США, Восток США, Северная Европа, Западная Европа, Запад Японии, Восток Японии, Юго-Восточная Азия и Восточная Азия.

После ее включения отключаются следующие конфигурации: **Пользовательские имена узлов** и **Контроль доступа**.

![Включение CDN для конечной точки потоковой передачи][streaming-endpoint-enable-cdn]


###Дополнительные замечания

- При включении CDN для конечной точки потоковой передачи клиенты не могут запрашивать содержимое непосредственно из источника. Если требуется возможность проверки содержимого с CDN или без нее, можно создать другую конечную точку потоковой передачи, для которой не включена CDN.
- Имя узла конечной точки потоковой передачи после включения CDN остается таким же. После включения CDN не требуется вносить никакие изменения в рабочий процесс служб мультимедиа. Например, если имя узла конечной точки потоковой передачи — strasbourg.streaming.mediaservices.windows.net, после включения CDN будет использоваться точно такое же имя узла.
- Для новых конечных точек потоковой передачи можно включить CDN просто путем создания новой конечной точки. Для существующих конечных точек потоковой передачи необходимо сначала остановить конечную точку, а затем включить CDN.
 

Дополнительные сведения см. в [разделе, анонсирующем интеграцию служб мультимедиа Azure с Azure CDN (сетью доставки содержимого)](http://azure.microsoft.com/blog/2015/03/17/announcing-azure-media-services-integration-with-azure-cdn-content-delivery-network/).
