# Создание веб-приложения для бизнеса в службе приложений Azure

[Служба приложений Azure](http://go.microsoft.com/fwlink/?LinkId=529714) отлично подходит для создания бизнес-приложений. Эти приложения являются приложениями интрасети, которые должны быть защищены для внутреннего использования. Как правило, они требуют аутентификации, обычно с использованием каталога организации. Кроме того, для их использования обычно требуется определенный уровень доступа и интеграции с локальными службами и данными.

Существует ряд преимуществ перемещения бизнес-приложений в веб-приложения службы приложений Azure, в частности, перечисленные далее.

-  Возможность масштабирования в сторону увеличения и уменьшения в соответствии с динамической нагрузкой, например для приложения, обрабатывающего данные ежегодных аттестаций. В период аттестаций трафик для большой компании резко возрастает. Azure предоставляет варианты масштабирования, позволяющие компании выполнить горизонтальное масштабирование, чтобы справиться с нагрузкой в период большого трафика, связанного с аттестациями, и экономить деньги в течение всего остального года благодаря обратному масштабированию. 
-  Возможность уделять больше внимания разработке приложений и меньше — приобретению инфраструктуры и управлению ею.
-  Предоставление сотрудникам и партнерам больше возможностей для использования приложений из любой точки мира. В этом случае пользователям не потребуется подключаться к корпоративной сети для использования приложения, а ИТ-подразделение будет избавлено от необходимости реализовывать сложные решения с обратным прокси. Существует несколько вариантов проверки подлинности, обеспечивающих защиту доступа к корпоративным приложениям.

Ниже приведен пример бизнес-приложения, работающего в веб-приложениях службы приложений Azure. В нем демонстрируются возможности, которые обеспечивает объединение веб-приложений с другими службами при минимальных инвестициях в технологии. **Щелкните элемент в топографии, чтобы узнать о нем больше.**

## Перенос имеющихся ресурсов

В веб-приложения службы приложений Azure можно переносить существующие веб-ресурсы, написанные на разных языках и расположенные на разных платформах.

Существующие веб-ресурсы можно запускать в веб-приложениях службы приложений, независимо от языка и платформы (.NET, PHP, Java, Node.js или Python). Их можно перемещать в веб-приложения с помощью привычных средств для работы с [FTP] или системы управления версиями. Веб-приложения поддерживают прямую публикацию из популярных систем управления версиями, таких как [Visual Studio], [Visual Studio Online] и [Git] (локальная система, GitHub, BitBucket, DropBox, Mercurial и т. д.).

## Защита ресурсов

Защищайте ресурсы с помощью шифрования, выполняйте аутентификацию корпоративных пользователей локально и удаленно и авторизуйте использование ими различных ресурсов.

Защищайте внутренние ресурсы от нарушителей с помощью [HTTPS]. Доменное имя ***.azurewebsites.net** предоставляется сразу с SSL-сертификатом, а при использовании личного домена его SSL-сертификат можно перенести в веб-приложения службы приложений. За использование каждого SSL-сертификата ежемесячно взимается почасовая плата. Дополнительные сведения см. в статье [Сведения о ценах на службу приложений].

[Проверяйте подлинность пользователей] по каталогу организации. Веб-приложения службы приложений могут проверять подлинность пользователей с помощью локальных поставщиков удостоверений, например служб федерации Active Directory (AD FS) или клиента Azure Active Directory, синхронизированного с корпоративным развертыванием службы Active Directory. Пользователи могут получать доступ к веб-свойствам в веб-приложениях с помощью единого входа — локально или удаленно. Существующие службы, такие как Office 365 или Microsoft Intune, уже используют Azure Active Directory. Благодаря [простой авторизации] включить проверку подлинности на том же клиенте Azure Active Directory для веб-приложения очень легко.

[Авторизуйте пользователей] для использования ими веб-свойств. Ту же локальную схему кода ASP.NET можно перенести в веб-приложения службы приложений, добавив минимальный дополнительный код и используя, например, оформление `[Authorize]`. При этом сохраняется такая же гибкость для мелкоструктурного управления доступом, как и для приложений, управление которыми осуществляется локально.

## Подключение к локальным ресурсам ##

Подключайтесь к данным или ресурсам веб-приложения, где бы они ни находились: в облаке для повышения производительности или в локальном расположении для обеспечения соответствия требованиям. Дополнительную информацию о хранении данных в Azure см. в статье [Центр управления безопасностью Azure.].

В службе Azure предлагается широкий выбор внутренних серверов баз данных, в том числе для [Базы данных SQL Azure] и [MySQL], которые способны удовлетворить требования веб-приложений. Защита данных в Azure позволяет хранить данные рядом с местом размещения веб-приложения и оптимизировать его производительность.

Тем не менее, иногда компании должны хранить свои данные локально. Веб-приложения службы приложений позволяют с легкостью установить [гибридное подключение] к локальному ресурсу, например ко внутреннему серверу базы данных. Если вам нужно единое решение по управлению локальными подключениями, можно интегрировать несколько веб-приложений с одной [виртуальной сетью Azure] с VPN типа «сеть-сеть». После этого у вас будет доступ к локальным ресурсам, как если бы ваши веб-приложения были расположены локально.

## Оптимизация

Автоматическое масштабирование с помощью Autoscale, кэширование с помощью кэша Redis для Azure, запуск фоновых задач с помощью заданий WebJob и высокая доступность благодаря Traffic Manager Azure помогут оптимизировать бизнес-приложения.

Возможность [вертикального и горизонтального масштабирования] веб-приложений службы приложений позволяет удовлетворить требования бизнес-приложения независимо от размера рабочей нагрузки. Разверните веб-приложение вручную с помощью [портала управления Azure], программным путем с помощью [API управления службами] или [сценариев PowerShell] либо автоматически с помощью функции автомасштабирования. В рамках уровня **Standard** автомасштабирование позволяет развернуть веб-приложение автоматически исходя из уровня использования ЦП. Рекомендации см. в статье [Троя Ханта] [10 принципов быстрого масштабирования веб-приложений с помощью Azure].

Увеличьте скорость отклика веб-приложения с помощью [кэша Redis для Azure]. Используйте его, чтобы кэшировать данные из серверных баз данных и другие элементы, например [состояние сеанса ASP.NET] и [кэш вывода].

Поддерживайте высокую доступность веб-приложения с помощью [диспетчера трафика Azure]. Если на основном сайте возникает проблема, при использовании метода **отработки отказа** диспетчер трафика Azure автоматически направляет трафик на дополнительный сайт.

## Мониторинг и анализ

Средства Azure и сторонних разработчиков позволяют следить за производительностью веб-приложения. Вы сможете получать оповещения о критически важных событиях в веб-приложении. Благодаря Application Insights и средствам аналитики веб-журналов HDInsight у вас будет полное представление о пользователях.

[Ознакомьтесь] с текущими показателями производительности веб-приложения и квотами ресурсов в колонке веб-приложения на [портале предварительной версии Azure](http://go.microsoft.com/fwlink/?LinkId=529715). Чтобы получить полное представление о доступности, производительности и использовании приложений, можно воспользоваться инструментом [Azure Application Insights]. Это средство позволяет быстро и эффективно устранять неполадки, проводить диагностику и получать данные об использовании. Также существует возможность вести расширенный мониторинг данных ваших веб-приложений с помощью средств сторонних разработчиков, например [New Relic].

В рамках уровня **Standard** посредством почтовых уведомлений можно получать мониторинг отклика приложения, как только оно перестает отвечать. Дополнительные сведения см. в разделе [Практическое руководство. Получение уведомлений об оповещениях и управление правилами оповещения в Azure].
