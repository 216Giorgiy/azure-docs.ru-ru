# Создание кампании цифрового маркетинга на основании веб-приложений службы приложений Azure
Веб-приложения [службы приложений Azure](http://go.microsoft.com/fwlink/?LinkId=529714) отлично подходят для создания цифровых маркетинговых кампаний. Цифровые маркетинговые кампании обычно являются краткосрочными и предназначены для достижения кратковременных маркетинговых целей. Необходимо рассмотреть два основных сценария. В первом сценарии сторонняя маркетинговая фирма создает кампанию для своего клиента и управляет этой кампанией в течение рекламной акции. Второй сценарий задействует маркетинговую фирму, создающую ресурсы цифровой маркетинговой кампании и затем передающую своему клиенту право владения этими ресурсами. Далее клиент запускает цифровую маркетинговую кампанию и управляет этой кампанией на свое усмотрение. Веб-сайты Azure отлично подходят для обоих сценариев.

>[AZURE.NOTE]Чтобы приступить к работе со службой приложений Azure до создания учетной записи Azure, перейдите к разделу [Пробное использование службы приложений](http://go.microsoft.com/fwlink/?LinkId=523751), где вы можете быстро создать кратковременное веб-приложение начального уровня в службе приложений. Никаких кредитных карт и обязательств.

Ниже приведен пример глобальной, многоканальной цифровой маркетинговой кампании с использованием веб-приложений службы приложений Azure. В нем демонстрируются возможности, которые обеспечивает объединение веб-приложений службы приложений с другими службами при минимальных инвестициях в технологии. **Щелкните элемент в топографии, чтобы узнать о нем больше.**

## Создание с нуля или перенос существующих ресурсов

Быстро создавайте новые веб-приложения на основе популярных систем управления содержимым (CMS) из коллекции или переносите существующие веб-ресурсы в веб-приложения службы приложений благодаря поддержке различных языков и платформ.

На сайте Azure Marketplace есть шаблоны из популярных систем управления содержимым (CMS), таких как [Orchard], [Umbraco], [Drupal] и [WordPress]. Вы сможете создать веб-приложение с помощью выбранной версии CMS. Широкий выбор внутренних серверов для баз данных, в частности для [баз данных SQL Azure] и [MySQL], позволяет подобрать сервер в соответствии со своими потребностями.

Существующие веб-ресурсы можно запускать в веб-приложениях, независимо от языка и платформы (.NET, PHP, Java, Node.js или Python). Их можно перемещать в веб-приложения с помощью известных [FTP-инструментов]. Если вы часто создаете цифровые маркетинговые кампании, возможно, что у вас уже есть веб-ресурсы в системе управления версиями. Веб-приложения можно развернуть непосредственно из популярных систем управления версиями, таких как [Visual Studio], [Visual Studio Online] и [Git] (локальная система, GitHub, BitBucket, DropBox, Mercurial и т. д.).

## Гибкость

Будьте гибкими — постоянно публикуйте непосредственно из существующей системы управления версиями и запускайте тестирование A/B в веб-приложениях службы приложений.

Во время планирования, создания прототипов и на ранних этапах разработки веб-приложения вы и ваш клиент можете просматривать реальные рабочие версии кампании перед ее запуском путем [развертывания в промежуточном слоте] веб-приложения. Интеграция системы управления версиями с веб-приложениями службы приложений позволяет [постоянно публиковать] содержимое в промежуточном слоте, а когда все будет готово, перенести содержимое в рабочую среду без каких-либо простоев.

Кроме того, при планировании изменений для действующего веб-приложения вы можете легко [выполнить A/B-тестирование] предложенных обновлений (с помощью функции тестирования в рабочей среде) и проанализировать поведение реальных пользователей. Такая возможность позволит принимать обоснованные решения относительно проекта приложения.


## Использование социальных сетей

Цифровую маркетинговую кампанию в веб-приложениях службы приложений можно интегрировать с социальными сетями путем аутентификации у таких известных поставщиков, как Facebook и Twitter. Пример использования такого подхода с приложением ASP.NET см. в статье [Создание приложения ASP.NET MVC с проверкой подлинности и базой данных SQL и развертывание службы приложений Azure].

Кроме того, каждый сайт социальной сети обычно предоставляет информацию о других способах интеграции с .NET и прочими платформами.

## Использование мультимедиа и поддержка всех устройств

Улучшите цифровую маркетинговую кампанию с помощью других служб Azure, таких как:

-  отправка и потоковая передача видео по глобальным каналам с помощью [службы мультимедиа Azure];
-  отправка электронных сообщений пользователям с помощью [службы SendGrid в Azure Marketplace];
-  присутствие на устройства с ОС Windows, iOS и Android с помощью [мобильных служб];
-  отправка push-уведомлений на большое количество устройств с помощью [концентратора уведомлений].

## Глобализация

Перейдите на глобальный уровень, обслуживая региональные сайты с помощью диспетчера Azure Traffic Manager и мгновенной доставки содержимого с помощью Azure CDN.

Чтобы обслуживать глобальных клиентов в соответствующих областях, используйте [диспетчер трафика Azure]. С его помощью вы сможете направлять посетителей на соответствующие региональные сайты, что будет обеспечивать оптимальную производительность. Кроме того, можно равномерно распределить нагрузку размещенного в нескольких регионах сайта среди нескольких копий веб-приложений.

Мгновенно доставляйте статическое содержимое пользователям по всему миру благодаря [интеграции веб-приложения с Azure CDN]. Azure CDN кэширует статическое содержимое в ближайшем к пользователю [узле CDN], что сводит к минимуму задержку и количество подключений к веб-приложению.

## Оптимизация

Функция автоматического масштабирования, кэширование с помощью кэша Redis для Azure, запуск фоновых задач с помощью веб-заданий и высокая доступность благодаря диспетчеру трафика Azure помогут оптимизировать веб-приложение.

Возможность [масштабирования и развертывания] веб-приложений службы приложений идеально подходит для непрогнозируемых рабочих нагрузок, в частности для цифровых маркетинговых кампаний. Развернуть веб-приложение можно вручную с помощью [портала предварительной версии Azure](http://go.microsoft.com/fwlink/?LinkId=529715), программным путем с помощью [API управления службами] или [сценариев PowerShell] либо автоматически с помощью функции автоматического масштабирования. В рамках уровня **Standard** автомасштабирование позволяет развернуть веб-приложение автоматически исходя из уровня использования ЦП. Эта функция позволяет максимально увеличить гибкость и свести к минимуму затраты за счет развертывания веб-приложения по мере необходимости (в зависимости от активности пользователей). Рекомендации см. в статье [Троя Ханта] [10 принципов быстрого масштабирования веб-приложений с помощью Azure].

Увеличьте скорость отклика веб-приложения с помощью [кэша Redis для Azure]. Используйте его, чтобы кэшировать данные из серверных баз данных и другие элементы, например [состояние сеанса ASP.NET] и [кэш вывода].

Поддерживайте высокую доступность веб-приложения с помощью [диспетчера трафика Azure]. Если на основном сайте возникает проблема, при использовании метода **отработки отказа** диспетчер трафика Azure автоматически направляет трафик на дополнительный сайт.

## Мониторинг и анализ

Средства Azure и сторонних разработчиков позволяют следить за производительностью веб-приложения. Вы сможете получать оповещения о критически важных событиях в веб-приложении. Благодаря Application Insights и средствам аналитики веб-журналов HDInsight у вас будет полное представление о пользователях.

[Ознакомьтесь] с текущими показателями производительности веб-приложения и квотами ресурсов в колонке веб-приложения на [портале предварительной версии Azure](http://go.microsoft.com/fwlink/?LinkId=529715). Чтобы получить полное представление о доступности, производительности и использовании приложений, можно воспользоваться инструментом [Azure Application Insights]. Это средство позволяет быстро и эффективно устранять неполадки, проводить диагностику и получать данные об использовании. Также существует возможность вести расширенный мониторинг данных ваших веб-приложений с помощью средств сторонних разработчиков, например [New Relic].

В рамках уровня **Standard** существует возможность мониторинга отклика: вы получаете уведомления электронной почты, если ваше веб-приложение перестает отвечать. Дополнительные сведения см. в разделе [Практическое руководство. Получение уведомлений об оповещениях и управление правилами оповещения в Azure].
