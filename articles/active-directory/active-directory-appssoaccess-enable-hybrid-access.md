# Гибридный доступ с прокси-сервером приложений
С прокси-сервером приложений Microsoft Azure Active Directory (AD) вы можете предоставлять безопасный доступ к приложениям, размещенным в частной сети, из любого места и с любого устройства. После установки в своей среде соединителя прокси-сервера приложения его можно легко настроить с Azure AD.

Существует только одно требование для включения доступа к веб-приложению: веб-приложение должно быть доступно серверу, на котором установлен соединитель. Другими словами, соединитель можно установить либо на самих серверах приложений, либо на любом другом сервере в вашей среде, так чтобы соединителю было доступно веб-приложение.

##Принцип работы
### Краткий обзор
1. Соединители разворачиваются в локальной сети. (Для обеспечения избыточности и масштабирования можно развернуть несколько соединителей.)
2. Соединитель подключается к облачной службе.
3. Соединитель и облачная служба направляют пользовательский трафик к приложениям.

 ![Схема прокси-сервера приложений AzureAD](./media/active-directory-appssoaccess-whatis/azureappproxxy.png)

### Более подробный взгляд
1. Пользователь обращается к приложению через прокси-сервер приложений и перенаправляется на страницу входа в Azure AD для проверки подлинности.
2. После успешного входа в систему создается маркер и отправляется пользователю.
3. Пользователь отправляет маркер прокси-серверу приложений, который извлекает из маркера имя субъекта пользователя (UPN) и имя субъекта безопасности (SPN) и направляет запрос соединителю.
4. Соединитель от имени пользователя запрашивает билет Kerberos, который может использоваться для внутренней проверки подлинности (Windows). Это так называемое ограниченное делегирование Kerberos.
5. Active Directory выдает билет Kerberos.
6. Билет отправляется серверу приложений и проверяется.
7. Ответ отправляется конечному пользователю через прокси-сервер приложений.
