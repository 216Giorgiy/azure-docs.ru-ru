# Публикация приложений с помощью прокси приложения Azure AD

> [AZURE.NOTE]Прокси приложения — это функция, которая доступна только после обновления до выпуска Premium или Basic службы Azure Active Directory. Дополнительные сведения см. в статье [Выпуски Azure Active Directory](https://msdn.microsoft.com/library/azure/dn532272.aspx).

После включения прокси приложения Microsoft Azure AD можно публиковать приложения, чтобы сделать их доступными для пользователей за пределами частной сети.

В этом разделе описываются действия, которые необходимо выполнить для публикации приложений, работающих в локальной сети, к которым вы хотите разрешить безопасный удаленный доступ извне сети.

> [AZURE.NOTE]Чтобы убедиться, что соединитель работает правильно, первым приложением, которое вы публикуете, должен быть любой веб-сайт, доступный в частной сети. Так вы сможете убедиться, что он доступен пользователям из Интернета, прежде чем действительно опубликовать приложение.

## Публикация приложения с помощью мастера

1. Откройте предпочитаемый вами браузер и перейдите на портал управления Azure.
2. В левой области на портале управления Azure перейдите на вкладку «Active Directory».
3. Выберите каталог, в котором включен прокси приложения и для которого требуется опубликовать приложение (например, Wingtip Toys).
4. Щелкните **Приложения**, а затем нажмите кнопку **Добавить**, расположенную в нижней части экрана. ![Добавление приложения](http://i.imgur.com/wEeZ7IR.png)
5. В диалоговом окне «Что необходимо сделать?» щелкните **Опубликовать приложение для доступа извне сети**. ![Новое приложение, которое будет доступно извне вашей сети](http://i.imgur.com/Wj3vkyD.png)

6. Следуйте указаниям на экране, чтобы предоставить следующие сведения о приложении: ![Свойства приложения](http://i.imgur.com/MkPNc93.png)

**Параметр** | **Дополнительные сведения**
---|---
Внешний URL-адрес | Это URL-адрес облачной службы, используемый для доступа к приложению извне частной сети. URL-адрес создается автоматически на основе указанного имени с суффиксом msappproxy.net.
Метод предварительной проверки подлинности | <p>Задайте тип метода предварительной аутентификации, используемой приложением: </p><p>а. Microsoft Azure Active Directory (Microsoft Azure AD) — задает метод предварительной аутентификации Microsoft Azure Active Directory (Microsoft Azure AD). Когда пользователь пытается получить доступ к приложению, прокси приложения будет перенаправлять пользователя для входа в среду Azure AD, которая аутентифицирует его. Это гарантирует, что у пользователя есть необходимые разрешения для каталогов и приложения.</p> <p>б. Транзитная предварительная аутентификация не выполняется.</p>
Протокол внешнего URL-адреса | <p>По умолчанию приложения публикуются с использованием протокола HTTPS.</p> <p>Чтобы включить протокол HTTP для внутреннего приложения, вам следует задать транзитную предварительную аутентификацию, и затем вы сможете изменить протокол внешнего URL-адреса с HTTPS на HTTP.</p> <p>Обратите внимание, что публикация приложений с помощью протокола HTTP может вызвать проблемы безопасности для приложения и пользователей.</p> <p>Можно добавить пользовательский домен вместо используемого по умолчанию msappproxy.net. Дополнительную информацию см. ниже.</p>
Внутренний URL-адрес | Это внутренний URL-адрес, используемый соединителем прокси приложения для внутреннего доступа к приложению. Это должен быть URL-адрес опубликованного приложения, который используется для доступа к приложению из частной сети. Это допустимый URL-адрес без пути, а также без пробелов и символов.

Чтобы завершить работу мастера, установите флажок в нижней части экрана. Теперь приложение определено в Azure AD.



## Назначение пользователей и групп для приложения

1. Для приложений с предварительной аутентификацией необходимо назначить пользователей и группы, которые будут иметь доступ к приложению. <p>Приложения, использующие транзитную аутентификацию, доступны всем пользователям. Тем не менее, чтобы пользователь видел приложение в своем списке приложений, вы должны назначить это приложение пользователю.
2. После завершения работы мастера добавления приложения отображается страница быстрого запуска прокси приложения. Чтобы назначить пользователей, щелкните **Назначить пользователей**. ![Экран быстрого запуска прокси приложения](http://i.imgur.com/OmuWElV.png)
3. Выберите пользователей или группы, которым необходимо назначить это приложение, и щелкните **Назначить**. 

> [AZURE.NOTE]Для приложений со встроенной проверкой подлинности Windows можно назначить только пользователей и группы, которые синхронизированы в локальной службе Active Directory. Для приложений, опубликованных с помощью прокси приложения Azure Active Directory, нельзя назначить пользователей, использующих для входа учетную запись Майкрософт, а также гостей. Убедитесь, что назначаемые пользователи входят в систему с помощью своих учетных данных, содержащихся в том же домене, что и публикуемое приложение.

## Расширенная конфигурация

1. На странице **Настройка** можно изменить опубликованные приложения или настроить дополнительные параметры, например единый вход для локальных приложений.![Расширенная конфигурация](http://i.imgur.com/FhbRvoq.png)

2. Выберите приложение и щелкните **Настройка**. Доступны следующие параметры:

**Параметр** | **Дополнительные сведения**
---|---
Имя | Введите описательное имя приложения.
Внешний URL-адрес | Это URL-адрес облачной службы, используемый для доступа к приложению извне частной сети. URL-адрес создается автоматически на основе указанного имени с суффиксом **msappproxy.net**.
Метод предварительной проверки подлинности | Задайте тип метода предварительной аутентификации:<p>Active Directory — задает метод предварительной аутентификации Active Directory. Когда пользователь пытается получить доступ к приложению, прокси приложения будет перенаправлять пользователя для входа в прокси приложения, который аутентифицирует его. Это гарантирует, что у пользователя есть необходимые разрешения для каталогов и приложения.</p><p> Транзитная предварительная аутентификация не выполняется.</p>
Внешний URL-адрес | По умолчанию приложения публикуются с использованием протокола HTTPS. Чтобы включить протокол HTTP для внутреннего приложения, вам следует задать транзитную предварительную аутентификацию, и затем вы сможете изменить протокол внешнего URL-адреса с HTTPS на HTTP. Обратите внимание, что публикация приложений с помощью протокола HTTP может вызвать проблемы безопасности для приложения и пользователей. Можно задать пользовательский домен вместо используемого по умолчанию msappproxy.net. Дополнительную информацию см. в разделе «Работа с личными доменами».
Внутренний URL-адрес | Это внутренний URL-адрес, используемый соединителем прокси приложения для внутреннего доступа к приложению. Это должен быть URL-адрес опубликованного приложения, который используется для доступа к приложению из частной сети. Это допустимый URL-адрес без пути, а также без пробелов и символов.
Перевод веб-страниц в заголовках | Для приложений (например, некоторых конфигураций SharePoint), требующих, чтобы заголовки HTTP узлов не преобразовывались, задайте значение «Нет». Это отключит преобразование заголовков запросов и ответов.
Метод внутренней проверки подлинности | Если для предварительной аутентификации используется прокси приложения, можно задать метод внутренней аутентификации, чтобы пользователи могли использовать единый вход для этого приложения. <p> Выберите **встроенную проверку подлинности Windows (IWA)**, если ваше приложение использует встроенную проверку подлинности Windows и вы можете настроить ограниченное делегирование Kerberos (KCD), чтобы включить единый вход для этого приложения. <p> Выберите **Нет**, если ваше приложение не использует встроенную проверку подлинности Windows. <p> Приложения, использующие встроенную проверку подлинности Windows, должны быть настроены с использованием KCD. В противном случае прокси приложения не сможет опубликовать эти приложения. <p> Дополнительную информацию см. в разделе «Единый вход для локальных приложений со встроенной проверкой подлинности Windows с использованием KCD и прокси приложения».
Внутреннее имя субъекта-службы приложения | Это имя субъекта-службы (SPN) внутреннего приложения, настроенное в локальном прокси приложения. Имя субъекта-службы используется соединителем прокси приложения для получения токенов Kerberos для приложения, использующего ограниченное делегирование Kerberos (KCD). <p> Дополнительную информацию см. в разделе «Единый вход для локальных приложений со встроенной проверкой подлинности Windows с использованием KCD и прокси приложения».

После публикации приложений с помощью прокси приложения Azure Active Directory они отображаются в списке приложений в Azure AD, и вы можете управлять ими. Если отключить службы прокси приложения после публикации приложений, приложения не будут удалены, но станут недоступны извне частной сети. Чтобы просмотреть приложение и убедиться, что оно доступно, дважды щелкните имя приложения. Если служба прокси приложения отключена и приложение недоступно, в верхней части экрана появится предупреждающее сообщение. Чтобы удалить приложение, выберите его из списка и щелкните **Удалить**.
