# <a name="eligible-assignments-and-resource-visibility-in-pim"></a>Допустимые назначения и видимость ресурсов в PIM

Управление привилегированными пользователями (PIM) для ролей ресурсов Azure предоставляет расширенное управление безопасностью для организаций с критически важными ресурсами Azure. Администраторы ресурсов могут использовать PIM для назначения допустимых членов для ролей ресурса. Дополнительные сведения о типах и состояниях назначений для ролей ресурсов Azure представлены в следующих разделах. 

## <a name="assignment-types"></a>Типы назначений

Управление привилегированными пользователями для ресурсов Azure поддерживает два типа назначений:

- допустимое;
- Активна

Назначение допустимой роли означает, что член роли должен выполнить определенное действие для использования этой роли. Поддерживаются такие действия, как прохождение многофакторной идентификации, предоставление коммерческого обоснования или запрос утверждения от назначенных утверждающих.

Назначение активной роли не предусматривает дополнительных действий члена для использования роли. Члены, назначенные в качестве активных, постоянно имеют все полномочия, присвоенные роли.

## <a name="assignment-duration"></a>Длительность назначений

Администраторы ресурсов могут выбирать из двух вариантов для каждого типа назначения в процессе настройки параметров управления привилегированными пользователями для роли. Эти параметры определяют максимальную длительность по умолчанию для членов, которым назначается эта роль в управлении привилегированными пользователями. 

Администратор может выбрать один из этих типов назначения:

- Разрешить постоянную допустимую роль
- Разрешить постоянную активную роль

Или же администратор может выбрать один из этих типов назначения:

- Установить срок действия допустимой роли
- Установить срок действия активной роли

Если администратор ресурса выбрал вариант **Разрешить постоянную допустимую роль** или **Разрешить постоянную активную роль**, все администраторы, назначающие членов роли для этого ресурса, смогут присваивать постоянное членство.

Если же администратор ресурса выбрал вариант **Установить срок действия допустимой роли** или **Установить срок действия активной роли**, он сможет контролировать жизненный цикл назначения, устанавливая обязательные даты для начала и окончания назначений.

> [!NOTE] 
> Администраторы ресурса могут обновлять назначения с указанной датой окончания. Кроме того, члены могут самостоятельно создавать запросы на [продление или обновление назначений](pim-resource-roles-renew-extend.md).


## <a name="assignment-states"></a>Состояние назначений

Управление привилегированными пользователями для ресурсов Azure поддерживает два состояния назначений, которые отображаются на вкладке **Активные роли** в представлениях **Мои роли**, **Роли** и **Члены**. Возможны следующие состояния:

- назначено;
- активировано.

При просмотре списка членов в представлении **Активные роли** значение в столбце **Состояние** позволяет различать пользователей, которым назначена активная роль (**Назначено**) и которые выполнили действие активации для допустимой роли (**Активировано**).

## <a name="azure-resource-role-approval-workflow"></a>Рабочий процесс утверждения роли ресурса Azure

С помощью рабочего процесса утверждения в PIM для ролей ресурсов Azure администраторы могут дополнительно защитить критически важные ресурсы или ограничить доступ к ним. С этой целью они могут затребовать утверждение для активации назначения ролей.

Для ролей ресурсов Azure существует уникальная концепция иерархии ресурсов. Благодаря этой иерархии все нижестоящие дочерние ресурсы в родительском контейнере могут наследовать назначения ролей от объекта родительского ресурса. 

Например, администратор ресурсов Кирилл использует PIM, чтобы назначить Алису в качестве правомочного участника роли "Владелец" в подписке Contoso. После применения такого назначения Алиса становится владельцем всех контейнеров группы ресурсов в рамках подписки Contoso. Она также является владельцем всех ресурсов (например, виртуальных машин), содержащихся во всех группах ресурсов подписки.

Предположим, что в подписке Contoso есть три группы ресурсов: Fabrikam Test, Fabrikam Dev и Fabrikam Prod. Каждая из этих групп ресурсов содержит одну виртуальную машину.

Параметры PIM настраиваются для каждой роли ресурса. В отличие от назначений, эти параметры не наследуются и применяются строго к роли ресурса.

В примере выше Кирилл использует PIM, чтобы потребовать от всех участников в роли "Владелец" подписки Contoso отправки запроса на утверждение активации. Чтобы защитить ресурсы в группе ресурсов Fabrikam Prod, Кирилл также требует утверждения участников роли "Владелец" этого ресурса. Роли владельца в Fabrikam Test и Fabrikam Dev не требуют утверждения для активации.

Когда Алиса отправляет запрос на активацию своей роли "Владелец" в подписке Contoso, утверждающий должен одобрить или отклонить этот запрос. Если Алиса решит [расширить область активации](pim-resource-roles-activate-your-roles.md) на группу ресурсов Fabrikam Prod, утверждающий также должен утвердить или отклонить этот запрос. Однако, если Алиса решит расширить область активации на группу Fabrikam Test, Fabrikam Dev или обе группы, утверждение не требуется.

Рабочий процесс утверждения может не понадобиться для всех участников роли. Предположим, ваша компания привлекла нескольких консультантов для помощи в разработке приложения, которое будет выполняться в подписке Azure. Вы как администратор ресурсов хотите, чтобы сотрудники получали доступ без обязательного утверждения, но партнеры по контракту должны отправлять запрос на утверждение. Чтобы настроить рабочий процесс утверждения только для партнеров по контракту, вы можете создать настраиваемую роль с теми же разрешениями, что и для роли, назначенной сотрудникам. Вы можете запросить утверждение для активации этой настраиваемой роли. [Дополнительные сведения о настраиваемых ролях](pim-resource-roles-custom-role-policy.md).
