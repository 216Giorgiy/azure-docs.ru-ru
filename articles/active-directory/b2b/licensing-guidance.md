# <a name="azure-active-directory-b2b-collaboration-licensing-guidance"></a>Руководство по лицензированию службы совместной работы Azure Active Directory B2B

Благодаря службе совместной работы Azure Active Directory (Azure AD) B2B вы можете приглашать внешних пользователей (или "гостевых пользователей") использовать ваши платные службы Azure AD. Для каждой платной лицензии Azure AD, которую можно присвоить пользователю, вы можете пригласить до пяти гостевых пользователей в пределах квоты внешних пользователей.

Гостевой пользователь — это лицо, которое не является членом вашей организации или ее дочерних компаний. Гостевые пользователи определяются по их связи с вашей организацией, а не по учетным данным, которые они используют для входа. Гостевой пользователь может войти с помощью внешнего идентификатора или учетных данных, принадлежащих вашей организации.

Ниже приведены пользователи, которые *не* являются гостевыми:
- Сотрудники, внутренние подрядчики или агенты организации.
- Сотрудники, внутренние подрядчики или агенты дочерних компаний организации.

На основе соотношения 1:5 предусмотрено автоматическое вычисление лицензирования гостевых пользователей B2B и формирование соответствующего отчета. В настоящее время невозможно присвоить лицензии гостевых пользователей B2B непосредственно для гостевых пользователей.

Существуют ситуации, когда для гостевого пользователя не формируется отчет с учетом квоты внешних пользователей 1:5. Если гостевой пользователь уже имеет платную лицензию Azure AD в собственной организации, для него не используется лицензия гостевого пользователя B2B. Гостевые пользователи также могут пользоваться бесплатными возможностями Azure AD без дополнительных требований лицензирования. У них есть доступ к бесплатным возможностям Azure AD, даже если у вас нет платных лицензий Azure AD. 

## <a name="examples-calculating-guest-user-licenses"></a>Примеры: вычисление лицензий гостевых пользователей
Когда вы определите количество гостевых пользователей, которым требуется доступ к вашим платным службам Azure AD, приобретите для них достаточно платных лицензий Azure AD в требуемом соотношении 1:5. Ниже приведены некоторые примеры:

- Вы хотите пригласить 100 гостевых пользователей в ваши приложения или службы Azure AD и присвоить права управления доступом и подготовки для всех гостевых пользователей. Вам также требуется Многофакторная идентификация и условный доступ для 50 из этих гостевых пользователей. Для этой комбинации вам потребуется 10 лицензий Azure AD Basic и 10 лицензий Azure AD Premium P1. Если вы планируете использовать для гостевых пользователей функции защиты идентификации, вам потребуются лицензии Azure AD Premium P2 в соотношении 1:5 для охвата гостевых пользователей.
- Вы хотите пригласить 60 гостевых пользователей, всем из которых требуется MFA, поэтому у вас должно быть не менее 12 лицензий Azure AD Premium P1. У вас есть 10 сотрудников с лицензиями Azure AD Premium P1. Значит при соотношении лицензирования 1:5 в группе может быть до 50 гостевых пользователей. Необходимо также приобрести две дополнительные лицензии Premium P1, чтобы охватить 10 дополнительных гостевых пользователей.

## <a name="using-the-b2b-collaboration-api-to-invite-users-from-your-affiliates"></a>Использование API службы совместной работы B2B для приглашения пользователей из дочерних компаний

По определению гостевой пользователь B2B является внешним пользователем, которого вы приглашаете для использования ваших платных приложений и служб Azure AD. Сотрудник, внутренний подрядчик или агент организации либо одной из ее дочерних компаний не будет соответствовать требованиям для совместной работы B2B, даже если используются возможности B2B. Ниже приведены некоторые примеры: 
- Вы хотите использовать внешние учетные данные (например, удостоверение социальной сети), чтобы пригласить пользователя, который является сотрудником организации. Этот сценарий не соответствует требованиям лицензирования, и его нельзя использовать. Сотрудник не становится внешним пользователем, если он использует внешние учетные данные.  
- Вы хотите использовать API B2B для приглашения пользователя из одной из дочерних компаний организации. Несмотря на то что в этом сценарии для приглашения пользователя используется API B2B, это не считается совместной работой B2B. Это не соответствует требованиям лицензирования, так как пользователь из дочерней компании не является внешним пользователем. 

В обеих ситуациях лучше использовать API B2B для приглашения пользователей как членов (invitedUserType = Member) и присвоить им по лицензии Azure AD.
