# <a name="conditional-access-for-b2b-collaboration-users"></a>Условный доступ пользователей в службе совместной работы B2B

## <a name="multi-factor-authentication-for-b2b-users"></a>Многофакторная идентификация для пользователей B2B
Благодаря службе совместной работы Azure AD B2B организации имеют уникальную возможность применять политики Многофакторной идентификации к пользователям B2B. Эти политики можно применять на уровне клиента, приложения или отдельных пользователей. Кроме того, эти политики можно применять к штатным сотрудникам и членам организации. Политики MFA применяются на уровне организации ресурсов.

Это означает следующее:
1. Администратор или информационный работник компании А приглашает пользователя из компании Б в приложение Foo, принадлежащее компании A.
2. При доступе в приложение *Foo*, принадлежащее компании A, настроено обязательное прохождение MFA.
3. Когда пользователь из компании Б пытается получить доступ к приложению Foo из клиента компании А, то ему предлагается пройти Многофакторную идентификацию согласно требованиям политики MFA компании А.
4. Пользователь может настроить параметры прохождения MFA, согласованные с компанией А.
5. Эти настройки будут действительны для любого идентификатора (например, Azure AD или MSA, если пользователи из компании Б используют для аутентификации внешние идентификаторы).
6. У компании А должны быть номера SKU Azure AD уровня "Премиум", поддерживающие MFA. Пользователь из компании Б будет использовать эту лицензию компании А.

Таким образом, ответственность за применение MFA по отношению к пользователям службы совместной работы B2B из партнерской организации *всегда* лежит на приглашающем клиенте, а не на партнерской организации (даже если она имеет возможности применения MFA).

### <a name="mfa-reset-for-b2b-collaboration-users"></a>Сброс настроек MFA для пользователей службы совместной работы B2B
В настоящее время администратор может потребовать от пользователей службы совместной работы B2B пройти повторную проверку только с помощью приведенных ниже командлетов PowerShell. Поэтому следует использовать эти командлеты, когда требуется сбросить настройки метода проверки для пользователя службы совместной работы B2B.

1. Подключение к Azure AD

  ```
  $cred = Get-Credential
  Connect-MsolService -Credential $cred
  ```
2. Получение сведений о методах проверки для всех пользователей.

  ```
  Get-MsolUser | where { $_.StrongAuthenticationMethods} | select UserPrincipalName, @{n="Methods";e={($_.StrongAuthenticationMethods).MethodType}}
  ```
  Пример:

  ```
  PS C:\Users\tjwasserGet-MsolUser | where { $_.StrongAuthenticationMethods} | select UserPrincipalName, @{n="Methods";e={($_.StrongAuthenticationMethods).MethodType}}
  ```

3. Сброс настроек метода MFA для определенного пользователя. Затем можно использовать полученное имя участника-пользователя (UserPrincipalName) для выполнения команды сброса настроек. Эта команда потребует от пользователя службы совместной работы B2B повторно настроить методы проверки. Пример:

  ```
  Reset-MsolStrongAuthenticationMethodByUpn -UserPrincipalName gsamoogle_gmail.com#EXT#@ WoodGroveAzureAD.onmicrosoft.com
  ```

### <a name="why-do-we-perform-mfa-at-the-resource-tenancy"></a>Зачем выполнять MFA на уровне клиента ресурса?

В текущем выпуске MFA всегда применяется на уровне клиента ресурса. Причина этого заключается в предсказуемости.

Предположим, что пользователь компании Contoso (Светлана) приглашен в компанию Fabrikam, в которой включена MFA для пользователей B2B.

В компании Contoso включена политика для приложения 1, но не для приложения 2. Если посмотреть на утверждение MFA Contoso в токене, чтобы определить, должна ли Светлана проходить MFA в Fabrikam, может возникнуть следующая проблема.

* День 1. Светлана прошла MFA в Contoso, так как она обращается к приложению 1. В таком случае запрос на выполнение MFA в Fabrikam не появится.

* День 2: Светлана обратилась к приложению 2 в Contoso. Теперь при обращении к Fabrikam, она должна зарегистрироваться для выполнения MFA в Fabrikam.

Это может запутать Светлану и, скорее всего, приведет к проблемам при входе в систему.

Кроме того, даже если в Contoso предусмотрена функция выполнения MFA, нет гарантии того, что Fabrikam будет доверять политике MFA в Contoso.

Наконец, клиент ресурса MFA также поддерживает управляемые учетные записи службы (MSA) и внешние идентификаторы партнерских организаций, в которых отсутствует MFA.

Таким образом, мы советуем всегда применять MFA к ресурсам для пользователей B2B. В некоторых случаях это может привести к двойной проверке, но всякий раз при обращении к клиенту ресурса Светлана должна зарегистрироваться для MFA в клиенте ресурса.

### <a name="device-location-and-risk-based-conditional-access-for-b2b-users"></a>Условный доступ на основе устройств, расположения и рисков для пользователей B2B

Если компания Contoso включает условный доступ на основе политик к корпоративным данным, доступ из неуправляемых устройств (устройств, которые не управляются компанией Contoso и не соответствуют ее политикам устройств) блокируется.

Если компания Contoso не управляет устройствами пользователей B2B, тогда доступ этих пользователей из организаций-партнеров будет заблокирован независимо от контекста применения этих политик.

Устройства пользователей из другой организации навряд ли находятся под управлением приглашающей организации. Таким образом, в будущих обновлениях мы сделаем так, что Contoso будет доверять состоянию соответствия устройства конкретного партнера. Это позволит Contoso применять политики, в которых пользователю из Fabrikam будут доступны ресурсы Contoso, если они также используют устройства, управляемые Fabrikam.

В то же время Contoso сможет создать списки исключений, содержащие данные о пользователях конкретного партнера из политики условного доступа на основе устройств.

#### <a name="location-based-conditional-access-for-b2b"></a>Условный доступ на основе расположения для пользователей B2B

Политики условного доступа на основе расположения можно применять к пользователям B2B, если приглашающая организация (например, Contoso) способна создать надежный периметр сети (то есть, диапазон IP-адресов), определяющий партнерские организации (например, Fabrikam).

#### <a name="risk-based-conditional-access-for-b2b"></a>Политики условного доступа на основе рисков для пользователей B2B

Сейчас политики выполнения входа на основе рисков невозможно применять к пользователям B2B, так как оценка рисков выполняется в домашней организации этих пользователей (другими словами, с помощью их клиентов удостоверений).

В будущих обновлениях мы планируем включить в федерацию оценки рисков от партнеров (с согласия партнера), чтобы компания Contoso могла защитить свои внешние приложения и данные не только от известных рисков, но и от рисков, о которых невозможно узнать, так как они могут возникнуть в любом месте.
