# <a name="azure-active-directory-activity-logs-in-azure-monitor-preview"></a>Журналы действий Azure Active Directory в Azure Monitor (предварительная версия)

Теперь можно направлять журналы действий Azure AD с помощью Azure Monitor в учетную запись хранения или концентратор событий. С помощью общедоступной предварительной версии журналов Azure Active Directory в Azur Monitor можно выполнять следующие действия.

* Архивировать журналы аудита для учетной записи хранения Azure, что позволяет сохранять данные в течение длительного времени
* Выполнять потоковую передачу журналов аудита в концентратор событий Azure для аналитики с помощью популярных средств SIEM, таких как Splunk и QRadar
* Интегрировать журналы аудита с собственными пользовательскими решениями журнала путем потоковой передачи их в концентратор событий

> [!VIDEO https://www.youtube.com/embed/syT-9KNfug8]

## <a name="supported-reports"></a>Поддерживаемые отчеты

С помощью этой функции можно направлять журналы действий аудита и входов в систему в учетную запись хранения Azure, концентратор событий или пользовательское решение. 

* **Журналы аудита**. [Отчет о действиях журналов аудита](active-directory-reporting-activity-audit-logs.md) обеспечивает доступ к истории каждой задачи, выполняемой в клиенте Azure.
* **Входы в систему**. [Отчет о действиях входов в систему](active-directory-reporting-activity-sign-ins.md) позволяет определить, кто выполнил задачи, указанные в отчете журналов аудита.

> [!NOTE]
> В настоящее время журналы действий аудита и входов в систему в режиме B2C не поддерживаются.
>

## <a name="prerequisites"></a>Предварительные требования

Для использования этой функции необходимо иметь следующее.

* Подписка Azure. Если у вас нет подписки Azure, можно [зарегистрироваться и получить бесплатную пробную версию](https://azure.microsoft.com/free/).
* [Лицензия](https://azure.microsoft.com/pricing/details/active-directory/) Azure AD Free, "Основная", "Премиум 1" или "Премиум 2" для доступа к журналам Azure AD на портале Azure. 

В зависимости от того, куда необходимо направить данные журнала аудита, необходимо одно из двух следующих.

* Учетная запись хранения Azure, для которой имеется разрешение*ListKeys*. Рекомендуется использовать общую учетную запись хранения вместо учетной записи хранения BLOB-объектов. С ценами на хранение можно ознакомиться на странице [Калькулятор цен](https://azure.microsoft.com/pricing/calculator/?service=storage). 
* Пространство имен концентраторов событий Azure для интеграции со сторонними решениями.

> [!NOTE]
> Для доступа к журналам действий Azure AD необходимо быть *Глобальным администратором* или *Администратором безопасности* в клиенте Azure AD.
>

## <a name="cost-considerations"></a>Рекомендации по стоимости

Если уже имеется лицензия Azure AD, то понадобится подписка Azure для настройки учетной записи хранения и концентратора событий. Подписка Azure предоставляется бесплатно, но необходимо будет оплатить за использование ресурсов Azure, включая учетную запись хранения, которая используется для архивирования, и концентратор событий, который используется для потоковой передачи. Количество данных и, следовательно, понесенные затраты будут значительно отличаться в зависимости от размера клиента.

### <a name="event-hub-messages-for-activity-logs"></a>Сообщения концентратора событий для журналов действий

События отправляются пакетами как одно сообщение примерно с пятиминутными интервалами и содержат все события этого временного интервала. Сообщение в концентраторе событий не может превышать 256 КБ, и если размер всех сообщений временного интервала превышает этот объем, то отправляются несколько сообщений. 

Например, для крупного клиента, насчитывающего более 100 000 пользователей, обычно происходит около 18 событий в секунду, что соответствует 5 400 событиям каждые 5 минут. Поскольку журналы аудита составляют около 2 КБ на каждое событие, это эквивалентно 10,8 МБ данных, следовательно за этот 5-минутный интервал в концентратор событий будут отправляться 43 сообщения. Используйте [Калькулятор цен](https://azure.microsoft.com/pricing/details/event-hubs/) для вычисления точной оценки объема данных, которые ожидаются для приложения.

## <a name="frequently-asked-questions"></a>Часто задаваемые вопросы

Этот раздел содержит часто задаваемые вопросы и известные проблемы с журналами Azure Active Directory в Azure Monitor.

**Вопрос. С чего начать?** 

**Ответ.** Чтобы получить сведения о том, как развертывать этот компонент, начните с раздела [Azure Active Directory activity logs in Azure Monitor (preview)](reporting-azure-monitor-diagnostics-overview.md) (Журналы действий Azure Active Directory в Azure Monitor (предварительный просмотр)). После ознакомления с необходимыми требованиями, просмотрите руководства по настройке и направлению ваших журналов в концентраторы событий.

---

**Вопрос. Какие журналы включены?**

**Ответ.** Журналы входа в систему и журналы аудита доступны для направления через эту функцию, несмотря на то, что события аудита в режиме B2C в настоящее время не поддерживаются. Ознакомьтесь с [Interpret the Azure Active Directory audit logs schema in Azure Monitor (preview)](reporting-azure-monitor-diagnostics-audit-log-schema.md) (Интерпретация схемы журналов аудита Azure Active Directory в Azure Monitor (предварительный просмотр)) и [Interpret the Azure Active Directory sign-in logs schema in Azure Monitor (preview)](reporting-azure-monitor-diagnostics-sign-in-log-schema.md) (Интерпретация схемы журналов входа Azure Active Directory в Azure Monitor (предварительный просмотр)), чтобы узнать, какие типы журналов и какие функциональные журналы поддерживаются в настоящее время. 

---

**Вопрос. Через какое время после выполнения действия соответствующие журналы будут отображаться в концентраторах событий?**

**Ответ.** Журналы должны отображаться в концентраторах событий в пределах от двух до пяти минут с момента выполнения действия. Дополнительные сведения о концентраторах событий см. на странице [Что такое концентраторы событий Azure?](/azure/event-hubs/event-hubs-what-is-event-hubs.md)

---

**Вопрос. Через какое время после выполнения действия соответствующие журналы будут отображаться в учетных записях хранения?**

**Ответ.** Для учетных записей хранения Azure время ожидания составляет от 5 до 15 минут с момента выполнения действия.

---

**Вопрос. Сколько будет стоить хранение данных?**

**Ответ.** Затраты на хранение зависят от размеров журналов, а также от сроков их хранения. Для приблизительной оценки затрат на хранение для клиентов в зависимости от объема созданных журналов ознакомьтесь с разделом [Azure Active Directory activity logs in Azure Monitor (preview)](reporting-azure-monitor-diagnostics-overview.md) (Журналы действий Azure Active Directory в Azure Monitor (предварительный просмотр)).

---

**Вопрос. Сколько будет стоить потоковая передача данных в концентраторы событий?**

**Ответ.** Стоимость потоковой передачи зависит от количества сообщений, получаемых в минуту. Чтобы получить дополнительные сведения о вычислении затрат, а также узнать приблизительную оценку на основе количества сообщений, ознакомьтесь с разделом [Azure Active Directory activity logs in Azure Monitor (preview)](reporting-azure-monitor-diagnostics-overview.md) (Журналы действий Azure Active Directory в Azure Monitor (предварительный просмотр)). 

---

**Вопрос. Какие средства SIEM поддерживаются в настоящее время?** 

**Ответ.** В настоящее время поддерживается Azure Monitor, Splunk, QRadar и Sumologic. Однако единственный инструмент SIEM, который поддерживается для журналов Azure Active Directory, — это Splunk. Дополнительные сведения о работе соединителей см. в разделе [Потоковая передача данных мониторинга Azure в концентратор событий для потребления внешним инструментом](/azure/monitoring-and-diagnostics/monitor-stream-monitoring-data-event-hubs).

---

**Вопрос. Возможно ли получить доступ к данным из концентратора событий без использования внешнего средства SIEM?** 

**Ответ.** Да, для доступа к журналам из своего пользовательского приложения можно использовать статью [Основные сведения о получении сообщений с помощью узла EventProcessorHost в .NET Standard](/azure/event-hubs/event-hubs-dotnet-standard-getstarted-receive-eph).
