# <a name="authenticating-identities-without-passwords-through-microsoft-passport"></a>Проверка подлинности удостоверений без использования паролей с помощью службы Microsoft Passport

Использования текущих способы проверки подлинности лишь с помощью паролей недостаточно для защиты пользователей. Пользователи имеют тенденцию менять пароли и забывать их. Пароли можно раскрыть, украсть, взломать и подобрать методом перебора. Они также трудны для запоминания и уязвимы для атак типа[pass the hash](https://technet.microsoft.com/dn785092.aspx)(передача хэша).

## <a name="about-microsoft-passport"></a>Общие сведения о Microsoft Passport
Microsoft Passport — это метод проверки подлинности на основе открытых/закрытых ключей и сертификатов для организаций и потребителей, обеспечивающий более надежную защиту, чем пароли. Такая форма аутентификации использует учетные данные пары ключей, которые могут заменить пароли и не подвержены таким уязвимостям, как нарушение безопасности, хищение и фишинг.

 Microsoft Passport позволяет пользователю проходить проверку подлинности с учетной записью Майкрософт, Windows Server Active Directory, Microsoft Azure Active Directory (Azure AD) или сторонних служб, поддерживающих проверку подлинности FIDO. После первоначальной двухэтапной проверки во время регистрации в Microsoft Passport служба настраивается на устройстве пользователя, затем пользователь задает жест (Windows Hello или ПИН-код). Для подтверждения своей личности пользователь предоставляет жест. Windows использует Microsoft Passport для проверки подлинности пользователей и предоставления им доступа к защищенным ресурсам и службам.

Закрытый ключ предоставляется исключительно через "жест пользователя", такой как ПИН-код, биометрические данные, удаленное устройство (например смарт-карта, которую пользователь использует для входа в устройство). Эта информация затем связывается с сертификатом или асимметричной парой ключей. Закрытый ключ заверяется аппаратными средствами, если устройство оснащено микросхемой доверенного платформенного модуля (TPM). Закрытый ключ всегда находится в устройстве.

Открытый ключ зарегистрирован в Azure Active Directory и Windows Server Active Directory (для локальных сред). Поставщики удостоверений (IDP) выполняют проверку пользователя путем сопоставления открытого ключа пользователя с закрытым ключом и предоставляют сведения для входа посредством одноразового пароля (OTP), PhoneFactor или иного механизма уведомления.

## <a name="why-enterprises-should-adopt-microsoft-passport"></a>Преимущества внедрения Microsoft Passport для предприятий

Используя Microsoft Passport, предприятия могут обеспечить более надежную защиту своих ресурсов за счет следующего:

* настройка Microsoft Passport с помощью предпочитаемого варианта оборудования (это означает, что ключи будут создаваться на базе TPM 1.2 или TPM 2.0 при наличии. Если доверенный платформенный модуль недоступен, ключ будет создаваться программным обеспечением);

* определение сложности и длины ПИН-кода, а также использования Hello в вашей организации;

* настройка Microsoft Passport для поддержки сценариев, подобных использованию смарт-карт, с помощью отношения доверия на основе сертификатов.

## <a name="how-microsoft-passport-works"></a>Принципы работы Microsoft Passport
1. Ключи генерируются на оборудовании с помощью модуля TPM или программного обеспечения. На многих устройствах имеется встроенная микросхема модуля TPM, обеспечивающая защиту оборудования благодаря интеграции криптографических ключей в устройства. Модуль TPM 1.2 или TPM 2.0 создает ключи или сертификаты, сформированные на основе созданных ключей.

2. Доверенный платформенный модуль подтверждает эти ключи с привязкой к оборудованию.

3. Для снятия блокировки устройства требуется один жест. Этот жест предоставляет доступ к нескольким ресурсам, если устройство присоединено к домену или Azure AD.

## <a name="how-the-microsoft-passport-lifecycle-works"></a>Принципы жизненного цикла Microsoft Passport

![Жизненный цикл Microsoft Passport](./media/active-directory-azureadjoin/active-directory-azureadjoin-microsoft-passport.png)

На предыдущей схеме показаны пара закрытого и открытого ключей и процедура проверки поставщиком удостоверений. Каждое из этих действий подробно рассмотрено ниже.

1. Пользователь подтверждает свою подлинность посредством нескольких встроенных методов проверки (жесты, физические смарт-карты, многофакторная проверка подлинности) и отправляет эти сведения поставщику удостоверений (например в Azure Active Directory или локальный каталог Active Directory).

2. После этого устройство создает ключ, проверяет его, оставляет у себя открытый ключ, добавляет в него операторы станции, подписывает и отправляет этот ключ поставщику удостоверений для регистрации.

4. Сразу после регистрации поставщиком открытой части ключа устройству предписывается выполнить подпись с использованием закрытой части ключа.

5. После этого поставщик удостоверений проверяет информацию и выдает маркер проверки подлинности, который позволяет пользователю и устройству получить доступ к защищенным ресурсам. Поставщики удостоверений могут создать приложения для нескольких платформ или с помощью поддержки браузера (через API JavaScript/Webcrypto) создавать и использовать учетные данные Microsoft Passport для пользователей.

## <a name="the-deployment-requirements-for-microsoft-passport"></a>Требования к развертыванию для Microsoft Passport
### <a name="at-the-enterprise-level"></a>На уровне предприятия

* Организация располагает подпиской Azure.

### <a name="at-the-user-level"></a>На уровне пользователей

* На компьютере пользователя выполняется Windows 10 "Профессиональная" или "Корпоративная".

Подробные инструкции по развертыванию см. в разделе [Включение Microsoft Windows Hello для бизнеса в организации](active-directory-azureadjoin-passport-deployment.md).
