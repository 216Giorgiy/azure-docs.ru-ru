# <a name="preview-managing-single-sign-on-for-enterprise-apps-in-the-new-azure-portal"></a>Управление параметрами единого входа для корпоративных приложений на новом портале Azure (предварительная версия)
> [!div class="op_single_selector"]
> * [Портал Azure](active-directory-enterprise-apps-manage-sso.md)
> * [Классический портал Azure](active-directory-sso-integrate-saas-apps.md)
> 
> 

В этой статье рассматривается управление параметрами единого входа для приложений, в частности тех, которые были добавлены из [коллекции приложений Azure Active Directory (Azure AD)](active-directory-appssoaccess-whatis.md#get-started-with-the-azure-ad-application-gallery), с помощью [портала Azure](https://portal.azure.com). Сейчас эта возможность управления параметрами единого входа с помощью Azure AD доступна в общедоступной предварительной версии. В этой статье описываются новые функции, а также несколько временных ограничений на период действия предварительной версии. [Что есть в предварительной версии?](active-directory-preview-explainer.md)

## <a name="finding-your-apps-in-the-new-portal"></a>Поиск приложений на новом портале
Начиная с сентября 2016 г. все приложения, для которых администратор каталога настроил единый вход в каталоге с помощью [коллекции приложений Azure Active Directory](active-directory-appssoaccess-whatis.md#get-started-with-the-azure-ad-application-gallery) на [классическом портале Azure](https://manage.windowsazure.com), теперь можно просматривать, а также управлять ими на портале Azure.

Эти приложения доступны в разделе **Корпоративные приложения** на портале Azure, ссылку на который можно найти в списке **Другие службы** на [портале](https://portal.azure.com). Корпоративные приложения — это приложения, развернутые и используемые пользователями внутри организации.

![Колонка "Корпоративные приложения"][1]

Выберите **Все приложения** , чтобы просмотреть список всех настроенных приложений, включая приложения, добавленные из коллекции. При выборе приложения загружается колонка ресурсов с отчетами для этого приложения и различными управляемыми параметрами.

Чтобы управлять параметрами единого входа, выберите **Единый вход**.

![Колонка ресурсов приложения][2]

## <a name="single-sign-on-modes"></a>Режимы единого входа
Вверху в колонке **Единый вход** расположено меню **Режим**, в котором можно настроить режим единого входа. Доступны следующие параметры.

* **Вход на основе SAML**. Этот параметр доступен, если приложение поддерживает федеративный единый вход с помощью Azure Active Directory по протоколу SAML 2.0.
* **Вход по паролю**. Этот параметр доступен, если Azure AD поддерживает заполняемую для приложения форму пароля.
* **Вход по ссылке** — ранее назывался "Существующий единый вход". Этот параметр позволяет администраторам помещать ссылку для перехода в приложение на пользовательскую панель доступа Azure AD или в средство запуска приложений Office 365.

Дополнительные сведения об этих режимах см. в разделе [Принцип работы единого входа с Azure Active Directory](active-directory-appssoaccess-whatis.md#how-does-single-sign-on-with-azure-active-directory-work).

## <a name="saml-based-sign-on"></a>Единый вход на основе SAML
При выборе параметра **SAML-based sign on** (Единый вход на основе SAML) откроется колонка, разделенная на четыре части.

### <a name="domains-and-urls"></a>Домены и URL-адреса
В этом разделе приведены все сведения о доменах и URL-адресах приложения, добавленных в каталог Azure AD. Все параметры, которые необходимо указать для включения в приложении единого входа, отображаются непосредственно на экране. Чтобы просмотреть все дополнительные параметры, установите флажок **Показать дополнительные параметры URL-адресов**. Ниже приведен список поддерживаемых параметров.

* **URL-адрес входа** , по которому пользователь переходит для входа в это приложение. Если приложение настроено для выполнения единого входа, инициированного поставщиком услуг, то при переходе пользователя на этот URL-адрес поставщик услуг выполняет необходимое перенаправление в Azure AD для проверки подлинности и входа пользователя. Если это поле заполнено, Azure AD будет использовать этот URL-адрес для запуска приложения из Office 365 и панели доступа Azure AD. Если это поле оставить пустым, Azure AD выполняет единый вход, инициированный поставщиком удостоверений, при запуске приложения из Office 365, с панели доступа Azure AD или при переходе по URL-адресу единого входа Azure AD.
* **Идентификатор**. Этот универсальный код ресурса должен однозначно определять приложение, для которого выполняется настройка единого входа. Это значение, которое Azure AD отправляет назад в приложение в качестве параметра "Аудитория" токена SAML, а приложение должно его проверить. Это значение также отображается как идентификатор сущности во всех метаданных SAML, предоставленных приложением.
* **URL-адрес ответа**. URL-адрес ответа указывает место, где приложение ожидает токен SAML. Он также называется URL-адресом службы обработчика утверждений (ACS). После ввода адресов нажмите кнопку "Далее", чтобы перейти на следующий экран. Этот экран сообщает, какие параметры вам нужно настроить на стороне приложения, чтобы оно принимало токен SAML от Azure AD.
* **Состояние ретранслятора** — это необязательный параметр, используемый, чтобы указать приложению, куда необходимо перенаправить пользователя после завершения аутентификации. Обычно в качестве значения для этого параметра используется допустимый URL-адрес приложения, но в некоторых приложениях это поле заполняется другими значениями. (Дополнительные сведения см. в документации по единому входу для приложения.) Возможность задать состояние ретрансляции — это новая возможность, доступная на новом портале Azure.

### <a name="user-attributes"></a>Атрибуты пользователя
В этом разделе администраторы могут просмотреть и изменить атрибуты, передаваемые в токене SAML, выдаваемом Azure AD каждый раз при входе в приложение.

В первом предварительном выпуске можно изменить только атрибут **Идентификатор пользователя** . Это поле в Azure AD, однозначно определяющее каждого пользователя в приложении. Например, если при развертывании приложения в качестве имени пользователя и уникального идентификатора используется значение адреса электронной почты, то в Azure AD значение необходимо задать в поле user.mail.

Возможность редактирования других атрибутов будет добавлена в последующем выпуске.

### <a name="saml-signing-certificate"></a>Сертификат для подписи токена SAML
В этом разделе приведены сведения о сертификатах, используемых Azure AD для подписи токенов SAML, выдаваемых приложению каждый раз, когда пользователь проходит проверку подлинности. Это позволяет проверять свойства имеющегося сертификата, в том числе дату окончания срока их действия.

Возможность смены сертификата и управления дополнительными параметрами сертификата будет добавлена в последующем выпуске. Обратите внимание, что возможности полного управления сертификатами все еще доступны на [классическом портале Azure](active-directory-sso-certs.md).

### <a name="application-configuration"></a>Конфигурация приложения
В последнем разделе представлена документация и (или) элементы управления, необходимые для настройки Azure Active Directory в качестве поставщика удостоверений для приложения.

В раскрывающемся меню **Настройка приложения** содержатся краткие встроенные инструкции по настройке приложения. Это еще одна новая уникальная функция на новом портале Azure.

## <a name="password-based-sign-on"></a>Единый вход на основе пароля
Если этот параметр поддерживается приложением, чтобы включить его, выберите режим единого входа на основе пароля, а затем щелкните **Сохранить**. Дополнительные сведения о развертывании единого входа на основе пароля см. в разделе [Принцип работы единого входа с Azure Active Directory](active-directory-appssoaccess-whatis.md#how-does-single-sign-on-with-azure-active-directory-work).

![Единый вход на основе пароля][4]

## <a name="linked-sign-on"></a>Связанный единый вход
Если этот параметр поддерживается приложением, чтобы добавить URL-адрес, на который будет перенаправляться пользователь после выбора ссылки на панели доступа Azure AD или в средстве запуска приложений Office 365, выберите режим связанного единого входа. Дополнительные сведения о развертывании единого входа по ссылке (ранее известного как "Существующий единый вход") см. в разделе [Принцип работы единого входа с Azure Active Directory](active-directory-appssoaccess-whatis.md#how-does-single-sign-on-with-azure-active-directory-work).

![Связанный единый вход][5]
