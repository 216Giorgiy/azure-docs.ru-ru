# <a name="properties-of-an-azure-active-directory-b2b-collaboration-user"></a>Свойства пользователя службы совместной работы Azure Active Directory B2B

## <a name="defining-a-b2b-collaboration-user"></a>Определение пользователя службы совместной работы B2B
Пользователь службы совместной работы B2B — это пользователь, у которого UserType = Guest, то есть гостевой пользователь. Этот пользователь обычно является представителем партнерской организации и по умолчанию имеет ограниченные привилегии в каталоге приглашающей организации. В зависимости от потребностей приглашающей организации пользователь службы совместной работы B2B может иметь одно из следующих состояний учетной записи:
1. Пользователь размещен во внешнем экземпляре Azure Active Directory (Azure AD) и представлен в главной организации как гостевой пользователь. В этом случае пользователь B2B входит в систему с помощью учетной записи Azure AD, принадлежащей его домашнему клиенту. Если внешняя организация, к которой принадлежит пользователь, во время приглашения не использует Azure AD, то гостевой пользователь в Azure AD создается по принципу JIT (just in time), когда пользователь активирует приглашение после проверки адреса электронной почты. Это также называется JIT-клиентом или вирусным клиентом.
2. Пользователь размещен в учетной записи Майкрософт и представлен в главной организации как гостевой пользователь. В этом случае гостевой пользователь входит в систему с помощью учетной записи Майкрософт. В обновлении общедоступной предварительной версии службы совместной работы Azure AD B2B внешний идентификатор (не принадлежащий корпорации Майкрософт) приглашенного пользователя, например google.com или аналогичный, создается как учетная запись Майкрософт по принципу JIT во время активации предложения.
3. Пользователь размещен в локальном каталоге Active Directory главной организации и синхронизируется с каталогом Azure AD главной организации. В этом выпуске атрибут UserType такого пользователя в облаке необходимо вручную изменить на guest (гость), используя PowerShell. В будущих выпусках это действие будет выполняться автоматически службой Azure AD Connect.
4. Пользователь размещен в каталоге Azure AD главной организации, его атрибут UserType = Guest, а учетные данные управляются главной организацией.

  ![отображение инициалов приглашающего](media/active-directory-b2b-user-properties/redemption-diagram.png)

## <a name="key-properties-of-the-b2b-collaboration-user"></a>Основные свойства пользователя службы совместной работы B2B

### <a name="usertype"></a>UserType
Этот атрибут указывает связь пользователя с главным клиентом. Он может иметь два значения:
- Member (участник). Сотрудник главной организации, пользователь в платежной ведомости организации. Например, это может быть пользователь, который имеет возможность заходить на сайты, доступные только для внутреннего использования. Этот пользователь не должен считаться внешним сотрудником.
- Guest (гость). Обозначает пользователя, который не считается в компании внутренним пользователем. Это может быть внешний сотрудник, партнер, клиент или подобный пользователь, который в принципе не должен получать служебных записок от Генерального директора или пользоваться социальным пакетом компании.

  > ![NOTE] Атрибут UserType не связан с тем, как пользователь входит в систему, или к какой роли каталога он принадлежит и т. д. Этот атрибут просто показывает связь пользователя с главной организацией и позволяет организации применять политики в зависимости от значения этого атрибута.

### <a name="source"></a>Источник
Как пользователь входит в систему.

- Приглашенный пользователь. Это пользователь, который был приглашен, но еще не активировал свое приглашение.

- Внешний каталог Active Directory. Этот пользователь размещен во внешней организации и проходит аутентификацию с помощью учетной записи Azure AD, принадлежащей другой организации. Это соответствует состоянию 1, описанному выше.

- Учетная запись Майкрософт. Этот пользователь размещен в MSA и проходит аутентификацию с помощью с учетной записи Майкрософт. Это соответствует состоянию 2, описанному выше.

- Windows Server AD. Этот пользователь синхронизируется из локального каталога Active Directory, принадлежащего этой организации. Это соответствует состоянию 3, описанному выше.

- Azure Active Directory. Этот пользователь проходит аутентификацию с помощью учетной записи Azure AD, принадлежащей этой организации. Это соответствует состоянию 4, описанному выше.

  > ![NOTE] Source и UserType — это независимые атрибуты. Значение атрибута Source не подразумевает определенное значение UserType.

## <a name="can-b2b-users-be-added-as-members-instead-of-guests"></a>Можно ли добавить пользователей службы B2B как участников (Member), а не гостей (Guest)?
Как правило, понятия "пользователь службы B2B" и "гостевой пользователь" являются синонимами. Поэтому пользователь службы совместной работы B2B добавляется как пользователь, у которого по умолчанию UserType = Guest. Однако в некоторых случаях партнерская организация является скорее частью более крупной организации (ассоциации), к которой также относится главная организация. Поэтому главной организации может потребоваться взаимодействовать с пользователями из партнерской организации как с участниками, а не гостями. В этом случае воспользуйтесь интерфейсами API диспетчера приглашений B2B, чтобы добавить или пригласить пользователей из партнерской организации в главную организацию как участников.

## <a name="converting-usertype"></a>Преобразование UserType
В настоящее время с помощью PowerShell можно преобразовывать тип пользователей, меняя значение атрибута UserType с Member на Guest и обратно. При этом предполагается, что свойство UserType представляет связь пользователя с организацией. Таким образом, изменение этого свойства приведет только к изменению связи пользователя с организацией. Если изменилась связь с пользователем, то следует ответить и на другие вопросы, в частности, изменять ли имя участника-пользователя? Должен ли пользователь сохранить доступ к ресурсам, который у него был? Нужно ли назначить почтовый ящик? Следовательно, изменять UserType с помощью PowerShell не рекомендуется, так как это вызывает атомарное действие. Кроме того, в будущих выпусках это свойство станет недоступным для изменения посредством PowerShell, поэтому мы не рекомендуем использовать зависимость от его значения.

## <a name="removing-guest-user-limitations"></a>Снятие ограничений для гостевых пользователей
В определенных случаях требуется предоставить гостевым пользователям привилегии более высокого уровня. Для этого можно добавить гостевого пользователя в любую роль и даже удалить заданные по умолчанию ограничения для гостевых пользователей в каталоге. Это позволит ему пользоваться такими же привилегиями, как у участников. Ознакомьтесь с дополнительными сведениями, чтобы узнать больше.

Ограничения для гостевых пользователей, заданные по умолчанию, можно отключить. При этом гостевые пользователи в каталоге компании получат те же разрешения, что и обычные пользователи (участники).
