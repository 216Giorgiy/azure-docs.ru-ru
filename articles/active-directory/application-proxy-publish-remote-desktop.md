# <a name="publish-remote-desktop-with-azure-ad-application-proxy"></a>Публикация удаленного рабочего стола с помощью прокси приложения Azure AD

В этой статье объясняется, как предоставить удаленным пользователям доступ к развернутым службам удаленных рабочих столов Windows. Развернутые службы удаленных рабочих столов могут располагаться в локальной среде или в частных сетях, например в развертываниях IaaS.

> [!NOTE]
> Прокси приложения — это функция, которая доступна только после обновления службы Azure Active Directory (Azure AD) до выпуска Premium или Basic. Дополнительные сведения см. в разделе [Выпуски Azure Active Directory](active-directory-editions.md).

Трафик по протоколу удаленного рабочего стола можно опубликовать с помощью прокси приложения Azure AD как сквозное приложение прокси. Такое решение обеспечивает не только подключение, но и базовую защиту, например буферизацию сети, усиленный интерфейс для доступа через Интернет и защиту от распределенных атак "отказ в обслуживании" (DDoS).

## <a name="remote-desktop-deployment"></a>Развертывание удаленного рабочего стола

В развернутой службе удаленного рабочего стола публикуется шлюз удаленных рабочих столов, чтобы обеспечить преобразование трафика "удаленный вызов процедур (RPC) по протоколу HTTPS" в трафик "RDP по протоколу UDP".

Вы можете настроить клиенты для использования удаленного рабочего стола, например MSTSC.exe, для доступа к прокси приложения Azure AD. Так вы создадите новое подключение HTTPS к шлюзу удаленных рабочих столов с помощью его соединителей. В результате шлюз не будет доступен из Интернета напрямую, и все запросы HTTPS будут сначала направляться в облако.

Эта топология показана на схеме ниже.

## <a name="configure-the-remote-desktop-gateway-url"></a>Настройка URL-адреса для шлюза удаленных рабочих столов

Если пользователи как обычно используют URL-адрес шлюза удаленных рабочих столов и активируют трафик RDP, они смогут получить доступ к файлам и другим методам.

Публикацию можно выполнить с помощью доменного имени, предоставленного прокси приложения (msappproxy.net), или с помощью пользовательского доменного имени, настроенного в Azure AD (например, rdg.contoso.com).

Если для клиентских устройств и RDP-файла уже настроен URL-адрес шлюза удаленных рабочих столов, вы можете просто использовать то же самое доменное имя, чтобы не изменять эти настройки. В таком случае сертификат для этого домена необходимо указать для прокси приложения, а его список отзыва сертификатов (CRL) должен быть доступен через Интернет.

Если URL-адрес шлюза удаленных рабочих столов не настроен, пользователи и администраторы могут указать его прямо в клиентах удаленного рабочего стола (MSTSC) в диалоговом окне подключения к удаленному рабочему столу, как показано ниже.

 ![Диалоговое окно подключения к удаленному рабочему столу](./media/application-proxy-publish-remote-desktop/remote-desktop-connection-advanced.png)

Чтобы открыть диалоговое окно **Параметры подключения**, на вкладке **Дополнительно** щелкните **Параметры**.

 ![Окно параметров подключения в диалоговом окне подключения к удаленному рабочему столу](./media/application-proxy-publish-remote-desktop/remote-desktop-connection-settings.png)

## <a name="remote-desktop-web-access"></a>Веб-доступ к удаленным рабочим столам

Если ваша организация использует портал веб-доступа к удаленным рабочим столам (RDWA), для публикации можно также использовать прокси приложения Azure AD. При публикации на этом портале можно использовать предварительную проверку подлинности и единый вход.

Топология со сценарием RDWA показана на схеме ниже.

 ![Схема сценария RDWA](./media/application-proxy-publish-remote-desktop/remote-desktop-web-access-portal1.png)

В приведенном выше примере для доступа к порталу RDWA пользователи будут проходить проверку подлинности в Azure AD. Если проверка подлинности в Azure AD выполнена ранее (например, при использовании Office 365), повторная проверка подлинности для входа на портал RDWA не требуется.

Когда пользователь начинает сеанс RDP, ему нужно еще раз пройти проверку подлинности через канал RDP. Это связано с тем, что единый вход между порталом RDWA и шлюзом удаленных рабочих столов основан на хранении учетных данных пользователя на клиентской стороне с использованием ActiveX. Этот процесс запускается при проверке подлинности через форму входа на портале RDWA. Если для проверки подлинности на портале RDWA используется Kerberos, форма входа для проверки подлинности не отображается, поэтому единый вход между порталом RDWA и RDP работать не будет.

Если необходим единый вход на портале RDWA для трафика RDP или форма проверки подлинности на портале RDWA была существенно изменена, можно не настраивать предварительную проверку подлинности при публикации на портале RDWA.

Топология с этим сценарием показана на схеме ниже.

 ![Схема сценария RDWA](./media/application-proxy-publish-remote-desktop/remote-desktop-web-access-portal2.png)

В приведенном выше примере пользователь должен пройти проверку подлинности на портале RDWA, используя форму входа, но проверку подлинности через подключение RDP проходить не нужно.

>[!NOTE]
>В обоих случаях для использования трафика RDP не нужна предварительная проверка подлинности. Поэтому пользователи могут обращаться к удаленному рабочему столу через портал RDWA.
