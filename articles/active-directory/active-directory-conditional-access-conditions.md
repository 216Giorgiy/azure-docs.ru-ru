# <a name="what-are-conditions-in-azure-active-directory-conditional-access"></a>Что собой представляют условия условного доступа Azure Active Directory 

С помощью [условного доступа Azure Active Directory (Azure AD)](active-directory-conditional-access-azure-portal.md) можно контролировать доступ авторизованных пользователей к облачным приложениям. В политике условного доступа определяется реакция ("сделать это") на конкретное условие, при котором активируется политика ("в этом случае"). 

![Контроль](./media/active-directory-conditional-access-conditions/10.png)


В контексте условного доступа:

- элемент **When this happens** (В этом случае) называется **условиями**; 
- элемент **Then do this** (Сделать это) называется **элементами управления доступом**.

Сочетание условий с элементами управления доступом представляет собой политику условного доступа.

![Контроль](./media/active-directory-conditional-access-conditions/61.png)


Условия, которые вы не настроили в политике условного доступа, не применяются. Некоторые условия [обязательны](active-directory-conditional-access-best-practices.md#whats-required-to-make-a-policy-work) для применения политики условного доступа к среде. 

В этой статье представлены общие сведения об условиях и о том, как они используются в политике условного доступа. 

## <a name="users-and-groups"></a>Пользователи и группы

Условие пользователей и групп является обязательным в политике условного доступа. В политике можно либо выбрать параметр **Все пользователи**, либо указать определенных пользователей и группы.

![Контроль](./media/active-directory-conditional-access-conditions/111.png)

Если вы выбрали:

- Параметр **Все пользователи**, то политика применяется ко всем пользователям в каталоге, включая гостевых пользователей.

- Параметр **Select users and groups** (Выбор пользователей и групп), вы можете задать следующие параметры:

    - **All guest users** (Все гостевые пользователи) — позволяет указать политику для гостевых пользователей B2B. Это условие соответствует любой учетной записи пользователя с указанным значением *гость* для атрибута *userType*. Этот параметр можно использовать в тех случаях, когда политика должна применяться сразу после создания учетной записи в потоке приглашения в Azure AD.

    - **Роли каталога** — позволяет указать политику на основе назначения ролей пользователя. Это условие поддерживает роли каталога, такие как *Глобальный администратор* или *Администратор паролей*.

    - **Users and groups** (Пользователи и группы) — позволяет указать определенные наборы пользователей. Например, можно выбрать группу, содержащую всех сотрудников отдела кадров, если приложение отдела кадров является облачным.

Это может быть группа любого типа в Azure AD, включая динамические группы, назначенные группы безопасности и группы рассылки.

Можно также исключить определенных пользователей или группы из политики. Как правило, это могут быть учетные записи службы, если политика требует применения многофакторной проверки подлинности (MFA). 

Нацеливание на конкретные наборы пользователей удобно использовать для развертывания новой политики. В новой политике следует указать только начальный набор пользователей, чтобы проверить ее работу. 



## <a name="cloud-apps"></a>Облачные приложения 

Облачное приложение — это веб-сайты или служба. Сюда также входят веб-сайты, защищенные с помощью прокси приложения Azure. Подробное описание поддерживаемых облачных приложений см. в разделе [Назначения облачных приложений](active-directory-conditional-access-technical-reference.md#cloud-apps-assignments).    

Условие облачных приложений является обязательным в политике условного доступа. В политике можно либо выбрать параметр **Все облачные приложения**, либо указать определенные приложения.

![Контроль](./media/active-directory-conditional-access-conditions/03.png)

Можно выбрать:

- Параметр **Все облачные приложения**, чтобы базовая политика применялась ко всей организации. Распространенным вариантом использования этого параметра является политика, которая требует прохождения многофакторной проверки подлинности, если обнаружен риск при входе для любого облачного приложения. Политика, применяемая ко **всем облачным приложениям**, распространяется на доступ ко всем веб-сайтам и службам. Этот параметр не ограничивается облачными приложениями, которые отображаются в списке **выбора облачных приложений**.

- Отдельные облачные приложения для нацеливания политики на определенные службы. Например, можно требовать наличия у пользователей [соответствующего требованиям устройства](active-directory-conditional-access-policy-connected-applications.md) для доступа к SharePoint Online. Эта политика также применяется к другим службам (например, к Microsoft Teams), когда они обращаются к содержимому SharePoint. 

Можно также исключить определенные приложения из политики. Тем не менее, эти приложения по-прежнему регулируются политиками, применяемыми к службам, к которым они обращаются. 



## <a name="sign-in-risk"></a>Риск при входе

Уровень риска при входе ("Высокий", "Средний" или "Низкий") указывает, насколько вероятно то, что попытка входа выполнена пользователем, который не является правомочным владельцем учетной записи. Azure AD вычисляет уровень риска при входе во время входа пользователя. Вычисляемый уровень риска можно использовать как условие в политике условного доступа. 

![Условия](./media/active-directory-conditional-access-conditions/22.png)

Чтобы использовать это условие, необходимо включить службу [Защита идентификации Azure Active Directory](active-directory-identityprotection.md).
 
Ниже приведены распространенные варианты использования данного условия политики.

- Блокирование пользователей с высоким уровнем риска при входе для предотвращения доступа потенциально неправомочных пользователей к облачным приложениям. 
- Требование прохождения многофакторной проверки подлинности пользователями со средним уровнем риска при входе. Применяя многофакторную проверку подлинности, можно обеспечить дополнительное подтверждение того, что вход выполняет правомочный владелец учетной записи.

Дополнительные сведения см. в разделе [Вход, представляющий риск](active-directory-identityprotection.md#risky-sign-ins).  

## <a name="device-platforms"></a>Платформы устройств

Платформа устройства характеризуется операционной системой, установленной на устройстве. Azure AD определяет платформу с помощью сведений, предоставляемых устройством, например агентом пользователя. Так как эти сведения не проверены, рекомендуется применять ко всем платформам политику, которая блокирует доступ, требует соответствия политикам Intune или наличия устройства, присоединенного к домену. По умолчанию политика применяется ко всем платформам устройств. 


![Условия](./media/active-directory-conditional-access-conditions/24.png)

Полный список поддерживаемых платформ устройств см. в разделе [Условие платформы устройства](active-directory-conditional-access-technical-reference.md#device-platform-condition).


Распространенным вариантом использования этого условия является политика, которая ограничивает доступ к облачным приложениям и предоставляет его только [управляемым устройствам](active-directory-conditional-access-policy-connected-applications.md#managed-devices). Дополнительные сценарии, включая использование условия платформы устройств, приведены в разделе [Условный доступ на основе приложений Azure Active Directory](active-directory-conditional-access-mam.md).



## <a name="device-state"></a>Состояние устройства

Условие состояния устройства позволяет исключить из политики условного доступа гибридные устройства, присоединенные к Azure AD, и устройства, помеченные как совместимые. Это полезно, если политика должна применяться только к неуправляемому устройству для обеспечения дополнительной безопасности сеанса. Например, выполнение только принудительного управления сеансом Microsoft Cloud App Security, если устройство неуправляемо. 


![Условия](./media/active-directory-conditional-access-conditions/112.png)

Если вы хотите заблокировать доступ для неуправляемых устройств, необходимо реализовать [условный доступ на основе устройства](active-directory-conditional-access-policy-connected-applications.md).


## <a name="locations"></a>Расположения

Расположения позволяют определить условия, которые основаны на том, из какого места была предпринята попытка подключения. 
     
![Условия](./media/active-directory-conditional-access-conditions/25.png)

Ниже приведены распространенные варианты использования данного условия политики.

- Требование многофакторной проверки подлинности для пользователей, которые пытаются получить доступ к службе, находясь вне корпоративной сети.  

- Блокировка доступа для пользователей, которые пытаются получить доступ к службе из определенных стран или регионов. 

Дополнительные сведения см. в статье [Условия расположения в условном доступе Azure Active Directory](active-directory-conditional-access-locations.md).


## <a name="client-apps"></a>Клиентские приложения

Условие клиентских приложений позволяет применить политику к различным типам приложений, включая:

- веб-сайты и службы;
- мобильные и классические приложения. 



Приложение классифицируется следующим образом.

- Приложение является веб-сайтом или службой, если оно использует веб-протоколы единого входа, SAML, WS-Fed или OpenID Connect для конфиденциального клиента.

- Приложение является мобильным или классическим приложением, если оно использует OpenID Connect для мобильных приложений для собственного клиента.

Полный список клиентских приложений, используемых в политике условного доступа, см. в разделе [Условие клиентских приложений](active-directory-conditional-access-technical-reference.md#client-apps-condition) в техническом справочнике по условному доступу в Azure Active Directory.

Ниже приведены распространенные варианты использования данного условия политики.

- Требование наличия [соответствующего требованиям устройства](active-directory-conditional-access-policy-connected-applications.md) для мобильных и классических приложений, которые скачивают большие объемы данных на устройство и поддерживают доступ через браузер с любого устройства.

- Блокирование доступа из веб-приложений, но разрешение доступа из мобильных и классических приложений.

Помимо использования веб-службы единого входа и современных протоколов аутентификации, это условие можно применить к почтовым приложениям, использующим Exchange ActiveSync, например к собственным почтовым приложениям на большинстве смартфонов. В настоящее время клиентские приложения, использующие устаревшие протоколы, нужно защищать с помощью AD FS.

Это условие можно выбрать, если в качестве облачного приложения указано только **Office 365 Exchange Online**.

![Облачные приложения](./media/active-directory-conditional-access-conditions/32.png)

Вы можете выбрать **Exchange ActiveSync** в качестве условия клиентских приложений, если в настроенной политике нет других условий. Но вы можете сузить область действия этого условия, чтобы оно применялось только к поддерживаемым платформам.

 
![Поддерживаемые платформы](./media/active-directory-conditional-access-conditions/33.png)

Применение этого условия только к поддерживаемым платформам равнозначно выбору параметра "Все платформы" в [условии платформы устройств](active-directory-conditional-access-conditions.md#device-platforms).

![Поддерживаемые платформы](./media/active-directory-conditional-access-conditions/34.png)

### <a name="legacy-authentication"></a>Устаревшие способы аутентификации  

Теперь условный доступ применяется к устаревшим клиентам Office, которые не поддерживают современные способы аутентификации, а также к клиентам, использующим почтовые протоколы, в том числе POP, IMAP, SMTP и т. д. Это позволяет настроить такие политики, как **блокировка доступа для других клиентов**.


![Устаревшие способы аутентификации](./media/active-directory-conditional-access-conditions/160.png)
 



#### <a name="known-issues"></a>Известные проблемы

- Если задать для политики параметр **Другие клиенты**, доступ из определенных клиентов, например SPConnect, заблокируется для всей организации. Так происходит из-за того, что такие устаревшие клиенты выполняют аутентификацию непредвиденным образом. Эта проблема не касается основных приложений Office, таких как устаревшие клиенты Office. 

- Чтобы политика вступила в силу, может потребоваться до 24 часов. 


#### <a name="frequently-asked-questions"></a>Часто задаваемые вопросы

**Заблокируются ли веб-службы Exchange (EWS)?**

Это зависит от протокола аутентификации, который используется EWS. Если приложение EWS использует современные способы аутентификации, доступом будет управлять клиентское приложение "Мобильные приложения и настольные клиенты". Если же приложение EWS использует обычную аутентификацию, доступом будет управлять клиентское приложение "Другие клиенты".


**Какие элементы управления можно использовать для клиентского приложения "Другие клиенты"?**

Для этого клиентского приложения можно настроить любой элемент управления. Но, при взаимодействии с пользователем доступ будет заблокирован во всех случаях. Клиентское приложение "Другие клиенты" не поддерживает элементы управления, связанные с MFA, совместимыми устройствами, присоединением к домену и т. д. 
 
**Какие условия можно использовать для клиентского приложения "Другие клиенты"?**

Для этого приложения можно настроить любые условия.

**Поддерживает ли Exchange ActiveSync все условия и элементы управления?**

Нет. Exchange ActiveSync (EAS) поддерживает следующие возможности:

- EAS разрешает выбирать только целевых пользователей и групп. Этот протокол не разрешает выбирать гостей и роли. Если же условие настроено для ролей или гостей, доступ будет заблокирован для всех пользователей, так как невозможно определить, к кому нужно применить политику.

- EAS работает только с Exchange, как облачным приложением. 

- Этот протокол не поддерживает другие условия, кроме самого клиентского приложения.

- EAS можно настроить с использованием любого элемента управления (использование всех, кроме тех, что связаны с соответствием устройства политике, приведет к блокировке).

**Можно ли примять политики ко всем клиентским приложениям по умолчанию?**

Нет. Стандартное поведение политик не изменилось. По умолчанию политики по-прежнему применяются к браузеру, а также клиентам мобильных и классических приложений.
