# Azure AD Connect Sync: общие сведения о сценарии использования с несколькими лесами

Во многих организациях есть среды, которые включают несколько локальных лесов Active Directory. Существуют разные причины для развертывания нескольких локальных лесов Active Directory. Типичными примерами являются проекты с лесами ресурсов учетных записей, лесами, связанными со слияниями и поглощениями, а также лесами, используемыми для передачи данных внешним исполнителям.

Майкрософт предоставляет решение DirSync для сценариев с одним лесом, а также решение FIM для сценариев с несколькими лесами. При этом настройка решения FIM может быть сложной и требующей много времени.

С помощью Azure AD Connect Sync можно не только существенно упростить конфигурацию, но и сделать ее более прогнозируемой.

В конфигурации по умолчанию, предоставляемой Azure AD Connect Sync, предполагается следующее.

1. У каждого пользователя есть только одна включенная учетная запись. Лес, в котором располагается эта учетная запись, используется для создания федерации с пользователем.
2. У каждого пользователя есть только один почтовый ящик.
3. Лес, в котором размещаются почтовые ящики пользователей, характеризуется наилучшим качеством данных для атрибутов, отображаемых в глобальном списке адресов Exchange (GAL). Если у пользователя нет почтового ящика, для передачи значений этих атрибутов может использоваться любой лес.


В этой статье рассматривается несколько распространенных сценариев. Кроме того, здесь показывается, как эти сценарии представлены в Azure AD Connect Sync:

1. Отдельные технологии 
2. Полная сетка с дополнительным решением GALSync 
3. Лес ресурсов учетной записи 


## Отдельные технологии

В этой среде все локальные леса обрабатываются как отдельные сущности. Ни один пользователь не может присутствовать в каком-либо другом лесу.<br> У каждого леса есть своя организация Exchange; синхронизация GAL между лесами отсутствует. Такая ситуация может возникать после слияний и поглощений, а также в организации, в которой все подразделения функционируют изолированно друг от друга.

![Отдельные технологии][1]

На этом рисунке каждый объект в каждом лесу будет представлен в метавселенной в единственном экземпляре с объединением в целевом каталоге Azure AD. Конечный результат может быть таким же, как и при подключении одного сервера DirSync к каждому исходному лесу AD.
 




## Полная сетка с дополнительным решением GALSync

Топология полной сетки позволяет размещать пользователей и ресурсы в любом лесу. Как правило, между такими лесами образуются двусторонние отношения доверия.

Если Exchange присутствует в нескольких лесах, может быть доступным дополнительное решение GALSync. Это решение представляет пользователя из одного леса как контакт во всех остальных лесах.

В этом сценарии объекты удостоверений обычно соединяются с помощью атрибута mail. В результате пользователь с почтовым ящиком в одном лесу соединяется с контактами в других лесах. Группы рассылки и группы безопасности могут находиться в каждом лесу. Также они могут содержать пользователей, контакты и участников внешней безопасности.

## Лес ресурсов учетной записи
В топологии леса ресурсов учетных записей есть один или несколько лесов учетных записей с активными учетными записями пользователей.<br> В этом сценарии есть один лес с отношениями доверия со всеми лесами учетных записей. Обычно у такого леса расширенная схема AD с Exchange и Lync. Как и другие общие службы, все службы Exchange и Lync располагаются в этом лесу. У пользователей в этом лесу есть отключенная учетная запись; почтовый ящик связан с лесом учетных записей.
