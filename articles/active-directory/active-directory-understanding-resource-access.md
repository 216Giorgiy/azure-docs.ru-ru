# <a name="understanding-resource-access-in-azure"></a>Основные сведения о доступе к ресурсам в Azure
> [!NOTE]
> В этой статье рассматривается управление доступом к ресурсам на всем портале Azure с использованием администраторов подписок. Кроме того, портал предварительной версии Azure поддерживает [управление доступом на основе ролей](role-based-access-control-configure.md) , позволяющее контролировать доступ к ресурсам Azure более точно.
> 
> 

В октябре 2013 года классический портал Azure и API управления службами были интегрированы в Azure Active Directory. Это было сделано для того, чтобы заложить фундамент для повышения удобства работы пользователей при управлении доступом к ресурсам Azure. Azure Active Directory уже предоставляет широкие функциональные возможности, включая управление пользователями, локальную синхронизацию каталогов, многофакторную проверку подлинности и управление доступом к приложениям. Разумеется, все эти функции должны быть доступны для управления всеми ресурсами Azure.

Управление доступом в Azure начинается с оплаты. Владелец учетной записи Azure, доступ к которой осуществляется через [Центр управления учетной записью Azure](https://account.windowsazure.com/subscriptions), является администратором учетной записи (АУ). Подписки — это контейнеры для оплаты, которые в то же время обеспечивают защиту: у каждой подписки есть администратор службы (АС), который может добавлять, удалять и изменять ресурсы Azure в этой подписке через [классический портал Azure](https://manage.windowsazure.com/). По умолчанию АУ новой подписки становится и ее АС, однако АУ может изменить АС в центре учетных записей Azure.

<br><br>![Учетные записи Azure][1]

С подпиской также связан каталог. Каталог определяет набор пользователей. Это могут быть пользователи рабочих или учебных учетных записей, создавшие каталог, или внешние пользователи (т. е. пользователи учетных записей Майкрософт). Подписки доступны подмножеству пользователей каталога, которые были назначены администраторами служб (АС) или соадминистраторами (СА). Единственное исключение для обеспечения обратной совместимости: учетные записи Майкрософт (ранее Windows Live ID) можно назначать в качестве АС или СА, даже если они отсутствуют в каталоге.

<br><br>![Управление доступом в Azure][2]

Функциональные возможности классического портала Azure позволяют СА, которые вошли с использованием учетной записи Майкрософт, перейти в каталог, связанный с подпиской, с помощью команды **Изменить каталог** на станице **Подписки** в диалоговом окне **Параметры**. Обратите внимание, что эта операция влияет на управление доступом в соответствующей подписке.

> [!NOTE]
> Команда **Изменить каталог** в классическом портале Azure недоступна для пользователей, вошедших в систему с помощью рабочей или учебной учетной записи, поскольку такие пользователи могут входить только в тот каталог, к которому они относятся.
> 
> 

<br><br>![Простой алгоритм входа пользователя в систему][3]

В самом простом случае организация (например, Contoso) принудительно внедряет выставление счетов и управление доступом в один и тот же набор подписок. То есть каталог привязывается к подпискам, принадлежащим одной и той же учетной записи Azure. После успешного входа в классический портал Azure пользователи видят две коллекции ресурсов (на предыдущей иллюстрации выделены оранжевым цветом).

* Каталоги, в которых существует их учетная запись пользователя (созданная или добавленная как внешний субъект). Обратите внимание, что каталог, используемый для входа в систему, в данном случае не имеет значения, поэтому ваши каталоги будут отображаться всегда, где бы вы ни выполняли вход.
* Ресурсы, составляющие часть подписок, связанных с каталогом, который используется для входа в систему и к которому пользователь может получить доступ (будь то АС или СА).

<br><br>![Пользователь с несколькими подписками и каталогами][4]

Пользователи с подписками в нескольких каталогах могут переключать текущий контекст классического портала Azure, используя для этого фильтр подписок. При этом осуществляется отдельный вход в другой каталог, хотя за счет единого входа пользователь этого не замечает.

В связи с единым видом каталога такие операции, как перемещение ресурсов между подписками, могут усложняться. Чтобы выполнить перенос ресурсов, может потребоваться впервые использовать команду **Изменить каталог** на странице "Подписки" диалогового окна **Параметры**, чтобы привязать подписки к одному и тому же каталогу.
