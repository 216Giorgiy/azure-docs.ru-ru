# Основные сведения о доступе к ресурсам в Azure


> [AZURE.NOTE] В этой статье рассматривается управление доступом к ресурсам на всем портале Azure с использованием администраторов подписок. Кроме того, портал предварительной версии Azure поддерживает [управление доступом на основе ролей](role-based-access-control-configure.md), позволяющее контролировать доступ к ресурсам Azure более точно.

В октябре 2013 года портал управления Azure и API управления службами были интегрированы в Azure Active Directory. Это было сделано для того, чтобы заложить фундамент для повышения удобства работы пользователей при управлении доступом к ресурсам Azure. Azure Active Directory уже предоставляет широкие функциональные возможности, включая управление пользователями, локальную синхронизацию каталогов, многофакторную проверку подлинности и управление доступом к приложениям. Разумеется, все эти функции должны быть доступны для управления всеми ресурсами Azure.

Управление доступом в Azure начинается с оплаты. Владелец учетной записи Azure, доступ к которой осуществляется через [центр учетных записей Azure](https://account.windowsazure.com/subscriptions), является администратором учетной записи (АУ). Подписки — это контейнеры для оплаты, которые в то же обеспечивают защиту: у каждой подписки есть администратор службы (АС), который может добавлять, удалять и изменять ресурсы Azure в этой подписке через [портал управления Azure](https://manage.windowsazure.com/). По умолчанию АУ новой подписки становится и ее АС, однако АУ может изменить АС в центре учетных записей Azure.
 
<br><br>![Учетные записи Azure][1]

Подписки имеют также связь с каталогом. Каталог определяет набор пользователей. Это могут быть пользователи рабочих или учебных учетных записей, создавшие каталог, или внешние пользователи (т. е. пользователи учетных записей Майкрософт). Подписки доступны подмножеству пользователей каталога, которые были назначены администратора службы (АС) или соадминистратором (СА); единственное исключение состоит в том, что для целей обратной совместимости учетные записи Майкрософт (ранее Windows Live ID) могут назначаться в качестве АС или СА, даже если они не присутствуют в каталоге.

<br><br>![Управление доступом в Azure][2]

 
Функциональные возможности портала управления Azure позволяют АС, вошедшим в систему с помощью учетной записи Майкрософт, изменять каталог, с которым связана подписка, с помощью команды **Изменить каталог** на странице **Подписки** раздела **Параметры**. Обратите внимание, что эта операция влияет на управление доступом в соответствующей подписке.



> [AZURE.NOTE] Команда **Изменить каталог** на портале управления Azure недоступна для пользователей, вошедших в систему с помощью рабочей или учебной учетной записи, поскольку такие пользователи могут входить только в тот каталог, к которому они относятся.

<br><br>![Простой алгоритм входа пользователя в систему][3]

В самом простом случае организация (например, Contoso) принудительно внедряет выставление счетов и управление доступом в один и тот же набор подписок. То есть каталог привязывается к подпискам, принадлежащим одной и той же учетной записи Azure. После успешного входа в портал управления Azure пользователи видят две коллекции ресурсов (на предыдущей иллюстрации выделены оранжевым цветом).


- Каталоги, в которых существует их учетная запись пользователя (созданная или добавленная как внешний субъект). Обратите внимание, что каталог, используемый для входа в систему, в данном случае не имеет значения, поэтому ваши каталоги будут отображаться всегда, где бы вы ни выполняли вход.

- Ресурсы, составляющие часть подписок, связанных с каталогом, который используется для входа в систему и к которому пользователь может получить доступ (будь то АС или СА).


<br><br>![Пользователь с несколькими подписками и каталогами][4]


Пользователи с подписками в нескольких каталогах могут переключать текущий контекст портала управления Azure, используя для этого фильтр подписок. При этом осуществляется отдельный вход в другой каталог, хотя за счет единого входа пользователь этого не замечает.

В связи с единым видом каталога такие операции, как перемещение ресурсов между подписками, могут усложняться. Возможно, что для передачи ресурсов нужно будет предварительно воспользоваться командой **Изменить каталог** на страницу "Подписки" в разделе **Параметры**, чтобы привязать подписки к тому же каталогу.
