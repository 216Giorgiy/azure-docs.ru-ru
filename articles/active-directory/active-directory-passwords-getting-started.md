## <a name="enable-users-to-reset-their-azure-ad-passwords"></a>Разрешение пользователям сбрасывать пароли Azure AD
В этом разделе пошагово рассматривается включение самостоятельного сброса паролей для облачного каталога AAD, регистрация пользователей для самостоятельного сброса паролей и, наконец, тестирование самостоятельного сброса пароля от имени пользователя.

* [Необходимые условия для сброса паролей в режиме самообслуживания](#prerequisites)

* [Шаг 3. Сброс пароля от имени пользователя](#step-3-reset-your-azure-ad-password-as-a-user)

### <a name="prerequisites"></a>Предварительные требования
Прежде чем можно будет включить и использовать самостоятельный сброс пароля, необходимо выполнить следующие предварительные требования:

* Создать клиента AAD. Дополнительные сведения см. на странице [начала работы с подпиской Azure](https://azure.microsoft.com/trial/get-started-active-directory/).
* Получите подписку Azure. Дополнительные сведения см. в разделе [Что такое клиент Azure AD?](active-directory-administer.md#what-is-an-azure-ad-tenant)
* Связать клиент AAD с подпиской Azure. Дополнительные сведения см. в статье [Связь между подписками Azure и службой Azure Active Directory](https://msdn.microsoft.com/library/azure/dn629581.aspx).
* Выполнить обновление до Azure AD Premium или Basic либо использовать платную лицензию на Office 365. Дополнительные сведения см. на странице со [сведениями о ценах на выпуски Azure Active Directory](https://azure.microsoft.com/pricing/details/active-directory/).
  
  > [!NOTE]
  > Чтобы включить самостоятельный сброс пароля для пользователей облака, необходимо выполнить обновление до Azure AD Premium или Azure AD Basic либо воспользоваться платной лицензией на Office 365.  Чтобы включить самостоятельный сброс пароля для локальных пользователей, необходимо выполнить обновление до Azure AD Premium. Дополнительные сведения см. на странице со [сведениями о ценах на выпуски Azure Active Directory](https://azure.microsoft.com/pricing/details/active-directory/). Эта информация включает подробные инструкции по регистрации в Azure AD Premium или Basic, по активации плана лицензирования и активации доступа к Azure AD, а также по назначению доступа для учетных записей администратора и пользователя.
  > 
  > 
* Создайте по крайней мере одну учетную запись администратора и одну учетную запись пользователя в вашем каталоге AAD.
* Назначьте созданным учетным записям администратора и пользователя платную лицензию AAD Premium, AAD Basic или Office 365.

### <a name="step-1-configure-password-reset-policy"></a>Шаг 1. Настройка политики сброса паролей
Чтобы настроить политику сброса паролей пользователями, выполните следующие действия.

1. Откройте любой браузер и перейдите на [классический портал Azure](https://manage.windowsazure.com).
2. На [классическом портале Azure](https://manage.windowsazure.com) найдите **расширение Active Directory** на панели навигации слева.
   
   ![Управлении паролями в Azure AD][001]
3. На вкладке **Каталог** щелкните каталог, в котором следует настроить политику сброса паролей пользователями, например Wingtip Toys.
   
    ![][002]
4. Выберите вкладку **Настройка** .
   
   ![][003]
5. На вкладке **Настройка** прокрутите вниз до раздела **Политика сброса пароля пользователя**.  Здесь настраивается каждый аспект политики сброса паролей пользователей для заданного каталога.  
   
   > [!NOTE]
   > Эта **политика применяется только к конечным пользователям в организации, а не к администраторам**. По соображениям безопасности Майкрософт управляет политикой сброса паролей для администраторов. Если этот раздел не отображается, убедитесь, что вы уже зарегистрировались в Azure Active Directory Premium или Basic и **назначили лицензию** для учетной записи администратора, настраивающего эту функцию.
   > 
   > 
   
   ![][004]
6. Чтобы настроить политику сброса паролей пользователями, передвиньте переключатель **Пользователям разрешен сброс пароля** на значение **Да**.  При этом откроются еще несколько элементов управления, которые позволяют настроить порядок работы этой функции в вашем каталоге.  Вы можете настроить сброс паролей по своему усмотрению.  Дополнительные сведения о функциях каждого из элементов управления политикой сброса паролей см. в статье [Настройка компонентов управления паролями в соответствии с требованиями организации](active-directory-passwords-customize.md).
   
   ![][005]
7. После настройки политики сброса паролей пользователями для вашего клиента нажмите кнопку **Сохранить** внизу экрана.
   
   > [!NOTE]
   > Мы советуем установить политику сброса паролей пользователями с использованием двух запросов, чтобы можно было увидеть, как эта возможность работает в самом сложном случае.
   > 
   > 
   
   ![][006]

### <a name="step-2-add-contact-data-for-your-test-user"></a>Шаг 2. Добавление контактных данных для тестового пользователя
Для пользователей в организации данные для сброса пароля можно задать несколькими способами.

* Изменение данных пользователей на [классическом портале Azure](https://manage.windowsazure.com) или на [портале администрирования Office 365](https://portal.microsoftonline.com).
* Использование AAD Connect для синхронизации свойств пользователей в Azure AD из локального домена Active Directory
* Изменение свойств пользователей с помощью Windows PowerShell.
* Разрешение пользователям регистрировать свои собственные данные на портале по адресу [http://aka.ms/ssprsetup](http://aka.ms/ssprsetup)
* Чтобы потребовать от пользователей регистрацию для сброса пароля при входе в панель доступа по адресу [http://myapps.microsoft.com](http://myapps.microsoft.com), установите для параметра конфигурации SSPR **Запрашивать регистрацию у пользователей при входе в панель доступа** значение **Да**.

Если вы хотите узнать больше о том, какие данные используются при сбросе паролей, а также о правилах форматирования этих данных, см. статью [Дополнительные сведения об управлении паролями](active-directory-passwords-learn-more.md#what-data-is-used-by-password-reset).

#### <a name="to-add-user-contact-data-via-the-user-registration-portal"></a>Добавление контактных данных пользователя на портале регистрации пользователей
1. Для использования портала регистрации сброса паролей необходимо предоставить пользователям в организации ссылку на эту страницу ([http://aka.ms/ssprsetup](http://aka.ms/ssprsetup)) или включить параметр требования автоматической регистрации пользователей.  После открытия этой ссылки пользователям будет предложено выполнить вход с учетной записью из организации.  После этого они увидят следующую страницу:
   
   ![][007]
2. Здесь пользователи могут предоставить и проверить свой номер мобильного телефона, альтернативный адрес электронной почты или вопросы безопасности.  Вот как выглядит проверка по мобильному телефону.
   
   ![][008]
3. После того как пользователь укажет эту информацию, страница обновится, показывая, что информация верна (ниже она затемнена).  После нажатия кнопок **Готово** или **Отмена** пользователь переместится на панель доступа.
   
   ![][009]
4. Когда пользователь проверит обе части информации, его профиль будет обновлен указанными данными.  В этом примере **Рабочий телефон** указан вручную, поэтому пользователь может также использовать его в качестве способа связи для сброса своего пароля.
   
   ![][010]

### <a name="step-3-reset-your-azure-ad-password-as-a-user"></a>Шаг 3. Сброс пароля AD от имени пользователя
Теперь, когда вы настроили пользовательскую политику сброса и указали контактные сведения для пользователя, этот пользователь может выполнить самостоятельный сброс пароля.

#### <a name="to-perform-a-self-service-password-reset"></a>Сброс пароля в режиме самообслуживания
1. Если перейти на сайт типа [**portal.microsoftonline.com**](http://portal.microsoftonline.com), отобразится экран входа, подобный изображенному ниже.  Откройте ссылку **Не можете получить доступ к своей учетной записи?** , чтобы протестировать пользовательский интерфейс для сброса пароля.
   
   ![][011]
2. После открытия ссылки **Не удается получить доступ к своей учетной записи?** вы будете перемещены на новую страницу, которая запросит **идентификатор пользователя**, для которого требуется сбросить пароль.  Введите здесь свой тестовый **идентификатор пользователя**, пройдите проверку CAPTCHA и нажмите кнопку **Далее**.
   
   ![][012]
3. Так как пользователь в этом случае указал **рабочий телефон**, **мобильный телефон** и **альтернативный адрес электронной почты**, ему предоставляются все эти варианты для прохождения первой проверки.
   
   ![][013]
4. В этом случае выберите сначала **вызов** по **рабочему телефону**.  Обратите внимание, что при выборе метода на основе телефона пользователям будет предложено **подтвердить свой номер телефона** , прежде чем они смогут сбрасывать пароли.  Это позволяет предотвратить спам злоумышленниками телефонных номеров пользователей в организации.
   
   ![][014]
5. После подтверждения пользователем своего номера телефона при нажатии кнопки вызова появится счетчик, а на телефоне раздастся звонок.  После приема вызова будет воспроизведено сообщение о том, что **пользователю следует нажать клавишу #** для подтверждения своей учетной записи.  Нажатие этой клавиши автоматически подтвердит, что пользователь прошел первую проверку, и переключит пользовательский интерфейс на второй шаг проверки.
   
   ![][015]
6. После прохождения первой проверки пользовательский интерфейс автоматически обновляется, удаляя ее из списка вариантов выбора.  Так как сначала был использован **рабочий телефон**, допустимыми вариантами для второго этапа проверки останутся только **мобильный телефон** и **альтернативный адрес электронной почты**.  Щелкните параметр **Отправить письмо на дополнительный адрес электронной почты** .  После этого при нажатии кнопки электронной почты будет отправлено сообщение по запасному адресу электронной почты.
   
   ![][016]
7. Ниже приведен образец сообщения, которое увидят пользователи. Обратите внимание на фирменную символику клиента:
   
   ![][017]
8. Когда почтовое сообщение придет, страница обновится, и вы сможете ввести проверочный код из сообщения в поле ввода, показанное ниже.  После ввода соответствующего кода будет подсвечена кнопка "Далее", и вы сможете пройти второй этап проверки.
   
   ![][018]
9. После выполнения требований политики организации вам будет разрешено выбрать новый пароль.  Пароль проверяется на основе соответствия требованиям AAD к "надежному" паролю (см. [Политика паролей в Azure AD](https://msdn.microsoft.com/library/azure/jj943764.aspx)), и появляется указатель надежности, показывая, соответствует ли введенный пароль этой политике.
   
   ![][019]
10. После ввода пароля, соответствующего политике организации, пароль будет сброшен и можно будет сразу выполнить вход с новым паролем.
    
    ![][020]

### <a name="writeback-prerequisites"></a>Необходимые условия для обратной записи
Прежде чем можно будет включить и использовать обратную запись паролей, необходимо выполнить следующие предварительные требования:

* Имеется в наличии клиент Azure AD с включенными возможностями Azure AD Premium.  Дополнительные сведения см. на странице со [сведениями о ценах на выпуски Azure Active Directory](active-directory-editions.md).
* Сброс пароля настроен и включен в вашем клиенте.  Дополнительные сведения см. в разделе [Разрешение пользователям сбрасывать пароли Azure AD](#enable-users-to-reset-their-azure-ad-passwords).
* Имеется по крайней мере одна учетная запись администратора и одна тестовая учетная запись пользователя с лицензией Azure AD Premium, на которой можно протестировать данную функцию.  Дополнительные сведения см. на странице со [сведениями о ценах на выпуски Azure Active Directory](active-directory-editions.md).
  
  > [!NOTE]
  > Убедитесь в том, что учетная запись администратора, используемая для включения обратной записи паролей, является учетной записью администратора облачной службы (созданной в Azure AD), а не федеративной учетной записью (созданной в локальной службе AD и синхронизированной с Azure AD).
  > 
  > 
* На локальной площадке развернута служба AD с одним или несколькими лесами под управлением Windows Server 2008, Windows Server 2008 R2, Windows Server 2012 или Windows Server 2012 R2 с установленными новейшими пакетами обновления.
  
  > [!NOTE]
  > При использовании более старой версии Windows Server 2008 или 2008 R2 эту функцию по-прежнему можно использовать, но перед принудительным включением локальной политики паролей AD в облаке необходимо будет [загрузить и установить обновление KB 2386717](https://support.microsoft.com/kb/2386717) .
  > 
  > 
* Установлено средство Azure AD Connect и среда AD подготовлена к синхронизации с облаком.  Дополнительные сведения см. в статье [Интеграция локальных удостоверений с Azure Active Directory](connect/active-directory-aadconnect.md).
  
  > [!NOTE]
  > Прежде чем проверять компонент обратной записи паролей, выполните полный импорт и синхронизацию из Active Directory и Azure Active Directory в службе Azure AD Connect.
  > 
  > 
* При использовании Azure AD Sync или Azure AD Connect требуется открыть исходящий **TCP-порт 443** (и в некоторых случаях **TCP-порты 9350–9354**).  Дополнительные сведения см. в разделе [Шаг 3. Настройка брандмауэра](#step-3-configure-your-firewall). Использование DirSync для этого сценария больше не поддерживается.  Если вы по-прежнему используете DirSync, выполните обновление до последней версии Azure AD Connect перед развертыванием компонента обратной записи паролей.
  
  > [!NOTE]
  > Всем пользователям средств Azure AD Sync или DirSync настоятельно рекомендуется выполнить обновление до новейшей версии Azure AD Connect для обеспечения наиболее удобной работы и новых возможностей по мере их выпуска.
  > 
  > 

### <a name="step-1-download-the-latest-version-of-azure-ad-connect"></a>Шаг 1. Загрузка новейшей версии Azure AD Connect
Обратная запись паролей доступна в выпусках Azure AD Connect или в средстве Azure AD Sync с номером версии **1.0.0419.0911** и выше.  Обратная запись паролей с автоматической разблокировкой учетной записи доступна в выпусках Azure AD Connect или в средстве Azure AD Sync с номером версии **1.0.0485.0222** и выше. Если вы используете старую версию, выполните обновление как минимум до указанной выше версии, прежде чем продолжить. [Щелкните здесь, чтобы загрузить новейшую версию Azure AD Connect](connect/active-directory-aadconnect.md#install-azure-ad-connect).

#### <a name="to-check-the-version-of-azure-ad-sync"></a>Проверка версии Azure AD Sync
1. Перейдите в папку **%ProgramFiles%\Azure Active Directory Sync\**.
2. Найдите исполняемый файл **ConfigWizard.exe** .
3. Щелкните правой кнопкой мыши исполняемый файл и выберите в контекстном меню пункт **Свойства** .
4. Выберите вкладку **Сведения** .
5. Найдите поле **Версия файла** .
   
   ![][021]

Если номер версии больше или равен **1.0.0419.0911** или вы устанавливаете Azure AD Connect, можно перейти сразу к разделу [Шаг 2. Включение обратной записи паролей в Azure AD Connect с помощью пользовательского интерфейса или Powershell и проверка](#step-2-enable-password-writeback-in-azure-ad-connect).

> [!NOTE]
> Если вы впервые устанавливаете средство Azure AD Connect, рекомендуется следовать некоторым практическим советам для подготовки среды к синхронизации службы каталогов.  Перед установкой средства Azure AD Connect необходимо активировать синхронизацию каталогов либо на [портале администрирования Office 365](https://portal.microsoftonline.com), либо на [классическом портале Azure](https://manage.windowsazure.com).  Дополнительные сведения см. в статье [Дальнейшие действия и управление Azure AD Connect](active-directory-aadconnect-whats-next.md).
> 
> 

### <a name="step-2-enable-password-writeback-in-azure-ad-connect"></a>Шаг 2. Включение обратной записи паролей в Azure AD Connect
Теперь, когда средство Azure AD Connect загружено, все готово к включению обратной записи паролей.  Это можно сделать одним из двух способов.  Вы можете включить обратную запись паролей либо на экране дополнительных возможностей мастера установки Azure AD Connect, либо через Windows PowerShell.

#### <a name="to-enable-password-writeback-in-the-configuration-wizard"></a>Включение обратной записи паролей в мастере настройки
1. На **компьютере синхронизации каталогов** откройте мастер настройки **Azure AD Connect**.
2. Пропускайте шаги, пока не попадете на экран настройки **дополнительных возможностей** .
3. Установите флажок **Обратная запись пароля** .
   
   ![][022]
4. Завершите работу мастера, на последней странице будет содержаться сводка изменений, в которую будет включено изменение конфигурации обратной записи паролей.

> [!NOTE]
> Обратную запись паролей можно в любой момент отключить, либо запустив еще раз данный мастер и отменив выбор функции, либо установив для параметра **Write Passwords Back to On-Premises Directory** (Записывать пароли обратно в локальный каталог) значение **Нет** в разделе **Политика сброса пароля пользователя** (вкладка **Настройка** вашего каталога на [классическом портале Azure](https://manage.windowsazure.com)).  Дополнительные сведения о настройке процесса сброса паролей см. в статье [Настройка компонентов управления паролями в соответствии с требованиями организации](active-directory-passwords-customize.md).
> 
> 

#### <a name="to-enable-password-writeback-using-windows-powershell"></a>Включение обратной записи паролей с помощью Windows PowerShell
1. На **компьютере синхронизации каталогов** откройте новое **окно Windows PowerShell с повышенными привилегиями**.
2. Если модуль не загружен, введите команду `import-module ADSync` для загрузки командлетов Azure AD Connect в текущий сеанс.
3. Получите список соединителей Azure AD в системе, выполнив командлет `Get-ADSyncConnector` и сохранив результаты в `$aadConnectorName`, например `$connectors = Get-ADSyncConnector|where-object {$\_.name -like "\*AAD"}`.
4. Чтобы получить текущее состояние обратной записи для текущего соединителя, выполните следующий командлет: `Get-ADSyncAADPasswordResetConfiguration –Connector $aadConnectorName.name`
5. Включите обратную запись паролей, выполнив командлет `Set-ADSyncAADPasswordResetConfiguration –Connector $aadConnectorName.name –Enable $true`

> [!NOTE]
> Если появится запрос учетных данных, убедитесь в том, что учетная запись администратора, указанная для AzureADCredential, является **учетной записью администратора облачной службы (созданной в Azure AD)**, а не федеративной учетной записью (созданной в локальной службе AD и синхронизированной с Azure AD).
> 
> [!NOTE]
> Обратную запись паролей можно отключить с помощью PowerShell, повторив те же инструкции, но передав `$false` или присвоив параметру **Write Passwords Back to On-Premises Directory** (Записывать пароли обратно в локальный каталог) значение **Нет** в разделе **Политика сброса пароля пользователя** (вкладка **Настройка** вашего каталога на [классическом портале Azure](https://manage.windowsazure.com)).
> 
> 

#### <a name="verify-that-the-configuration-was-successful"></a>Проверка успешного выполнения настройки
После успешно выполнения настройки появится сообщение "Функция обратной записи при сбросе пароля включена" в окне Windows PowerShell или сообщение об успешном выполнении в пользовательском интерфейсе конфигурации.

Кроме того, можно проверить правильность установки службы, открыв средство просмотра событий, перейдя в журнал событий приложений и отыскав событие **31005 - OnboardingEventSuccess** из источника **PasswordResetService**.

  ![][023]

### <a name="step-3-configure-your-firewall"></a>Шаг 3. Настройка брандмауэра
После включения компонента обратной записи паролей необходимо убедиться в том, что компьютер, на котором запущено средство Azure AD Connect, может подключиться к облачным службам Майкрософт для получения запросов от этого компонента. Этот этап подразумевает обновление правил подключения на сетевых устройствах (прокси-серверах, брандмауэрах и т. д.) для разрешения исходящих подключений к указанным URL- и IP-адресам, принадлежащим корпорации Майкрософт, на определенных сетевых портах. Эти изменения могут отличаться в зависимости от версии средства Azure AD Connect. Дополнительные сведения см. в разделе [Как работает обратная запись паролей](active-directory-passwords-learn-more.md#how-password-writeback-works) и [Модель безопасности обратной записи паролей](active-directory-passwords-learn-more.md#password-writeback-security-model).

#### <a name="why-do-i-need-to-do-this"></a>Зачем это нужно делать?

Чтобы обеспечить правильную работу компонента обратной записи паролей, компьютер, на котором запущено средство Azure AD Connect, должен иметь возможность устанавливать исходящие HTTPS-подключения к **.servicebus.windows.net* и определенному IP-адресу, используемому службой Azure, как определено в [списке диапазонов IP-адресов центров обработки данных Microsoft Azure](https://www.microsoft.com/download/details.aspx?id=41653).

Для средства Azure AD Connect версии 1.0.8667.0 и выше

- **Вариант 1.** Разрешить все исходящие HTTPS-подключения через порт 443 (используя URL- или IP-адрес).
    - Сценарии использования:
        - Используйте этот вариант, если требуется самая простая конфигурация, которую не нужно обновлять в случае изменения диапазонов IP-адресов центра обработки данных Azure.
    - Необходимые действия:
        - Разрешите все исходящие HTTPS-подключения через порт 443, используя URL- или IP-адрес.
<br><br>
- **Вариант 2.** Разрешить исходящие HTTPS-подключения к определенным диапазонам IP-адресов и URL-адресам.
    - Сценарии использования:
        - Рассмотрите этот вариант, если вы используете сетевую среду с ограниченным доступом и не хотите разрешать исходящие подключения.
        - Чтобы обеспечить работу компонента обратной записи паролей в этой конфигурации, на сетевых устройствах необходимо еженедельно обновлять IP-адреса из списка диапазонов IP-адресов центра обработки данных Microsoft Azure. Эти диапазоны IP-адресов доступны в виде XML-файла, который обновляется каждую среду (по тихоокеанскому времени). Изменения вступают в силу в следующий понедельник (по тихоокеанскому времени).
    - Необходимые действия:
        - Разрешите все исходящие HTTPS-подключения к *.servicebus.windows.net.
        - Разрешите все исходящие HTTPS-подключения ко всем IP-адресам в списке диапазонов IP-адресов центра обработки данных Microsoft Azure и обновляйте эту конфигурацию еженедельно.

> [!NOTE]
> Если вы настроили компонент обратной записи паролей, используя инструкции выше, и не видите сообщений об ошибках в журнале событий Azure AD Connect, но при проверке возникают ошибки подключения, это может быть связано с тем, что сетевое устройство в вашей среде блокирует HTTPS-подключения к IP-адресам. Например, хотя подключение к *https://*.servicebus.windows.net* разрешено, подключение к определенному IP-адресу может блокироваться. Чтобы устранить эту проблему, необходимо либо разрешить в сетевой среде все исходящие HTTPS-подключения через порт 443 ко всем URL- или IP-адресам (см. вариант 1), либо обратиться к группе сетевых администраторов, чтобы явным образом разрешить HTTPS-подключения к определенным IP-адресам (см. вариант 2).

**Для предыдущих версий:**

- Разрешить исходящие TCP-подключения через порт 443, 9350–9354 и 5671. 
- Разрешить исходящие подключения к *https://ssprsbprodncu-sb.accesscontrol.windows.net/*.

> [!NOTE]
> При использовании средства Azure AD Connect версии до 1.0.8667.0 мы настоятельно советуем установить [последнюю версию](https://www.microsoft.com/download/details.aspx?id=47594), в которую добавлено несколько улучшений компонента обратной записи паролей, упрощающих настройку.

После настройки сетевых устройств перезагрузите компьютер, на котором запущено средство Azure AD Connect.

### <a name="step-4-set-up-the-appropriate-active-directory-permissions"></a>Шаг 4. Настройка соответствующих разрешений Active Directory
Для каждого леса, содержащего пользователей, чьи пароли будут сброшены, если X — это учетная запись, которая была указана для этого леса в мастере настройки (при начальной конфигурации), то учетной записи X должны быть предоставлены разрешения **Сброс пароля**, **Смена пароля**, **Write Permissions** (Разрешения на запись) в `lockoutTime` и **Write Permissions** (Разрешения на запись) в `pwdLastSet`, расширенные права на корневой объект каждого домена в этом лесу. Право должно быть помечено как наследуемое всеми объектами пользователя.  

Если вы не уверены, к какой учетной записи оно относится, откройте интерфейс настройки Azure Active Directory Connect и щелкните **Обзор решения** .  Учетная запись, для которой необходимо добавить разрешение, подчеркнута красным цветом на следующем снимке экрана.

**<font color="red">Задайте это право для каждого домена в каждом лесу в системе, в противном случае компонент обратной записи паролей будет работать неправильно.</font>**

  ![][032]

  Установка этих разрешений позволит учетной записи службы MA для каждого леса управлять паролями от имени учетных записей пользователей в этом лесу. Если не назначить эти разрешения, то даже при внешне правильной настройке обратной записи паролей пользователи столкнутся с ошибками при попытке управления своими локальными паролями из облака. Ниже приведены подробные инструкции о том, как можно сделать это с помощью оснастки управления **Пользователи и компьютеры Active Directory** .

> [!NOTE]
> Репликация этих разрешений по всем объектам вашего каталога может занять до часа времени.
> 
> 

#### <a name="to-set-up-the-right-permissions-for-writeback-to-occur"></a>Настройка необходимых разрешений для произведения обратной записи
1. Откройте оснастку **Пользователи и компьютеры Active Directory** с учетной записью, обладающей разрешениями на администрирование соответствующего домена.
2. В параметре **Меню "Вид"** включите параметр **Дополнительные функции**.
3. На левой панели щелкните правой кнопкой мыши объект, который представляет корень домена.
4. Выберите вкладку **Безопасность** .
5. Затем нажмите **Дополнительно**.
   
   ![][024]
6. На вкладке **Разрешения** нажмите кнопку **Добавить**.
   
   ![][025]
7. Выберите учетную запись, которой следует предоставить разрешения (это та же учетная запись, которая была указана при настройке синхронизации для этого леса).
8. В раскрывающемся списке сверху выберите **Дочерние пользовательские объекты**.
9. В диалоговом окне **Запись разрешения** установите флажки **Сброс пароля**, **Смена пароля**, **Write Permissions** (Разрешения на запись) в `lockoutTime` и **Write Permissions** (Разрешения на запись) в `pwdLastSet`.
   
   ![][026]
   ![][027]
   ![][028]
10. Затем нажимайте кнопки **Применить/ОК** во всех открытых диалоговых окнах.

### <a name="step-5-reset-your-ad-password-as-a-user"></a>Шаг 5. Сброс пароля от имени пользователя
Теперь, когда обратная запись паролей включена, можно проверить ее работу, сбросив пароль пользователя, чья учетная запись синхронизирована с облачным клиентом.

#### <a name="to-verify-password-writeback-is-working-properly"></a>Проверка правильности работы обратной записи паролей
1. Перейдите по адресу [https://passwordreset.microsoftonline.com](https://passwordreset.microsoftonline.com) или откройте любой экран входа с идентификатором организации и выберите ссылку **Не можете получить доступ к своей учетной записи?**
   
   ![][029]
2. Теперь должна открыться новая страница с запросом идентификатора пользователя, для которого требуется сбросить пароль. Введите идентификатор тестового пользователя и пройдите процесс сброса пароля.
3. После сброса пароля вы увидите экран, который выглядит примерно так. Это означает, что ваш пароль успешно сброшен в локальном и/или облачном каталогах.
   
   ![][030]
4. Чтобы проверить успешность выполнения операции или диагностировать возможные ошибки, перейдите на **компьютер синхронизации каталогов**, откройте **средство просмотра событий**, перейдите в **журнал событий приложений** и найдите событие **31002 - PasswordResetSuccess** из источника **PasswordResetService** для тестового пользователя.
   
   ![][031]
