### <a name="tip-1-licensing---make-sure-you-understand-the-licensing-requirements"></a>Совет 1. Лицензирование. Убедитесь, что вы понимаете условия лицензионного соглашения
Для работы функции сброса пароля Azure AD в вашей организации должна быть назначена по крайней мере одна лицензия. Лицензирование каждого пользователя при сбросе пароля необязательно. Тем не менее, если использовать эту функцию без лицензии, это будет считаться несоответствующим лицензионному соглашению Майкрософт и повлечет необходимость лицензирования этих пользователей.

Ниже приведены ссылки на некоторые статьи и разделы, которые помогут понять, какие лицензии необходимы для сброса пароля.

* [Цены и доступность](https://docs.microsoft.com/azure/active-directory/active-directory-passwords#pricing-and-availability)
* [Сценарии, поддерживаемые для обратной записи паролей](https://docs.microsoft.com/azure/active-directory/active-directory-passwords-learn-more#scenarios-supported-for-password-writeback)

### <a name="tip-2-testing---test-with-an-end-user-not-an-administrator-and-pilot-with-a-small-set-of-users"></a>Совет 2. Тестирование. Выполняйте тестирование в качестве конечного пользователя (не администратора) и начинайте с небольшого набора пользователей
При тестировании с правами администратора представленная ниже политика сброса паролей для администраторов обязательна.  Это означает, что вы не увидите ожидаемых результатов политики, настроенной для конечных пользователей.

Политики, настроенные в администраторском интерфейсе, применяются только к конечным пользователям, но не администраторам. Чтобы обеспечить безопасность организации, корпорация Майкрософт реализует строгие политики сброса паролей по умолчанию для администраторов, которые могут отличаться от политик, заданных для конечных пользователей.

#### <a name="administrator-password-reset-policy"></a>Политика сброса пароля администратора
* **Применяется к**: роли администратора (глобального администратора, администратора службы технической поддержки, администратора паролей и т. д.).
* **Применяется политика, разрешающая только один способ входа:**
 * на протяжении 30 дней после начала использования пробной версии **или**
 * когда отсутствует личный домен **и** Azure AD Connect не выполняет синхронизацию удостоверений.
 * **_Требуется_** задать **один** из следующих параметров: адрес электронной почты для проверки подлинности, запасной адрес электронной почты, телефон для проверки подлинности, мобильный или рабочий телефон.
* **Применяется политика, разрешающая два способа входа:** 
 * спустя 30 дней после окончания срока действия пробной версии **или**
 * когда имеется личный домен **или** 
 * в Azure AD Connect включена синхронизация удостоверений из локальной среды.
 * _**Требуется**_ задать **два** из следующих параметров: адрес электронной почты для проверки подлинности, запасной адрес электронной почты, телефон для проверки подлинности, мобильный или рабочий телефон.

### <a name="tip-3-deployment---pre-populate-data-for-your-users-so-they-dont-have-to-register"></a>Совет 3. Развертывание. Предварительно заполните данные пользователей, чтобы им не пришлось регистрироваться
Многие не понимают, что пользователей не обязательно регистрировать для сброса пароля, чтобы использовать эту функцию.  Задав свойства номера телефона или адреса электронной почты для пользователей заранее, можно немедленно развернуть сброс паролей для всей организации **без участия пользователей.**

Чтобы узнать, как это сделать с помощью API, PowerShell или Azure AD Connect, ознакомьтесь с документацией ниже.
* [Дополнительные сведения об управлении паролями](https://docs.microsoft.com/azure/active-directory/active-directory-passwords-learn-more#deploying-password-reset-without-requiring-end-user-registration)
* [Какие данные используются при сбросе пароля](https://docs.microsoft.com/azure/active-directory/active-directory-passwords-learn-more#what-data-is-used-by-password-reset)

### <a name="tip-4-deployment---use-password-reset-to-obviate-the-need-to-communicate-temporary-passwords"></a>Совет 4. Развертывание. Используйте сброс пароля, чтобы избежать необходимости сообщать временные пароли
Это пометка к совету 3. После предварительной настройки пользователей для сброса паролей представьте себе ситуацию, когда сотрудник присоединяется к компании в первый раз. Вам не нужно передавать сотруднику временный пароль, теперь он может перейти на [портал сброса паролей Azure AD](https://passwordreset.microsoftonline.com) и сбросить пароль.

Если пользователь использует [присоединенное к домену Azure AD устройство под управлением Windows 10](https://docs.microsoft.com/azure/active-directory/active-directory-azureadjoin-devices-group-policy), он может сделать это прямо в стандартном окне входа Windows 10 и получить доступ к новому ПК без вашей помощи.

Чтобы узнать, как это сделать с помощью API, PowerShell или Azure AD Connect, ознакомьтесь с документацией ниже. После предварительного заполнения этих данных попросите пользователей сбросить пароли, чтобы сразу получить доступ к своим учетным записям.
* [Дополнительные сведения об управлении паролями](https://docs.microsoft.com/azure/active-directory/active-directory-passwords-learn-more#deploying-password-reset-without-requiring-end-user-registration)

### <a name="tip-5-writeback---look-at-the-application-event-log-on-your-aad-connect-machine-to-troubleshoot-password-writeback"></a>Совет 5. Обратная запись. Просматривайте журнал событий приложения на компьютере AAD Connect для устранения неполадок обратной записи паролей
Журнал событий приложения Azure AD Connect содержит широкий набор сведений, описывающих активность службы обратной записи паролей в режиме реального времени. Чтобы получить доступ к этому журналу, выполните следующие действия:

1. Войдите в систему компьютера **Azure AD Connect**.
2. Откройте **средство просмотра событий Windows**, нажав кнопку **Пуск** и введя **просмотр событий**.
3. Откройте журнал событий **приложения**.
4. Проверьте наличие событий из источников **PasswordResetService** или **ADSync** для получения дополнительных сведений о возможной проблеме.

Полный список событий, которые могут присутствовать в этом журнале, а также множество дополнительных сведений об устранении неполадок обратной записи паролей см. в следующих статьях:

### <a name="tip-6-writeback---ensure-you-enable-the-correct-permissions-firewall-rules-and-connection-settings-for-password-writeback"></a>Совет 6. Обратная запись. Включите правильные разрешения, правила брандмауэра и параметры подключения для обратной записи паролей
Чтобы обратная запись работала правильно, необходимо следующее:

1. Правильно настроенные с помощью функции обратной записи паролей **разрешения Active Directory**, позволяющие пользователям самостоятельно изменять свои пароли и разблокировать учетные записи в AD.
2. Открыть правильные **порты брандмауэра**, позволяющие службе обратной записи паролей безопасно обмениваться данными с внешним миром, используя исходящее подключение.
3. Внести правильные **исключения брандмауэра** для URL-адресов основных служб сброса пароля, например служебной шины.
4. **Прокси-сервер или брандмауэр не завершает простаивающие исходящие подключения.** Рекомендуется завершать их через 10 или больше минут простоя.

Полный список рекомендаций по устранению неполадок и конкретные рекомендации по настройке разрешений и правил брандмауэра для обратной записи паролей см. в следующих разделах:

### <a name="tip-7-reporting---see-who-is-registering-or-resetting-passwords-with-the-azure-ad-sspr-audit-logs"></a>Совет 7. Отчеты. Просматривайте, кто регистрируется или сбрасывает пароли с помощью журналов аудита Azure AD SSPR 
После того как сброс пароля развернут и работает, необходимо проанализировать его работу и понять, кого необходимо зарегистрировать, с какими проблемами могут столкнуться пользователи при сбросе паролей и как максимально эффективно использовать это решение.

Используя журналы аудита сброса паролей Azure AD, все это можно сделать на портале Azure, из PowerBI, с помощью API уведомления о событиях Azure AD или из PowerShell.  Дополнительные сведения о том, как использовать эти функции уведомлений, см. в следующих материалах:

Щелкните параметр **Отправить письмо на дополнительный адрес электронной почты** .  После этого при нажатии кнопки электронной почты будет отправлено сообщение по запасному адресу электронной почты.

   ![][017]

[Политика паролей в Azure AD](https://msdn.microsoft.com/library/azure/jj943764.aspx)), и появляется указатель надежности, показывая, соответствует ли введенный пароль этой политике.

### <a name="writeback-prerequisites"></a>Необходимые условия для обратной записи
Прежде чем можно будет включить и использовать обратную запись паролей, необходимо выполнить следующие предварительные требования:

* Имеется в наличии клиент Azure AD с включенными возможностями Azure AD Premium.  Дополнительные сведения см. на странице со [сведениями о ценах на выпуски Azure Active Directory](active-directory-editions.md).
* Azure AD Connect необходимо подключить к эмулятору контроллера основного домена, чтобы обеспечить работу компонента обратной записи паролей.  При необходимости можно настроить Azure AD Connect для использования с контроллером основного домена. Для этого щелкните правой кнопкой мыши **Свойства** для соединителя синхронизации Active Directory, а затем выберите **Настройка разделов каталога**. Здесь найдите раздел **Параметры подключения контроллера домена** и установите флажок **Использовать только предпочитаемые контроллеры домена**.  Примечание. Если предпочитаемый контроллер домена не является эмулятором основного контроллера домена, Azure AD Connect все равно будет связываться с основным контроллером домена для обратной записи паролей.
* Сброс пароля настроен и включен в вашем клиенте.  Дополнительные сведения см. в разделе [Разрешение пользователям сбрасывать пароли Azure AD](#enable-users-to-reset-their-azure-ad-passwords).
* Имеется по крайней мере одна учетная запись администратора и одна тестовая учетная запись пользователя с лицензией Azure AD Premium, на которой можно протестировать данную функцию. ### <a name="step-2-enable-password-writeback-in-azure-ad-connect"></a>Шаг 2. Включение обратной записи паролей в Azure AD Connect
Теперь, когда средство Azure AD Connect загружено, все готово к включению обратной записи паролей.  Это можно сделать одним из двух способов.  Вы можете включить обратную запись паролей либо на экране дополнительных возможностей мастера установки Azure AD Connect, либо через Windows PowerShell.

#### <a name="to-enable-password-writeback-in-the-configuration-wizard"></a>Включение обратной записи паролей в мастере настройки
1. На **компьютере синхронизации каталогов** откройте мастер настройки **Azure AD Connect**.
2. Пропускайте шаги, пока не попадете на экран настройки **дополнительных возможностей** .
3. Установите флажок **Обратная запись пароля** .

   ![][022]
4. Завершите работу мастера, на последней странице будет содержаться сводка изменений, в которую будет включено изменение конфигурации обратной записи паролей.

> [!NOTE]
> Обратную запись паролей можно в любой момент отключить, либо запустив еще раз данный мастер и отменив выбор функции, либо установив для параметра **Write Passwords Back to On-Premises Directory** (Записывать пароли обратно в локальный каталог) значение **Нет** в разделе **Политика сброса пароля пользователя** (вкладка **Настройка** вашего каталога на [классическом портале Azure](https://manage.windowsazure.com)).  Дополнительные сведения о настройке процесса сброса паролей см. в статье [Настройка компонентов управления паролями в соответствии с требованиями организации](active-directory-passwords-customize.md).
>
>

#### <a name="to-enable-password-writeback-using-windows-powershell"></a>Включение обратной записи паролей с помощью Windows PowerShell
1. На **компьютере синхронизации каталогов** откройте новое **окно Windows PowerShell с повышенными привилегиями**.
2. Если модуль не загружен, введите команду `import-module ADSync` для загрузки командлетов Azure AD Connect в текущий сеанс.
3. Получите список соединителей Azure AD в системе, выполнив командлет `Get-ADSyncConnector` и сохранив результаты в `$aadConnectorName`, например `$aadConnectorName = Get-ADSyncConnector|where-object {$_.name -like "*AAD"}`.
4. Чтобы получить текущее состояние обратной записи для текущего соединителя, выполните следующий командлет: `Get-ADSyncAADPasswordResetConfiguration –Connector $aadConnectorName.name`
5. Включите обратную запись паролей, выполнив командлет `Set-ADSyncAADPasswordResetConfiguration –Connector $aadConnectorName.name –Enable $true`

> [!NOTE]
> Если появится запрос учетных данных, убедитесь в том, что учетная запись администратора, указанная для AzureADCredential, является **учетной записью администратора облачной службы (созданной в Azure AD)**, а не федеративной учетной записью (созданной в локальной службе AD и синхронизированной с Azure AD).
>
> [!NOTE]
> Обратную запись паролей можно отключить с помощью PowerShell, повторив те же инструкции, но передав `$false` или присвоив параметру **Write Passwords Back to On-Premises Directory** (Записывать пароли обратно в локальный каталог) значение **Нет** в разделе **Политика сброса пароля пользователя** (вкладка **Настройка** вашего каталога на [классическом портале Azure](https://manage.windowsazure.com)).
>
>

После успешно выполнения настройки появится сообщение "Функция обратной записи при сбросе пароля включена" в окне Windows PowerShell или сообщение об успешном выполнении в пользовательском интерфейсе конфигурации.

Кроме того, можно проверить правильность установки службы, открыв средство просмотра событий, перейдя в журнал событий приложений и отыскав событие **31005 - OnboardingEventSuccess** из источника **PasswordResetService**.

### <a name="step-3-configure-your-firewall"></a>Шаг 3. Настройка брандмауэра
После включения компонента обратной записи паролей необходимо убедиться в том, что компьютер, на котором запущено средство Azure AD Connect, может подключиться к облачным службам Майкрософт для получения запросов от этого компонента. Этот этап подразумевает обновление правил подключения на сетевых устройствах (прокси-серверах, брандмауэрах и т. д.) для разрешения исходящих подключений к указанным [URL- и IP-адресам, принадлежащим корпорации Майкрософт](https://support.office.com/article/Office-365-URLs-and-IP-address-ranges-8548a211-3fe7-47cb-abb1-355ea5aa88a2?ui=en-US&rs=en-US&ad=US), на определенных сетевых портах. Эти изменения могут отличаться в зависимости от версии средства Azure AD Connect. Дополнительные сведения см. в разделе [Как работает обратная запись паролей](active-directory-passwords-learn-more.md#how-password-writeback-works) и [Модель безопасности обратной записи паролей](active-directory-passwords-learn-more.md#password-writeback-security-model).

#### <a name="why-do-i-need-to-do-this"></a>Зачем это нужно делать?

Чтобы обеспечить надлежающую работу функции обратной записи паролей, компьютер под управлением Azure AD Connect должен иметь возможность обращаться к службе сброса паролей и служебной шине Azure.

Для средства Azure AD Connect версии **1.1.443.0** и выше:

- Для последней версии средства Azure AD Connect понадобится **исходящий HTTPS-доступ** к:
    - *passwordreset.microsoftonline.com*;
    - *servicebus.windows.net*

Для средства Azure AD Connect от версии **1.0.8667.0** до **1.1.380.0**:

- **Вариант 1.** Разрешить все исходящие HTTPS-подключения через порт 443 (используя URL- или IP-адрес).
    - Сценарии использования:
        - Используйте этот вариант, если требуется самая простая конфигурация, которую не нужно обновлять в случае изменения диапазонов IP-адресов центра обработки данных Azure.
    - Необходимые действия:
        - Разрешите все исходящие HTTPS-подключения через порт 443, используя URL- или IP-адрес.
<br><br>
- **Вариант 2.** Разрешить исходящие HTTPS-подключения к определенным диапазонам IP-адресов и URL-адресам.
    - Сценарии использования:
        - Рассмотрите этот вариант, если вы используете сетевую среду с ограниченным доступом и не хотите разрешать исходящие подключения.
        - Чтобы обеспечить работу компонента обратной записи паролей в этой конфигурации, на сетевых устройствах необходимо еженедельно обновлять IP-адреса из списка диапазонов IP-адресов центра обработки данных Microsoft Azure. Эти диапазоны IP-адресов доступны в виде XML-файла, который обновляется каждую среду (по тихоокеанскому времени). Изменения вступают в силу в следующий понедельник (по тихоокеанскому времени).
    - Необходимые действия:
        - Разрешите все исходящие HTTPS-подключения к *.servicebus.windows.net.
        - Разрешите все исходящие HTTPS-подключения ко всем IP-адресам в списке диапазонов IP-адресов центра обработки данных Microsoft Azure и обновляйте эту конфигурацию еженедельно. Список доступен для загрузки [здесь](https://www.microsoft.com/download/details.aspx?id=41653).

> [!NOTE]
> Если вы настроили компонент обратной записи паролей, используя инструкции выше, и не видите сообщений об ошибках в журнале событий Azure AD Connect, но при проверке возникают ошибки подключения, это может быть связано с тем, что сетевое устройство в вашей среде блокирует HTTPS-подключения к IP-адресам. Например, хотя подключение к *https://*.servicebus.windows.net* разрешено, подключение к определенному IP-адресу может блокироваться. Чтобы устранить эту проблему, необходимо либо разрешить в сетевой среде все исходящие HTTPS-подключения через порт 443 ко всем URL- или IP-адресам (см. вариант 1), либо обратиться к группе сетевых администраторов, чтобы явным образом разрешить HTTPS-подключения к определенным IP-адресам (см. вариант 2).

**Для предыдущих версий:**

- Разрешить исходящие TCP-подключения через порт 443, 9350–9354 и 5671.
- Разрешить исходящие подключения к *https://ssprsbprodncu-sb.accesscontrol.windows.net/*.

> [!NOTE]
> При использовании средства Azure AD Connect версии до 1.0.8667.0 мы настоятельно советуем установить [последнюю версию](https://www.microsoft.com/download/details.aspx?id=47594), в которую добавлено несколько улучшений компонента обратной записи паролей, упрощающих настройку.

После настройки сетевых устройств перезагрузите компьютер, на котором запущено средство Azure AD Connect.

#### <a name="idle-connections-on-azure-ad-connect-114430-and-up"></a>Неактивные подключения в Azure AD Connect (версии 1.1.443.0 и более поздней)
Чтобы проверять состояние подключений, средство Azure AD Connect отправляет периодические запросы проверки связи и активности конечным точкам служебной шины. Если средство обнаруживает, что слишком много подключений обрываются, оно автоматически увеличивает частоту запросов проверки связи к конечной точке. Самый минимальный интервал проверки связи, при котором подключение прерывается, составляет 1 раз в 60 секунд, однако **настоятельно рекомендуется, чтобы прокси-серверы или брандмауэры позволяли сохранять неактивные подключения в течение по крайней мере 2–3 минут.** \*Для предыдущих версий рекомендуется интервал в 4 минуты или больше.

### <a name="step-4-set-up-the-appropriate-active-directory-permissions"></a>Шаг 4. Настройка соответствующих разрешений Active Directory
Для каждого леса, содержащего пользователей, чьи пароли будут сброшены, если X — это учетная запись, которая была указана для этого леса в мастере настройки (при начальной конфигурации), то учетной записи X должны быть предоставлены разрешения **Сброс пароля**, **Смена пароля**, **Write Permissions** (Разрешения на запись) в `lockoutTime` и **Write Permissions** (Разрешения на запись) в `pwdLastSet`, расширенные права на корневой объект каждого домена в этом лесу или для подразделения пользователя, который должен находиться в области SSPR.  Последний вариант можно использовать, чтобы предоставить разрешения на сброс паролей только определенной группе пользовательских объектов, если это невозможно сделать в корне домена. Право должно быть помечено как наследуемое всеми объектами пользователя.  

Если вы не уверены, к какой учетной записи оно относится, откройте интерфейс настройки Azure Active Directory Connect и щелкните **Обзор решения** .  Учетная запись, для которой необходимо добавить разрешение, подчеркнута красным цветом на следующем снимке экрана.

**<font color="red">Задайте это право для каждого домена в каждом лесу в системе, в противном случае компонент обратной записи паролей будет работать неправильно.</font>**

  ![][032]

  Установка этих разрешений позволит учетной записи службы MA для каждого леса управлять паролями от имени учетных записей пользователей в этом лесу. Если не назначить эти разрешения, то даже при внешне правильной настройке обратной записи паролей пользователи столкнутся с ошибками при попытке управления своими локальными паролями из облака. Ниже приведены подробные инструкции о том, как можно сделать это с помощью оснастки управления **Пользователи и компьютеры Active Directory** .

> [!NOTE]
> Репликация этих разрешений по всем объектам вашего каталога может занять до часа времени.
>
>

#### <a name="to-set-up-the-right-permissions-for-writeback-to-occur"></a>Настройка необходимых разрешений для произведения обратной записи
1. Откройте оснастку **Пользователи и компьютеры Active Directory** с учетной записью, обладающей разрешениями на администрирование соответствующего домена.
2. В параметре **Меню "Вид"** включите параметр **Дополнительные функции**.
3. На левой панели щелкните правой кнопкой мыши объект, который представляет корень домена.
4. Выберите вкладку **Безопасность** .
5. Затем нажмите **Дополнительно**.

   ![][024]
6. На вкладке **Разрешения** нажмите кнопку **Добавить**.

   ![][025]
7. Выберите учетную запись, которой следует предоставить разрешения (это та же учетная запись, которая была указана при настройке синхронизации для этого леса).
8. В раскрывающемся списке сверху выберите **Дочерние пользовательские объекты**.
9. В диалоговом окне **Запись разрешения** установите флажки **Сброс пароля**, **Смена пароля**, **Write Permissions** (Разрешения на запись) в `lockoutTime` и **Write Permissions** (Разрешения на запись) в `pwdLastSet`.

   ![][026]
   ![][027]
   ![][028]
10. Затем нажимайте кнопки **Применить/ОК** во всех открытых диалоговых окнах.

### <a name="step-5-reset-your-ad-password-as-a-user"></a>Шаг 5. Сброс пароля от имени пользователя
Теперь, когда обратная запись паролей включена, можно проверить ее работу, сбросив пароль пользователя, чья учетная запись синхронизирована с облачным клиентом.

#### <a name="to-verify-password-writeback-is-working-properly"></a>Проверка правильности работы обратной записи паролей
1. Перейдите по адресу [https://passwordreset.microsoftonline.com](https://passwordreset.microsoftonline.com) или откройте любой экран входа с идентификатором организации и выберите ссылку **Не можете получить доступ к своей учетной записи?**

   ![][029]
2. Теперь должна открыться новая страница с запросом идентификатора пользователя, для которого требуется сбросить пароль. Введите идентификатор тестового пользователя и пройдите процесс сброса пароля.
3. После сброса пароля вы увидите экран, который выглядит примерно так. Это означает, что ваш пароль успешно сброшен в локальном и/или облачном каталогах.

   ![][030]
4. Чтобы проверить успешность выполнения операции или диагностировать возможные ошибки, перейдите на **компьютер синхронизации каталогов**, откройте **средство просмотра событий**, перейдите в **журнал событий приложений** и найдите событие **31002 - PasswordResetSuccess** из источника **PasswordResetService** для тестового пользователя.

   ![][031]
