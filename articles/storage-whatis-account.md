#Что такое учетная запись хранения?

Хранилище Azure включает три службы: хранилище BLOB-объектов, хранилище таблиц и хранилище очередей. Эти службы включаются в каждую учетную запись хранения. Учетная запись хранения предоставляет уникальное пространство имен для работы с BLOB-объектами, очередями и таблицами.

Учетная запись хранения может содержать до 200 ТБ BLOB-объектов, данных в очередях и таблицах, если была создана 8 июня 2012 г. или позже; емкость учетных записей хранения, созданных до этой даты, не превышает 100 ТБ. В рамках одной подписки можно создать до 20 учетных записей хранения с уникальными именами. Дополнительные сведения об учетных записях хранения см. в разделе [Цели масштабируемости хранилища и производительности Azure](http://msdn.microsoft.com/ru-ru/library/dn249410.aspx).

Все сведения об учетной записи хранения, включая дату ее создания, доступны на странице **Панель мониторинга** для **хранилищ** на портале управления.

Затраты на хранение основываются на четырех факторах: емкость хранилища, схема репликации, транзакции хранилища и передача данных. Объем хранилища определяет, какая часть выделенного объема учетной записи хранения используется для хранения данных. Стоимость простого хранения данных определяется в зависимости от объема хранящихся данных и способа их репликации. Транзакции – это все операции чтения и записи в хранилище Azure. Передача данных означает данные, переданные из региона Azure. Когда к данным вашей учетной записи хранения обращается приложение, выполняемое в другом регионе, то независимо от типа приложения (облачная служба или приложение другого типа) вам начисляется плата за исходящие данные. (Для служб Azure можно сгруппировать данные и службы в одном центре обработки данных, чтобы уменьшить или исключить плату за передачу данных.)  

Подробные сведения о ценах на объем хранилища, репликацию и транзакции см. на странице [Сведения о ценах на хранилище](http://www.windowsazure.com/ru-ru/pricing/details/#storage). Подробные сведения о ценах на исходящие данные см. на странице [Сведения о ценах на передачу данных](http://www.windowsazure.com/ru-ru/pricing/details/data-transfers/).

##Основные понятия

- **геоизбыточное хранилище (GRS)**   Геоизбыточное хранилище предоставляет высочайший уровень надежности хранения путем органичной репликации данных в дополнительное местоположение, находящееся в том же регионе. Это обеспечивает переход на другой ресурс в случае сбоя в основном расположении. Второе расположение находится в сотнях километров от основного. GRS реализуется через функцию *георепликации*, которая включена для учетной записи хранения по умолчанию, однако может быть отключена, если она окажется ненужной (например, если политики компании запрещают ее применение). Дополнительные сведения см. в разделе [Введение в георепликацию для хранилищ Azure](http://blogs.msdn.com/b/windowsazurestorage/archive/2011/09/15/introducing-geo-replication-for-windows-azure-storage.aspx). 

- **локально избыточное хранилище (LRS)**   Локально избыточное хранилище предоставляет очень надежное и доступное хранилище, расположенное в одном местоположении. Для локально избыточного хранилища данные учетной записи реплицируются три раза в одном и том же центре обработки данных. Все хранилища Azure являются локально избыточными. Для дополнительной надежности можно включить георепликацию. Локально избыточное хранилище предоставляется по ценам со скидками. Сведения о ценах см. в разделе [Сведения о ценах](http://www.windowsazure.com/ru-ru/pricing/details/#storage). 

- **территориальная группа**   *Территориальная группа* — это географическое группирование развертываний облачных служб и учетных записей хранения в Azure. Территориальная группа может повысить производительность служб, если расположить нагрузку, связанную с вычислениями и хранением данных, в одном центре обработки данных или рядом с целевой аудиторией пользователей. Кроме того, плата за передачу данных не взимается, если доступ к данным в учетной записи хранения получается через службу, работающую в той же территориальной группе.

- **конечные точки учетной записи хранения**   *Конечные точки* учетной записи хранения представляют высочайший уровень пространства имен для доступа к BLOB-объектам, таблицам и очередям. Конечные точки для учетной записи хранения по умолчанию имеют следующие форматы: 

    - Служба BLOB-объектов: http://*моя_уч_запись_хранения*.blob.core.windows.net

    - Служба таблиц: http://*моя_уч_запись_хранения*.table.core.windows.net

    - Служба очередей: http://*моя_уч_запись_хранения*.queue.core.windows.net

- **URL-адреса учетной записи хранения**   URL-адрес для доступа к объекту в учетной записи хранения строится путем добавления местоположения объекта в учетной записи хранения к конечной точке. Например, адрес blob-объекта может иметь следующий формат: http://*моя_уч_запись_хранения*.blob.core.windows.net/*мой_контейнер*/*мой_blob_объект*

- **ключи доступа к хранилищу**   При создании учетной записи хранения Azure создает два 512-битных кода доступа к хранилищу, которые используются для проверки подлинности при доступе к учетной записи хранения. Предоставляя два ключа доступа к хранилищу, Azure позволяет повторно создавать ключи без прерывания предоставления службы хранилища или доступа к этой службе.

- **минимальные и подробные показатели**   Можно настроить минимальные или подробные показатели в параметрах мониторинга учетной записи хранения. *Минимальные показатели* собирают данные по таким показателям, как исходящие и входящие данные, доступность, задержка и процент успешных операций, который агрегируется для служб BLOB, таблиц и очередей. *Подробные показатели* собирают операционные сведения в дополнения к сводным данным уровня обслуживания применительно к тем же показателям. Подробные метрики позволяют осуществлять более тщательный анализ проблем, возникающих во время работы приложений. Полный список доступных показателей см. в разделе [Схема таблицы показателей аналитики хранения](http://msdn.microsoft.com/ru-ru/library/windowsazure/hh343264.aspx) Дополнительные сведения о мониторинге хранилищ см. в разделе [Общие сведения о показателях аналитики хранилищ](http://msdn.microsoft.com/ru-ru/library/windowsazure/hh343258.aspx).

- **Ведение журналов**   Ведение журналов — настраиваемая функция учетных записей хранения, которая позволяет регистрировать запросы на чтение, запись и удаление BLOB-объектов, таблиц и очередей. Ведение журналов настраивается на портале управления Azure, однако можно просматривать журналы на портале управления. Журналы хранятся в учетной записи хранения в контейнере $logs. Дополнительные сведения см. в разделе [Обзор аналитики хранения](http://msdn.microsoft.com/ru-ru/library/windowsazure/hh343268.aspx).
