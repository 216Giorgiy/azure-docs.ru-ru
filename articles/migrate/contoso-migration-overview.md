# <a name="contoso-migration-overview"></a>Миграция Contoso: общие сведения


В этой статье показано, как вымышленная организация Contoso переносит локальную инфраструктуру в облако [Microsoft Azure](https://azure.microsoft.com/overview/what-is-azure/). 

Это первый документ из цикла статей о том, как вымышленная компания Contoso выполняет миграцию в Azure. Этот цикл содержит сведения и сценарии, которые иллюстрируют настройку инфраструктуры миграции и выполнение миграции различных типов. Сценарии описаны в порядке возрастания сложности. Со временем мы добавим в этот цикл и другие статьи. В статьях описано, как компания Contoso выполняет собственную задачу миграции, но в материалах также включены дополнительные материалы и конкретные инструкции.

## <a name="introduction"></a>Введение

Azure предоставляет доступ к комплексному набору облачных служб. С помощью этих служб вы, как разработчики и ИТ-специалисты, можете создавать, развертывать и администрировать приложения с помощью разных средств и инфраструктур в глобальной сети центров обработки данных. Если ваша компания столкнется с проблемами, связанными с переходом на цифровые технологии, облако Azure поможет вам оптимизировать ресурсы и операции, наладить взаимодействие с клиентами и сотрудниками и трансформировать свои продукты.

Однако мы признаем, что даже при всех преимуществах облака Azure, таких как скорость и гибкость, минимальные затраты, высокая производительность и надежность, у многих организаций в какой-то момент возникнет потребность в запуске локальных центров обработки данных. В ответ на препятствия, возникающие в процессе внедрения облачных технологий, Azure предоставляет стратегию гибридного облака, которая создает мосты между локальными центрами обработки данных и общедоступным облаком Azure. Это, например, использование облачных ресурсов Azure, таких как служба Backup, для защиты локальных ресурсов, или получение информации о локальных рабочих нагрузках с помощью служб аналитики Azure. 

В рамках стратегии гибридного облака Azure предоставляет постоянно расширяющиеся решения для переноса локальных приложений и рабочих нагрузок в облако. С помощью простых шагов вы можете в полном объеме оценить локальные ресурсы, чтобы понять, как они будут работать в облаке Azure. Затем, с учетом этой оценки, можно без опасений переносить ресурсы в Azure. Когда ресурсы будут доступны и запущены в Azure, вы сможете оптимизировать их для сохранения и улучшения доступа, повышения гибкости, безопасности и надежности.

## <a name="migration-strategies"></a>Стратегии миграции

Стратегий миграции в облако четыре: повторное размещения, рефакторинг, перепроектирование и перестроение. Используемая стратегия зависит от бизнес-факторов и целей миграции. Вы можете реализовать несколько стратегий. Например, можно выбрать повторное размещение (lift-and-shift) простых приложений или приложений, не являющихся критически важными для бизнеса, а более сложные и важные — перепроектировать. Давайте рассмотрим стратегии.


**Стратегия** | **Определение** | **Сценарии использования** 
--- | --- | --- 
**Повторное размещение** | Иногда такую миграцию называют lift-and-shift. Этот вариант не требует изменений кода, позволяя быстро перенести существующие приложения в Azure. Каждое приложение переносится в облачную среду как есть, без риска и затрат, связанных с изменениями кода. | При необходимости быстро переместить приложения в облако.<br/><br/> При необходимости переместить приложение без его изменения.<br/><br/> Если архитектура приложений спроектирована так, что они могут использовать возможности масштабирования [Azure IaaS](https://azure.microsoft.com/overview/what-is-iaas/) после миграции.<br/><br/> Если приложения важны для бизнеса, но немедленные изменения в их возможностях не требуются.
**Рефакторинг** | Рефакторинг, часто называемый перепаковкой, требует минимальных изменений в приложениях, чтобы они могли подключаться к [Azure PaaS](https://azure.microsoft.com/overview/what-is-paas/)и использовать облачные предложения.<br/><br/> Например, можно перенести существующие приложения в Службу приложений Azure или Службу Azure Kubernetes (AKS).<br/><br/> Или же можно выполнить рефакторинг реляционных и нереляционных баз данных в такие варианты структур, как Управляемый экземпляр Базы данных SQL, База данных Azure для MySQL, База данных Azure для PostgreSQL, а также Azure Cosmos DB. | Если ваше приложение можно легко перепаковать для работы в Azure.<br/><br/> Если требуется применить инновационные рекомендации DevOps, предоставляемые Azure, или же вы рассматриваете реализацию DevOps с использованием стратегии контейнеров для рабочих нагрузок.<br/><br/> Перед применением рефакторинга необходимо оценить переносимость существующей базы кода и имеющиеся навыки разработки.
**Перепроектирование** | Перепроектирование для миграции предусматривает изменение и расширение функциональности приложения и базы кода, чтобы оптимизировать архитектуру приложения для обеспечения масштабируемости в облаке.<br/><br/> Например, можно разбить монолитное приложение на группу микрослужб, которые работают совместно и легко масштабируются.<br/><br/> Или же можно перепроектировать реляционные и нереляционные базы данных в полностью управляемые DBaaS-решения, такие как Управляемый экземпляр Базы данных SQL Azure, База данных Azure для MySQL, База данных Azure для PostgreSQL и Azure Cosmos DB. | Если приложениям требуется масштабная переработка для внедрения новых возможностей или для более эффективной работы на облачной платформе.<br/><br/> Если необходимо использовать существующие инвестиции в приложения, соблюдать требования к масштабированию, применять инновационные методы Azure DevOps и свести к минимуму использование виртуальных машин.
**Перестроение** | Перестроение предусматривает еще один шаг — сборку приложения заново с использованием облачных технологий Azure.<br/><br/> Например, можно создавать приложения с нуля, используя облачные технологии, такие как бессерверные приложения, ИИ Azure, Управляемый экземпляр Базы данных SQL Azure и Azure Cosmos DB. | Если требуется быстрая разработка, а существующие приложения обладают ограниченной функциональностью и временем существования.<br/><br/> Если вы готовы ускорить бизнес-инновации (включая рекомендации по DevOps от Azure), построить новые приложения с помощью облачных технологий и воспользоваться преимуществами прогресса ИИ, блокчейна и Интернета вещей.

### <a name="demo-apps"></a>Демонстрационные приложения

В статьях используются два демонстрационных приложения — SmartHotel и osTicket.

- **SmartHotel360.** Это приложение разработано корпорацией Майкрософт как тестовое приложение, которое можно использовать при работе с Azure. Оно предоставляется с открытым исходным кодом и доступно для скачивания с сайта [GitHub](https://github.com/Microsoft/SmartHotel360). Это приложение ASP.NET, подключенное к базе данных SQL Server. Сейчас приложение работает на двух виртуальных машинах VMware под управлением Windows Server 2008 R2 и с SQL Server 2008 R2. Виртуальные машины приложения размещаются в локальной среде и управляются сервером vCenter Server.
- **osTicket.** Приложение с открытым исходным кодом для создания запросов в службу поддержки, которое выполняется на платформе Linux. Скачать эту версию можно с [GitHub](https://github.com/osTicket/osTicket). Сейчас приложение работает на двух виртуальных машинах VMware под управлением Ubuntu 16.04LTS, с использованием Apache 2, PHP 7.0 и MySQL 5.7
