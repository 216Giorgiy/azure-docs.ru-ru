# <a name="azure-app-service-plans-in-depth-overview"></a>Подробный обзор планов службы приложений Azure
План службы приложений представляет собой набор компонентов и возможностей, которые можно совместно использовать в нескольких приложениях. В рамках плана службы приложений Azure доступны все веб-приложения, мобильные приложения, приложения-функции и приложения API, входящие в состав [службы приложений Azure](http://go.microsoft.com/fwlink/?LinkId=529714). Эти планы поддерживают пять ценовых категорий: *Бесплатный*, *Общий*, *Базовый*, *Стандартный* и *Премиум*. Каждая категория имеет определенные возможности и ограничения. Приложения в одной подписке и одном географическом расположении могут совместно использовать план. Во всех приложениях, совместно использующих план, могут применяться все возможности и компоненты, определенные для этого плана в соответствии с ценовой категорией. Все приложения, связанные с конкретным планом, выполняются на ресурсах, которые определены этим планом.

Например, если план настроен на использование двух небольших экземпляров стандартного уровня, все приложения, связанные с этим планом, выполняются на обоих экземплярах с возможностью доступа к соответствующим функциям. Экземпляры плана, на которых выполняются приложения, являются полностью управляемыми и высокодоступными.

В этой статье рассматриваются ключевые характеристики плана службы приложений, такие как ценовая категория и масштаб, а также их роль в управлении вашими приложениями.

## <a name="apps-and-app-service-plans"></a>Приложения и планы службы приложений
Приложение службы приложений в конкретный момент времени может быть сопоставлено лишь с одним планом службы приложений.

Как приложения, так и планы находятся в группе ресурсов. Группа ресурсов определяет пределы жизненного цикла для каждого содержащегося в ней ресурса. Группы ресурсов помогают управлять всеми частями приложения в комплексе.

Так как одна группа ресурсов могут включать несколько планов службы приложений, вы можете назначать разным приложениям разные физические ресурсы. Например, можно разделить ресурсы между средами разработки, тестирования и производства. Наличие отдельных сред для выполнения рабочих процессов, а также процессов разработки и тестирования позволяет изолировать ресурсы. В таком случае нагрузочное тестирование новой версии вашего приложения не будет конкурировать за одни и те же ресурсы с рабочим приложением, которое отвечает за обслуживание реальных клиентов.

Наличие нескольких планов в одной группе ресурсов позволяет создавать приложения, которые охватывают несколько географических регионов. Например, высокодоступное приложение, запущенное в двух регионах, будет включать по крайней мере два плана — по одному для каждого региона, при этом каждый план будет сопоставлен с одним приложением. В этом случае все копии приложения будут находиться в одной группе ресурсов. Наличие группы ресурсов с несколькими планами и несколькими приложениями облегчает проверку и просмотр состояния приложения, а также управление им.

## <a name="create-an-app-service-plan-or-use-existing-one"></a>Создание плана службы приложений или использование имеющегося
При создании приложения мы рекомендуем создавать группу ресурсов. С другой стороны, если создаваемое приложение будет компонентом более крупного решения, такое приложение следует создавать в группе ресурсов, которая выделена крупному приложению.

Новое приложение может быть как независимым проектом, так и частью более крупного приложения. Важно то, что для его размещения вы можете выбрать существующий план службы приложений или создать новый. Это решение, скорее, вопрос емкости и ожидаемой нагрузки.

Если новое приложение будет потреблять много ресурсов, а условия его масштабирования будут отличаться от условий для приложений в существующем плане, мы рекомендуем выделить для нового приложения отдельный план.

Создав план, вы сможете не только выделить для приложения новый набор ресурсов, но и в целом более уверенно управлять назначением ресурсов, ведь каждый план получит отдельный набор экземпляров.

Также вы сможете перемещать приложения между планами, изменяя способы выделения ресурсов для более крупного приложения.

Наконец, если требуется создать приложение в другом регионе и для этого региона отсутствует план, в нем необходимо создать план, чтобы получить возможность разместить в нем приложение.

## <a name="create-an-app-service-plan"></a>Создание плана службы приложений
> [!TIP]
> Документацию по определенной среде службы приложений см. в разделе [Создание плана службы приложений](../app-service-web/app-service-web-how-to-create-a-web-app-in-an-ase.md#createplan).
> 
> 

План службы приложений можно создать отдельно с помощью средства просмотра плана службы приложений или же при создании приложения.

На [портале Azure](https://portal.azure.com) щелкните **Создать** > **Интернет+мобильные устройства**, выберите **Веб-приложение** или другое приложение службы приложений.
![Создание приложения на портале Azure.][createWebApp]

Затем можно выбрать или создать план службы приложений для нового приложения.

 ![Создание плана службы приложений.][createASP]

Чтобы создать план службы приложений, щелкните **+Создать**, введите имя в поле **План службы приложений** и выберите соответствующее **расположение**. Щелкните **Ценовая категория**и выберите для службы соответствующую категорию. Щелкните **Просмотреть все**, чтобы просмотреть другие варианты тарификации, например **Бесплатный** и **Общий**. После выбора ценовой категории нажмите кнопку **Выбрать** .

## <a name="move-an-app-to-a-different-app-service-plan"></a>Перемещение приложения в другой план службы приложений
Приложения можно переместить в другой план службы приложений на [портале Azure](https://portal.azure.com). Приложения можно перемещать между планами, если планы находятся в одной группе ресурсов и в одном географическом регионе.

Чтобы переместить приложение в другой план, перейдите к приложению, которое требуется переместить. В меню **Параметры** щелкните **Изменить план служб приложений**.

Щелкнув **Изменить план служб приложений**, вы откроете средство выбора **План службы приложений**. Здесь можно выбрать существующий план или создать новый. Отображаются только допустимые планы (в одной группе ресурсов и географическом расположении).

![Селектор планов службы приложений.][change]

Каждый план имеет свою ценовую категорию. Например, при изменении уровня сайта с "Бесплатный" на "Стандартный" приложение получит доступ ко всем компонентам и ресурсам уровня "Стандартный".

## <a name="clone-an-app-to-a-different-app-service-plan"></a>Клонирование приложения в другой план служб приложений
Если вы хотите переместить приложение в другой регион, один из вариантов — клонировать его. Клонирование позволяет создавать копии приложения в новом или имеющемся плане служб приложений либо в среде службы приложений в любом регионе.

 ![Клонирование приложения.][appclone]

Функция **Клонирование приложений** доступна в меню **Средства**.

Клонирование имеет некоторые ограничения, о которых можно узнать из статьи [Клонирование приложений службы приложений Azure с помощью портала Azure](../app-service-web/app-service-web-app-cloning-portal.md).

## <a name="scale-an-app-service-plan"></a>Масштабирование плана службы приложений
Существует три способа масштабирования плана.

* **Изменение ценовой категории плана.** Например, уровень плана можно изменить с "Базовый" на "Стандартный" или "Премиум". Тогда все приложения, связанные с этим планом, смогут использовать возможности, доступные на новом уровне служб.
* **Изменение размера экземпляров плана**. Например, для плана уровня "Базовый", в рамках которого используются небольшие экземпляры, можно настроить использование крупных экземпляров. Все приложения, связанные с этим планом, смогут использовать дополнительные ресурсы ЦП и памяти, предоставляемые более крупными экземплярами.
* **Изменение числа экземпляров плана**. Например, масштабированный до трех экземпляров план уровня "Стандартный" можно масштабировать до десяти экземпляров. План категории "Премиум" можно масштабировать до 20 экземпляров (при условии доступности). Все приложения, связанные с этим планом, смогут использовать дополнительные ресурсы ЦП и памяти, предоставляемые большему количеству экземпляров.

Вы можете изменить ценовую категорию и размер экземпляров, щелкнув элемент **Увеличить масштаб** в разделе параметров приложения или плана службы приложений. Изменения будут применены к плану службы приложений и повлияют на все размещенные в нем приложения.

 ![Установка значений для увеличения масштаба приложения.][pricingtier]

## <a name="app-service-plan-cleanup"></a>Удаление плана службы приложений
За **планы службы приложений**, с которыми не связаны приложения, по-прежнему взимается плата, так как они продолжают резервировать вычислительную мощность, указанную в свойствах масштабирования для плана службы приложений.
Чтобы избежать непредвиденных расходов, после удаления последнего приложения, размещенного в плане службы приложений, также удаляется и пустой план.

## <a name="summary"></a>Сводка
Планы службы приложений представляют собой набор компонентов и возможностей, которые доступны всем приложениям. Планы службы приложений позволяют гибко выделять наборы ресурсов для отдельных приложений, оптимизируя использование ресурсов Azure. Например, чтобы снизить затраты на тестовую среду, вы можете использовать один план для нескольких приложений. Можно также увеличить пропускную способность для рабочей среды путем ее масштабирования в рамках нескольких регионов и планов.
