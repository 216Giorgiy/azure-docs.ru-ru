# <a name="what-are-classic-alerts-in-microsoft-azure"></a>Что такое классические оповещения в Microsoft Azure?

> [!NOTE]
> В этой статье объясняется, как создавать классические оповещения метрик прежней версии. Azure Monitor теперь поддерживает [новые оповещения метрик практически в реальном времени и новые функции оповещений](monitoring-overview-unified-alerts.md). 
>

С помощью оповещений можно настроить условия для данных и получать уведомления, когда последние данные мониторинга соответствуют этим условиям.


## <a name="alerts-on-azure-monitor-data"></a>Оповещения для данных Azure Monitor
Существуют два типа классических оповещений: оповещения метрик и оповещения журнала действий.

* **Классические оповещения метрик**. Это оповещение активируется, когда значение указанной метрики выходит за рамки заданного порога. Оповещение формирует уведомление, когда оно "активировано" (при достижении заданного порога и выполнении условий оповещения). Кроме того, оно создается после "разрешения" (когда значение снова соответствует порогу и условие больше не выполняется). 

* **Классические оповещения журнала действий**. Оповещение журнала потоковой передачи, которое активируется, когда создается событие журнала действий, соответствующее настроенным условиям фильтров. Эти предупреждения имеют только одно состояние, "Активировано", так как обработчик оповещений просто применяет условия фильтра к любому новому событию. Данные оповещения могут использоваться для получения уведомлений, когда возникает новый инцидент службы "Работоспособность служб" или когда пользователь или приложение выполняет какую-либо операцию в подписке, например "Удаление виртуальной машины".

Чтобы получать данные журналов диагностики из Azure Monitor, направьте данные в Log Analytics (прежнее название — Operations Management Suite) и используйте оповещения о запросах Log Analytics. Теперь в Log Analytics используется [новый метод оповещения](monitoring-overview-unified-alerts.md).

## <a name="taxonomy-of-azure-monitor-alerts-classic"></a>Классификация классических оповещений Azure Monitor
Для описания классических оповещений и их функций в Azure используются следующие термины.
* **Оповещение**. Определение условий (одно или несколько правил либо условий) для активации.
* **Активное**. Состояние, когда выполняются условия, определенные в классическом оповещении.
* **Разрешено**. Состояние, когда условия классического оповещения больше не выполняются после того, как они были выполнены.
* **Уведомление**. Действие, которое выполняется при активации классического оповещения.
* **Действие**. Определенный вызов, отправляемый получателю уведомления (например, сообщение электронной почты или запрос POST на URL-адрес веб-перехватчика). Обычно уведомления могут активировать несколько действий.

## <a name="how-do-i-receive-notifications-from-an-azure-monitor-classic-alert"></a>Как получать уведомления о классическом оповещении Azure Monitor?
Исторически так сложилось, что оповещения Azure от различных служб использовали собственные встроенные методы уведомления. 

Теперь Azure Monitor предлагает возможность группирования многократно используемых уведомлений с помощью *групп действий*. Группы действий позволяют указать получателей уведомления. Когда активируется оповещение, которое ссылается на группу действий, все получатели получают соответствующее уведомление. Это позволяет многократно использовать группы получателей (например, список дежурных инженеров) для многих объектов оповещений. Группы действий поддерживают уведомление с помощью различных методов. Это может быть отправка запроса POST на URL-адрес веб-перехватчика, отправка сообщений электронной почты, текстовых сообщений, а также ряд других действий. Дополнительные сведения см. в разделе [Создание групп действий и управление ими на портале Azure](monitoring-action-groups.md). 

Более старые классические оповещения журнала действий используют группы действий.

Однако более старые оповещения метрик не используют группы действий. Вместо этого можно настроить следующие действия: 
* отправка уведомлений по электронной почте администратору службы, соадминистраторам или на дополнительные электронные адреса, указанные вами;
* вызов webhook, который позволяет запускать дополнительные автоматизированные действия.

Веб-перехватчики позволяют автоматизировать работу и вносить исправления, например с помощью следующих служб:
- Runbook службы автоматизации Azure
- Функции Azure
- Приложение логики Azure
- сторонняя служба.
