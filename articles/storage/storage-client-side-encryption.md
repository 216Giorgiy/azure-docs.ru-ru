# Шифрование на стороне клиента для службы хранилища Microsoft Azure (предварительная версия)

## Обзор

Добро пожаловать на страницу с описанием [предварительной версии новой клиентской библиотеки хранилища Azure для .NET](https://www.nuget.org/packages/WindowsAzure.Storage/4.4.1-preview). Эта предварительная версия библиотеки содержит новые функции, позволяющие разработчикам шифровать данные в клиентских приложениях перед их загрузкой в хранилище Azure и расшифровывать их во время выгрузки. Предварительная версия библиотеки также поддерживает интеграцию с [хранилищем ключей](http://azure.microsoft.com/services/key-vault/) Azure для управления ключами учетной записи хранения.

## Зачем требуется шифрование на стороне клиента?

Шифрование на стороне клиента имеет значительно больше преимуществ, чем шифрование на стороне сервера, поскольку у вас есть полный контроль над ключами доступа к учетной записи. Шифрование на стороне клиента обеспечивает максимальную безопасность для приложений, поскольку ключи невидны для хранилища Azure и данные никогда не могут быть расшифрованы. Предварительная версия библиотеки находится в открытом доступе на [GitHub](https://github.com/Azure/azure-storage-net/tree/preview), поэтому вы сразу видите, как библиотека шифрует данные в соответствии с вашими стандартами.

## Почему библиотека поддерживает шифрование на стороне клиента?

Любой разработчик умеет шифровать свои данные на стороне клиента перед их загрузкой,тем не менее процесс шифрования требует специальных знаний. Необходимо также продумывать аспекты безопасности и производительности. Все разработчики могли бы создать собственное решение по шифрованию, но поскольку каждое решение отличалось бы, все они были бы несовместимы.

Предварительная версия библиотеки предназначена для следующих целей:

- Реализовать рекомендации по безопасности
- Добиться максимальной производительности
- Обеспечить простоту использования распространенных сценариев
- Поддерживать взаимодействие между языками Данные, зашифрованные с помощью клиентской библиотеки .NET, впоследствии смогут быть расшифрованы с помощью клиентских библиотек для других поддерживаемых языков, включая Java, Node.js и C++ (и наоборот).

## Что доступно уже сейчас?

Предварительная версия библиотеки в настоящее время поддерживает шифрование больших двоичных объектов, таблиц и очередей на основе конвертного шифрования. Шифрование и расшифровка с помощью асимметричных ключей требует больших вычислительных ресурсов. При конвертном шифровании сами данные шифруются не с помощью таких ключей, а с помощью случайного симметричного ключа шифрования содержимого. Этот ключ шифрования содержимого затем шифруется с помощью открытого ключа. Поддержка хранилища ключей Azure помогает эффективно управлять ключами.

Использовать шифрование на стороне клиента просто. Можно указать параметры запроса с соответствующей политикой шифрования (большие двоичные объекты, очереди или таблицы) и передать ее в API-интерфейсы загрузки и выгрузки данных. Клиентская библиотека автоматически шифрует данные на стороне клиента при загрузке в хранилище Azure и расшифровывает их при вызове. Дополнительные сведения и примеры кода см. в записи блога [Приступая к работе с шифрованием на стороне клиента для службы хранилища Microsoft Azure](http://blogs.msdn.com/b/windowsazurestorage/archive/2015/04/29/getting-started-with-client-side-encryption-for-microsoft-azure-storage.aspx).

Некоторые дополнительные сведения о шифровании на стороне клиента:

- **Безопасность**: зашифрованные данные недоступны для чтения, даже если будут взломаны ключи учетной записи хранения.
- **Фиксированное шифрование**: зашифрованные данные будут иметь прогнозируемый размер в зависимости от исходного размера.
- **Автономное шифрование:** каждый большой двоичный объект, сущность таблицы или сообщения очереди сохраняют все данные о шифровании или в самом объекте, или в его метаданных. Единственным необходимым внешним значением является ключ шифрования.
- **Смена ключей**: пользователи могут менять ключи сами, при этом сменить можно несколько ключей.
- **Понятный вариант обновления**: дополнительные алгоритмы шифрования и версии протокола будут поддерживаться в будущем без каких-либо значительных изменений кода.
- **Поддержка шифрования больших двоичных объектов**:
	- **Полная загрузка больших двоичных объектов**: можно полностью зашифровать и загружать файлы, например документы, фотографии и видео.
	- **Полная выгрузка или выгрузка больших двоичных объектов на основе диапазонов**: можно выгрузить и расшифровать большой двоичный объект как полностью, так и в диапазонах.


>[AZURE.IMPORTANT]Учтите следующие важные моменты при использовании предварительной версии библиотеки:
>
>- Не используйте предварительную версию библиотеки для рабочих данных. Будущие изменения библиотеки повлияют на используемые схемы. Расшифровка данных, которые были зашифрованы с помощью предварительной версии библиотеки, в будущих версиях не гарантируется.  
>- При чтении или записи в зашифрованном большом двоичном объекте используйте команды загрузки полного большого двоичного объекта и команды диапазонной либо полной выгрузки. Избегайте написания зашифрованного большого двоичного объекта с помощью операций протокола, таких как Put Block, Put Block List, Write Pages или Clear Pages, иначе это может привести к повреждению зашифрованного большого двоичного объекта, что сделает его нечитабельным.
>- Для таблиц существуют аналогичные ограничения. Будьте внимательны и не обновляйте зашифрованные свойства без обновления метаданных шифрования.
>- Если задать метаданные в зашифрованном большом двоичном объекте, могут быть перезаписаны метаданные, относящиеся к шифрованию и необходимые для расшифровки, поскольку настройку метаданных добавить нельзя. Это также относится к моментальным снимкам: не рекомендуется указывать метаданные во время создания моментального снимка зашифрованного большого двоичного объекта.
