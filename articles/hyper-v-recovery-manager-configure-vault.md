# Приступая к работе с Azure Site Recovery: защита из локальной среды в локальную среду


Используйте Azure Site Recovery, чтобы организовать защиту виртуальных машин на локальных серверах узла Hyper-V, расположенных в облаках VMM. Вы можете настроить следующие конфигурации:

-   **Защита из локальной среды в локальную среду** — репликация локальных виртуальных машин на другой локальный сайт. Настройка и включение параметров защиты выполняется в хранилищах Azure Site Recovery. Данные виртуальной машины реплицируются из одного локального сервера Hyper-V на другой. Дополнительную информацию об этом сценарии см. в разделе [Приступая к работе с Azure Site Recovery: защита из локальной среды в локальную среду][Приступая к работе с Azure Site Recovery: защита из локальной среды в локальную среду].
-   **Защита из локальной среды в среду Azure** — репликация из локальной виртуальной машины в Azure. Настройка и включение параметров защиты выполняется в хранилищах Azure Site Recovery. Данные виртуальной машины реплицируются из локального сервера Hyper-V в службу хранилища Azure. Дополнительную информацию об этом сценарии см. в разделе [Приступая к работе с Azure Site Recovery: защита из локальной среды в среду Azure][Приступая к работе с Azure Site Recovery: защита из локальной среды в среду Azure].

## <span id="about"></span></a>О данном учебнике

Используйте этот учебник, чтобы быстро подготовить эксперимент для Azure Site Recovery при развертывании из локальной среды в Azure. По возможности используются самый быстрый путь и значения по умолчанию. Вы создадите хранилище Azure Site Recovery, установите поставщик Azure Site Recovery на исходном сервере VMM, настроите параметры облачной защиты, включите защиту виртуальных машин и протестируете свое развертывание.

Если вам нужна информация о полном развертывании, см.:

-   [Планирование развертывания Azure Site Recovery][Планирование развертывания Azure Site Recovery] — описывает шаги по планированию, которые необходимо выполнить перед тем, как начинать полное развертывание.

## <span id="before"></span></a>Перед началом работы

Перед началом работы с этим учебником проверьте выполнение необходимых условий.

### <span id="HVRMPrereq"></span></a>Предварительные требования

-   **Учетная запись Azure** — вам понадобится учетная запись Azure. Если у вас ее нет, см. раздел [Бесплатная пробная версия Azure][Бесплатная пробная версия Azure]. Информацию по ценам см. в разделе [Azure Site Recovery Manager Pricing Details][Azure Site Recovery Manager Pricing Details] (Цены на диспетчер Azure Site Recovery).
-   **Сервер VMM** — вам понадобится по крайней мере один сервер VMM под управлением System Center 2012 с пакетом обновления 1 (SP1) или System Center 2012 R2.
-   **Облака VMM** — вам необходимо не менее одного облака на исходном сервере VMM, который вы хотите защитить, и одно на конечном сервере VMM. Если вы используете один сервер VMM, для него необходимо настроить два облака. Основное облако, которые вы хотите защитить, должно содержать следующее:
    -   Одна или несколько групп узлов VMM
    -   Один или несколько серверов или кластеров узлов Hyper-V в каждой группе узлов.
    -   Одна или несколько виртуальных машин, расположенных на исходном сервере Hyper-V в облаке.

## Шаг 1. Создание хранилища

1.  Выполните вход на [Портал управления][Портал управления].

2.  Разверните элементы **Службы данных** и **Службы восстановления**, а затем щелкните **Site Recovery Vault** (Хранилище Site Recovery).

3.  Щелкните **Создать**, а затем **Быстрое создание**.

4.  В поле **Имя** введите понятное имя для идентификации хранилища.

5.  В поле **Область** выберите географический регион для хранилища архивации. Чтобы проверить поддерживаемые регионы, см. географическую доступность в разделе [Azure Site Recovery Pricing Details][Azure Site Recovery Pricing Details] (Информация о ценах на Azure Site Recovery).

6.  Щелкните элемент **Создать хранилище**.

    ![Новое хранилище][Новое хранилище]

Проверьте строку состояния, чтобы убедиться, что хранилище успешно создано. На главной странице служб восстановления это хранилище будет указано с состоянием **Активно**.

<a name="upload"></a>

## Шаг 2. Настройка хранилища

1.  На странице **Службы восстановления** щелкните хранилище, чтобы открыть страницу быстрого запуска. Страницу быстрого запуска можно открыть и в любой другой момент, щелкнув этот значок.

	![Значок быстрого запуска][Значок быстрого запуска]

2.  В раскрывающемся списке выберите **Between two on-premises Hyper-V sites** (Между двумя локальными сайтами Hyper-V).
3.  В разделе **Prepare VMM Servers** (Подготовка серверов VMM) выберите пункт **Generate registration key** (Создать регистрационный ключ). Ключ действителен в течение пяти дней после создания. Скопируйте файл на сервер VMM. Он вам потребуется при настройке поставщика.

    ![Регистрационный ключ][Регистрационный ключ]

<a name="download"></a>

## Шаг 3. Установка поставщика Azure Site Recovery

1.  На странице **Быстрый запуск** в разделе **Prepare VMM servers** (Подготовка серверов VMM) выберите пункт **Download Microsoft Azure Site Recovery Provider for installation on VMM servers** (Скачать поставщик Microsoft Azure Site Recovery для установки на серверы VMM), чтобы получить последнюю версию установочного файла приложения поставщика.

2.  Запустите этот файл на исходном и целевом серверах VMM.

3.  В разделе **Pre-requirements Check** (Проверка предварительных требований) выберите остановку службы VMM, чтобы начать установку приложения поставщика. Служба остановится и будет перезапущена автоматически после завершения установки.

    ![Предварительные требования][Предварительные требования]

4.  В **Центре обновления Майкрософт** вы можете настроить обновления. Если этот параметр включен, то обновления поставщика будут устанавливаться в соответствии с вашей политикой Центра обновления Майкрософт.

    ![Центр обновления Майкрософт][Центр обновления Майкрософт]

После установки приложения поставщика продолжите установку, чтобы зарегистрировать сервер в хранилище.

1.  На странице «Подключение к Интернету» укажите, как запущенный на сервере VMM поставщик подключается к Интернету. Выберите параметр **Use default system proxy settings** (Использовать настройки прокси-сервера по умолчанию), чтобы использовать настройки подключения к Интернету, по умолчанию настроенные на сервере.

    ![Параметры Интернета][Параметры Интернета]

2.  В поле **Регистрационный ключ** укажите, что вы скачали его с Azure Site Recovery и скопировали на сервер VMM.
3.  В поле **Имя хранилища** проверьте имя хранилища, в котором будет зарегистрирован сервер.
4.  В поле **Имя сервера** укажите понятное имя, описывающее сервер VMM в хранилище.

    ![Регистрация сервера][Регистрация сервера]

5.  В поле синхронизации **Initial cloud metadata** (Первоначальные метаданные в облаке) выберите, нужно ли синхронизировать метаданные для всех облаков сервера VMM в хранилище. На каждом сервере это действие требуется выполнять только один раз. Если не требуется синхронизировать все облака, можно оставить этот флажок снятым и синхронизировать каждое облако индивидуально в свойствах облака в консоли VMM.

6.  В разделе **Шифрование данных** вы создаете сертификат, который используется для шифрования данных, защищенных в Azure.
    Этот параметр не важен, если вы проводите репликацию из одного локального узла на другой.

    ![Регистрация сервера][1]

7.  Для завершения процесса нажмите кнопку **Зарегистрировать**. После регистрации метаданные с сервера VMM извлекаются службой Azure Site Recovery. Сервер отображается на вкладке **Ресурсы** на странице **Серверы** в хранилище.

## <span id="clouds"></span></a>Шаг 4: настройка параметров защиты облаков

После регистрации серверов VMM можно настроить параметры защиты облачных систем. Вы включили параметр **Синхронизировать облачные данные с хранилищем**, когда установили поставщик, так что все облака на сервере VMM появятся на вкладке **Защищенные элементы** в хранилище.

![Опубликованное облако][Опубликованное облако]

1.  На странице быстрого запуска щелкните **Set up protection for VMM clouds** (Настроить защиту для облаков VMM).
2.  На вкладке **Защищенные элементы** выберите облако, которое нужно настроить, и перейдите на вкладку **Конфигурация**. Обратите внимание на следующее:
3.  В разделе **Цель** выберите **VMM**.
4.  В поле **Целевое расположение** выберите локальный сервер VMM, управляющий облаком, которое требуется использовать для восстановления.
5.  В поле **Целевое облако** укажите целевое облако, которое требуется использовать для отработки отказа виртуальных машин в исходном облаке. Обратите внимание на следующее:

    -   Рекомендуется выбрать целевое облако, удовлетворяющее требованиям к восстановлению для виртуальных машин, которые планируется защитить.
    -   Облако может принадлежать относиться только к одной паре облаков — в качестве основного либо в качестве целевого.

6.  В поле **Периодичность копирования** оставьте значение по умолчанию. Это значение указывает, как часто следует синхронизировать данные между исходным и конечным расположениями. Оно применяется только при работе узла Hyper-V в Windows Server 2012 R2. Для других серверов используется значение по умолчанию, равное 5 минутам.
7.  В поле **Дополнительные точки восстановления** оставьте значение по умолчанию. Это значение указывает, следует ли создавать дополнительные точки восстановления. При использовании нулевого значения по умолчанию на сервере узла реплики хранится только последняя точка восстановления для основной виртуальной машины.
8.  В поле **Периодичность создания соответствующих приложению снимков** оставьте значение по умолчанию. Это значение указывает, как часто создаются моментальные снимки. Снимки используют службу теневого копирования томов (VSS), чтобы обеспечить согласованное состояние приложение при создании моментального снимка. Если вы хотите задать это значение при работе с учебником, убедитесь, что оно меньше количества заданных дополнительных точек восстановления.
    ![Настройка параметров защиты][Настройка параметров защиты]
9.  В поле **Data transfer compressed** (Сжатие данных при передаче) укажите, требуется ли сжимать передаваемые реплицированные данные.
10. В поле **Проверки подлинности** укажите, как осуществляется проверка подлинности трафика между серверами основного узла и узла восстановления Hyper-V. При прохождении данного учебника рекомендуется выбрать протокол HTTPS, если у вас не настроена рабочая среда Kerberos. Azure Site Recovery автоматически настроит сертификаты для аутентификации HTTPS. Никакой дополнительной настройки не требуется. Обратите внимание, что этот параметр используется только при запуске серверов узлов Hyper-V в Windows Server 2012 R2.
11. В поле **Порт** оставьте значение по умолчанию. Это значение задает номер порта, через который исходный и конечный компьютеры узла Hyper-V прослушивают трафик репликации.
12. В поле **Метод репликации** укажите, как будет осуществляться начальная репликация данных из исходного расположения в целевое перед началом обычной репликации.
    -   **По сети** — копирование данных через сеть может требовать много времени и ресурсов. Рекомендуется использовать этот параметр, если облако содержит виртуальные машины с относительно небольшими виртуальными жесткими дисками, а основной сервер VMM подключен к дополнительному серверу VMM через широкополосное соединение. Можно немедленно начать копирование или выбрать соответствующее время. При использовании репликации по сети рекомендуется запланировать ее на часы наименьшей нагрузки.
    -   **Автономно** — этот метод указывает, что начальная репликация будет выполняться с помощью внешнего носителя. Это удобно, если требуется избежать снижения производительности сети, а также для местоположений, находящихся на большом расстоянии друг от друга. Для использования этого метода необходимо указать расположение экспорта в исходном облаке и расположение импорта в целевом облаке. При включении защиты для виртуальной машины виртуальный жесткий диск копируется в указанное расположение экспорта. Вы отправляете его на целевой сайт и копируете в расположение импорта. Система копирует импортированные сведения на виртуальные машины реплики. Полный список необходимых условий репликации в автономном режиме см. в разделе [Шаг 3. Настройка параметров защиты для облаков VMM][Шаг 3. Настройка параметров защиты для облаков VMM] руководства по развертыванию.
13. Выберите **Delete Replica Virtual Machine** (Удалить виртуальную машину реплики), чтобы указать, что виртуальная машина реплики должна быть удалена, когда защита виртуальной машины прекращена в результате выбора параметра **Delete protection for the virtual machine** (Удалить защиту виртуальной машины) на вкладке «Виртуальные машины» в свойствах облака. Когда этот параметр включен, то при отключении защиты виртуальная машина удаляется из Azure Site Recovery, параметры Site Recovery для виртуальной машины удаляются из консоли VMM, также удаляется реплика.
    ![Настройка параметров защиты][2]

После сохранения параметров создается задание, которое можно отслеживать на вкладке **Задания**. Все сервера узлов Hyper-V в исходном облаке VMM могут быть настроены для репликации. Параметры облака можно изменить на вкладке **Настройка**. Чтобы изменить целевое расположение или целевое облако после сохранения конфигурации, необходимо удалить эту конфигурацию облака и затем повторно настроить облако.

## <span id="enablevirtual"></span></a>Шаг 7: включение защиты виртуальной машины

После успешной настройки серверов, облаков и сетей можно включить защиту для виртуальных машин в облаке.

1.  На вкладке **Виртуальные машины** в облаке, в котором расположена виртуальная машина, щелкните **Включить защиту** и выберите пункт **Добавить виртуальные машины**.
2.  Из списка виртуальных машин в облаке выберите те, которые хотите защитить.

![Включение защиты виртуальной машины][Включение защиты виртуальной машины]

Ход выполнения действия «Включение защиты» можно наблюдать на вкладке «Задания», в том числе и первоначальную репликацию. После запуска задачи финализации защиты виртуальная машина готова к отработке отказа. После включения защиты и репликации виртуальных машин вы сможете просматривать их в Azure.

![Задание защиты виртуальной машины][Задание защиты виртуальной машины]

## <span id="recoveryplans"></span></a>Шаг 8: тестирование развертывания

Чтобы протестировать развертывание, можно запустить тестовую отработку отказа для отдельной виртуальной машины или создать план восстановления, состоящий из нескольких виртуальных машин, и запустить тестовую отработку отказа плана. Тестовая обработка отказа имитирует механизм отработки отказа и восстановления в изолированной сети.

-   Инструкции по созданию плана восстановления см. в разделе [Создание и настройка планов восстановления: из локальной среды в среду Azure][Создание и настройка планов восстановления: из локальной среды в среду Azure].
-   Указания по запуску тестовой отработки отказа см. в разделе [Тестирование развертывания из локальной среды в локальную среду][Тестирование развертывания из локальной среды в локальную среду].

### <span id="runtest"></span></a>Мониторинг активности

Можно использовать вкладку **Задания** и **Панель мониторинга**, чтобы просматривать и отслеживать выполнение основных заданий в хранилище Azure Site Recovery, в том числе настройку защиты для облака, включение и отключение защиты для виртуальной машины, выполнение отработки отказа (запланированной, незапланированной или тестовой) и выполнение незапланированной отработки отказа.

На вкладке **Задания** можно просматривать задания, их подробности и ошибки, запускать запросы извлечения заданий, отвечающих конкретным критериям, экспортировать задания в Excel и перезапускать неудавшиеся задания.

Из **Панели мониторинга** можно скачивать последние версии установочных файлов поставщика и агента, получать информацию о конфигурации хранилища, просматривать количество виртуальных машин, защитой которых управляет хранилище, просматривать последние задания, управлять сертификатом хранилища и проводить повторную синхронизацию виртуальных машин.

Дополнительную информацию о взаимодействии с заданиями и панелью мониторинга см. в [руководстве по эксплуатации и мониторингу][руководстве по эксплуатации и мониторингу].
