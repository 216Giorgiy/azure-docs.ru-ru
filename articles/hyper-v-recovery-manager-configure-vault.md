# Настройка диспетчера восстановления Hyper-V Hyper-V Azure



<div class="dev-callout"> 

<P>Диспетчер восстановления Hyper-V Azure координирует и контролирует защиту виртуальных машин Hyper-V, находящихся в частных облаках на серверах диспетчера виртуальных машин (VMM) в System Center 2012 с пакетом обновления 1 (SP1) или System Center 2012 R2. Диспетчер восстановления Hyper-V организует отработку отказа для этих виртуальных машин с одного локального сервера размещения Hyper-V на другой. Хранилища диспетчера восстановления Hyper-V в Azure используются для хранения конфигурации защиты.</P>

<h3><a id="HVRMPrereq"></a>Необходимые условия для диспетчера восстановления Hyper-V</h3>

<UL>
<LI>**Учетная запись Azure** — вам потребуется учетная запись Azure с включенным компонентом служб восстановления Azure. Если такая учетная запись отсутствует или компонент не зарегистрирован, см. разделы <a href="http://aka.ms/try-azure">Бесплатная пробная версия Azure</a> и <a href="http://go.microsoft.com/fwlink/?LinkId=378268">Сведения о ценах на диспетчер восстановления Hyper-V</a>.</LI>
<LI>**Сертификат (CER)** — для регистрации серверов VMM в хранилище диспетчера восстановления Hyper-V потребуется отправить сертификат управления (CER), содержащий открытый ключ, в хранилище. Обратите внимание на следующее:<UL>
	<LI>В рамках данного учебника вы можете использовать самозаверяющий сертификат, созданный с помощью средства Makecert.exe. Для полного развертывания можно использовать любой действительный сертификат SSL, который соответствует сертификат необходимым условиям, описанным в разделе <a href=" http://go.microsoft.com/fwlink/?LinkId=386485">Необходимые условия и поддержка</a> руководства по планированию.</LI>
	<LI>Одновременно с каждым хранилищем сопоставлен только один сертификат CER. При необходимости можно отправить новый сертификат, чтобы перезаписать существующий.</LI>
</UL></LI>
<LI>**Сертификат (PFX-файл)** — сертификат CER необходимо экспортировать в PFX-файл (содержащий закрытый ключ). Этот файл импортируется на каждый сервер VMM, содержащий виртуальные машины, которые необходимо защитить. Затем в процессе развертывания при установке агента поставщика диспетчера восстановления Hyper-V на каждом сервере VMM вы выбираете этот PFX-файл, чтобы зарегистрировать сервер VMM для хранилища.</LI>

</UL>

<h3><a id="VMMPrereq"></a>Необходимые условия для VMM</h3>

<UL>
<LI>По крайней мере один сервер VMM с System Center 2012 SP1 или System Center 2012 R2.</LI>
<LI>Частные облака VMM. Если вы используете один сервер VMM, для него необходимо настроить два облака. При наличии двух и более серверов VMM потребуется хотя бы одно облако на исходном сервере VMM, которое вы хотите защитить, и хотя бы одно облако на конечном сервере VMM, которое будет использоваться для восстановления. Основное облако, которые вы хотите защитить, должно содержать следующее:<UL>
	<LI>Одна или несколько групп узлов VMM</LI>
	<LI>Один или несколько серверов узла Hyper-V в каждой группе узлов.</LI>
		</UL></LI>
<LI>Если требуется подключение виртуальных машин к сети виртуальных машин после отработки отказа, настройте сетевое сопоставление в диспетчере восстановления Hyper-V. Перед началом развертывания проверьте выполнение следующих условий:<UL>
	<LI>Виртуальная машина в облаке на исходном сервере VMM подключена к сети виртуальных машин. Сеть виртуальных машин должна быть связана с логической сетью, которая сопоставлена с облаком.</LI>
	<LI>Целевое облако на конечном сервере VMM имеет соответствующую сеть виртуальных машин. Сеть виртуальных машин должна быть связана с соответствующей логической сетью, которая сопоставлена с целевым облаком.</LI>
	</UL></LI>
	
	<P>Для получения дополнительных сведений о сетевом сопоставлении см. раздел <a href="http://go.microsoft.com/fwlink/?LinkId=324817">Подготовка к сетевому сопоставлению</a> руководства по планированию.</P>
</UL>

<h3><a id="obtaincert"></a>Получение самозаверяющего сертификата (CER)</h3>
<P>Используйте средство MakeCert для создания сертификата CER x.509, который соответствует всем требованиям к сертификату, воспользовавшись следующей процедурой:</P>
<ol>
<LI>
На компьютере, где требуется запустить MakeCert, загрузите последнюю версию пакета <a href="http://go.microsoft.com/fwlink/?LinkId=378269">Windows SDK</a> Обратите внимание, что команда makecert.exe является частью ядра пакета средств разработки программного обеспечения Windows, поэтому вам не требуется загружать и устанавливать пакет целиком.</LI>
<LI>На странице "Выбор расположения" выберите пункт **Установить пакет средств разработки программного обеспечения Windows для Windows 8.1 на данном компьютере**.</LI>
<LI>На странице "Принять участие в программе улучшения качества программного обеспечения (CEIP)" выберите предпочтительный параметр.</LI>
<LI>На странице "Лицензионное соглашение" выберите элемент **Принимаю**, чтобы принять условия.</LI>
<LI>На странице "Выберите компоненты, которые вы хотите установить" отмените выбор всех параметров, кроме параметра **Пакет средств разработки программного обеспечения Windows**.</LI>
<LI>После установки убедитесь, что программа makecert.exe появилась в папке C:\ProgramFiles (x 86) \Windows Kits\<i>версия_Windows</i>\bin\x64.</LI>
<LI>Откройте командную строку (cmd.exe) с правами администратора и перейдите к папке с makecert.exe.</LI> 
<LI>> Выполните следующую команду, чтобы создать самозаверяющий сертификат. Замените значение CertificateName на имя, которое вы хотите использовать для сертификата, и укажите фактический срок действия сертификата после параметра -e:</LI>
<code>
makecert.exe -r -pe -n CN=CertificateName -ss my -sr localmachine -eku 1.3.6.1.5.5.7.3.2 -len 2048 -e 01/01/2016 CertificateName.cer</code>
</ol>
<P>Сертификат создается и сохраняется в той же папке. Вам может потребоваться переместить его в более удобное место, чтобы выполнить экспорт в следующем шаге.</P>

<h3><a id="exportcert"></a>Экспорт сертификата в формате PFX-файла</h3>
<P>Выполните эту процедуру для экспорта сертификата CER в формате PFX-файла.</P>
<ol>
<li>Введите **mmc.exe** на начальном экране для запуска консоли управления (MMC).</li>
<li>В меню **Файл** выберите пункт **Добавить или удалить оснастку** Отображается диалоговое окно **Добавление и удаление оснасток**.</li>
<li>В области **Доступные оснастки** выберите элемент **Сертификаты** и нажмите кнопку **Добавить**.</li>
<li>Выберите элемент **Учетная запись компьютера** и нажмите кнопку **Далее**.</li>
<li>Выберите элемент **Локальный компьютер** и нажмите кнопку **Готово**.</li>
<li>В дереве консоли MMC разверните узел **Сертификаты** и затем узел **Личные**.</li>
<li>В области сведений щелкните сертификат, которым вы хотите управлять.</li>
<li>В меню **Действие** выберите пункт **Все задачи** и щелкните элемент **Экспорт**. Запускается мастер экспорта сертификатов. Нажмите кнопку **Далее**.</li>
<li>На странице **Экспортирование закрытого ключа** щелкните элемент **Да**, чтобы экспортировать закрытый ключ. Нажмите кнопку **Далее**. Обратите внимание, что это требуется только в том случае, когда нужно экспортировать закрытый ключ на другие серверы после установки.</li>
<li>На странице "Формат экспортируемого файла" выберите **Файл обмена личной информацией - PKCS #12 (.PFX)**. Нажмите кнопку **Далее**.</li>
<li>На странице **Пароль** введите и подтвердите пароль, используемый для шифрования закрытого ключа. Нажмите кнопку **Далее**.</li>
<li>Следуйте указаниям на страницах мастера, чтобы экспортировать сертификат в формате PFX-файла.</li>
</ol>


<h3><a id="importcert"></a>Импорт сертификата PFX на серверы VMM</h3>
<p>После экспорта сертификата скопируйте его на каждый сервер VMM, который требуется зарегистрировать в хранилище, и затем импортируйте его следующим образом. Обратите внимание, что при запуске средства MakeCert.exe на сервере VMM импортировать сертификат на этот сервер не нужно.</p>
 
<ol>
<li>Скопируйте файлы сертификатов (PFX) закрытого ключа в папку на локальном сервере.</li>
<li>В оснастке MMC для сертификатов выберите элемент ** Учетная запись компьютера** и нажмите кнопку "Далее".</li>
<li>Выберите элемент "Локальный компьютер" и нажмите кнопку **Готово**. Нажмите кнопку **ОК** в диалоговом окне **Добавление и удаление оснасток**. </li>
<li>В консоли MMC разверните узел **Сертификаты**, щелкните правой кнопкой мыши элемент **Личные**, выберите пункт "Все задачи" и щелкните элемент **Импорт** для запуска мастера импорта сертификатов.</li>
<li>На странице приветствия мастера импорта сертификатов нажмите кнопку **Далее**.</li>
<li>На странице "Импортируемый файл" нажмите кнопку **Обзор** и перейдите в папку с PFX-файлом того сертификата, который требуется импортировать. Выберите нужный файл и нажмите кнопку **Открыть**.</li>
<li>В поле **Пароль** на странице "Пароль" введите пароль для файла закрытого ключа, который указан в предыдущей процедуре, а затем нажмите кнопку **Далее**.</li>
<li>На странице "Хранилище сертификатов" выберите пункт **Поместить все сертификаты в следующее хранилище**, нажмите кнопку **Обзор**, выберите **Личное** хранилище, нажмите кнопку **ОК** и кнопку "Далее".</li>
<li>На последней странице мастера импорта сертификатов нажмите кнопку **Готово**.</li>
</ol>

После выполнения этих действий можно будет выбрать сертификат CER для отправки в хранилище, а также выбрать PFX-файл сертификата при регистрации сервера VMM во время установки поставщика.
</div>

	![Новое хранилище](./media/hyper-v-recovery-manager-configure-vault/RS_hvvault.png)

<a name="download"></a> <h2>Шаг 3. Загрузка и установка поставщика</h2>
Установите поставщик диспетчера восстановления Hyper-V на каждом сервере VMM, который хотите зарегистрировать в хранилище. Самая новая версия установочного файла поставщика хранится в центре загрузки Azure. При запуске файла на сервере VMM выполняется установка поставщика, а сам сервер VMM регистрируется в хранилище.

1. На странице **Быстрый запуск** щелкните элемент **Загрузить поставщик**, чтобы получить установочный EXE-файл поставщика. Запустите этот файл на сервере VMM, чтобы начать установку поставщика.

	![Загрузить агент](./media/hyper-v-recovery-manager-configure-vault/RS_installwiz.png)

2. Следуйте инструкциям для завершения установки поставщика.

	![Завершение установки](./media/hyper-v-recovery-manager-configure-vault/RS_SetupComplete.png)

3. После завершения установки поставщика следуйте инструкциям мастера, чтобы зарегистрировать сервер VMM в хранилище.
4. На странице "Подключение к Интернету" укажите, как запущенный на сервере VMM поставщик подключается к Интернету. Поставщик может использовать доступные на сервере параметры подключения к Интернету по умолчанию, либо выберите элемент **Использование прокси-сервера для интернет-запросов** для использования настраиваемых параметров.
	
	![Параметры Интернета](./media/hyper-v-recovery-manager-configure-vault/RS_ProviderProxy.png)

	Если вы хотите использовать настраиваемые параметры в данном пошаговом руководстве, ознакомьтесь с информацией в разделе <a href="http://go.microsoft.com/fwlink/?LinkId=378266">Шаг 2. Установка поставщика и регистрация серверов VMM</a> руководства по развертыванию.


5. На странице регистрации сертификатов выберите PFX-файл, соответствующий сертификату CER, отправленному в хранилище. 

	![Регистрация сертификатов](./media/hyper-v-recovery-manager-configure-vault/RS_CertReg1.png)

На каждом сервере это действие требуется выполнять только один раз. Если вы не хотите синхронизировать все облака, можно опубликовать каждое облако отдельно, чтобы синхронизировать его перед настройкой параметров защиты облака.

<h2><a id="clouds"></a>Шаг 4. Настройка параметров защиты облачных систем</h2>

После регистрации серверов VMM можно настроить параметры защиты облачных систем. Если при установке поставщика на сервере VMM не был включен параметр **Синхронизировать облачные данные с хранилищем**, необходимо опубликовать облако в диспетчере восстановления Hyper-V из консоли VMM. После обнаружения облаков диспетчером восстановления Hyper-V можно настроить параметры защиты. 

<h3><a id="publishclouds"></a>Публикация облака</h3>

1. Откройте в консоли VMM рабочую область **Виртуальные машины (VM) и службы**.
2. В области **Виртуальные машины (VM) и службы** откройте облака, которые требуется опубликовать.
3. На странице свойств облака **Общие** выберите **Отправить данные конфигурации об этом облаке в диспетчер восстановления Hyper-V Azure** для публикации облака. После публикации облака оно отображается в хранилище.

	![Облака](./media/hyper-v-recovery-manager-configure-vault/RS_PublishCloud.png)

	![Опубликованное облако](./media/hyper-v-recovery-manager-configure-vault/RS_Clouds.png)

<h3><a id="configureclouds"></a>Настройка облаков</h3>

Чтобы настроить облака в целях защиты, выполните следующие действия: 

1. На странице быстрого запуска щелкните элемент **Настроить параметры защиты**.
2. На вкладке **Защищенные элементы** выберите облако, которое нужно настроить, и перейдите на вкладку **Конфигурация**.

	![Конфигурация облака](./media/hyper-v-recovery-manager-configure-vault/RS_CloudConfig.png)


3. В поле **Целевое расположение** укажите сервер VMM, управляющий облаком, которое требуется использовать для восстановления.
4. В поле **Целевое облако** укажите целевое облако, которое требуется использовать для отработки отказа виртуальных машин в исходном облаке.
5. В поле **Периодичность копирования** оставьте значение по умолчанию. Это значение указывает, как часто следует синхронизировать данные между исходным и конечным расположениями. Оно применяется только при работе узла Hyper-V в Windows Server 2012 R2. Для других серверов используется значение по умолчанию, равное 5 минутам.
6. В поле **Дополнительные точки восстановления** оставьте значение по умолчанию. Это значение указывает, следует ли создавать дополнительные точки восстановления. При использовании нулевого значения по умолчанию на сервере узла реплики хранится только последняя точка восстановления для основной виртуальной машины. 
7. В поле **Периодичность создания соответствующих приложению снимков** оставьте значение по умолчанию. Это значение указывает, как часто создаются моментальные снимки. Снимки используют службу теневого копирования томов (VSS), чтобы обеспечить согласованное состояние приложение при создании моментального снимка.  Если вы хотите задать это значение при работе с учебником, убедитесь, что оно меньше количества заданных дополнительных точек восстановления.
8. В поле **Сжатие данных при передаче** укажите, требуется ли сжимать передаваемые реплицированные данные. 
9. В поле **Проверки подлинности** укажите, как осуществляется проверка подлинности трафика между серверами основного узла и узла восстановления Hyper-V. При отсутствии настроенной рабочей среды Kerberos при работе с данным пошаговым руководством рекомендуется выбрать протокол HTTPS. При выборе HTTPS серверы узла будут осуществлять взаимную проверку подлинности с помощью сертификата сервера, а трафик будет зашифрован. Диспетчер восстановления Hyper-V автоматически настраивает сертификаты, используемые для проверки подлинности HTTPS. Никакой дополнительной настройки не требуется. Обратите внимание, что этот параметр используется только при запуске серверов узлов Hyper-V в Windows Server 2012 R2.
10. В поле **Порт** оставьте значение по умолчанию. Это значение задает номер порта, через который исходный и конечный компьютеры узла Hyper-V прослушивают трафик репликации. 
11. В поле **Параметры начальной репликации** укажите, как будет осуществляться начальная репликация данных из исходного расположения в конечное перед началом обычной репликации. 
	- **По сети** — копирование данных через сеть может требовать много времени и ресурсов. Рекомендуется использовать этот параметр, если облако содержит виртуальные машины с относительно небольшими виртуальными жесткими дисками, а основной сервер VMM подключен к дополнительному серверу VMM через широкополосное соединение. Можно немедленно начать копирование или выбрать соответствующее время. При использовании репликации по сети рекомендуется запланировать ее на часы наименьшей нагрузки.
	- **Автономно** — этот метод указывает, что начальная репликация будет выполняться с помощью внешнего носителя. Это удобно, если требуется избежать снижения производительности сети, а также для местоположений, находящихся на большом расстоянии друг от друга. Для использования этого метода необходимо указать расположение экспорта в исходном облаке и расположение импорта в целевом облаке. При включении защиты для виртуальной машины виртуальный жесткий диск копируется в указанное расположение экспорта. Вы отправляете его на целевой сайт и копируете в расположение импорта. Система копирует импортированные сведения на виртуальные машины реплики. Полный список необходимых условий репликации в автономном режиме см. в разделе <a href="http://go.microsoft.com/fwlink/?LinkId=323469">Шаг 3. Настройка параметров защиты для облаков VMM</a> руководства по развертыванию.

<h4><a id="cloudsettingd"></a>Конфигурирование после настройки защиты</h4>
После настройки облака производится настройка всех кластеров и серверов узла, заданных в исходном и целевом облаках, для репликации. В частности, настраивается следующее:

- Задаются правила брандмауэра, используемые репликой Hyper-V, чтобы были открыты порты для трафика репликации. 
- Устанавливаются сертификаты, необходимые для репликации.
- Задаются параметры реплики Hyper-V.

Параметры облака можно изменить на вкладке **Настройка**. Обратите внимание на следующее:

- Рекомендуется выбрать целевое облако, удовлетворяющее требованиям к восстановлению для виртуальных машин, которые планируется защитить. 
- Облако может принадлежать относиться только к одной паре облаков — в качестве основного либо в качестве целевого.
- После сохранения конфигурации облака создается задание, которое можно отслеживать на вкладке **Задания**. Чтобы изменить целевое расположение или целевое облако после сохранения конфигурации, необходимо удалить эту конфигурацию облака и затем повторно настроить облако.




<h2><a id="networks"></a>Действие 5. Настройка сетевого сопоставления</h2>

На вкладке **Сети** осуществляется настройка сопоставления между сетями виртуальных машин на исходном и конечном серверах VMM. Сетевое сопоставление обеспечивает подключение виртуальных машин реплики к соответствующим сетям после отработки отказа, а также размещение виртуальных машин реплики на узлах, которым доступны сети виртуальных машин. Проверьте выполнение требований к сети VMM, описанных в разделе <a href=" http://go.microsoft.com/fwlink/?LinkId=386485">Необходимые условия и поддержка</a> руководства по планированию. Рекомендуется сопоставлять сети следующим образом: 

- Сопоставляйте сети, используемые каждым облаком, для которого настроена защита.
- Выполняйте сетевое сопоставление до включения защиты виртуальных машин, так как сопоставление используется при размещении виртуальных машин реплики. Сопоставление будет работать правильно и при выполнении настройки после включения защиты, однако в этом случае может потребоваться миграция некоторых виртуальных машин реплики.  

Чтобы сопоставить сети, выполните следующие действия:

1. На странице быстрого запуска щелкните элемент **Настроить сетевое сопоставление**.
4. В поле **Исходное расположение** выберите исходный сервер VMM. 
5. В поле **Целевое расположение** выберите целевой сервер VMM. При развертывании диспетчера восстановления Hyper-V на одном сервере VMM исходное и целевое расположения будут совпадать.
Отображается список исходных сетей виртуальных машин и назначенных им целевых сетей виртуальных машин. Для сетей, которые не сопоставляются, отображается пустое значение.
6. Выберите несопоставленную запись в списке исходных и целевых расположений и щелкните элемент **Сопоставить**. Служба обнаруживает сети виртуальных машин на целевом сервере и отображает их.  

	![Управление сертификатом](./media/hyper-v-recovery-manager-configure-vault/RS_networks.png)

7. На странице "Выберите целевую сеть" выберите целевую сеть виртуальных машин, которую хотите использовать на целевом сервере VMM.
	![Целевая сеть](./media/hyper-v-recovery-manager-configure-vault/RS_TargetNetwork.png) 

8. Щелкните значки сведений рядом с именами исходной и целевой сетей для просмотра подсетей и типа для каждой сети.

	При выборе конечной сети отображаются защищенные облака, использующие исходную сеть, а также указывается доступность целевых сетей виртуальных машин, сопоставленных с облаками. Рекомендуется выбрать целевую сеть, которая доступна для всех облаков, используемых для защиты.

8. Установите флажок для завершения процесса сопоставления. Запускается задание, отслеживающее ход выполнения сопоставления. Его можно просмотреть на вкладке **Задания**.

	При этом любые существующие виртуальные машины реплики, соответствующие виртуальным машинам, подключенным к исходной сети виртуальных машин, подключаются к целевой сети виртуальных машин. Кроме того, новые виртуальные машины реплики, создаваемые после включения репликации, на виртуальных машинах, подключенных к исходной сети виртуальных машин, подключаются к целевой сети виртуальных машин.

<h3><a id="modifynetworks"></a>Изменение сетевых сопоставлений</h3>
Сетевые сопоставления можно изменить или удалить на вкладке **Сети**. Обратите внимание на следующее:
<UL>
<LI>В случае отмены выбранного сопоставления оно удаляется, а виртуальные машины реплики отключаются от сети, к которой они были подключены во время сопоставления.</LI>
<LI>При изменении выбранного сопоставления изменения обновляются, а виртуальные машины реплики подключаются с использованием новых указанных сетевых настроек.</LI>
</UL>

<h2><a id="virtualmachines"></a>Шаг 6. Включение защиты для виртуальных машин</h2>

<p>После успешной настройки серверов, облаков и сетей можно включить защиту для виртуальных машин в облаке. Для включения защиты в консоли VMM щелкните правой кнопкой мыши каждую виртуальную машину, которую необходимо защитить, и выберите пункт **Включить восстановление**.</p>

<p>После включения защиты виртуальная машина отображается в списке виртуальных машин облака. Ход выполнения операции по включению защиты можно просмотреть на вкладке **Задания**.</p>

![Виртуальные машины](./media/hyper-v-recovery-manager-configure-vault/RS_Clouds.png)

<P>При включении защиты виртуальных машин создаются три задания. Запускается задание включения защиты. Затем после завершения начальной репликации запускаются два дополнительных задания финализации защиты. Виртуальная машина считается готовой к отработке отказа только после успешного завершения этих трех заданий.</P>

<h2><a id="recoveryplans"></a>Шаг 7. Настройка и запуск планов восстановления</h2>
План восстановления собирает виртуальные машины в группы, чтобы их можно было обрабатывать как единое целое, и указывает порядок групп ля отработки отказа. Кроме определения групп виртуальных машин, можно настроить планы восстановления для запуска автоматических скриптов или ожидания подтверждения выполняемых вручную действий. Чтобы создать план восстановления, выполните следующие действия: 

1. На странице быстрого запуска щелкните элемент **Создать план восстановления**.
2. На странице "Указание имени конечного расположения страницы восстановления" введите имя, а также исходный и целевой серверы VMM. При развертывании диспетчера восстановления Hyper-V на одном сервере VMM исходный и целевой сервера будут совпадать. Исходный сервер VMM должен содержать виртуальные машины с включенной защитой.

	![Создание плана восстановления](./media/hyper-v-recovery-manager-configure-vault/RS_RecoveryPlan1.png)
3. На странице выбора виртуальных машин выберите виртуальные машины, которые хотите добавить в план восстановления. Они добавляются в группу плана восстановления по умолчанию (группа 1).
4. Установите флажок, чтобы создать план восстановления. Создаваемые планы восстановления можно удалить на вкладке **Планы восстановления**.

	![Виртуальные машины плана восстановления](./media/hyper-v-recovery-manager-configure-vault/RS_RecoveryPlan2.png)


После создания плана восстановления можно выполнять следующие действия:

- Настройка плана. Можно добавить в план восстановления новые группы или добавить виртуальные машины в группу. Можно определить настраиваемые действия в виде скриптов и ручных действий, которые выполняются до и после указанных групп. Имена групп задаются по принципу приращения. После стандартной группы 1 плана восстановления создается группа 2, затем группа 3 и так далее. Отработка отказа для виртуальных машин осуществляется в соответствии с порядком групп. Дополнительные сведения о настройке планов восстановления см. в разделе <a href="http://go.microsoft.com/fwlink/?LinkId=378271">Шаг 6. Создание и настройка планов восстановления</a> руководства по развертыванию.
- Запуск тестовой отработки отказа.


<h3><a id="run"></a>Запуск планов восстановления</h3>
Планы восстановления можно запустить в рамках упреждающего теста или запланированной отработки отказа, а также во время незапланированной отработки отказа. В этом пошаговом руководстве описывается выполнение тестовой обработки отказа. Сведения о других типах отработки отказа см. в <a href="hhttp://go.microsoft.com/fwlink/?LinkId=378272">руководстве по операциям и мониторингу</a>.

<h3><a id="protect"></a>Запустите план.</h3>

Запустите тестовую обработку отказа, чтобы проверить работу плана восстановления в тестовой среде. Тестовая обработка отказа используется для проверки правильной работы плана восстановления и стратегии обработки отказа для виртуальных машин.  Тестовая обработка отказа имитирует механизм отработки отказа и восстановления в изолированной сети. 

<h5><a id="networkrecommendations"></a>Перед началом работы</h5>
При активации тестовой обработки отказа вам требуется указать, как виртуальные машины должны быть подключены к сетям после отработки отказа. 
<UL>
<LI>Если требуется использовать существующую сеть, рекомендуется создать для этого отдельную логическую сеть, которая не используется рабочей среде.</LI>
<LI>Если выбрать параметр для автоматического создания тестовой сети виртуальных машин, посте выполнения тестовой обработки отказа выполняется автоматическое удаление временных сетей и тестовых виртуальных машин.</LI>
<LI>При использовании логической сети на базе виртуальной локальной сети убедитесь, что добавляемые в логическую сеть узлы являются изолированными. </LI>
<LI>При использовании логической сети на базе виртуализации сети Windows диспетчер восстановления Hyper-V автоматически создает изолированные сети виртуальных машин.</LI>
</UL>


<h5><a id="runtest"></a>Запуск тестовой отработки отказа</h5>
Запустите тестовую отработку отказа для плана восстановления следующим образом:

<OL>
<LI>На вкладке **Планы восстановления** выберите нужный план восстановления.</LI>
<LI>Чтобы инициировать отработку отказа, нажмите кнопку **Тестовая отработка отказа**.</LI>
<LI>На странице "Подтвердить тестовую отработку отказа" укажите, как виртуальные машины должны быть подключены к сетям после тестовой отработки отказа:</LI>
<UL>
<LI>**Нет** — выберите этот параметр, чтобы указать, что сети виртуальных машин не следует использовать при тестовой отработке отказа. Используйте этот параметр, если требуется протестировать отдельные виртуальные машины, а не всю сетевую конфигурацию. Он также позволят быстро получить представление о том, как работает отработка отказа. После отработки отказа тестовые виртуальные машины не будут подключены к сети.</LI>
<LI>**Использовать существующий** — используйте этот параметр, если вы уже создали и изолировали сеть виртуальных машин для тестовой отработки отказа. После отработки отказа все тестовые виртуальные машины, используемые для тестовой отработки отказа, будут подключены к сети, указанной в поле **Сеть виртуальных машин**.</LI>
<LI>**Создать автоматически** — выберите этот параметр, чтобы указать, что диспетчер восстановления Hyper-V должен автоматически создавать сеть виртуальных машин на основе параметров, указанных для логической сети и связанных с ней сетевых узлов. Используйте этот параметр, если план восстановления использует несколько сетей виртуальных машин. Для сетей на базе виртуализации сети Windows этот параметр можно использовать для автоматического создания сетей виртуальных машин с теми же настройками (подсети и пулы IP-адресов), что и сеть виртуальной машины реплики. Эти сети виртуальных машин автоматически удаляются после завершения тестовой отработки отказа.</LI>
</UL>
</OL>


<P>После завершения тестовой отработки отказа выполните следующие действия:</P>
<OL>
<LI>Проверьте, успешно ли выполнен запуск виртуальных машин.</LI>
<LI>После проверки успешного запуска виртуальных машин выполните тестовую отработку отказа для очистки изолированной среды. Если выбрано автоматическое создание сетей виртуальных машин, при очистке удаляются все тестовые виртуальные машины и тестовые сети.</LI>
<LI>Щелкните элемент **Примечания** для регистрации и сохранения любых наблюдений, связанных с тестовой отработкой отказа.</LI>
<LI>После запуска тестовой отработки отказа теста для плана восстановления процесс отработки отказа отображается на странице подробных сведений для плана восстановления. Можно просмотреть отдельные действия тестовой отработки отказа и их состояние, а также просматривать или создавать примечания, связанные с тестовой отработкой отказа. Кроме того, можно узнать подробности о заданиях отработки отказа, просмотрев задание на вкладке **Задания**. В сведениях указываются все задания, связанные с отработкой отказа, их состояние, время запуска и длительность. Задание из списка отработки отказа можно экспортировать в таблицу Excel.</LI>
</OL>


Обратите внимание на следующее:

- На странице "Подтвердить тестовую отработку отказа" отображаются сведения о сервере VMM, на котором будут созданы тестовые виртуальные машины.  
- За выполнением заданий тестовой отработки отказа можно следить на вкладке **Задания**. 






<h2><a id="jobsdashboard"></a>Шаг 8. Мониторинг</h2>
<h3><a id="jobsdashboard"></a>Использование вкладки "Задания" и панели мониторинга</h3>

<h4><a id="jobs"></a>Задания</h4>
На вкладке **Задания** отображаются основные задачи, выполняемые хранилищем диспетчера восстановления Hyper-V, включая настройку защиты для облака, включение и отключение защиты для виртуальной машины, выполнение отработки отказа (запланированной, незапланированной или тестовой), выполнение незапланированной отработки отказа, а также настройку обратной репликации. 

На вкладке **Задания** можно выполнить следующие задачи:



- **Выполнить запрос заданий** — можно выполнить запрос для получения заданий, соответствующих указанным критериям. Можно отфильтровать задания с использованием следующих параметров:

- 	Можно осуществлять поиск заданий, которые выполнялись на определенном сервере VMM.
- 	Можно осуществлять поиск заданий, которые выполнялись в облаке, на виртуальной машине, в сети, в плане восстановления в облаке или во всех этих расположениях.
- 	Можно определять задания, которые были вызваны за определенный временной период или диапазон дат.

Обратите внимание, что запрос может возвратить более 200 заданий, поэтому рекомендуется уточнить параметры запроса для возврата меньшего числа результатов.

- **Получить сведения о задании** — задание в списке **Задания** можно щелкнуть для получения дополнительных сведений. Эти сведения включают в себя имя задания, его состояние, время запуска и длительность выполнения. На вкладке сведений о задании можно экспортировать сведения о задании в электронную таблицу Excel или попытаться перезапустить задание, которые завершилось с ошибкой, было пропущено или отменено. 
- **Экспорт заданий** — результаты запроса заданий можно экспортировать в электронную таблицу Excel.
- **Перезагрузить** — можно перезапустить задания, которые завершились с ошибкой, чтобы попытаться выполнить их еще раз.
- **Сведения об ошибке** — для завершенных с ошибкой заданий можно щелкнуть элемент **Сведения об ошибке**, чтобы получить список ошибок. Для каждой ошибки отображаются возможные причины и рекомендации. Описание ошибки можно скопировать в буфер обмена для устранения неполадок.


>Вкладка **Задания** содержит следующие сведения:
<UL>
<LI>**Имя** — имя задания.</LI>
<LI>**Элемент** — имя облака, плана восстановления, сервера VMM или виртуальной машины, где было запущено задание.</LI>
<LI>**Состояние** — состояние задания: выполнено, отменено, завершено с ошибкой или другое.</LI>
<LI>**Тип** — тип задания.</LI>
<LI>**Запущено** — время запуска задания.</LI>
<LI>**Время задачи** — длительность выполнения задания.</LI>
</UL>

<h4><a id="protect"></a>Выполнение запроса заданий</h4>

Вы можете выполнять запросы для получения заданий, соответствующих указанным критериям, включая следующее:
<UL>
	<LI>Задания, которые выполнялись на определенном сервере VMM.</LI>
	<LI>Задания, которые выполнялись в определенном облаке, на виртуальной машине, в сети, в плане восстановления в облаке или во всех этих расположениях.</LI>
	<LI>Задания, которые были вызваны за определенный временной период или диапазон дат.</LI>
</UL>
	

Для запуска запроса заданий выполните следующие действия:
<OL>
<LI>Откройте вкладку **Задания**.</LI>
<LI>В поле **Сервер** укажите сервер VMM, на котором было запущено задание.</LI>
<LI>В поле **Тип** укажите, было ли задание выполнено в облаке, на виртуальной машине, в сети или в плане восстановления в облаке.</LI>
<LI>В поле **Состояние** укажите состояние задания.</LI>
<LI>В поле **Длительность** укажите интервал времени, в котором было вызвано задание.</LI>
</OL>


<h4><a id="dashboard"></a>Панель мониторинга</h4>
<P>На вкладке **Панель мониторинга** приведен краткий обзор использования диспетчера восстановления Hyper-V в вашей организации.  Оно предоставляет централизованный шлюз для просмотра виртуальных машин, защищенных хранилищами диспетчера восстановления Hyper-V. Панель мониторинга обладает следующими функциональными возможностями:</P>

На вкладке **Панель мониторинга** можно выполнить следующие задачи:



- **Загрузить поставщик** — загрузите поставщик диспетчера восстановления Hyper-V для установки на сервере VMM.
- **Управление сертификатом** — измените параметры для сертификата, связанного с хранилищем.
- **Удалить** — удалите хранилище.
- **Повторная синхронизация виртуальных машин** — если диспетчер восстановления Hyper-V обнаруживает, что какие-либо основные виртуальные машины и виртуальные машины восстановления не синхронизированы должным образом, можно просмотреть список соответствующих виртуальных машин и выбрать любую из них для повторения попытки синхронизации из панели мониторинга. 


Вкладка **Панель мониторинга** содержит следующие сведения:
<UL>
<LI**>Обзор использования** — показывает количество виртуальных машин с защитой, управляемых диспетчером восстановления Hyper-V.</LI>
<LI>**Сводка** — отображает важные сведения о конфигурации для служб восстановления и хранилища диспетчера восстановления Hyper-V. Указывает, работает ли хранилище, какой ему назначен сертификат, когда истекает его срока действия, а также предоставляет сведения о подписках для данной службы.</LI>
<LI>**Недавние задания** — показывает задания, которые были выполнены (успешно или с ошибками) в течение последних 24 часов, или задания, которые выполняются или ожидают обработки.</LI>
<LI>**Проблемы** — панель мониторинга отображает сведения о проблемах, связанных с подключениями сервера VMM, параметрами конфигурации облака или синхронизацией репликации виртуальных машин. Вы можете просмотреть подробные сведения о проблеме, задания, связанные с этой проблемой, а также попытаться выполнить повторную синхронизацию виртуальных машин.</LI>
</UL>
