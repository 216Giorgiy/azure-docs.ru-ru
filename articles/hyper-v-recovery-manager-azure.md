# Приступая к работе с Azure Site Recovery: защита локальной среды в Azure

<div class="dev-callout"> 

<p>Используйте Azure Site Recovery для защиты виртуальных машин, работающих на узлах Hyper-V, которые размещены в облаках System Center Virtual Machine Manager (VMM). Вы можете настроить:</p>

<ul>
<li><b>Защита локальной среды в Azure</b> — репликация виртуальных машин, размещенных на серверах узлов Hyper-V облаков VMM в Azure. Параметры защиты настраиваются и включаются в хранилищах Azure Site Recovery. Репликация данных виртуальных машин выполняется с локального сервера Hyper-V в хранилище Azure.</li>

<li><b>Защита локальной среды в локальной среде</b>— Репликация виртуальных машин, размещенных на серверах узлов Hyper-V облаков VMM, из одного локального сайта на другой. Параметры защиты настраиваются и включаются в хранилищах Azure Site Recovery. Репликация данных виртуальных машин выполняется с одного локального сервера Hyper-V на другой. Azure Site Recovery выполняет оркестрацию этого процесса.
Познакомьтесь с этим сценарием в разделе <a href="http://go.microsoft.com/fwlink/?LinkId=398765">Начиная работу с Azure Site Recovery: Защита локальной среды в локальной среде</a>.</li>

</ul>
 
<h2><a id="about"></a>Сведения о данном учебнике</h2>


<P>Этот учебник предназначен для оказания помощи при развертывании Azure Site Recovery для быстрого проведения эксперимента. В учебнике используется самый быстрый путь и параметры по умолчанию, когда это возможно. Рассматриваются следующие шаги:
<ul>
<li>Настройка хранилища Azure Site Recovery — получение сертификата, отправленного в хранилище и выполнение настройки на исходном сервере VMM, а также создание ключа хранилища. </li>
<li>Настройка исходного сервера VMM и серверов узла Hyper-V — установка поставщика Azure Site Recovery на исходном сервере VMM и установка агента Azure Recovery Services на серверах узла Hyper-V.</li>
<li>Настройка облаков VMM — настройка параметров защиты облаков на исходном сервере VMM.</li>
<li>Включение виртуальных машин — Включение защиты виртуальных машин</li>
<li>Выполнение отработки отказа — создание плана восстановления и выполнение тестовой отработки отказа.</li>
</ul>


Информацию о полной развертке см. в разделе:</P>

<h3><a id="obtaincert"></a>Получение самозаверяющего сертификата (CER)</h3>
<P>Создайте сертификат .cer x.509, соответствующий всем требованиям, которые предъявляются к сертификатам:</P>
<ol>
<LI>
На компьютере, где требуется запустить MakeCert, загрузите последнюю версию пакета <a href="http://go.microsoft.com/fwlink/?LinkId=378269">Windows SDK</a> Установка полного пакет SDK не потребуется.</LI>
<ol>
<LI>На странице &quot;Выбор расположения&quot; выберите пункт <b>Установить пакет средств разработки программного обеспечения Windows для Windows 8.1 на данном компьютере</b>.</LI>
<LI>На странице &quot;Выберите компоненты, которые вы хотите установить&quot; отмените выбор всех параметров, кроме параметра <b>Пакет средств разработки программного обеспечения Windows</b>.</LI>
<LI>После установки убедитесь, что программа makecert.exe появилась в папке C:\ProgramFiles (x 86) \Windows Kits\<i>версия_Windows</i>\bin\x64.</LI>
<LI>Откройте командную строку (cmd.exe) с правами администратора и перейдите к папке с makecert.exe.</LI> 
<LI> Выполните следующую команду, чтобы создать самозаверяющий сертификат. Замените значение CertificateName на имя, которое вы хотите использовать для сертификата, и укажите фактический срок действия сертификата после параметра -e:</LI>
<code>
makecert.exe -r -pe -n CN=CertificateName -ss my -sr localmachine -eku 1.3.6.1.5.5.7.3.2 -len 2048 -e 01/01/2016 CertificateName.cer</code>
</ol>
<P>Успешное состояние указывает на то, что сертификат был создан. Он хранится в той же папке, что и makecert.exe. При необходимости его можно переместить в более доступное место для экспорта.</P>

<h3><a id="exportcert"></a>Экспорт сертификата в формате PFX-файла</h3>
<P>Выполните эту процедуру для экспорта сертификата CER в формате PFX-файла.</P>
<ol>
<li>Успешное состояние указывает на то, что сертификат был создан. Он хранится в той же папке, что и makecert.exe. При необходимости его можно переместить в более доступное место для экспорта.</li>
<li>В области сведений щелкните сертификат, которым вы хотите управлять.</li>
<li>В меню <b>Действие</b> выберите элемент <b>Все задачи</b> и щелкните <b>Экспорт</b>. Запускается мастер экспорта сертификатов. Нажмите кнопку <b>Далее</b>.</li>
<li>На странице <b>Экспортирование закрытого ключа</b> щелкните <b>Да</b>, чтобы экспортировать закрытый ключ. Нажмите кнопку <b>Далее</b>. Обратите внимание, что это требуется только в том случае, когда нужно экспортировать закрытый ключ на другие серверы после установки.</li>
<li>На странице &quot;Формат экспортируемого файла&quot; выберите <b>Файл обмена личной информацией – PKCS #12 (.PFX)</b>. Нажмите кнопку <b>Далее</b>.</li>
<li>На странице <b>Пароль</b> введите и подтвердите пароль, используемый для шифрования закрытого ключа. Нажмите кнопку <b>Далее</b>.</li>
<li>Следуйте указаниям на страницах мастера, чтобы экспортировать сертификат в формате PFX-файла.</li>
</ol>

<h3><a id="importcert"></a>Импорт сертификата PFX на серверы VMM</h3>
<p>После экспорта скопируйте PFX-файл на сервер VMM, а затем импортируйте его. Обратите внимание, что при запуске средства MakeCert.exe на сервере VMM импортировать сертификат на этот сервер не нужно.</p>
 
<ol>
<li>Скопируйте файлы сертификатов (PFX) закрытого ключа в папку на локальном сервере.</li>
<li>В оснастке MMC для сертификатов выберите элемент <b>Учетная запись</b> компьютера и нажмите кнопку <b>Далее</b>.</li>
<li>Выберите <b>Локальный компьютер</b> и нажмите кнопку <b>Готово</b>.</li>
<li>В консоли MMC разверните узел <b>Сертификаты</b>, щелкните правой кнопкой мыши элемент <b>Личные</b>, выберите пункт <b>Все задачи</b> и щелкните элемент <b>Импорт</b> для запуска мастера импорта сертификатов.</li>
<li>На странице &quot;Импортируемый файл&quot; нажмите кнопку <b>Обзор</b> и перейдите в папку с PFX-файлом того сертификата, который требуется импортировать. Выберите нужный файл и нажмите кнопку <b>Открыть</b>.</li>
<li>На странице «Пароль» в текстовом поле <b>Пароль</b> введите пароль для PFX-файла закрытого ключа и щелкните <b>Далее</b>.</li>
<li>На странице &quot;Хранилище сертификатов&quot; выберите пункт <b>Поместить все сертификаты в следующее хранилище</b>, нажмите кнопку <b>Обзор</b>, выберите <b>Личное</b> хранилище, нажмите кнопку <b>OK</b> и кнопку <b>Далее</b>. Завершите выполнение мастера.</li>
</ol>

После выполнения этих действий можно будет выбрать сертификат CER для отправки в хранилище во время настройки, а также выбрать PFX-файл сертификата при регистрации сервера VMM во время установки поставщика.
</div>

## Шаг 2. Создание хранилища

</p>
1.  Выполните вход на [Портал управления][Портал управления].

2.  Разверните **Data Services** (Службы данных), разверните **Recovery Services** (Службы восстановления), и щелкните **Site Recovery Vault** (Хранилище Site Recovery).

3.  Щелкните **Создать**, затем **Быстрое создание**.

4.  В поле **Имя** введите понятное имя для идентификации хранилища.

5.  В поле **Область** выберите географический регион для хранилища архивации. Доступные географические регионы: Восточная Азия, Западная Европа, Запад США, Восток США, Северная Европа, Юго-Восточная Азия.
6.  Щелкните элемент **Создать хранилище**.

    ![Новое хранилище][Новое хранилище]

Проверьте строку состояния, чтобы убедиться в успешном создании хранилища. Хранилище будет включено в список как **активное** на главной странице служб восстановления.

<a name="upload"></a>

## Шаг 3. Настройка хранилища

</p>
1.  На странице **Службы восстановления** щелкните хранилище, чтобы открыть страницу быстрого запуска. Страницу быстрого запуска можно также открыть в любое время с помощью значка.

    ![Значок быстрого запуска][Значок быстрого запуска]

2.  В раскрывающемся списке **Setup Recovery** (Настройка восстановления) выберите элемент **Between an on-premises site and Microsoft Azure** (Между локальным сайтом и Microsoft Azure).
3.  Чтобы отправить сертификат (.cer) в хранилище, щелкните **Управление сертификатом**.

    ![Быстрый запуск][Быстрый запуск]

4.  В диалоговом окне **Управление сертификатом** щелкните **Поиск файла** и выберите файл .cer.

    ![Управление сертификатом][Управление сертификатом]

5.  Чтобы создать ключ для хранилища, щелкните **Get the vault key** (Получить ключ хранилища). Ключ создается автоматически. При повторном создании ключа он переписывает предыдущий ключ. Обратите внимание, что этот ключ понадобится позднее при установке поставщика Azure Site Recovery на сервер VMM.

    ![Управление сертификатом][1]

## Шаг 4. Установка поставщика Azure Site Recovery

1.  На странице **Быстрый запуск** щелкните элемент **Загрузить поставщик**, чтобы получить последнюю версию файла установки поставщика.

    ![Загрузите файл поставщика][Загрузите файл поставщика]

2.  Запустите этот файл на исходном сервере VM.

    ![Загрузить агент][Загрузить агент]

3.  После установки поставщика продолжите настройку, чтобы зарегистрировать сервер в хранилище.
    ![Установка завершена][Установка завершена]
4.  На странице "Подключение к Интернету" укажите, как запущенный на сервере VMM поставщик подключается к Интернету. Щелкните **Далее**, чтобы использовать настроенные на сервере параметры подключения к Интернету по умолчанию.
    ![Параметры Интернета][Параметры Интернета]
5.  На странице регистрации хранилища выполните следующие действия:

    -   Выберите закрытый ключ (.pfx), который был импортирован в сервер VMM.
    -   Выберите хранилище, в котором необходимо зарегистрировать сервер.
    -   Укажите ключ хранилища. Это тот ключ хранилища, который был создан раньше. Вырежьте и вставьте значение ключа со страницы быстрого запуска.

    </p>
    ![Регистрация сертификатов][Регистрация сертификатов]

6.  На странице «Шифрование данных» указывается, необходимо ли использовать параметр шифрования данных во время репликации в определенное облако. При выборе этого параметра автоматически создается сертификат SSL. Во время выполнения отработки отказа необходимо выбирать этот сертификат. После включения данного параметра можно включать или выключать шифрование данных для облака на портале Azure Site Recovery. При выполнении данного учебника оставьте значение по умолчанию и нажмите **Далее**.

    ![Хранилище сертификатов][Хранилище сертификатов]

7.  На странице сервера VMM выполните следующие действия:

    -   Укажите простое имя для сервера VMM. Это имя используется для идентификации сервера в консоли Azure Site Recovery.
    -   Выберите **Синхронизировать облачные метаданные с хранилищем**, чтобы синхронизировать информацию об облаках VMM с хранилищем Azure Site Recovery. На каждом сервере это действие требуется выполнять только один раз. Если вы не хотите синхронизировать все облака, можно опубликовать каждое облако отдельно, чтобы синхронизировать его перед настройкой параметров защиты облака.

8.  Для завершения процесса нажмите кнопку **Зарегистрировать**.

    ![PublishCloud][PublishCloud]

После успешной регистрации сервера его понятное имя будет отображаться на вкладке **Ресурсы** страницы "Серверы" в хранилище.

## <span id="storage"></span></a>Шаг 5: Установка агента служб восстановления Azure

Установите агента служб восстановления Azure на каждый сервер узла Hyper-V, размещенный в облаках VMM, которые необходимо защитить.

1.  На странице быстрого запуска щелкните **Загрузить агента служб Azure Site Recovery и установить на узлах**, чтобы получить последнюю версию файла установки агента.

    ![Установка агента служб восстановления Azure][Установка агента служб восстановления Azure]

2.  Запустите файл установки на каждом сервере узла Hyper-V, размещенный в облаках VMM, которые необходимо защитить.
3.  На странице проверки предварительных требований нажмите кнопку **Далее**. Все отсутствующие предварительные требования будут установлены автоматически.

![Предварительные требования для агента служб восстановления][Предварительные требования для агента служб восстановления]

1.  На странице параметров установки задайте место для установки агента и выберите местоположение кэша, в котором будут установлены метаданные резервной копии. Нажмите **Install** (Установить).

На странице "Подключение к Интернету" укажите, как запущенный на сервере VMM поставщик подключается к Интернету. Щелкните **Далее**, чтобы использовать настроенные на сервере параметры подключения к Интернету по умолчанию.![Местоположение агента служб восстановления][Местоположение агента служб восстановления]

## <span id="clouds"></span></a>Шаг 6: Настройка параметров защиты облака

После регистрации серверов VMM можно настроить параметры защиты облачных систем. Параметр **Синхронизировать облачные данные с хранилищем** был включен при установке поставщика, и все облака на сервере VMM появятся на вкладке **Защищенные элементы** в хранилище.

![Опубликованное облако][Опубликованное облако]

Настройте параметры защиты следующим образом:

1.  На странице быстрого запуска щелкните **Set up protection for VMM clouds** (Установить защиту для облаков VMM).

![Включение зашиты облака][Включение зашиты облака]

1.  На вкладке **Защищенные элементы** выберите облако, которое нужно настроить, и перейдите на вкладку **Конфигурация**.
2.  В элементе **Цель** выберите **Microsoft Azure**.
3.  В элементе **Учетная запись хранения** выберите хранилище Azure, которое будет использоваться для хранения виртуальных машин Azure.
4.  Задайте параметру **Encrypt stored data** (Шифрование сохраненных данных) значение **Off** (выкл.). Этот параметр определяет, что данные должны шифроваться при репликации между локальным сайтом и Azure.
5.  В поле **Периодичность копирования** оставьте значение по умолчанию. Это значение указывает, как часто следует синхронизировать данные между исходным и конечным расположениями.
6.  В поле **Сохранения точек восстановления для** оставьте значение по умолчанию. При использовании нулевого значения по умолчанию на сервере узла реплики хранится только последняя точка восстановления для основной виртуальной машины.
7.  В поле **Периодичность создания соответствующих приложению моментальных снимков** оставьте значение по умолчанию. Это значение указывает, как часто создаются моментальные снимки. Снимки используют службу теневого копирования томов (VSS), чтобы обеспечить согласованное состояние приложение при создании моментального снимка. Если это значение задается, оно должно быть меньше количества дополнительных точек восстановления, которое было настроено.

    ![Конфигурация облака][Конфигурация облака]

8.  В поле **Время запуска репликации** укажите, когда должна запускаться начальная репликация данных на Azure. Используется часовой пояс сервера узла Hyper-V. Рекомендуется запланировать начальную репликацию на часы наименьшей нагрузки.

    ![Параметры репликации облака][Параметры репликации облака]

После сохранения параметров создается задание, которое можно отслеживать на вкладке **Задания**. Все серверы узлов Hyper-V в исходном облаке VMM будут настроены для репликации. После сохранения параметры облака можно изменить на вкладке **Настройка**. Чтобы изменить целевое расположение или целевое хранилище необходимо удалить конфигурацию облака, а затем повторно его настроить. Обратите внимание, что при изменении учетной записи хранения, изменение применяется только к виртуальным машинам, которые включены для защиты после изменения учетной записи хранения. Существующие виртуальные машины в новую учетную запись хранения не переносятся.

## <span id="networkmapping"></span></a>Шаг 7: Настройка сетевого сопоставления

Дополнительно можно настроить сетевое сопоставление для сопоставления исходных сетей виртуальных машин с целевыми виртуальными сетями Azure. Если сетевое сопоставление не создается, то в Azure могут подключаться друг к другу только виртуальные машины, которые отрабатывают отказ в рамках одного плана восстановления. Если сетевое сопоставление создается, тогда все виртуальные машины, которые отрабатывают отказ в одной и той же сети, могут подключаться друг к друг вне зависимости от того, к какому плану восстановления они относятся. Кроме того, при наличии сетевого шлюза в целевой сети Azure виртуальные машины могут подключиться к локальным виртуальным машинам. При желании настроить сетевое сопоставление в рамках данного учебника см. раздел [Настройка сетевого сопоставления][Настройка сетевого сопоставления] руководства по развертыванию.

## <span id="virtualmachines"></span></a>Шаг 8: Включение защиты для виртуальных машин

После успешной настройки серверов, облаков и сетей можно включить защиту для виртуальных машин в облаке.

Перед включением защиты виртуальных машин проверьте и обновите параметры, если это требуется. Например, гостевой операционной системой на виртуальных машинах должна быть Windows Server 2008 (или выше) или Linux. Виртуальные машины должны быть только поколения 1. Полный список требований Azure Site Recovery см. в разделе [Предварительные требования и поддержка][Предварительные требования и поддержка] руководства по планированию.

В консоли VMM проверьте и обновите параметры. Изменение параметров операционной системы виртуальной машины выполняется на странице «Общие» свойств виртуальной машины. Параметры диска операционной системы обновляются на странице «Настройка оборудования».

![Изменение свойств виртуальной машины][Изменение свойств виртуальной машины]

![Изменение свойств виртуальной машины][2]

1.  Чтобы включить защиту, на вкладке **Виртуальные машины** облака, в котором расположена виртуальная машина, щелкните **Включить защиту**, а затем выберите **Добавить виртуальные машины**.
2.  Из списка виртуальных машин в облаке выберите ту, которую необходимо защитить.

![Включение защиты виртуальной машины][Включение защиты виртуальной машины]

После включения защиты создаются два задания. Запускается задание включения защиты. Затем после завершения начальной репликации запускаются два дополнительных задания финализации защиты. Виртуальная машина считается готовой к отработке отказа только после успешного завершения этих трех заданий. Ход выполнения заданий можно отслеживать на вкладке **Задания**.

![Задание защиты виртуальной машины][Задание защиты виртуальной машины]

## <span id="recoveryplans"></span></a>Шаг 9: Настройка и выполнение плана восстановления

План восстановления собирает виртуальные машины в группы для того, чтобы они могли отрабатывать отказ, как единое целое. Чтобы создать план восстановления, выполните следующие действия:

1.  На вкладке **Планы восстановления** щелкните **Создать**.
2.  На странице "Указание имени и конечного расположения страницы восстановления" выберите исходный сервер VMM и Azure как цель.

    ![Создание плана восстановления][Создание плана восстановления]

3.  На странице выбора виртуальных машин выберите виртуальные машины, которые хотите добавить в план восстановления. Отражаются только виртуальные машины с включенной защитой. Они добавляются в группу плана восстановления по умолчанию (группа 1).
4.  Установите флажок, чтобы создать план восстановления.

    ![Виртуальные машины плана восстановления][Виртуальные машины плана восстановления]

### <span id="run"></span></a>Тестирование отработки отказа

Планы восстановления можно запустить в рамках упреждающего теста или запланированной отработки отказа, а также во время незапланированной отработки отказа. В этом пошаговом руководстве описывается выполнение теста отработки отказа из VMM в Azure, чтобы проверить правильность работы стратегии отработки отказа. Тестовая обработка отказа имитирует механизм отработки отказа и восстановления в изолированной сети. Обратите внимание на следующее:

-   При необходимости подключиться к виртуальной машине в Azure с использованием удаленного рабочего стола после отработки отказа, включите подключение удаленного рабочего стола на виртуальной машине до запуска тестирования отработки отказа.
-   После отработки отказа будет использоваться общедоступный IP-адрес, чтобы подключиться к виртуальной машине в Azure с использованием удаленного рабочего стола. Если необходимо это сделать, убедитесь в отсутствии политик домена, которые не допустили бы подключение к виртуальной машине с использованием общедоступного адреса.

</p>

<p>
<span id="runtest"></span></a>Запуск отработки отказа

</h5>
Запустите тестовую отработку отказа для плана восстановления следующим образом:

</p>
1.  Перед запуском плана восстановления можно проверить параметры виртуальных машин в плане. Чтобы это сделать, на странице свойств облака щелкните виртуальную машину. На странице «Свойства источника и цели для отработки отказов» проверьте параметры. В частности убедитесь, что предложенный размер целевой виртуальной машины в Azure выбран точно, а параметры сети установлены правильно. Полный список предварительных требований к виртуальной машине см. в разделе [Предварительные требования и поддержка][3].

![Свойства виртуальной машины][Свойства виртуальной машины]

1.  На вкладке **Планы восстановления** выберите требуемый план восстановления.
2.  Чтобы инициировать отработку отказа, нажмите кнопку **Тестовая отработка отказа**.
3.  На странице «Подтверждение тестовой отработки отказа» выберите сеть Azure, к которой будут подключаться виртуальные машины после отработки отказа. Дополнительно можно указать **Без сети**. При установке этого параметра виртуальные машины не будут подключаться к сети после отработки отказа.

![Тестирование отработки отказа][Тестирование отработки отказа]

За выполнением заданий тестовой отработки отказа можно следить на вкладке **Задания**. После завершения тестовой отработки отказа выполните следующие действия:

1.  Проверьте, успешно ли выполнен запуск виртуальных машин в Azure.
2.  Щелкните элемент **Примечания** для регистрации и сохранения любых наблюдений, связанных с тестовой отработкой отказа.
3.  Кроме сведений на вкладке **Задания**, при выполнении тестовой отработки отказа для плана восстановления процесс отображается на странице сведений плана восстановления. Можно просматривать этапы и состояние отработки отказа, просматривать и создавать примечания для тестовой отработки отказа.

<h5><span id="runtest"></span></a>Отслеживание операций</h5>
<br>



Вкладку <b>Задания</b> и <b>Панель мониторинга</b> можно использовать для просмотра и отслеживания основных заданий, которые выполняются хранилищем Azure Site Recovery, в том числе настройки защиты облака, включения и выключения зашиты виртуальной машины, запуска отработки отказа (запланированного, незапланированного, тестового), а также исполнения незапланированной отработки отказа.

</p>
На вкладке **Задания** можно просматривать задания, детализировать сведения о задании и ошибки, выполнять запросы задания для извлечения заданий, отвечающих определенным критериям, экспортировать задания в Excel, а также перезапускать задания, завершившиеся ошибкой.

На **Панели мониторинга** можно загружать последние версии файлов установки поставщика и агента, получать информацию конфигурации для хранилища, просматривать количество виртуальных машин, защитой которых управляет хранилище, просматривать недавние задания, управлять сертификатом хранилища и повторно синхронизировать виртуальные машины.

Для получения дополнительной информации о взаимодействии с заданиями и панелью мониторинга см. раздел [Руководство по операциям и мониторингу][Руководство по операциям и мониторингу].
