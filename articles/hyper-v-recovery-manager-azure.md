# Приступая к работе с Azure Site Recovery: защита локальной среды в Azure

<div class="dev-callout">

Используйте Azure Site Recovery, чтобы организовать защиту виртуальных машин на локальных серверах узла Hyper-V, расположенных в облаках VMM. Вы можете настроить:

-   **Защита из локальной среды в среду Azure** — репликация из локальной виртуальной машины в Azure. Параметры защиты настраиваются и включаются в хранилищах Azure Site Recovery. Данные виртуальной машины реплицируются из локального сервера Hyper-V в службу хранилища Azure.
-   **Защита из локальной среды в локальную среду** — репликация локальных виртуальных машин на другой локальный сайт. Параметры защиты настраиваются и включаются в хранилищах Azure Site Recovery. Данные виртуальной машины реплицируются из одного локального сервера Hyper-V на другой. Дополнительную информацию об этом сценарии см. в разделе [Приступая к работе с Azure Site Recovery: защита из локальной среды в локальную среду][Приступая к работе с Azure Site Recovery: защита из локальной среды в локальную среду].

## <span id="about"></span></a>О данном учебнике

Используйте этот учебник, чтобы быстро подготовить эксперимент для Azure Site Recovery при развертывании из локальной среды в Azure. По возможности используются самый быстрый путь и значения по умолчанию. Вы создадите хранилище Azure Site Recovery, установите поставщик Azure Site Recovery на исходном сервере VMM, установите агент служб восстановления Azure на локальных серверах узла Hyper-V, расположенных в облаках VMM, настроите параметры облачной защиты, включите защиту виртуальных машин и протестируете свое развертывание.

Если вам нужна информация о полном развертывании, см.:

-   [Планирование развертывания Azure Site Recovery][Планирование развертывания Azure Site Recovery]— описывает шаги по планированию, которые необходимо выполнить перед тем, как начинать полное развертывание.
-   [Развертывание Azure Site Recovery: Защита локальной среды в Azure][Развертывание Azure Site Recovery: Защита локальной среды в Azure] — Пошаговые указания для полного развертывания.

При возникновении проблем с данным учебником см. вики-статью [Azure Site Recovery: распространенные ошибки и решения][Azure Site Recovery: распространенные ошибки и решения] или задайте свои вопросы на [форуме службы восстановления Azure][форуме службы восстановления Azure].

### <span id="HVRMPrereq"></span></a>Предварительные требования

-   **Учетная запись Azure** — вам понадобится учетная запись Azure. Если у вас ее нет, см. раздел [Бесплатная пробная версия Azure][Бесплатная пробная версия Azure]. Информацию по ценам см. в разделе [Azure Site Recovery Manager Pricing Details][Azure Site Recovery Manager Pricing Details] (Цены на диспетчер Azure Site Recovery).
-   **Учетная запись хранения Azure**— вам понадобится учетная запись хранения Azure для хранения данных, реплицированных в Azure. На учетной записи необходимо включить георепликацию. Она должна находиться в том же регионе, что и служба Azure Site Recovery, и быть связана с той же подпиской. Более подробную информацию о настройке хранилища Azure см. в разделе [Введение в хранилище Microsoft Azure][Введение в хранилище Microsoft Azure].
-   **Сервер VMM**— Сервер VMM, работающий под управлением System Center 2012 R2.
-   **Облака VMM**— как минимум одно облако на сервере VMM. Облако должно содержать:
    -   Одна или несколько групп узлов VMM
    -   Один или несколько серверов или кластеров узлов Hyper-V в каждой группе узлов.
    -   Одна или несколько виртуальных машин, расположенных на исходном сервере Hyper-V в облаке. Виртуальные машины должны быть поколения 1.
-   **Виртуальная машина** — Необходимо использовать виртуальные машины, которые соответствуют требованиям, предъявляемым Azure. Подробности приводятся в разделе [Необходимые условия и поддержка][Необходимые условия и поддержка] руководства по планированию.

## Шаг 1. Создание хранилища

</p>
1.  Выполните вход на [Портал управления][Портал управления].

2.  Разверните элементы **Службы данных** и **Службы восстановления**, а затем щелкните **Site Recovery Vault** (Хранилище Site Recovery).

3.  Щелкните **Создать**, а затем **Быстрое создание**.

4.  В поле **Имя** введите понятное имя для идентификации хранилища.

5.  В поле **Область** выберите географический регион для хранилища архивации. Доступные географические регионы: Восточная Азия, Западная Европа, Запад США, Восток США, Северная Европа, Юго-Восточная Азия.
6.  Щелкните элемент **Создать хранилище**.

    ![Новое хранилище][Новое хранилище]

Проверьте строку состояния, чтобы убедиться, что хранилище успешно создано. На главной странице служб восстановления это хранилище будет указано с состоянием **Активно**.

<a name="download"></a>

## Шаг 2. Создание регистрационного ключа и установка поставщика Azure Site Recovery

</p>
1.  На странице **Службы восстановления** щелкните хранилище, чтобы открыть страницу быстрого запуска. Страницу быстрого запуска можно открыть и в любой другой момент, щелкнув этот значок.

    ![Значок быстрого запуска][Значок быстрого запуска]

2.  В раскрывающемся списке выберите элемент **Between an on-premises Hyper-V site and Microsoft Azure (Между локальным сайтом и Microsoft Azure)**.
3.  В разделе **Prepare VMM Servers** (Подготовка серверов VMM) выберите пункт **Generate registration key** (Создать регистрационный ключ). Ключ действителен в течение пяти дней после создания. Скопируйте файл на сервер VMM. Он вам потребуется при настройке поставщика.

    ![Регистрационный ключ][Регистрационный ключ]

4.  На странице **Быстрый запуск** в разделе **Prepare VMM servers** (Подготовка серверов VMM) выберите пункт **Download Microsoft Azure Site Recovery Provider for installation on VMM servers** (Скачать поставщик Microsoft Azure Site Recovery для установки на серверы VMM), чтобы получить последнюю версию установочного файла приложения поставщика.

5.  Запустите этот файл на исходном сервере VM.

6.  В разделе **Pre-requirements Check** (Проверка предварительных требований) выберите остановку службы VMM, чтобы начать установку приложения поставщика. Служба остановится и будет перезапущена автоматически после завершения установки.

    ![Предварительные требования][Предварительные требования]

7.  В **Центре обновления Майкрософт** вы можете настроить обновления. Если этот параметр включен, то обновления поставщика будут устанавливаться в соответствии с вашей политикой Центра обновления Майкрософт.

    ![Центр обновления Майкрософт][Центр обновления Майкрософт]

После установки приложения поставщика продолжите установку, чтобы зарегистрировать сервер в хранилище.

1.  На странице **Подключение к Интернету** укажите, как запущенный на сервере VMM поставщик подключается к Интернету. Выберите пункт **Use default system proxy settings** (Использовать настройки прокси-сервера по умолчанию), чтобы использовать настройки подключения к Интернету, по умолчанию заданные на сервере.

    ![Параметры Интернета][Параметры Интернета]

2.  В поле **Регистрационный ключ** укажите, что вы скачали его с Azure Site Recovery и скопировали на сервер VMM.
3.  В поле **Имя хранилища** проверьте имя хранилища, в котором будет зарегистрирован сервер.
4.  В поле **Имя сервера** укажите понятное имя, описывающее сервер VMM в хранилище.

    ![Регистрация сервера][Регистрация сервера]

5.  В поле синхронизации **Initial cloud metadata** (Первоначальные метаданные в облаке) выберите, нужно ли синхронизировать метаданные для всех облаков сервера VMM в хранилище. На каждом сервере это действие требуется выполнять только один раз. Если не требуется синхронизировать все облака, можно оставить этот флажок снятым и синхронизировать каждое облако индивидуально в свойствах облака в консоли VMM.

6.  В параметре **Data Encryption** (Шифрование данных) следует указать расположение сертификата SSL, который создается автоматически для выполнения этой операции. Этот сертификат используется при активации шифрования данных для облачных сред, защищенных Azure на портале Azure Site Recovery. Данный сертификат следует хранить в безопасном месте. Во время перехода на другой ресурс в Azure при сбое необходимо его указывать для расшифровки зашифрованных данных.
    Этот параметр является недействительным при репликации между локально размещенными сайтами.

    ![Регистрация сервера][1]

7.  Для завершения процесса нажмите кнопку **Зарегистрировать**. После регистрации метаданные с сервера VMM извлекаются службой Azure Site Recovery. Сервер отображается на вкладке **Ресурсы** на странице **Серверы** в хранилище.

Используйте этот учебник, чтобы быстро подготовить эксперимент для Azure Site Recovery при развертывании из локальной среды в Azure. По возможности используются самый быстрый путь и значения по умолчанию. Вы создадите хранилище Azure Site Recovery, установите поставщик Azure Site Recovery на исходном сервере VMM, установите агент служб восстановления Azure на локальных серверах узла Hyper-V, расположенных в облаках VMM, настроите параметры облачной защиты, включите защиту виртуальных машин и протестируете свое развертывание.

## <span id="agent"></span></a>Шаг 4: Установка агента служб восстановления Azure на узлах Hyper-V

Установите агента служб восстановления Azure на каждый сервер узла Hyper-V, размещенный в облаках VMM, которые необходимо защитить.

1.  На странице быстрого запуска щелкните **Загрузить агент служб Azure Site Recovery и установить на узлах**, чтобы получить последнюю версию установочного файла агента.

    ![Установка агента служб восстановления Azure][Установка агента служб восстановления Azure]

2.  Запустите файл установки на каждом сервере узла Hyper-V, размещенный в облаках VMM, которые необходимо защитить.
3.  На странице **Проверка предварительных требований** нажмите кнопку **Далее**. Все отсутствующие предварительные требования будут установлены автоматически.

    ![Предварительные требования для агента служб восстановления][Предварительные требования для агента служб восстановления]

4.  На странице **Параметры установки** задайте место для установки агента и выберите местоположение кэша, в котором будут установлены метаданные резервной копии. Нажмите **Install** (Установить).

## <span id="clouds"></span></a>Шаг 5: настройка параметров защиты облаков

После регистрации серверов VMM можно настроить параметры защиты облачных систем. Вы включили параметр **Синхронизировать облачные данные с хранилищем**, когда установили поставщик, так что все облака на сервере VMM появятся на вкладке **Защищенные элементы** в хранилище.

![Опубликованное облако][Опубликованное облако]

1.  На странице быстрого запуска щелкните **Set up protection for VMM clouds** (Настроить защиту для облаков VMM).

2.  На вкладке **Защищенные элементы** выберите облако, которое нужно настроить, и перейдите на вкладку **Конфигурация**.
3.  В элементе **Цель** выберите **Microsoft Azure**.
4.  В элементе **Учетная запись хранения** выберите хранилище Azure, которое будет использоваться для хранения виртуальных машин Azure.
5.  Задайте параметру **Encrypt stored data** (Шифрование сохраненных данных) значение **Off** (выкл.). Этот параметр определяет, что данные должны шифроваться при репликации между локальным сайтом и Azure.
6.  В поле **Периодичность копирования** оставьте значение по умолчанию. Это значение указывает, как часто следует синхронизировать данные между исходным и конечным расположениями.
7.  В поле **Сохранения точек восстановления для** оставьте значение по умолчанию. При использовании нулевого значения по умолчанию на сервере узла реплики хранится только последняя точка восстановления для основной виртуальной машины.
8.  В поле **Периодичность создания соответствующих приложению снимков** оставьте значение по умолчанию. Это значение указывает, как часто создаются моментальные снимки. Снимки используют службу теневого копирования томов (VSS), чтобы обеспечить согласованное состояние приложение при создании моментального снимка. Если это значение задается, оно должно быть меньше количества дополнительных точек восстановления, которое было настроено.
9.  В поле **Время запуска репликации** укажите, когда должна запускаться начальная репликация данных на Azure. Используется часовой пояс сервера узла Hyper-V. Рекомендуется запланировать начальную репликацию на часы наименьшей нагрузки.

    ![Параметры репликации облака][Параметры репликации облака]

После сохранения параметров создается задание, которое можно отслеживать на вкладке **Задания**. Все сервера узлов Hyper-V в исходном облаке VMM могут быть настроены для репликации. После сохранения параметры облака можно изменить на вкладке **Настройка**. Чтобы изменить целевое расположение или целевое хранилище необходимо удалить конфигурацию облака, а затем повторно его настроить. Обратите внимание, что при изменении учетной записи хранения, изменение применяется только к виртуальным машинам, которые включены для защиты после изменения учетной записи хранения. Существующие виртуальные машины в новую учетную запись хранения не переносятся.

## <span id="virtualmachines"></span></a>Шаг 7: Включение защиты для виртуальных машин

После успешной настройки серверов, облаков и сетей можно включить защиту для виртуальных машин в облаке. Обратите внимание на следующее:

-   Виртуальные машины должны соответствовать требованиям, предъявляемым Azure. Проверьте выполнение требований, описанных в разделе [Необходимые условия и поддержка][Необходимые условия и поддержка], руководства по планированию.
-   Для обеспечения защиты должны быть заданы соответствующие свойства операционной системы и диска с операционной системой. Данные свойства можно установить при создании виртуальной машины в VMM с помощью шаблона виртуальной машины. Помимо этого, для уже существующих виртуальных машин эти свойства можно настроить на вкладках \*\*General\*\* (Общие свойства) и \*\*Hardware Configuration\*\* (Настройка оборудования) в свойствах виртуальной машины. Если установка этих свойств в VMM не была выполнена, сделать это можно на портале Azure Site Recovery.

![Создание виртуальной машины][Создание виртуальной машины]

![Изменение свойств виртуальной машины][Изменение свойств виртуальной машины]

1.  Чтобы включить защиту, на вкладке **Виртуальные машины** облака, в котором расположена виртуальная машина, щелкните **Включить защиту**, а затем выберите **Добавить виртуальные машины**
2.  Из списка виртуальных машин в облаке выберите те, которые хотите защитить.

    ![Включение защиты виртуальной машины][Включение защиты виртуальной машины]

3.  Проверьте свойства виртуальной машины и внесите в них необходимые изменения.

    ![Проверка виртуальных машин][Проверка виртуальных машин]

Ход выполнения действия «Включение защиты» можно наблюдать на вкладке «Задания», в том числе и первоначальную репликацию. После запуска задачи финализации защиты виртуальная машина готова к отработке отказа. После включения защиты и репликации виртуальных машин вы сможете просматривать их в Azure.

![Задание защиты виртуальной машины][Задание защиты виртуальной машины]

## <span id="test"></span></a>Шаг 8: тестирование развертывания

Чтобы протестировать развертывание, можно запустить тестовую отработку отказа для отдельной виртуальной машины или создать план восстановления, состоящий из нескольких виртуальных машин, и запустить тестовую отработку отказа плана. Тестовая обработка отказа имитирует механизм отработки отказа и восстановления в изолированной сети. Обратите внимание на следующее:

-   При необходимости подключиться к виртуальной машине в Azure с использованием удаленного рабочего стола после отработки отказа, включите подключение удаленного рабочего стола на виртуальной машине до запуска тестирования отработки отказа.
-   После отработки отказа будет использоваться общедоступный IP-адрес, чтобы подключиться к виртуальной машине в Azure с использованием удаленного рабочего стола. Если необходимо это сделать, убедитесь в отсутствии политик домена, которые не допустили бы подключение к виртуальной машине с использованием общедоступного адреса.

-   Инструкции по созданию плана восстановления см. в разделе [Создание и настройка планов восстановления: из локальной среды в среду Azure][Создание и настройка планов восстановления: из локальной среды в среду Azure].
-   Инструкции по запуску тестовой отработки отказа см. в разделе [Тестирование развертывания из локальной среды в Azure][Тестирование развертывания из локальной среды в Azure].

### <span id="runtest"></span></a>Мониторинг активности

Можно использовать вкладку **Задания** и **Панель мониторинга**, чтобы просматривать и отслеживать выполнение основных заданий в хранилище Azure Site Recovery, в том числе настройку защиты для облака, включение и отключение защиты для виртуальной машины, выполнение отработки отказа (запланированной, незапланированной или тестовой) и выполнение незапланированной отработки отказа.

На вкладке **Задания** можно просматривать задания, их подробности и ошибки, запускать запросы извлечения заданий, отвечающих конкретным критериям, экспортировать задания в Excel и перезапускать неудавшиеся задания.

Из **Панели мониторинга** можно скачивать последние версии установочных файлов поставщика и агента, получать информацию о конфигурации хранилища, просматривать количество виртуальных машин, защитой которых управляет хранилище, просматривать последние задания, управлять сертификатом хранилища и проводить повторную синхронизацию виртуальных машин.

Дополнительную информацию о взаимодействии с заданиями и панелью мониторинга см. в [руководстве по эксплуатации и мониторингу][руководстве по эксплуатации и мониторингу].
