# Перемещение каналов ExpressRoute из классической модели развертывания в модель Resource Manager

В этой статье описывается перемещение канала ExpressRoute из классической модели развертывания в модель Resource Manager.

[AZURE.INCLUDE [vpn-gateway-sm-rm](../../includes/vpn-gateway-sm-rm-include.md)]

Один канал ExpressRoute может использоваться для подключения к виртуальным сетям, развернутым как в классической модели, так и в модели Resource Manager. Канал ExpressRoute (независимо от метода создания) теперь можно связать с виртуальными сетями в обеих моделях развертывания.

![](./media/expressroute-move/expressroute-move-1.png)

## Каналы ExpressRoute, созданные с помощью классической модели развертывания

Чтобы обеспечить возможность подключения и к классической модели развертывания, и к модели Resource Manager, каналы ExpressRoute, созданные с помощью классической модели, сначала необходимо переместить в модель развертывания Resource Manager. Перемещение не сопровождается потерей или прерыванием подключения. Все связи между виртуальными сетями канала в классической модели развертывания (в пределах одной подписки и между разными подписками) будут сохранены. Когда перемещение завершится, каналы ExpressRoute будут выглядеть и функционировать так же, как и созданные с помощью модели развертывания Resource Manager. Теперь вы сможете подключаться к виртуальным сетям в модели развертывания Resource Manager.

После перемещения канала ExpressRoute в модель развертывания Resource Manager можно управлять жизненным циклом канала только с помощью этой модели. Это означает, что такие операции, как добавление, обновление или удаление пирингов, обновление свойств канала, например его пропускной способности, SKU и типа выставления счетов, а также удаление каналов, можно выполнять только в модели развертывания Resource Manager. Сведения об управлении доступом к обеим моделям развертывания см. в разделе ниже, посвященном каналам, созданным в модели развертывания Resource Manager.

Перемещение можно выполнять независимо от поставщика услуг подключения.

## Каналы ExpressRoute, созданные с помощью модели развертывания Resource Manager

Можно сделать каналы ExpressRoute, созданные с помощью модели развертывания Resource Manager, доступными для обеих моделей развертывания. Обеспечить доступ к любому каналу ExpressRoute в подписке можно из обеих моделей развертывания.

- У каналов ExpressRoute, созданных с помощью модели развертывания Resource Manager, по умолчанию не будет доступа к классической модели развертывания.
- Каналы ExpressRoute, перемещенные из классической модели развертывания в модель Resource Manager, по умолчанию будут доступны для обеих моделей развертывания.
- Канал ExpressRoute всегда будет иметь доступ к модели развертывания Resource Manager независимо от того, с помощью какой модели он создан, классической или Resource Manager. Это означает, что можно устанавливать подключения к виртуальным сетям, созданным с помощью модели развертывания Resource Manager, следуя инструкциям в разделе [Связывание виртуальных сетей](expressroute-howto-linkvnet-arm.md). 
- Доступ к классической модели развертывания контролируется параметром allowClassicOperations в канале ExpressRoute. 

>[AZURE.IMPORTANT] Применяются все квоты, описанные на странице [Ограничения службы](../azure-subscription-service-limits.md). Например, в стандартном канале может быть установлено не более 10 связей или подключений между виртуальными сетями в классической модели развертывания и модели Resource Manager.


### Управление доступом к классической модели развертывания

Можно включить один канал ExpressRoute для связи с виртуальными сетями в обеих моделях развертывания, задав для канала параметр allowClassicOperations.

Задав для параметра allowClassicOperations значение TRUE, можно связать виртуальные сети обеих моделей развертывания с каналом ExpressRoute. Чтобы установить связь с виртуальными сетями в классической модели развертывания, следуйте инструкциям в разделе [Установка связи с виртуальными сетями в классической модели развертывания](expressroute-howto-linkvnet-classic.md). Чтобы установить связь с виртуальными сетями в модели развертывания Resource Manager, следуйте инструкциям в разделе [Установка связи с виртуальными сетями в модели развертывания Resource Manager](expressroute-howto-linkvnet-arm.md).

При установке для параметра allowClassicOperations значения FALSE доступ к каналу из классической модели развертывания блокируется. При этом все связи виртуальных сетей в классической модели развертывания сохраняются. В этом случае канал ExpressRoute будет недоступен для классической модели развертывания.

### Поддерживаемые операции в классической модели развертывания

При установке для параметра allowClassicOperations значения TRUE в канале ExpressRoute поддерживаются описанные ниже классические операции.

 - Получение сведений о канале ExpressRoute
 - Создание, обновление, получение и удаление связей с виртуальными сетями в классической модели
 - Создание, обновление, получение и удаление авторизации связей с виртуальными сетями для подключений между подписками

При установке для параметра allowClassicOperations значения TRUE не будут поддерживаться указанные ниже операции.

 - Создание, обновление, получение и удаление частных и общедоступных пирингов BGP Azure, а также пиринга BGP Майкрософт
 - Удаление канала ExpressRoute

## Связь между классической моделью и моделью Resource Manager

Канал ExpressRoute функционирует как мост между классической моделью развертывания и моделью Resource Manager. Трафик между виртуальными машинами, развернутыми в виртуальных сетях в классической модели и в модели Resource Manager, будет проходить через ExpressRoute, если виртуальные сети обеих моделей связаны с одним каналом ExpressRoute. Общая пропускная способность будет ограничена пропускной способностью шлюза виртуальной сети. В таких случаях трафик не переходит в сети поставщика подключения или ваши собственные сети. Поток трафика между виртуальными сетями будет полностью содержаться в сети Майкрософт.

## Доступ к ресурсам общедоступного пиринга Azure и пиринга Майкрософт

Доступ к ресурсам, который обычно осуществляется при помощи общедоступного пиринга Azure и пиринга Майкрософт, не будет сопровождаться прерываниями.

## Поддерживаемые функции

В этом разделе описаны функции, которые поддерживаются при использовании этой возможности

 - Один канал ExpressRoute можно использовать для подключения к виртуальным сетям, развернутым в классической модели и в модели Resource Manager.
 - Канал ExpressRoute можно переместить из классической модели развертывания в модель Resource Manager После перемещения канал ExpressRoute будет иметь те же вид, производительность и функции, что и любой другой канал, созданный с помощью модели развертывания Resource Manager.
 - Перемещение поддерживает только канал ExpressRoute. Эта операция не предусматривает перемещение связей, виртуальных сетей и VPN-шлюзов канала.
 - После перемещения канала ExpressRoute в модель развертывания Resource Manager можно управлять жизненным циклом канала только с помощью этой модели. Это означает, что такие операции, как добавление, обновление или удаление пирингов, обновление свойств канала, например его пропускной способности, SKU и типа выставления счетов, а также удаление каналов, можно выполнять только в модели развертывания Resource Manager.
 - Канал ExpressRoute функционирует как мост между классической моделью развертывания и моделью Resource Manager. Трафик между виртуальными машинами, развернутыми в виртуальных сетях в классической модели и в модели Resource Manager, будет проходить через ExpressRoute, если виртуальные сети обеих моделей связаны с одним каналом ExpressRoute. 
 - Подключение между подписками в классической модели развертывания и модели Resource Manager.

## Какие функции НЕ поддерживаются

В этом разделе описаны функции, которые не поддерживаются при использовании этой возможности

 - Перемещение связей, шлюзов и виртуальных сетей из классической модели развертывания в модель Resource Manager.
 - Управление жизненным циклом канала ExpressRoute в классической модели развертывания.
 - Поддержка RBAC для классической модели развертывания. Элементы управления RBAC для канала в классической модели развертывания не поддерживаются. Любой администратор или соадминистратор подписки сможет установить или отменить связь между виртуальными сетями и каналом.
