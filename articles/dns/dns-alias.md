# <a name="azure-dns-alias-records-overview"></a>Обзор записей псевдонимов Azure DNS

Записи псевдонимов Azure DNS являются квалификацией на набор записей DNS, который позволяет ссылаться на другие ресурсы Azure из вашей зоны DNS. Например, вы можете создать набор записей псевдонимов, который ссылается на публичный IP-адрес Azure вместо записи A. Так как псевдоним набора записей динамически указывает на экземпляр сервиса общедоступного IP-адреса Azure динамически, псевдоним набора записей просто обновляется во время разрешения DNS.

Набор записей псевдонимов поддерживается для следующих типов записей в DNS-зоне Azure: A, AAAA и CNAME. 

> [!NOTE]
> Записи псевдонимов для типов записей A или AAAA для диспетчера трафика поддерживаются только для типов внешней конечной точки. Необходимо указать IPv4 или IPv6-адрес (в идеале статические IP-адреса) для внешних конечных точек в диспетчере трафика.

## <a name="capabilities"></a>Возможности

- **Выберите ресурс общедоступного IP-адреса из набора DNS-записей A/AAAA**. Можно создать набор записей A/AAAA и сделать его набором записей псевдонимов, указывающим на ресурс общедоступного IP-адреса.
- **Укажите на профиль диспетчера трафика из набора DNS-записей A/AAAA/CNAME**. Помимо возможности указания записи CNAME профиля диспетчера трафика (например: contoso.trafficmanager.net) из набора записей DNS CNAME, вы можете теперь также указать профиль диспетчера трафика, который имеет внешние конечные точки, из набора записей A или AAAA в вашей DNS-зоне.
   > [!NOTE]
   > Записи псевдонимов для типов записей A или AAAA для диспетчера трафика поддерживаются только для типов внешней конечной точки. Необходимо указать IPv4 или IPv6-адрес (в идеале статические IP-адреса) для внешних конечных точек в диспетчере трафика.
- **Выберите другой набор DNS-записей в пределах той же зоны**. Записи псевдонимов могут ссылаться на другие наборы записей того же типа. Например, вы можете иметь набор записей DNS CNAME, как псевдоним для другого набора записей CNAME того же типа. Это полезно, если вы хотите иметь некоторые наборы записей в качестве псевдонимов, а другие не как псевдонимы по поведению.

## <a name="scenarios"></a>Сценарии
Существует несколько распространенных сценариев для записей псевдонимов:

### <a name="prevent-dangling-dns-records"></a>Предотвратить несвязанные DNS-записи
Из зоны Azure DNS записи псевдонимов можно использовать для плотного отслеживания жизненного цикла ресурсов Azure, например общедоступного IP-адреса и профиля диспетчера трафика. Одной из распространенных проблем с традиционными DNS-записями являются "несвязанные записи", особенно с типами записей A/AAAA или CNAME. 

В традиционной записи зоны DNS, если целевой IP-адрес или CNAME больше не существуют, запись зоны DNS не знает об этом и должна быть обновлена вручную. В некоторых организациях это ручное обновление может не произойти вовремя или может быть проблематично из-за разделения ролей и соответствующих уровней доступа.

Например, роль, имеющая права на удаление записи CNAME или IP-адрес приложения, может не иметь прав на обновление записи DNS, указывающей на эти объекты. В результате могут быть задержки между моментом удаления IP-адреса или CNAME и записи DNS, которая указывает на нее, что потенциально может привести к сбою для конечных клиентов.

Записи псевдонимов устраняют сложности, связанные с этим сценарием, и помогают предотвратить появление несвязанных ссылок. Если запись DNS является псевдонимом, которая указывает на общедоступный IP-адрес или профиль менеджера трафика, и если эти ресурсы удаляются, запись псевдонима удаляется одновременно с ними. Это гарантирует, что конечные пользователи никогда не страдают от сбоя.

### <a name="update-dns-zones-automatically-when-application-ips-change"></a>Автоматическое обновление зоны DNS при изменении IP-адреса приложения

Как и в предыдущем сценарии, при перемещении приложения или если базовая виртуальная машина перезапускается, запись псевдонима обновляется автоматически при изменении IP-адреса для основного общедоступного ресурса. Вы можете избежать проблем безопасности, которые возникают, если конечные пользователи имеют доступ к другому приложению со старым IP-адресом.

### <a name="host-load-balanced-applications-at-the-zone-apex"></a>Размещайте приложения с балансировкой нагрузки в апексе зоны

Протокол DNS запрещает назначение записей, отличных от A или AAAA, в апексе зоны (например: contoso.com). Это создает проблемы для владельцев приложений, которые размещают приложения с балансировкой нагрузки за диспетчером трафика, так как невозможно было ссылаться на профиль диспетчера трафика из записи апекса зоны. В результате владельцам приложений приходилось использовать обходной путь. Например, перенаправление на уровне приложения для перенаправления из апекса зоны в другой домен (из contoso.com на www.contoso.com). Это дает точку сбоя в плане функциональности переадресации.

С помощью записей псевдонимов эта проблема устранена. Владельцы приложений теперь могут направить свои записи в апексе зоны на профиль диспетчера трафика, который имеет внешние конечные точки. Благодаря этой возможности владельцы приложений могут указывать на тот же профиль диспетчера трафика, который используется для любого другого домена в их зоне DNS. Например, contoso.com и www.contoso.com могут оба указывать на один и тот же профиль диспетчера трафика до тех пор, пока профиль диспетчера трафика содержит только настроенные внешние конечные точки.
