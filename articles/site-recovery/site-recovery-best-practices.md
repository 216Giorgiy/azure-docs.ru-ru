# Рекомендации по развертыванию службы Site Recovery


<a id="overview" name="overview" href="#overview"></a>

## Об этой статье

Эта статья содержит рекомендации, с которыми вам следует ознакомиться перед развертыванием Azure Site Recovery. Если вам нужны общие сведения о Site Recovery и соответствующих сценариях развертывания см. в статье [Обзор Site Recovery](site-recovery-overview.md).

Если после прочтения статьи у вас возникнут какие-либо вопросы, вы можете задать их на [форуме, посвященном службам восстановления Azure](https://social.msdn.microsoft.com/forums/azure/home?forum=hypervrecovmgr).


## Подготовка Azure

- **Учетная запись Azure**. Вам потребуется учетная запись [Microsoft Azure](http://azure.microsoft.com/). Если у вас ее нет, начните с получения [бесплатной пробной версии](pricing/free-trial/).
- Ознакомьтесь с [ценами](pricing/details/site-recovery/) на службу Site Recovery. 
- **Служба хранилища Azure**. При развертывании Site Recovery с репликацией на Azure вам потребуется учетная запись хранения Azure. Вы можете настроить ее при развертывании или подготовить до начала работы. Для учетной записи необходимо включить георепликацию. Она должна находиться в том же регионе, что и хранилище Azure Site Recovery, и быть связана с той же подпиской. См. статью [Знакомство со службой хранилища Microsoft Azure](../storage/storage-introduction.md). 

## Виртуальные машины

Если вам требуется выполнить репликацию в службу хранилища Azure, обратите внимание на следующее:

- **Поддержка операционных систем**. Вы можете развернуть Site Recovery для организации защиты виртуальных машин, работающих на любой операционной системе, поддерживаемой Azure. Поддерживаются большинство версий Windows и Linux.
- **Поддержка VHDX**. Хотя Azure пока что не поддерживает формат VHDX, служба Site Recovery автоматически преобразует VHDX в VHD при отработке отказа с перемещением в Azure. При восстановлении к локальной системе виртуальные машины продолжают использовать формат VHDX.
- **Требования Azure**. Убедитесь, что защищаемые виртуальные машины соответствуют требованиям Azure.

## Серверы VMM

С помощью Site Recovery можно организовать репликацию виртуальных машин, размещенных на серверах узла Hyper-V в облаках System Center Virtual Machine Manager (VMM), следующим образом: репликация с локального сервера VMM в Azure (с помощью реплики Hyper-V) — репликация на дополнительный локальный сайт (с помощью реплики Hyper-V). Рекомендуем выполнять развертывание с серверами VMM на основном и дополнительном сайтах. Однако при необходимости вы можете [развернуть один сервер VMM](site-recovery-single-vmm.md) для обоих сайтов: репликация на дополнительный локальный сайт (с помощью сети SAN). Вам потребуется основной и дополнительный центр обработки данных с сервером VMM на каждом сайте. Если необходимо развернуть VMM с Site Recovery, вам потребуется настроить инфраструктуру VMM. Если у вас нет сервера VMM, подробные сведения см. [здесь](site-recovery-hyper-v-site-to-azure.md).

### Настройка инфраструктуры облака VMM

Независимо от того, какой сценарий вы собираетесь развернуть с Site Recovery, вам потребуется создать по крайней мере два частных облака (одно на исходном сервере VMM, а другое — на целевом) в VMM, чтобы получить преимущества модели частного облака при определении и оценке облака и его базовых ресурсов, а также управлении ими с помощью консоли VMM. Облако VMM собирает и представляет сводный набор ресурсов, использование которых ограничено параметрами емкости облака и квотами роли пользователя. Самостоятельные пользователи могут использовать ресурсы облака и управлять ими, не вникая в особенности его базовой инфраструктуры. Объем ресурсов можно быстро нарастить, чтобы сократить или увеличить гибкость облака. Используйте следующие ресурсы для настройки облаков.

- [VMM 2012 и облака](http://www.server-log.com/blog/2011/8/26/vmm-2012-and-the-clouds.html)
- [Настройка структуры облака VMM](https://msdn.microsoft.com/library/azure/dn469075.aspx#BKMK_Fabric)
- [Создание частного облака в VMM](https://technet.microsoft.com/ru-ru/library/jj860425.aspx)
- [Пошаговое руководство. Создание частных облаков с помощью VMM в System Center 2012 с пакетом обновления 1](http://blogs.technet.com/b/keithmayer/archive/2013/04/18/walkthrough-creating-private-clouds-with-system-center-2012-sp1-virtual-machine-manager-build-your-private-cloud-in-a-month.aspx).




## Поставщики и агенты

Поставщики и агенты установлены на локальных серверах, чтобы они могли подключаться к службе Site Recovery в Azure. Они подключаются через Интернет с помощью зашифрованного HTTPS-подключения.

- **Требования к поставщику**. В ходе развертывания вы установите несколько компонентов на свои локальные серверы.
	- **На серверах VMM** вы установите поставщика Azure Site Recovery на серверы VMM, которые требуется зарегистрировать хранилище.
	- **На серверах узла Hyper-V в облаке VMM** вы установите агента служб восстановления Azure.
	- **На серверах узла Hyper-V без сервера VMM** вы установите поставщика Azure Site Recovery и агента служб восстановления Azure на каждом сервере узла Hyper-V или узле кластера.
-  **Установка поставщика**. При установке поставщика и агентов обратите внимание на следующее.
	-  На всех серверах в хранилище Site Recovery должна быть установлена одна версия поставщика. Использование разных версий в одном хранилище не поддерживается.
	-  Если вы планируете выполнять репликацию в Azure с серверов Hyper-V в облаке VMM, на сервере VMM должна быть установлена платформа System Center 2012 R2. Если вы планируете выполнять репликацию на дополнительный центр обработки данных, на сервере VMM должна быть установлена платформа System Center 2012 с пакетом обновления 1 (SP1) или версии R2.
- **Прокси-сервер**. Вам не требуется добавлять исключения брандмауэра или создавать особый прокси-сервер. Если вы хотите использовать настраиваемый прокси-сервер для подключения поставщика, перед началом развертывания вам потребуется сделать следующее.

	- Настройте прокси-сервер перед установкой поставщика.
	- Разрешите доступ к следующим URL-адресам в брандмауэре:
		- *.hypervrecoverymanager.windowsazure.com - *.accesscontrol.windows.net - *.backup.windowsazure.com - *.blob.core.windows.net - *.store.core.windows.net 
	- Если вы развертываете Site Recovery с VMM и используете настраиваемый прокси-сервер, автоматически создается учетная запись запуска от имени VMM (DRAProxyAccount). В ней используются учетные данные прокси-сервера, которые вы указали в параметрах настраиваемого прокси-сервера на портале Site Recovery. Вам потребуется настроить прокси-сервер так, чтобы эта учетная запись могла успешно проходить проверку подлинности.



## Локальные возможности подключения

- **Подключение поставщиков**. Поставщики и агенты установлены на локальных серверах, чтобы они могли подключаться к службе Site Recovery. Они подключаются к службе Site Recovery через Интернет с помощью зашифрованного HTTPS-подключения. Вам не требуется добавлять исключения брандмауэра или создавать особый прокси-сервер.
- **Подключение к Интернету**. Вам потребуются следующие возможности подключения к Интернету.
	- Если вы организуете защиту виртуальных машин на серверах узла Hyper-V в облаке VMM, подключение к Интернету требуется только серверу VMM.
	- Если вы организуете защиту виртуальных машин на серверах узла Hyper-V без сервера VMM, подключение к Интернету требуется только серверам узла Hyper-V.
	- Защищаемым виртуальным машинам подключение к Интернету не требуется.
- **VPN типа "сеть-сеть"**. Вам не потребуется VPN-подключение типа "сеть-сеть" для подключения к Site Recovery. Однако при наличии подключения типа "сеть-сеть" у вас будет доступ к виртуальным машинам, для которых выполнена отработка отказа, как и до ее выполнения. Обратите внимание, что при репликации в Azure вы настраиваете сеть VPN типа "сеть-сеть" между локальным сайтом и определенной сетью Azure. Она не используется для зашифрованного трафика репликации. Этот трафик проходит через Интернет в учетную запись хранения Azure по вашей подписке.
- **Подключение после отработки отказа**. Чтобы гарантировать пользователям возможность подключения к виртуальным машинам после отработки отказа с перемещением в Azure, сделайте следующее:
	- Включите протокол удаленного рабочего стола для виртуальных машин до отработки отказа.
	- После отработки отказа используйте подключение типа "сеть-сеть", чтобы подключаться к ним прежним образом, либо включите конечную точку протокола удаленного рабочего стола для машины.

## Хранилище

- **Учетная запись хранения Azure**. При репликации в Azure вам понадобится учетная запись хранения Azure. Для учетной записи необходимо включить георепликацию. Она должна находиться в том же регионе, что и хранилище Azure Site Recovery, и быть связана с той же подпиской. Подробнее см. в статье [Знакомство со службой хранилища Microsoft Azure](../storage/storage-introduction.md).
- **Сопоставление хранилищ**. Если вы выполняете репликацию виртуальных машин на локальных серверах VMM, вы можете настроить сопоставление хранилищ, чтобы обеспечить надежное подключение виртуальных машин к хранилищу после отработки отказа. При репликации между двумя локальными сайтами VMM, реплика виртуальной машины по умолчанию хранится на указанном целевом сервере узла Hyper-V. Вы можете настроить сопоставление между классификациями хранилищ VMM на исходных и целевых серверах VMM. Если вы собираетесь использовать эту функцию, настройте классификации хранилищ до начала развертывания. [Дополнительные сведения](site-recovery-storage-mapping.md) о сопоставлении хранилищ.
- **Сеть SAN**. Если вы планируете выполнять репликацию между двумя локальными сайтами с помощью репликации сети SAN, обратите внимание на следующее:
	- Вы можете использовать репликацию сети SAN только для репликации виртуальных машин Hyper-V на дополнительный центр обработки данных. Репликация в Azure при это невозможна.
	- Вы можете использовать существующую среду SAN. Вам не придется вносить изменения в массивы SAN.
	- Убедитесь, что ваш массив SAN [поддерживается](http://social.technet.microsoft.com/wiki/contents/articles/28317.deploying-azure-site-recovery-with-vmm-and-san-supported-storage-arrays.aspx).
	- Вам потребуются два сервера VMM на исходном и целевом сайте для развертывания SAN.
	- Убедитесь, что на кластерах узлов Hyper-V используется Windows Server 2012 или 2012 R2, если вы планируете развертывать репликацию сети SAN. [Подробнее](https://technet.microsoft.com/library/cc794868%28v=ws.10%29.aspx) об операционных системах на виртуальной машине, поддерживаемых разными версиями Hyper-V.
	- Вам потребуется обнаружить и классифицировать хранилище SAN в VMM.
	- Если вы еще не начали репликацию, то после обнаружения вам потребуется создать логические номера устройств (LUN) и выделить хранилище в консоли VMM. Если вы уже приступили к репликации, вы можете пропустить этот шаг.
	- Подробные инструкции находятся в этой [статье](site-recovery-vmm-san.md).


## Сети

- **Рекомендации по настройке сети**: [Дополнительные сведения](http://azure.microsoft.com/blog/2014/09/04/networking-infrastructure-setup-for-microsoft-azure-as-a-disaster-recovery-site/) о рекомендациях по настройке сети.

- **Настройка сетевого сопоставления**. Сетевое сопоставление — важный элемент при развертывании VMM и Site Recovery. Эта функция оптимально размещает реплицированные виртуальные машины на целевых серверах узла Hyper-V и обеспечивает подключение реплицированных виртуальных машин к соответствующим сетям после отработки отказа. Вы можете настроить сетевое сопоставление при репликации в Azure или на дополнительный центр обработки данных.
	- **Сетевое сопоставление в Azure**. При репликации в Azure сетевое сопоставление гарантирует, что все виртуальные машины, для которых отработка отказа выполняется в одной и той же сети, могут подключаться друг к другу вне зависимости от того, к какому плану восстановления они относятся. Кроме того, при наличии сетевого шлюза в целевой сети Azure виртуальные машины могут подключаться к другим локальным виртуальным машинам. Если вы не настроите сетевое сопоставление, подключение будет возможно только между машинами, для которых отработка отказа выполняется по одному и тому же плану восстановления.
	- **Сетевое сопоставление на дополнительном сайте**. При репликации на дополнительный сайт VMM сетевое сопоставление гарантирует, что после отработки отказа виртуальные машины подключатся к соответствующим сетям, а также обеспечивает оптимальное размещение реплик виртуальных машин на серверах узла Hyper-V. Если вы не настроите сетевое сопоставление, то реплицированные машины не будут подключаться к сетям виртуальных машин.
	- [Дополнительные сведения](site-recovery-network-mapping) о сетевом сопоставлении.
- **Настройка сетей VMM**.
	- Верно настройте логические сети и сети виртуальных машин в VMM. Дополнительные сведения о [логических сетях](http://blogs.technet.com/b/scvmm/archive/2013/02/14/networking-in-vmm-2012-sp1-logical-networks-part-i.aspx) и [сетях виртуальных машин](https://technet.microsoft.com/library/jj721575.aspx).
	- Убедитесь, что все виртуальные машины на исходном сервере VMM подключены к сети виртуальных машин.
	- Убедитесь, что сети виртуальных машин соединены с логической сетью, связанной с облаком.
	- При выполнении репликации в Azure создайте виртуальные сети в Azure. Учтите, что несколько сетей виртуальных машин можно сопоставить с одной сетью Azure. См. статью [Задачи конфигурации виртуальной сети](../vpn-gateway/vpn-gateway-site-to-site-create.md)

## Оптимизация производительности

- **Размер тома операционной системы**. При репликации виртуальной машины в Azure размер тома операционной системы не должен превышать 1 ТБ. Если размер томов превышает это значения, вы можете вручную перенести их на другой диск перед началом развертывания.
- **Размер диска с данными**. При репликации в Azure вы можете разместить на виртуальной машине до 32 дисков с данными по 1 ТБ каждый. Вы можете успешно выполнить репликацию и отработать отказ для виртуальной машины объемом около 32 ТБ.
- **Ограничения плана восстановления**. Служба Site Recovery может включать тысячи виртуальных машин. Планы восстановления предназначены для приложений, отработка отказа которых должна проводиться одновременно. По этой причине максимальное число машин в плане восстановления — 50.
- **Ограничения служб Azure**. Каждая подписка Azure включает набор ограничений по умолчанию для ядер, облачных служб и т. д. Рекомендуем выполнить тестовую отработку отказа, чтобы проверить доступность ресурсов по вашей подписке. Вы можете изменить эти ограничения через службу поддержки Azure.
- **Планирование использования ресурсов**. В этом вам поможет [планировщик использования ресурсов для реплики Hyper-V](http://www.microsoft.com/en-in/download/details.aspx?id=39057).
- **Пропускная способность репликации**. При невысокой пропускной способности репликации обратите внимание на следующее.
	- **ExpressRoute**. Служба Site Recovery работает с Azure ExpressRoute и оптимизаторами глобальной сети, такими как Riverbed. [Дополнительные сведения](http://blogs.technet.com/b/virtualization/archive/2014/07/20/expressroute-and-azure-site-recovery.aspx) об ExpressRoute.
	- **Трафик репликации**. Служба Site Recovery выполняет интеллектуальную начальную репликацию, используя только блоки данных, а не весь виртуальный жесткий диск. Текущая репликация применяется только к изменениям.
	- **Сетевой трафик**. Вы можете контролировать сетевой трафик, используемый для репликации, путем настройки [качества обслуживания Windows](https://technet.microsoft.com/library/hh967468.aspx) с применением политики на основе IP-адреса и порта назначения. Кроме того, при репликации в Azure Site Recovery с использованием агента службы архивации Azure вы можете настроить регулирование для этого агента. [Подробная информация](https://support.microsoft.com/kb/3056159).
- **Целевое время восстановления (RTO)**. Если вам требуется измерить целевое время восстановления, которое можно ожидать от Site Recovery, рекомендуем выполнить тестовую отработку отказа и просмотреть задания Site Recovery, чтобы определить время, необходимое для выполнения операций. Чтобы достичь наилучшего целевого время восстановления при отработке отказа с перемещением в Azure, рекомендуем автоматизировать все ручные действия путем интеграции со службой автоматизации Azure и планами восстановления.
- **Целевая точка восстановления (RPO)**. Служба Site Recovery поддерживает практически синхронную целевую точку восстановления при репликации в Azure. Для этого необходима достаточная пропускная способность между вашим центром обработки данных и Azure.

## Отработка отказа
- **Сбой на основном сайте**. Если при репликации с одного локального центра обработки данных на другой оба центра обработки данных подвергнутся сбою на вашем основном сайте, выполните внеплановую отработку отказа через портал Site Recovery. Для отработки отказа не требуется подключение в основном центре обработки данных.
- **Сохранение IP-адреса после отработки отказа с перемещением на дополнительный сайт**. Если вы желаете сохранить IP-адрес исходной виртуальной машины, после того как для нее будет выполнена отработка отказа с перемещением в дополнительный центр обработки данных, следуйте инструкциям в [этой статье](http://blogs.technet.com/b/scvmm/archive/2014/04/04/retaining-ip-address-after-failover-using-hyper-v-recovery-manager.aspx).
- **Сохранение IP-адреса после отработки отказа в Azure**. Вы можете указать IP-адрес для виртуальной машины, для которой выполнена отработка отказа, во вкладке "Настройка" этой виртуальной машины. Дополнительные сведения см. в статье о [настройке сетевых свойств виртуальной машины](site-recovery-vmm-to-azure.md#step-8-enable-protection-for-virtual-machines).
- **Сохранение общедоступного IP-адреса**. Если вы желаете сохранить общедоступный IP-адрес после отработки отказа с перемещением на дополнительный сайт, служба Site Recovery не препятствует этому, если ваш поставщик услуг Интернета предоставляет такую возможность. Вы не сможете сохранить общедоступный IP-адрес после отработки отказа с перемещением в Azure.
- **Сохранение внутренних адресов, не оговоренных в документе RFC 1918, в Azure**Вы можете сохранить адресные пространства, не оговоренные в документе RFC 1918, после отработки отказа с перемещением в Azure.
- **Частичная отработка отказа с перемещением в дополнительный центр обработки данных**. Если при отработке отказа для части сайта с перемещением в дополнительный центр обработки данных вам необходимо вновь подключиться к основному сайту, вы можете использовать VPN-подключение типа "сеть-сеть", чтобы подключить приложение, для которого выполнена отработка отказа, в дополнительном центре обработки данных к компонентам инфраструктуры, работающим на основном сайте. Обратите внимание, что при отработке отказа для всей подсети вы можете сохранить IP-адрес виртуальной машины. При отработке отказа для части подсети вы не сможете сохранить IP-адрес виртуальной машины, так как невозможно разделить подсети между сайтами.
- **Частичная отработка отказа с перемещением в Azure**. Если при отработке отказа для части сайта с перемещением в Azure вам необходимо вновь подключиться к основному сайту, вы можете использовать VPN-подключение типа "сеть-сеть", чтобы подключить приложение, для которого выполнена отработка отказа, в Azure к компонентам инфраструктуры, работающим на основном сайте. Обратите внимание, что при отработке отказа для всей подсети вы можете сохранить IP-адрес виртуальной машины. При отработке отказа для части подсети вы не сможете сохранить IP-адрес виртуальной машины, так как невозможно разделить подсети между сайтами.
- **Сохранение буквы диска**. Если после отработки отказа вам необходимо сохранить буквы дисков на виртуальных машинах, вы можете **включить** политику SAN для виртуальной машины. [Подробная информация](https://support.microsoft.com/kb/3031135).
- **Маршрутизация клиентских запросов после отработки отказа с перемещением в Azure**. Служба Site Recovery работает с диспетчером трафика Azure, чтобы выполнять маршрутизацию клиентских запросов к вашему приложению после отработки отказа. Вы можете использовать сценарии в планах восстановления (со службой автоматизации Azure) для выполнения обновлений DNS.

## Интеграция

- **Интеграция с другими технологиями непрерывности бизнес-процессов и аварийного восстановления (BCDR)**. Служба Site Recovery интегрируется с другими технологиями непрерывности бизнес-процессов и аварийного восстановления. При выполнении репликации на основе приложений мы рекомендуем использовать компонент SQL Server AlwaysOn для репликации виртуальных машин, на которых работает база данных, и реплику Hyper-V для интерфейсных виртуальных машин. Для AlwaysOn требуются лицензии SQL Server Enterprise на основном и дополнительном сайте, а также действующая виртуальная машина Azure по подписке. Встроенную функцию репликации Site Recovery стоит использовать, если приложениям не страшен небольшой период простоя. В этом случае мы рекомендуем использовать Site Recovery для выполнения репликации баз данных, приложений и интерфейсных виртуальных машин. Такой подход позволяет сэкономить на требуемой версии SQL Server, числе лицензий и затратах на обеспечение непрерывной работы виртуальных машин в Azure.
