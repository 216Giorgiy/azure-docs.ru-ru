# Azure Site Recovery: вопросы и ответы

Прочитайте эту статью, чтобы ознакомиться с ответами на часто задаваемые вопросы о службе Azure Site Recovery.

Если после прочтения этой статьи у вас возникнут другие вопросы, вы можете опубликовать их внизу этой страницы или на [форуме, посвященном службам восстановления Azure](https://social.msdn.microsoft.com/Forums/azure/home?forum=hypervrecovmgr).

## Общие сведения

### Какие функции выполняет служба Site Recovery?

Служба Site Recovery помогает реализовать стратегии непрерывности бизнес-процессов и аварийного восстановления (BCDR), автоматизируя процессы репликации из локальных виртуальных машин и физических серверов в Azure или дополнительный центр обработки данных и осуществляя управление ими. [Подробнее](site-recovery-overview.md).

### Что можно реплицировать с помощью службы Site Recovery?

- **Виртуальные машины Hyper-V**. Служба Site Recovery может реплицировать любую рабочую нагрузку, запущенную на поддерживаемой виртуальной машине Hyper-V. 
- **Физические серверы**. Служба Site Recovery может реплицировать любые рабочие нагрузки, запущенные на поддерживаемых физических серверах Windows или Linux.
- **Виртуальные машины VMware**. Служба Site Recovery может реплицировать любую рабочую нагрузку, запущенную на поддерживаемой виртуальной машине VMware.


### Что нужно для работы с Hyper-V?

Требования для сервера узла Hyper-V зависят от сценария развертывания. Ознакомьтесь с предварительными требованиями к Hyper-V:

- [Репликация виртуальных машин Hyper-V в Azure (без использования VMM)](site-recovery-hyper-v-site-to-azure.md#before-you-start)
- [Репликация виртуальных машин Hyper-V в Azure (с помощью VMM)](site-recovery-vmm-to-azure.md#before-you-start)
- [Репликация виртуальных машин Hyper-V в дополнительный центр обработки данных](site-recovery-vmm-to-vmm.md#before-you-start)

Требования к гостевым операционным системам, запущенным на сервере узла Hyper-V:

- Если вы выполняете репликацию на вторичный сайт, прочтите о [поддерживаемых гостевых операционных системах для виртуальных машин Hyper-V](https://technet.microsoft.com/library/mt126277.aspx).
- При репликации в Azure служба Site Recovery поддерживает все гостевые операционные системы, [поддерживаемые Azure](https://technet.microsoft.com/library/cc794868%28v=ws.10%29.aspx).


### Можно ли защитить виртуальные машины при запуске Hyper-V на клиентской операционной системе?

Нет, виртуальные машины должны быть расположены на сервере узла Hyper-V, который работает на компьютере под управлением поддерживаемой версии Windows Server. Если вам нужно защитить клиентский компьютер, вы можете реплицировать его как физический компьютер в [Azure](site-recovery-vmware-to-azure-classic.md) или в [дополнительный центр обработки данных](site-recovery-vmware-to-vmware.md).


### Какие рабочие нагрузки можно защитить с помощью службы Site Recovery?

Служба Site Recovery может реплицировать любые рабочие нагрузки, запущенные на поддерживаемой виртуальной машине или физическом сервере. Служба Site Recovery также поддерживает репликацию на уровне приложения, позволяя восстанавливать приложения до программируемого состояния. Она интегрируется с приложениями Майкрософт, включая SharePoint, Exchange, Dynamics, SQL Server и Active Directory, а также тесно взаимодействует с решениями ведущих производителей, в том числе Oracle, SAP, IBM и Red Hat. [Подробнее](site-recovery-workload.md) о защите рабочей нагрузки.


### Чтобы реплицировать узлы Hyper-V с помощью службы Site Recovery, эти узлы обязательно должны находиться в облаках System Center VMM? 

Если требуется репликация в дополнительный центр обработки данных, виртуальные машины Hyper-V должны быть расположены на серверах узла Hyper-V в облаке VMM. Но если предполагается репликация в Azure, ее можно выполнить, реплицировав виртуальные машины на серверах узла Hyper-V как с использованием облаков VMM, так и без них. [Подробнее](site-recovery-hyper-v-site-to-azure.md).

### Можно ли развернуть службу Site Recovery с помощью VMM при наличии только одного сервера VMM? 

Да. Вы можете реплицировать виртуальные машины на серверах Hyper-V в облаке VMM в Azure или между облаками VMM на том же сервере. Для репликации между локальными средами мы рекомендуем использовать сервер VMM как на основном, так и на дополнительном сайте. [Дополнительные сведения](site-recovery-single-vmm.md)

### Какие физические серверы можно защитить?

Вы можете защитить физические серверы под управлением поддерживаемых ОС Windows и Linux. Для этого реплицируйте их в Azure или на дополнительный сайт. [Узнайте](site-recovery-vmware-to-azure-classic.md#before-you-start-deployment) о требованиях к операционной системе для репликации в Azure или на дополнительный сайт. Обратите внимание, что физические серверы, реплицированные в хранилище Azure, работают как виртуальные машины в Azure, когда происходит отработка отказа. Когда после сбоя происходит восстановление из среды Azure на локальный сайт, восстановление физического сервера не поддерживается. Автоматический переход обратно возможен только на виртуальную машину VMware.

### Какие виртуальные машины VMware можно защитить?

Для защиты виртуальных машин VMware вам понадобится низкоуровневая оболочка vSphere и виртуальные машины с запущенными инструментами VMware. Мы также рекомендуем использовать сервер VMware vCenter для управления низкоуровневыми оболочками. [См. дополнительные сведения](site-recovery-vmware-to-azure-classic.md#before-you-start-deployment) о точных требованиях к репликации серверов и виртуальных машин VMware в Azure или на дополнительный сайт.

### Можно ли управлять аварийным восстановлением в офисах филиалов с помощью службы Site Recovery?

Да. Используя службу Site Recovery для управления репликацией и отработкой отказа в офисах филиалов, вы можете оркестрировать и отображать все рабочие нагрузки офисов филиалов в центральном расположении. Таким образом, можно легко выполнить отработку отказа и управлять аварийным восстановлением для всех филиалов, не выходя из главного офиса.

## Безопасность


### Отправляются ли данные репликации в службу Site Recovery?

Нет, служба Site Recovery не перехватывает данные репликации и не получает сведения о компонентах, запущенных на виртуальных машинах или физических серверах.

Обмен данными репликации осуществляется между локальными узлами Hyper-V, низкоуровневыми оболочками VMware или физическими серверами и хранилищем Azure или дополнительным сайтом. Служба Site Recovery не обладает способностью перехватывать эти данные. В службу Site Recovery передаются только метаданные, необходимые для управления репликацией и отработкой отказа.

Site Recovery имеет сертификат ISO 27001:2005 и сейчас проходит проверки на соответствие HIPAA, DPA и FedRAMP JAB.

### В соответствии с требованиями даже метаданные из локальных сред должны оставаться в одном географическом регионе. Может ли служба Site Recovery обеспечить это?

Да. При создании хранилища Site Recovery в регионе мы обеспечиваем хранение всех метаданных, необходимых для управления репликацией и отработкой отказа, в пределах этого региона.

### Выполняет ли служба Site Recovery шифрование репликации?
При репликации виртуальных машин и физических серверов между локальными сайтами поддерживается шифрование в процессе передачи. При репликации виртуальных машин и физических серверов в среду Azure поддерживается как шифрование в процессе передачи, так и обычное шифрование в Azure.



## Репликация

### Существуют ли предварительные требования для репликации виртуальных машин в Azure?

Виртуальные машины, которые необходимо реплицировать в Azure, должны соответствовать [требованиям Azure](site-recovery-best-practices.md#azure-virtual-machine-requirements).

### Можно ли реплицировать виртуальные машины Hyper-V второго поколения в Azure?

Да. Во время отработки отказа служба Site Recovery преобразует виртуальные машины второго поколения в первое. При восстановлении размещения виртуальная машина снова преобразуется в VM второго поколения. [Подробная информация](https://azure.microsoft.com/blog/2015/04/28/disaster-recovery-to-azure-enhanced-and-were-listening/).

### Как осуществляется оплата за виртуальные машины при репликации в Azure? 

При обычной репликации данные реплицируются в геоизбыточное хранилище Azure, поэтому вам не нужно платить за использование виртуальных машин Azure IaaS. При отработке отказа в Azure служба Site Recovery автоматически создает виртуальные машины Azure IaaS, после чего вам будут выставляться счета за вычислительные ресурсы, используемые в Azure.

### Существует ли пакет SDK для автоматизации рабочих процессов службы Site Recovery?

Да. Рабочие процессы службы Site Recovery можно автоматизировать с помощью интерфейса REST API, PowerShell или пакета SDK для Azure. Узнайте больше о [развертывании службы Site Recovery с помощью PowerShell и Azure Resource Manager](site-recovery-deploy-with-powershell-resource-manager.md).

### Какая учетная запись хранения требуется при репликации в Azure?

Необходима учетная запись хранения для [геоизбыточного хранилища класса Standard](../storage/storage-introduction.md#replication-for-durability-and-high-availability). Хранилище класса Premium в настоящее время не поддерживается.

### Как часто можно реплицировать данные?

- **Hyper-V.** Виртуальные машины Hyper-V под управлением Windows Server 2012 R2 можно реплицировать каждые 30 секунд, 5 минут или 15 минут. При настройке репликации сети SAN репликация будет синхронной.
- **VMware и физические серверы.** Здесь частота репликации не имеет значения. Она будет непрерывной. 

### Можно ли расширить репликацию с существующего сайта восстановления до другого сайта восстановления?

Расширенная репликация и цепочка репликации не поддерживаются. [Оставьте свой отзыв](https://feedback.azure.com/forums/256299-site-recovery/suggestions/6097959-support-for-exisiting-extended-replication/) об этой функции.


### Можно ли выполнить первую процедуру репликации в Azure в автономном режиме? 

Эта возможность не поддерживается. [Оставьте свой отзыв](https://feedback.azure.com/forums/256299-site-recovery/suggestions/6227386-support-for-offline-replication-data-transfer-from/) об этой функции.

### Можно ли исключить из репликации отдельные диски?

Эта возможность не поддерживается. [Оставьте свой отзыв](https://feedback.azure.com/forums/256299-site-recovery/suggestions/6418801-exclude-disks-from-replication/) об этой функции.

### Можно ли реплицировать виртуальные машины с динамическими дисками?

Динамические диски поддерживаются при репликации виртуальных машин Hyper-V. Они также поддерживаются при репликации виртуальных машин VMware и физических компьютеров, если вы используете [расширенное развертывание](site-recovery-vmware-to-azure-classic.md). Обратите внимание, что диск операционной системы должен быть базовым диском.

### Можно ли регулировать пропускную способность, выделенную для трафика репликации Hyper-V?
- При выполнении репликации между двумя локальными сайтами виртуальных машин Hyper-V можно использовать функции качества обслуживания Windows. Вот пример сценария: 

    	New-NetQosPolicy -Name ASRReplication -IPDstPortMatchCondition 8084 -ThrottleRate (2048*1024)
    	gpupdate.exe /force

- При репликации виртуальных машин Hyper-V в Azure можно настроить регулирование, используя следующий пример командлета PowerShell:

    	Set-OBMachineSetting -WorkDay $mon, $tue -StartWorkHour "9:00:00" -EndWorkHour "18:00:00" -WorkHourBandwidth (512*1024) -NonWorkHourBandwidth (2048*1024)

- [Узнайте больше](https://support.microsoft.com/ru-RU/kb/3056159) о регулировании трафика Hyper-V.
- [Узнайте больше](site-recovery-vmware-to-azure-classic.md#capacity-planning) о регулировании репликации VMware в Azure.


## Отработка отказа

### Как получить доступ к виртуальным машинам Azure после отработки отказа в Azure? 

Доступ к виртуальным машинам Azure можно получить через безопасное подключение к Интернету, подключение VPN типа "сеть — сеть" или с помощью Azure ExpressRoute. Подключение VPN осуществляется через внутренние порты сети Azure, в которой расположена виртуальная машина. При подключении через Интернет для виртуальных машин выполняется сопоставление с общедоступными конечными точками в облачной службе Azure.

### Каким образом Azure обеспечивает отказоустойчивость данных при отработке отказа в Azure?

Среда Azure разработана для обеспечения отказоустойчивости. Site Recovery изначально рассчитана на отработку отказа в дополнительном центре обработки данных Azure в соответствии с соглашением об уровне обслуживания Azure (при необходимости). При обработке отказа Azure обеспечивает хранение метаданных и хранилищ в географическом регионе, выбранном для хранилища.

### Что будет, если в ходе репликации между двумя сайтами на основном сайте неожиданно произойдет сбой?

Можно активировать незапланированную отработку отказа на дополнительном сайте. Для отработки отказа службе Site Recovery не требуется подключение на основном сайте.

### Отработка отказа выполняется автоматически?

Отработка отказа не выполняется автоматически. Вы можете быстро активировать ее одним щелчком на портале или с помощью [командлетов PowerShell для Site Recovery](https://msdn.microsoft.com/library/dn850420.aspx). Отработка отказа — это последовательность действий на портале Site Recovery.

Чтобы автоматизировать выполнение этих процедур, можно использовать локальное решение Orchestrator или Operations Manager для обнаружения сбоя виртуальной машины, а затем активировать отработку отказа, используя пакет SDK.


##
### Я поставщик услуг. Работает ли служба Site Recovery для выделенных или общих моделей инфраструктуры?
Да, Site Recovery поддерживает как выделенную, так и общую модели инфраструктуры.

### Предоставляется ли удостоверение клиента поставщика услуг службе Site Recovery?
Нет. Клиентам не требуется доступ к порталу Site Recovery. Только администратор поставщика служб взаимодействует с порталом.


### Будут ли данные клиентов передаваться в Azure?

При репликации между сайтами, принадлежащими поставщику служб, данные приложения никогда не передаются в Azure. Данные шифруются при передаче и реплицируются непосредственно между сайтами поставщиков служб.

При репликации в Azure данные приложения передаются в службу хранилища Azure, а не в службу Site Recovery. Данные шифруются в процессе передачи и остаются зашифрованными в Azure.

### Будут ли клиенты получать счета за какие-либо службы Azure?

Нет. Azure выставляет счета только поставщику службы. Поставщики служб отвечают за создание специальных счетов для своих клиентов.

### Нужно ли всегда запускать виртуальные машины при репликации в Azure?

Нет. Данные реплицируются в учетной записи геоизбыточного хранилища Azure в рамках вашей подписки. Во время тестовой (учебной) или фактической отработки отказа служба Site Recovery автоматически создает в подписке виртуальные машины.

### Обеспечивается ли изоляция на уровне клиента при репликации в Azure?

Да.

### Какие платформы поддерживаются на данный момент?

Поддерживаются развертывания на основе Azure Pack, Cloud Platform System и System Center 2012 и более поздних версий. Дополнительные сведения об интеграции Azure Pack см. в статье [Настройка защиты для виртуальных машин](https://technet.microsoft.com/library/dn850370.aspx).

### Поддерживаются ли отдельные развертывания Azure Pack или односерверные развертывания VMM?

Да, можно выполнять репликацию виртуальных машин Hyper-V в Azure или между сайтами поставщиков служб. Обратите внимание, что при репликации между сайтами поставщиков служб интеграция Runbook Azure недоступна.
