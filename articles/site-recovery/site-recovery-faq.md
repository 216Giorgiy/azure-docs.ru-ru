# Azure Site Recovery: вопросы и ответы
## Об этой статье

Данная статья содержит ответы на часто задаваемые вопросы о службе Site Recovery. Если после прочтения этой статьи у вас возникнут какие-либо вопросы, вы можете задать их на [форуме, посвященном службам восстановления Azure](https://social.msdn.microsoft.com/Forums/azure/home?forum=hypervrecovmgr).


## Поддержка

###Какие функции выполняет служба Site Recovery?

Служба Site Recovery помогает реализовать стратегии непрерывности бизнес-процессов и аварийного восстановления (BCDR), автоматизируя процессы репликации из локальных виртуальных машин и физических серверов в Azure или дополнительный центр обработки данных и осуществляя управление ими. [Подробнее](site-recovery-overview.md).


### Что можно защитить с помощью службы Site Recovery?

- **Виртуальные машины Hyper-V**. Служба Site Recovery позволяет защитить любую рабочую нагрузку, запущенную на виртуальной машине Hyper-V.
- **Физические серверы**. Служба Site Recovery позволяет защитить физические серверы под управлением Windows или Linux.
- **Виртуальные машины VMware**. Служба Site Recovery позволяет защитить любую рабочую нагрузку, запущенную на виртуальной машине VMware.


###Какие виртуальные машины Hyper-V можно защитить?

Это зависит от сценария развертывания.

Просмотрите разделы с предварительными требованиями для сервера узла Hyper-V:

- [Репликация виртуальных машин Hyper-V в Azure (без использования VMM)](site-recovery-hyper-v-site-to-azure.md#before-you-start)
- [Репликация виртуальных машин Hyper-V в Azure (с помощью VMM)](site-recovery-vmm-to-azure.md#before-you-start)
- [Репликация виртуальных машин Hyper-V в дополнительный центр обработки данных](site-recovery-vmm-to-vmm.md#before-you-start)

Информация, касающаяся операционных систем на виртуальной машине

- При репликации в дополнительный центр обработки данных см. список поддерживаемых операционных систем на виртуальных машинах, работающих на серверах узла Hyper-V, в статье [Поддерживаемые операционные системы на виртуальных машинах Hyper-V](https://technet.microsoft.com/library/mt126277.aspx).
- При репликации в Azure служба Site Recovery поддерживает все гостевые операционные системы на виртуальной машине, [поддерживаемые Azure](https://technet.microsoft.com/library/cc794868%28v=ws.10%29.aspx).

Служба Site Recovery позволяет защитить любую рабочую нагрузку, запущенную на поддерживаемых виртуальных машинах.

### Можно ли защитить виртуальные машины при запуске Hyper-V на клиентской операционной системе?

Нет, Hyper-V не поддерживается. Для решения этой задачи необходимо реплицировать физический компьютер [в Azure](site-recovery-vmware-to-azure-classic.md) или [дополнительный центр обработки данных](site-recovery-vmware-to-vmware.md).


### Какие рабочие нагрузки можно защитить с помощью службы Site Recovery?

Службу Site Recovery можно использовать для защиты большинства рабочих нагрузок, запущенных на виртуальной машине или физическом сервере. С помощью службы Site Recovery можно выполнить развертывание аварийного восстановления с учетом особенностей приложения. Она интегрируется с приложениями Майкрософт, включая SharePoint, Exchange, Dynamics, SQL Server и Active Directory, и тесно взаимодействует с решениями ведущих производителей, в том числе Oracle, SAP, IBM и Red Hat. Можно настроить решение аварийного восстановления для каждого конкретного приложения. [Подробнее](site-recovery-workload.md) о защите рабочей нагрузки.


### Всегда ли требуется сервер System Center VMM для защиты виртуальных машин Hyper-V?

Нет. Вы можете реплицировать не только виртуальные машины Hyper-V, расположенные в облаках VMM, но и виртуальные машины Hyper-V в среде без развертывания VMM. [Подробнее](site-recovery-hyper-v-site-to-azure.md). Обратите внимание: если требуется репликация в дополнительный центр обработки данных, серверами узла Hyper-V необходимо управлять в облаках VMM.

### Можно ли развернуть службу Site Recovery с помощью VMM при наличии только одного сервера VMM?

Да. Можно выполнить репликацию виртуальных машин Hyper-V в облаке на сервере VMM в Azure или между облаками VMM на том же сервере. Обратите внимание, что для репликации между локальными средами на основном и дополнительном сайтах потребуется сервер VMM. [Дополнительные сведения](site-recovery-single-vmm.md)

### Какие физические серверы можно защитить?

Можно защитить физические серверы под управлением Windows и Linux, которые реплицируются в Azure или на дополнительный сайт. [Ознакомьтесь](site-recovery-vmware-to-azure-classic.md#before-you-start-deployment) с требованиями к операционной системе. Те же ограничения применяются независимо от конечного расположения репликации физических серверов (Azure или дополнительный сайт).

Обратите внимание, что если локальный сервер выйдет из строя, физические серверы запустятся в качестве виртуальных машин в Azure. Восстановление размещения на физическом локальном сервере пока не поддерживается. Вам придется восстанавливать размещение на виртуальной машине VMware.

### Какие виртуальные машины VMware можно защитить?

В этом случае необходим сервер VMware vCenter, низкоуровневая оболочка vSphere и виртуальные машины с запущенными инструментами VMware. [Ознакомьтесь](site-recovery-vmware-to-azure-classic.md#before-you-start-deployment) с точными требованиями. Те же ограничения применяются независимо от конечного расположения репликации физических серверов: Azure или дополнительный сайт.

### Существуют ли предварительные требования для репликации виртуальных машин в Azure?

Виртуальные машины, которые необходимо реплицировать в Azure, должны соответствовать [требованиям Azure](site-recovery-best-practices.md#azure-virtual-machine-requirements).

### Можно ли реплицировать виртуальные машины Hyper-V второго поколения в Azure?

Да. Во время отработки отказа Site Recovery преобразует машины второго поколения в первое. При восстановлении размещения машина преобразуется обратно во второе поколение. [Подробная информация](https://azure.microsoft.com/blog/2015/04/28/disaster-recovery-to-azure-enhanced-and-were-listening/).

### Как осуществляется оплата за виртуальные машины при репликации в Azure?

При регулярной репликации данные реплицируются в геоизбыточное хранилище Azure, поэтому нет необходимости платить за использование виртуальных машин Azure IaaS, что является значительным преимуществом. При отработке отказа в Azure служба Site Recovery автоматически создает виртуальные машины Azure IaaS, после чего вам будут выставляться счета за вычислительные ресурсы, используемые в Azure.

### Можно ли управлять аварийным восстановлением в офисах филиалов с помощью службы Site Recovery?

Да. Используя службу Site Recovery для управления репликацией и отработкой отказа в офисах филиалов, вы получаете объединенные возможности управления и представление для всех рабочих нагрузок офисов филиалов в центральном расположении. Таким образом, можно легко выполнить отработку отказа и управлять аварийным восстановлением для всех филиалов, не выходя из главного офиса.

### Существует ли пакет SDK для автоматизации рабочего процесса Site Recovery?

Да. Рабочие процессы службы Site Recovery можно автоматизировать с помощью интерфейса REST API, PowerShell или пакета SDK для Azure. Узнайте больше о [развертывании службы Site Recovery с помощью PowerShell и диспетчера ресурсов Azure](site-recovery-deploy-with-powershell-resource-manager.md).


## Безопасность и соответствие требованиям

### Данные отправляются в службу Site Recovery?

Нет, служба Site Recovery не перехватывает данные приложения и не обладает сведениями о том, что запущено на виртуальных машинах или физических серверах.

Обмен данными репликации осуществляется между узлами Hyper-V, низкоуровневыми оболочками VMware или физическими серверами в основных и дополнительных центрах обработки данных, а также между центром обработки данных и службой хранилища Azure. Служба Site Recovery не обладает способностью перехватывать эти данные. В службу Site Recovery передаются только метаданные, необходимые для управления репликацией и отработкой отказа.

Site Recovery имеет сертификат ISO 27001:2005 и сейчас проходит проверки на соответствие HIPAA, DPA и FedRAMP JAB.

### Для соответствия требованиям даже метаданные из локальных сред должны оставаться в одном географическом регионе. Может ли служба Site Recovery помочь нам?

Да. При создании хранилища Site Recovery в регионе мы обеспечиваем хранение всех метаданных, необходимых для управления репликацией и отработкой отказа, в пределах этого региона.

### Выполняет ли служба Site Recovery шифрование репликации?
При репликации виртуальных машин и физических серверов между локальными сайтами поддерживается шифрование в процессе передачи. При репликации виртуальных машин и физических серверов между локальными сайтами и Azure поддерживается как шифрование в процессе передачи, так и шифрование в Azure.




## Репликация и отработка отказа

### Какая учетная запись хранения требуется при репликации в Azure?

Необходима учетная запись хранения для [геоизбыточного хранилища класса Standard](../storage/storage-introduction.md#replication-for-durability-and-high-availability). Хранилище класса Premium в настоящее время не поддерживается.

### Как часто можно реплицировать данные?
- **Hyper-V.** Виртуальные машины Hyper-V под управлением Windows Server 2012 R2 можно реплицировать каждые 30 секунд, 5 минут или 15 минут. При настройке репликации сети SAN репликация будет синхронной.
- **VMware и физические серверы.** Здесь частота репликации не имеет значения. Она будет непрерывной.

### Можно ли расширить репликацию с существующего сайта восстановления до сайта третьего уровня?
Расширенная репликация и цепочка репликации не поддерживаются. Оставьте отзыв об этой функции на [форуме](https://feedback.azure.com/forums/256299-site-recovery/suggestions/6097959-support-for-exisiting-extended-replication/).


### Можно ли выполнить первую процедуру репликации в Azure в автономном режиме?

Эта возможность не поддерживается. Оставьте отзыв об этой функции на [форуме](https://feedback.azure.com/forums/256299-site-recovery/suggestions/6227386-support-for-offline-replication-data-transfer-from/).


### Можно ли исключить из репликации отдельные диски?

Эта возможность не поддерживается. Оставьте отзыв об этой функции на [форуме](https://feedback.azure.com/forums/256299-site-recovery/suggestions/6418801-exclude-disks-from-replication/).

### Можно ли реплицировать виртуальные машины с динамическими дисками?

Динамические диски поддерживаются при репликации виртуальных машин Hyper-V. Однако они не поддерживаются при репликации виртуальных машин VMware или физических серверов. Оставьте отзыв об этой функции на [форуме](https://feedback.azure.com/forums/256299-site-recovery/).

### Как получить доступ к виртуальным машинам Azure после отработки отказа в Azure?

Доступ к виртуальным машинам Azure можно получить через безопасное подключение к Интернету или подключение VPN типа "сеть — сеть" либо Azure ExpressRoute (при их наличии). Подключение VPN осуществляется через внутренние порты сети Azure, в которой расположена виртуальная машина. При подключении через Интернет для виртуальных машин выполняется сопоставление с общедоступными конечными точками в облачной службе Azure.

### Каким образом среда Azure обеспечивает отказоустойчивость данных при отработке отказа?

Среда Azure призвана обеспечивать отказоустойчивость службы. Site Recovery изначально рассчитана на отработку отказа в дополнительном центре обработки данных Azure в соответствии с соглашением об уровне обслуживания Azure (при необходимости). При обработке отказа Azure обеспечивает хранение метаданных и хранилищ в географическом регионе, выбранном для хранилища.

### Что произойдет, если при выполнении репликации между двумя центрами обработки данных в основном центре обработки данных неожиданно возникнет сбой?

Можно активировать незапланированную отработку отказа на дополнительном сайте. Для отработки отказа службе Site Recovery не требуется подключение на основном сайте.


### Отработка отказа выполняется автоматически?

Отработка отказа не выполняется автоматически. Чтобы активировать ее, необходимо щелкнуть в соответствующем месте на портале или воспользоваться [командлетами Site Recovery PowerShell](https://msdn.microsoft.com/library/dn850420.aspx). Восстановление размещения — еще одно простое действие, которое выполняется на портале Site Recovery. Чтобы автоматизировать выполнение этих процедур, можно использовать локальное решение Orchestrator или Operations Manager для обнаружения сбоя виртуальной машины, а затем активировать отработку отказа, используя пакет SDK.

### Можно ли регулировать пропускную способность, выделенную для трафика репликации Hyper-V, при репликации виртуальных машин Hyper-V?
- При выполнении репликации между двумя локальными сайтами виртуальных машин Hyper-V можно использовать функции качества обслуживания Windows. Вот пример сценария:

    	New-NetQosPolicy -Name ASRReplication -IPDstPortMatchCondition 8084 -ThrottleRate (2048*1024)
    	gpupdate.exe /force

- При репликации виртуальных машин Hyper-V в Azure можно настроить регулирование, используя следующий пример командлета PowerShell:

    	Set-OBMachineSetting -WorkDay $mon, $tue -StartWorkHour "9:00:00" -EndWorkHour "18:00:00" -WorkHourBandwidth (512*1024) -NonWorkHourBandwidth (2048*1024)



## Развертывание поставщика служб

### Работает ли служба Site Recovery для выделенных или общих моделей инфраструктуры?
Да, Site Recovery поддерживает как выделенную, так и общую модели инфраструктуры.

### Предоставляется ли удостоверение клиента службе Site Recovery?
Нет. На самом деле, клиентам не требуется доступ к порталу Site Recovery. Только администратор поставщика служб выполняет действия на портале.


### Будут ли данные клиентов передаваться в Azure?
При репликации между сайтами, принадлежащими поставщику служб, данные приложения никогда не передаются в Azure. Данные шифруются и реплицируются непосредственно между сайтами поставщиков служб.

При репликации в Azure данные приложения передаются в службу хранилища Azure, а не в службу Site Recovery. Данные шифруются в процессе передачи и остаются зашифрованными в Azure.

### Будут ли клиенты получать счета за какие-либо службы Azure?

Нет. Azure выставляет счета только поставщику службы. Поставщики служб отвечают за создание специальных счетов для своих клиентов.

### Нужно ли всегда запускать виртуальные машины при репликации в Azure?

Нет. Данные реплицируются в учетной записи геоизбыточного хранилища Azure вашей подписки. Во время тестовой (учебной) или фактической отработки отказа служба Site Recovery автоматически создает в подписке виртуальные машины.

### Обеспечивается ли изоляция на уровне клиента при репликации в Azure?

Да.

### Какие платформы поддерживаются на данный момент?

Поддерживаются развертывания на основе Azure Pack, системы облачной платформы и System Center 2012 и более поздних версий. Дополнительные сведения об интеграции Azure Pack и службы Site Recovery см. в статье [Настройка защиты для виртуальных машин](https://technet.microsoft.com/library/dn850370.aspx).

### Поддерживаются ли отдельные развертывания Azure Pack или односерверные развертывания VMM?

Да, можно выполнять репликацию виртуальных машин Hyper-V в Azure или между сайтами поставщиков служб. Обратите внимание, что при репликации между сайтами поставщиков служб интеграция Runbook Azure недоступна.
