# Azure Site Recovery: вопросы и ответы
## Об этой статье

Данная статья содержит часто задаваемые вопросы об Azure Site Recovery. Общие сведения о Site Recovery и соответствующих сценариях развертывания см. в статье [Обзор Site Recovery](site-recovery-overview.md).

Если после прочтения этой статьи у вас возникнут другие вопросы, вы можете задать их на [форуме, посвященном службам восстановления Azure](https://social.msdn.microsoft.com/Forums/azure/home?forum=hypervrecovmgr).


## Общие вопросы
### Безопасна ли служба ASR? Какие данные отправляются в Azure?

Да, служба ASR безопасна. Она не перехватывает данные приложения и не знает о том, что именно запущено на виртуальной машине. Подключение к Интернету не требуется ни на узлах Hyper-V, ни на виртуальных машинах.

Так как репликация выполняется между сайтами предприятия и поставщика службы, данные приложения не отправляются в Azure. В Azure отправляются только метаданные, необходимые для оркестрации отработки отказов, например имена виртуальных машин и облачных служб. ASR не способна перехватывать данные приложения, и эти данные всегда остаются на локальном компьютере.

ASR имеет сертификат ISO 27001:2005 и сейчас проходит проверки на соответствие HIPAA, DPA и FedRAMP JAB.

### Согласно нормативным требованиям даже метаданные из наших локальных сред должны оставаться в одном географическом регионе. Может ли служба ASR обеспечить соблюдение этого требования?

Да. При создании хранилища Site Recovery в выбранном регионе мы обеспечиваем хранение всех метаданных, необходимых для оркестрации аварийного восстановления, в пределах этого региона.

### Существует ли пакет SDK для автоматизации рабочего процесса ASR?

Да. Рабочие процессы ASR можно автоматизировать с помощью API REST, PowerShell или пакета SDK для Azure. Дополнительные сведения можно найти в публикации [Поддержка PowerShell для Azure Site Recovery](http://azure.microsoft.com/blog/2014/11/05/introducing-powershell-support-for-azure-site-recovery/) в блоге.

## Поддержка версий

### Какие версии узлов и кластеров Windows Server поддерживаются?
При выборе реплики Hyper-V для включения репликации и защиты между сайтами Hyper-V можно использовать Windows Server 2012 и Windows Server 2012 R2.


### Какие версии гостевых операционных систем поддерживают виртуальные машины Hyper-V?
Наиболее актуальный список поддерживаемых операционных систем на виртуальных машинах можно найти в разделе [О виртуальных машинах и гостевых операционных системах](https://technet.microsoft.com/library/cc794868%28v=ws.10%29.aspx).

### Можно ли настроить защиту виртуальной машины, когда Hyper-V работает на клиентской операционной системе?

В системе Hyper-V, работающей на клиентской операционной системе, нельзя настроить репликацию виртуальных машин в Azure. Реплика Hyper-V поддерживается только на серверных операционных системах. Физические компьютеры можно реплицировать в Azure, следуя инструкциям из [этой статьи](site-recovery-vmware-to-azure.md).

### Поддерживает ли служба ASR машины второго поколения?

Сейчас служба ASR поддерживает репликацию виртуальных машин Hyper-V второго поколения в Azure. Во время отработки отказа ASR преобразует машины из второго поколения в первое. При восстановлении после сбоя машина преобразуется обратно в первое поколение. [Узнайте больше](http://azure.microsoft.com/updates/azure-site-recovery-supports-gen-2-vm-protection-in-west-us-north-europe-and-japan-west/) о текущей поддержке.


## Развертывание между площадками поставщиков услуг 

### Работает ли это решение для выделенных или общих моделей инфраструктуры?
Да, ASR поддерживает как выделенную так и общую модели инфраструктуры.

### Предоставляется ли удостоверение клиента службе Azure?
Нет. На самом деле клиентам не требуется доступ к порталу ASR. Только администратор поставщика службы выполняет действия на портале ASR в Azure.

### Будут ли клиенты получать счета из Azure?
Нет. Microsoft Azure отправляет счета только поставщику службы. Поставщики служб отвечают за создание специальных счетов для своих клиентов.

### Будут ли данные клиентов передаваться в Azure?
При использовании сайта, принадлежащего поставщику службы, для аварийного восстановления данные приложения никогда не передаются в Azure. Данные шифруются и реплицируются непосредственно между сайтами поставщиков служб. При использовании Azure в качестве сайта аварийного восстановления данные приложения отправляются в Azure — хранящиеся и передающиеся данные остаются шифрованными. Для установки соединения с Azure также можно использовать VPN, ExpressRoute или общедоступное подключение к Интернету.

### Если Azure используется в качестве сайта аварийного восстановления, нужно ли всегда запускать виртуальные машины в Azure?
Нет, ASR разработана как решение DRaaS первого класса в общедоступном облаке. Данные реплицируются в учетной записи геоизбыточного хранилища Azure вашей подписки. Во время учебной или фактической отработки отказа ASR автоматически создает в подписке виртуальные машины.

### Обеспечивается ли изоляция на уровне клиента при выборе Azure в качестве сайта аварийного восстановления?
Да.

### Какие платформы поддерживаются на данный момент?
Поддерживаются Azure Pack, система облачной платформы, а также Hyper 2012 и более поздних версий на основе System Center. Дополнительные сведения об интеграции ASR и Azure Pack см. в статье [Настройка защиты для виртуальных машин](https://technet.microsoft.com/library/dn850370.aspx).

### Поддерживаются ли отдельные развертывания Azure Pack или односерверные развертывания VMM?
Да, отдельное развертывание SCVMM поддерживается для сценариев аварийного восстановления как между сайтами поставщиков служб, так и между сайтом поставщика служб и Azure.

Отдельное развертывание Azure Pack также поддерживается при включении аварийного восстановления между сайтами поставщиков служб. Тем не менее, интеграция ASR Runbook недоступна для этой топологии.

## Развертывание между сайтами Hyper-V и Azure (без VMM)

### Требует ли ASR VPN-подключение типа "сеть-сеть"?
Нет, это не обязательно. ASR работает и через общедоступное подключение к Интернету. Тем не менее, если у вас настроено VPN-подключение типа "сеть-сеть", то вы сможете получать доступ к восстановленным виртуальным машинам так же, как раньше. Дополнительные рекомендации по сетям при включении аварийного восстановления в Azure см. в статье [Настройка сетевой инфраструктуры для Microsoft Azure как сайта аварийного восстановления](http://azure.microsoft.com/blog/2014/09/04/networking-infrastructure-setup-for-microsoft-azure-as-a-disaster-recovery-site/).

### Нужно ли оплачивать что-то, кроме тарифа за защиту ASR?
В стабильном состоянии мы реплицируем изменения в геоизбыточном хранилище Azure, и вам не требуется оплачивать тарифы за виртуальные машины Azure IaaS (это значительное преимущество). При отработке отказа в Azure ASR автоматически создает виртуальные машины Azure IaaS, после чего вы оплачиваете вычислительные ресурсы, используемые в Azure.

### У меня есть филиал, где не установлен VMM. Можно ли включить защиту рабочих нагрузок в филиале через Azure? Каковы основные преимущества использования ASR для этой цели?
Да, ASR можно использовать для защиты виртуальных машин, работающих на узлах Hyper-V, которые не управляются с помощью SCVMM.

При использовании ASR для управления аварийным восстановлением всех филиалов вы получаете общий вид на все рабочие нагрузки филиалов. Кроме того, вы получаете возможность администрировать отработку отказов и аварийное восстановление одним щелчком для всех филиалов из главного офиса. В настоящий момент сопоставление сетей и шифрование хранящихся данных не поддерживаются для аварийного восстановления филиалов.

### Существует ли ограничение на количество защищенных виртуальных машин для филиалов?
Нет. Поддержка остается такой же, как и в случае предприятия.

### Нужно ли устанавливать какой-либо агент на виртуальные машины, которые необходимо защитить?

На виртуальных машинах агент не нужен. На каждом сервере Hyper-V, где запущены виртуальные машины, которые необходимо защитить, нужно установить поставщик Site Recovery и агент служб восстановления Azure. Эти два компонента устанавливаются с помощью файла поставщика, который можно скачать во время развертывания службы Site Recovery.

### Нужны ли для поставщика, запущенного на сервере Hyper-V, какие-то особые настройки брандмауэра?

Особые настройки не требуются. Компоненты поставщика на сервере взаимодействуют с Azure через HTTPS-соединение. Для подключения используются заданные на сервере узла Hyper-V стандартные настройки Интернета, если не указано иное.

### Должен ли сервер Hyper-V входить в домен?

Нет, сервер не должен входить в домен.

### Какие версии узлов и кластеров Windows Server поддерживаются?
При использовании ASR и репликации Hyper-V между сайтами Hyper-V и Azure можно использовать Windows Server 2012 и Windows Server 2012 R2.

### Какие версии операционных систем поддерживаются на виртуальных машинах?
Наиболее актуальный список поддерживаемых операционных систем на виртуальных машинах можно найти в статье [О виртуальных машинах и гостевых операционных системах](https://technet.microsoft.com/library/cc794868%28v=ws.10%29.aspx).

## Развертывание между двумя центрами данных VMM

### Можно ли реплицировать в Azure виртуальную машину, которая реплицируется на дополнительный сайт? 

Нет, такая цепочка репликации не поддерживается.


## Развертывание между двумя центрами данных VMM с SAN

### Что делать, если у меня уже настроена репликация на основе SAN и ее не нужно менять?
Без проблем. ASR поддерживает сценарии, в которых репликация уже настроена. В этом случае вы можете использовать ASR для оркестрации аварийного восстановления на виртуальных машинах, настройки сети аварийного восстановления, планирования восстановления с учетом особенностей используемых приложений, выполнения учебного аварийного восстановления, а также для обеспечения выполнения нормативных требований и требований к созданию отчетов об аудите.


### Нужен ли мне диспетчер VMM? Нужен ли диспетчер VMM с обеих сторон?
Да. Массив SAN должен быть поставлен под управление VMM с помощью предназначенного для массивов поставщика SMI-S.

Допустимо и отдельное высокодоступное развертывание VMM на основе типа массива, но для управления сайтами мы рекомендуем использовать отдельный сервер VMM.


### Что делать, если я сомневаюсь в администраторе хранилища?
Мы работаем с имеющейся репликацией, настроенной вашим администратором хранилища. Это означает, что администратору хранилища не нужно менять свои массивы. Тем не менее, если организации нужно автоматизировать управление хранилищем с помощью SCVMM, она может подготовить хранилище с помощью ASR и VMM.

### Поддерживается ли синхронная репликация? Кластеры виртуальных машин? Общее хранилище?
Да, да и да!

### "Сеть-сеть": что нужно остановить локально, чтобы это работало?
Если для обеспечения репликации и защиты на сайтах Hyper-V используется репликация на основе массивов (SAN), то вам достаточно установить поставщика ASR DR на серверах SCVMM. Кроме того, помните, что это *единственный* узел, которому требуется подключение к Интернету.


Ваши массивы также должны быть обнаружены SCVMM с помощью обновленного поставщика SMI-S, предоставленного соответствующими поставщиками хранилищ.

## Развертывание между физическими серверами и Azure

### Можно ли защитить мой физический локальный сервер в Azure?

Да, физический сервер можно реплицировать с помощью ASR. Если ваш локальный сервер выйдет из строя, вы сможете запустить его реплику как виртуальную машину Azure. Автоматический переход обратно на физический локальный сервер пока не поддерживается. Автоматический возврат возможен только на виртуальную машину Hyper-V или VMware.


## Отработка отказа

### Что делать, если Azure не работает? Можно ли вызвать отработку отказа из VMM?

Хотя среда Azure призвана обеспечивать отказоустойчивость службы, мы хотим быть готовыми к худшему. ASR уже рассчитана на отработку отказа во вспомогательном центре обработки данных Azure в соответствии с соглашением об уровне обслуживания Azure. Мы также гарантируем, что в этом случае ваши метаданные и хранилище ASR останутся в географическом регионе, выбранном для хранилища.


### Что делать, если поставщик услуг Интернета для основного центра обработки данных также подвергается сбою при аварии? Что делать, если поставщик услуг Интернета для вспомогательного центра обработки данных также подвергается сбою?

С помощью портала ASR можно выполнять незапланированную отработку отказа одним щелчком. Для отработки отказа не требуется подключение в основном центре обработки данных.

Возможно, приложениям, которые должны быть запущены во вспомогательном центре обработки данных, потребуется подключение к Интернету в том или ином виде. ASR предполагает, что, хотя авария затронула основной сайт и поставщика услуг Интернета, сервер SCVMM вспомогательного сайта по-прежнему подключен к ASR.

### Поддерживает ли ASR восстановление размещения?
Да. Восстановить Azure на локальный сайт с помощью портала ASR можно одним щелчком.

### Возможна ли автоматическая отработка отказа?

Автоматическая отработка отказа невозможна. Чтобы запустить отработку отказа, нужно либо нажать кнопку на портале, либо воспользоваться командлетами [PowerShell для Site Recovery](https://msdn.microsoft.com/library/dn850420.aspx). Вы можете отследить сбой виртуальной машины с помощью локального решения Orchestrator или Operations Manager, а затем запустить отработку отказа, используя пакет SDK. Так можно автоматизировать этот процесс.




## Мониторинг

### Какие решения позволяют отслеживать установку и выполняющуюся репликацию?
Вы можете подписаться на уведомления по электронной почте непосредственно из службы ASR, чтобы узнавать о проблемах, требующих вашего внимания. Вы также можете отслеживать работоспособность текущей репликации с помощью SCOM.
