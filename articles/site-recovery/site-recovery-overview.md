# <a name="what-is-site-recovery"></a>Что такое Site Recovery?
Вас приветствует служба Azure Site Recovery. В этой статье содержится общий обзор службы Site Recovery.

Всем организациям нужна стратегия обеспечения непрерывности бизнеса и аварийного восстановления (BCDR), которая обеспечит безопасность и доступность приложений и данных при запланированных и незапланированных простоях, а также поможет быстро вернуться в нормальный режим работы.

Site Recovery помогает реализовать стратегию BCDR, управляя процессами репликации локальных виртуальных машин и физических серверов. Серверы и виртуальные машины можно реплицировать из основного локального центра обработки данных в облако (Azure) или в дополнительный центр обработки данных.

При сбоях основного сайта выполняется отработка отказа на дополнительный сайт. Это обеспечивает доступность приложений и рабочих нагрузок. При восстановлении нормального режима работы исходного расположения происходит переключение на него.

## <a name="site-recovery-in-the-azure-portal"></a>Служба Site Recovery на портале Azure
В Azure предусмотрены две [модели развертывания](../azure-resource-manager/resource-manager-deployment-model.md) для создания ресурсов и работы с ними: модель Azure Resource Manager и классическая модель управления службой. Есть также две версии портала Azure: [классический портал Azure](https://manage.windowsazure.com/) и [портал Azure](https://portal.azure.com).

* Служба Site Recovery может быть развернута как на классическом портале, так и на портале Azure.
* На классическом портале Azure поддерживается классическая модель управления службой Site Recovery,
* а на портале Azure — классическая модель или модель Resource Manager.

Информация, приведенная в этой статье, относится как к классической модели развертывания, так и к модели развертывания с помощью портала Azure. Отличия будут описаны отдельно.

## <a name="why-deploy-site-recovery"></a>Зачем нужно развертывание Site Recovery?
Ниже перечислены преимущества использования службы Site Recovery для вашего бизнеса.

* **Упрощение стратегии BCDR.** Репликацию, отработку отказа и восстановление нескольких рабочих нагрузок можно выполнять в одном расположении на портале Azure. Site Recovery управляет репликацией и отработкой отказа, не мешая передаче данных приложений.
* **Гибкая репликация.** Вы можете реплицировать любые рабочие нагрузки, выполняющиеся на поддерживаемых виртуальных машинах Hyper-V, виртуальных машинах VMware и физических серверах Windows и Linux.
* **Устранение дополнительного центра обработки данных.** Рабочие нагрузки можно реплицировать в Azure, а не на дополнительный сайт. Это позволит сократить затраты и отказаться от обременительного обслуживания дополнительного центра обработки данных. Реплицированные данные хранятся в отказоустойчивой службе хранилища Azure. При отработке отказа виртуальные машины Azure заполняются реплицированными данными.
* **Простота тестирования репликации.** Вы можете легко выполнить тестовую отработку отказа в рамках учений по аварийному восстановлению, не влияя на рабочие среды.
* **Отработка отказа и восстановление.** Вы можете запускать запланированные процедуры отработки отказа без потери данных при ожидаемых простоях, а также незапланированные процедуры отработки отказа с минимальной потерей данных (в зависимости от частоты репликации) на случай непредвиденных сбоев. Вы сможете быстро переключиться на основной сайт, как только он снова станет доступен.
* **Отработки отказа нескольких виртуальных машин.** Вы можете настроить планы восстановления с использованием скриптов и модулей Runbook службы автоматизации Azure. Планы восстановления позволяют моделировать и настраивать процедуры отработки отказа и восстановления многоуровневых приложений, которые распределены между несколькими виртуальными машинами.
* **Интеграция с имеющимися технологиями BCDR.** Служба Site Recovery успешно интегрируется с другими компонентами технологий BCDR. Например, можно использовать Site Recovery для защиты серверной части SQL Server корпоративных рабочих нагрузок, включая встроенную поддержку для SQL Server AlwaysOn для управления отработкой отказа групп доступности.

## <a name="how-does-site-recovery-protect-workloads"></a>Как Site Recovery обеспечивает защиту рабочих нагрузок?
Служба Site Recovery реализует стратегию репликации с учетом особенностей приложений, обеспечивая согласованное выполнение рабочих нагрузок и приложений при сбоях.

* **Моментальные снимки, согласованные на уровне приложений.** Виртуальные машины реплицируются с использованием моментальных снимков, согласованных на уровне приложений для одноуровневых или многоуровневых приложений. В моментальных снимках, согласованных на уровне приложений, сохраняются не только данные, хранящиеся на диске, но и все данные в памяти и все выполняемые транзакции.
* **Практически синхронная репликация.** Частота репликации, обеспечиваемая службой Site Recovery для Hyper-V, составляет всего 30 секунд; для VMware поддерживается непрерывная репликация.
* **Гибкие планы восстановления.** Планы восстановления можно создавать и настраивать с использованием внешних скриптов и действий, выполняемых вручную. Интеграция с модулями Runbook службы автоматизации Azure позволяет выполнять восстановление всего стека приложений одним щелчком.
* **Интеграция с SQL Server AlwaysOn.** Отработкой отказа для групп доступности можно управлять с помощью планов восстановления.
* **Библиотека службы автоматизации.** Обширная библиотека службы автоматизации Azure содержит готовые к работе и учитывающие особенности приложений скрипты, которые можно скачать и интегрировать с помощью Site Recovery.
* **Простое управление сетью.** Дополнительные функции управления сетью в Site Recovery и Azure упрощают сетевые требования к приложениям, включая резервирование IP-адресов, настройку балансировщиков нагрузки и интеграцию диспетчера трафика Azure для эффективного переключения сетей.
