#  Перенос виртуальных машин Windows из Amazon Web Services (AWS) в Azure с помощью Azure Site Recovery

Служба Azure Site Recovery помогает реализовать стратегии непрерывности бизнес-процессов и аварийного восстановления, управляя процессами репликации, отработки отказа и восстановления виртуальных машин и физических серверов. Виртуальные машины можно реплицировать в Azure или во вторичный локальный центр обработки данных. Краткий обзор см. в статье [Что такое Azure Site Recovery?](site-recovery-overview.md)


## Обзор

В этой статье описано, как использовать Site Recovery для переноса или отработки отказа экземпляров Windows, работающих в AWS, в Azure. В ней кратко описаны шаги, которые полностью рассмотрены в статье [Репликация виртуальных машин VMware или физических серверов в Azure](site-recovery-vmware-to-azure-classic.md). В вышеупомянутой статье приведена последняя улучшенная версия сценария, позволяющая реплицировать виртуальные машины VMware или физические серверы Windows и Linux в Azure. Мы рекомендуем вам ознакомиться с подробными инструкциями по каждому этапу развертывания в вышеупомянутой статье.

>[AZURE.NOTE] Для переноса виртуальных машин Windows в AWS **не следует использовать** инструкции в этой [устаревшей статье](site-recovery-vmware-to-azure-classic-legacy.md).

## Приступая к работе

Ниже перечислены условия, которые необходимо выполнить перед началом работы.

- **Сервер управления**: локальная виртуальная машина под управлением Windows Server 2012 R2, которая выступает в качестве сервера управления. Установите компоненты Site Recovery (включая сервер конфигурации и сервер обработки) на этом сервере. Подробнее см. в [рекомендациях для сервера управления](site-recovery-vmware-to-azure-classic.md#management-server-considerations) и [предварительных требованиях к локальной среде](site-recovery-vmware-to-azure-classic.md#on-premises-prerequisites).
- **Экземпляры виртуальной машины EC2** — экземпляры, которые требуется перенести и защитить.

## Шаги по развертыванию

1. [Создание хранилища](site-recovery-vmware-to-azure-classic.md#step-1-create-a-vault)
2. [Развертывание сервера управления](site-recovery-vmware-to-azure-classic.md#Step-5-install-the-management-server). 
3. После развертывания сервера управления убедитесь, что он может обмениваться данными с экземплярами EC2, которые требуется перенести.
4. [Создание группы защиты](site-recovery-vmware-to-azure-classic.md#step-8-create-a-protection-group). Группа защиты содержит защищенные компьютеры, которые имеют одинаковые настройки репликации. Укажите параметры репликации для группы, и они будут применяться ко всем машинам, добавляемым в эту группу. 
5. [Установка службы Mobility Service](site-recovery-vmware-to-azure-classic.md#step-9-install-the-mobility-service). На каждую виртуальную машину, которую вы планируете защитить, требуется установить службу Mobility Service. Эта служба отправляет данные на сервер обработки. Службу Mobility Service можно установить вручную или передать и установить автоматически с помощью сервера обработки после включения защиты для виртуальной машины. Правила брандмауэра на экземплярах EC2, которые требуется перенести, необходимо настроить таким образом, чтобы разрешить принудительную установку этой службы. Группа безопасности для экземпляров EC2 должна содержать следующие правила:

	![Правила брандмауэра](./media/site-recovery-migrate-aws-to-azure/migrate-firewall.png)

6. [Включение защиты для виртуальных машин](site-recovery-vmware-to-azure-classic.md#step-10-enable-protection-for-a-machine). Добавьте виртуальные машины, которые вы хотите защитить, в группу репликации.

	![включение защиты](./media/site-recovery-migrate-aws-to-azure/migrate-add-machines.png)

7. Вы можете просмотреть экземпляры EC2, которые требуется перенести в Azure, с помощью их частных IP-адресов, которые можно получить в консоли EC2. В группе защиты можно добавить частный IP-адрес каждого экземпляра.

	![включение защиты](./media/site-recovery-migrate-aws-to-azure/migrate-machine-ip.png)

	После добавления виртуальной машины в группу защита будет включена, и запустится первоначальная репликация в соответствии с параметрами группы защиты.

8. [Запуск внеплановой отработки отказа](site-recovery-failover.md#run-an-unplanned-failover). После завершения первоначальной репликации вы можете запустить незапланированную отработку отказа из AWS в Azure для каждой виртуальной машины. При необходимости вы можете создать план восстановления и запустить внеплановую отработку отказа, чтобы перенести несколько виртуальных машин из AWS в Azure. [Узнайте подробнее](site-recovery-create-recovery-plans.md) о планах восстановления.
