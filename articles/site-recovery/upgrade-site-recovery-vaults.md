# <a name="upgrade-site-recovery-vaults-to-azure-resource-manager-based-recovery-services-vaults"></a>Обновление хранилищ Site Recovery до хранилищ служб восстановления на основе Azure Resource Manager

В этой статье описывается обновление хранилищ Site Recovery до хранилищ служб восстановления на основе Azure Resource Manager без влияния на выполняющуюся репликацию. Вы можете узнать больше о возможностях и преимуществах Azure Resource Manager [здесь](../azure-resource-manager/resource-group-overview.md).

## <a name="introduction"></a>Введение
Хранилище служб восстановления — это ресурс Azure Resource Manager для управления задачами архивации и аварийного восстановления в облаке. Это единое хранилище, которое можно использовать на новом портале Azure. Это замена классическому резервному хранилищу и хранилищу Site Recovery.

Хранилища служб восстановления предоставляют возможность воспользоваться множеством функций, включая следующие:

-   Поддержка Azure Resource Manager. Вы можете обеспечить защиту и отработку отказа виртуальных машин и физических компьютеров в стеке Azure Resource Manager.

-   Исключение диска. Если у вас имеются временные файлы или быстро изменяющиеся данные, на которые вы не хотите использовать пропускную способность, тома можно исключить из репликации. Эта возможность недавно включена в сценарии развертывания VMware в Azure и Hyper-V в Azure. Скоро она будет включена и в другие сценарии.

- Поддержка для хранилища класса Premium и локально избыточного хранилища (LRS). Теперь защиту серверов можно обеспечить в учетных записях хранения класса Premium, которые позволяют защитить приложения с высокой скоростью выполнения операций ввода-вывода. Эта возможность недавно включена в сценарий развертывания VMware в Azure.

-   Оптимизация начала работы. Теперь, когда вы приступаете к работе, у вас есть дополнительные возможности, которые помогут упростить настройку аварийного восстановления.

- Управление архивацией и Site Recovery из одного хранилища. Теперь вы можете обеспечить защиту серверов для аварийного восстановления или выполнить архивацию в одном хранилище и значительно снизить расходы на управление.

Дополнительные сведения об обновленных возможностях и компонентах см. в этом [блоге](https://azure.microsoft.com/blog/azure-site-recovery-now-available-in-a-new-experience-with-support-for-arm-and-csp/).

## <a name="salient-features"></a>Ключевые возможности

- **Отсутствие влияния на выполняющуюся репликацию.** Во время и после обновления выполняющаяся репликация продолжается без перерывов.

- **Отсутствие дополнительных затрат.** Воспользуйтесь полным набором новых возможностей без дополнительных затрат.

- **Сохранение данных.** Так как это обновление, а не миграция, имеющиеся сведения о репликации (точки восстановления, параметры репликации и т. д.) остаются неизменными во время и после обновления.


## <a name="what-happens-during-the-upgrade"></a>Что происходит во время обновления?

Выполнение операций (регистрация нового сервера, включение репликации для виртуальной машины и т. д.) во время обновления запрещено. Любые операции, предполагающие считывание или запись данных в хранилище, такие как репликация защищенных элементов в хранилище, продолжают выполняться без перерывов.

## <a name="changes-to-your-automation-and-tooling-after-vault-upgrade"></a>Изменение возможностей автоматизации и набора инструментов после обновления хранилища
При обновлении типа хранилища с классической модели на модель развертывания с помощью Resource Manager требуется обновить существующую службу автоматизации или набор инструментов, чтобы обеспечить их работу после обновления.

## <a name="preparing-your-environment-for-vault-upgrade"></a>Подготовка среды к обновлению хранилища

1.  Установите PowerShell версии 5 или более поздней версии (или выполните обновление до этой версии), используя эту [ссылку](https://www.microsoft.com/download/details.aspx?id=50395).

2.  Установите последнюю версию Azure PowerShell MSI [отсюда](https://github.com/Azure/azure-powershell/releases).

3.  [Скачайте](https://aka.ms/vaultupgradescript) сценарий для обновления хранилища.

## <a name="prerequisites-for-upgrade"></a>Предварительные требования для обновления
Для обновления хранилищ Site Recovery до хранилищ службы восстановления на основе Azure Resource Manager выполните следующие предварительные требования.

- Минимальная версия агента. Для обновления требуется установить на сервере поставщик Azure Site Recovery версии не ниже 5.1.1700.0.

- Поддерживаемая конфигурация. Не настраивайте для хранилища сеть SAN и группы доступности AlwaysOn SQL. Остальные конфигурации поддерживаются.

>[!NOTE]
> Управление сопоставлением хранилищ через PowerShell можно настроить только после обновления.

- Поддерживаемый сценарий развертывания. В вашем хранилище не должна быть реализована старая модель развертывания VMware в Azure.  Прежде чем продолжить, перейдите к расширенной модели развертывания.

- Отсутствие активных заданий, инициированных пользователем, включающих операции на плоскости управления. Так как во время обновления доступ к плоскости управления ограничен, необходимо завершить все операции на плоскости управления, а затем активировать обновление. Это не касается выполняющейся репликации.

## <a name="frequently-asked-questions"></a>Часто задаваемые вопросы

- Влияет ли обновление на выполняющуюся репликацию?

  Нет. Во время и после обновления репликация продолжается без перерыва.

- Что происходит с параметрами сети (VPN типа "сеть — сеть", параметры IP-адресов и т. д.)?

  Обновление не влияет на параметры сети. Все соединения Azure с локальной средой не меняются.
- Что произойдет с моими хранилищами, если я не планирую обновление в ближайшем будущем?

  Начиная с сентября будет прекращена поддержка хранилища Site Recovery на старом портале Azure. Поэтому мы настоятельно рекомендуем пользователям использовать функцию обновления для перехода к новому порталу.

- Как изменятся существующие средства с этим планом миграции?  

  Одним из важнейших изменений, которое следует учесть в плане обновления, является обновление набора инструментов до модели развертывания с помощью Resource Manager, на которой основаны хранилища служб восстановления.

- Сколько времени продлится простой плоскости управления?

  Он займет 15-30 минут (максимум час).

- Возможен ли откат после обновления?

  Нет. Откат после успешного обновления ресурсов не поддерживается.

- Можно ли проверить подписку или ресурсы, чтобы узнать, подходят ли они для обновления?

  Да. Первый шаг при обновлении с поддержкой платформы — убедиться, что ресурсы поддерживают обновление. В случае сбоя проверки предварительных требований отобразится соответствующая ошибка или предупреждения.

- Как сообщить о проблеме с обновлением?

  Если вы столкнулись с какими-либо сбоями во время обновления, запишите идентификатор операции, указанный в сообщении об ошибке. Служба поддержки Майкрософт примет профилактические меры для устранения данной проблемы. Вы можете обратиться в службу поддержки, указав идентификатор подписки, имя хранилища и идентификатор операции. Мы приступим к устранению проблемы как можно скорее. Не повторяйте операцию, если это явно не указано в инструкциях корпорации Майкрософт.

## <a name="how-to-run-the-script"></a>Как выполнить сценарий?

В командной строке PowerShell выполните следующую команду:

    PS > .\RecoveryServicesVaultUpgrade-1.0.0.ps1 -SubscriptionID <subscriptionID>  -VaultName <vaultname> -Location <location> -ResourceType HyperVRecoveryManagerVault -TargetResourceGroupName <rgname>

- SubscriptionID. Идентификатор подписки, связанный с обновляемым хранилищем.
- VaultName. Имя обновляемого хранилища.
- Location. Расположение обновляемого хранилища.
- ResourceType. HyperVRecoveryManagerVault для хранилищ Site Recovery.
- TargetResourceGroupName. Группа ресурсов, в которую будет помещено обновленное хранилище. TargetResourceGroupName может быть имеющейся группой ресурсов Azure Resource Manager или новой. Если указанное значение TargetResourceGroupName не существует, оно создается в рамках обновления в том же расположении, что и хранилище. Дополнительные сведения о группах ресурсов см. [здесь](../azure-resource-manager/resource-group-overview.md#resource-groups):

>[!NOTE]
>Имена групп ресурсов имеют ограничения. Следуйте им, так как их невыполнение может привести к сбою обновления хранилища.

Пример:

    .\RecoveryServicesVaultUpgrade-1.0.0.ps1 -SubscriptionId 1234-54123-354354-56416-8645 -VaultName gen2dr -Location "north europe" -ResourceType hypervrecoverymanagervault -TargetResourceGroupName abc


Кроме того, можно выполнить сценарий, как показано ниже, и вам будет предложено задать входные значения всех необходимых параметров.

    PS > .\RecoveryServicesVaultUpgrade-1.0.0.ps1
    cmdlet RecoveryServicesVaultUpgrade-1.0.0.ps1 at command pipeline position 1

    Supply values for the following parameters:
    SubscriptionId:
    VaultName:
    Location:
    ResourceType:
    TargetResourceGroupName:

1.  Сценарий PowerShell предложит ввести учетные данные. Учетные данные нужно ввести дважды — для учетной записи ASM и учетной записи Azure Resource Manager.

2.  После ввода учетных данных сценарий запускает проверку предварительных требований, чтобы убедиться, что настройка инфраструктуры соответствует необходимым требованиям, указанным в документе выше.

3.  После этой проверки вам будет предложено подтвердить продолжение обновления хранилища. После подтверждения начнется обновление хранилища. Это может занять 15–30 минут.

4.  По завершении обновления можно войти в обновленное хранилище на новом портале Azure.

### <a name="how-to-view-your-replication-settings"></a>Просмотр параметров репликации

В хранилище Site Recovery для каждой группы защиты настроены параметры репликации (частота копирования, хранение точки восстановления, согласованные на уровне приложения моментальные снимки и т. д.). В хранилище служб восстановления эти параметры настраиваются как политика репликации. Имя политики является именем группы защиты или primarycloud_Policy.

Дополнительные сведения о политике репликации см. [здесь](site-recovery-setup-replication-settings-vmware.md).
