#  Настройка защиты с одним сервером VMM

## Обзор

Служба Azure Site Recovery помогает реализовать стратегии непрерывности бизнеса и восстановления после сбоев (BCDR), управляя процессами репликации, отработки отказа и восстановления виртуальных машин в ряде сценариев развертывания. Полный список сценариев развертывания см. в статье [Обзор Azure Site Recovery](site-recovery-overview.md).

При наличии в инфраструктуре только одного сервера VMM можно развернуть Site Recovery для репликации виртуальных машин из облаков VMM в Azure или организовать репликацию между облаками на одном сервере VMM. Это рекомендуется делать только в том случае, если не удается развернуть два сервера VMM (по одному на каждом сайте), поскольку отработка отказа и восстановление в таком развертывании затруднены. Для восстановления потребуется вручную выполнить отработку отказа сервера VMM вне консоли Azure Site Recovery (с помощью реплики Hyper-V в консоли диспетчера Hyper-V).

Настроить репликацию с помощью одного сервера VMM можно несколькими способами.

### Автономное развертывание

Разверните автономный сервер VMM в качестве виртуальной машины на основном сайте и реплицируйте эту виртуальную машину на дополнительный сайт с помощью Site Recovery и реплики Hyper-V. Чтобы сократить время простоя, сервер SQL Server можно установить на виртуальную машину VMM. Если VMM использует удаленный экземпляр SQL Server, прежде чем восстанавливать сервер VMM, потребуется сначала восстановить этот экземпляр.

![Автономный виртуальный сервер VMM](./media/site-recovery-single-vmm/SingleVMMStandalone.png)

### Развертывание кластера

Для обеспечения высокой доступности VMM его можно развернуть в качестве виртуальной машины в отказоустойчивом кластере Windows. Это полезно в случаях, если для управления критическими рабочими нагрузками используется VMM, поскольку при этом гарантируется доступность рабочей нагрузки и защита от отработки отказа оборудования узла, на котором выполняется VMM. Для развертывания одного сервера VMM с помощью Site Recovery виртуальная машина VMM должна быть развернута на кластере, распределенном по географически удаленным узлам. База данных SQL Server, используемая VMM, должна быть защищена с помощью групп доступности SQL Server AlwaysOn с репликой на дополнительном сайте. В случае аварии сервер VMM и его соответствующая база данных SQL Server переносятся на дополнительный сайт, где они становятся доступны.

![Кластеризованный виртуальный сервер VMM](./media/site-recovery-single-vmm/SingleVMMCluster.png)


## Перед началом работы

- Ниже приведены пошаговые инструкции по развертыванию Site Recovery с одним автономным сервером VMM.
- Прежде чем приступить к развертыванию, убедитесь, что у вас есть все [необходимые компоненты](site-recovery-vmm-to-vmm.md/#before-you-start).
- На каждом сервере VMM должно быть настроено по крайней мере два облака. Одно облако будет выступать в качестве защищаемого облака, а другое выполняет функции защиты.
- Облака, для которых вы настраиваете защиту, должны содержать следующие компоненты:
	- Одна или несколько групп узлов VMM
	- один или несколько серверов узлов Hyper-V в каждой группе узлов;
	- одну или несколько виртуальных машин Hyper-V на каждом сервере узла.

Если у вас возникают проблемы, опубликуйте свои вопросы на [форуме по Azure Recovery Services](https://social.msdn.microsoft.com/Forums/azure/home?forum=hypervrecovmgr).



## Настройка развертывания одного сервера

1. Если VMM не развернут, установите VMM на виртуальную машину с базой данных SQL Server. Ознакомьтесь с [требованиями к системе](https://technet.microsoft.com/library/dn771747.aspx). 
2. Настройте на сервере хотя бы два облака. Дополнительные сведения:

	- [Новые функции частного облака в System Center 2012 R2 VMM](http://channel9.msdn.com/Events/TechEd/NorthAmerica/2013/MDC-B357#fbid=) и [VMM 2012 и облачные среды](http://www.server-log.com/blog/2011/8/26/vmm-2012-and-the-clouds.html). 
	- [Настройка структуры облака VMM](https://msdn.microsoft.com/library/azure/dn469075.aspx#BKMK_Fabric).
	- [Создание частного облака в VMM](https://technet.microsoft.com/library/jj860425.aspx) и [Пошаговое руководство: создание частных облаков с помощью VMM в System Center 2012 с пакетом обновления 1 (SP1)](http://blogs.technet.com/b/keithmayer/archive/2013/04/18/walkthrough-creating-private-clouds-with-system-center-2012-sp1-virtual-machine-manager-build-your-private-cloud-in-a-month.aspx).
3. Добавьте исходный сервер узла Hyper-V, на котором находится подлежащая защите виртуальная машина, в облако, которое вы собираетесь защитить (в облако-источник). Добавьте целевой сервер узла Hyper-V в облако на сервере VMM, который будет обеспечивать защиту.
4. [Создайте](site-recovery-vmm-to-vmm.md/#step-1-create-a-site-recovery-vault) хранилище Azure Site Recovery и создайте ключ регистрации хранилища.
4. [Установите](site-recovery-vmm-to-vmm.md/#step-3-install-the-azure-site-recovery-provider) поставщик Azure Site Recovery на сервер VMM и зарегистрируйте сервер в хранилище. 
5. Убедитесь, что облака отображаются на портале Site Recovery, а затем [настройте параметры защиты облака](site-recovery-vmm-to-vmm.md/#step-4-configure-cloud-protection-settings).
	- В полях **Исходное расположение** и **Целевое расположение** укажите имя одного VMM-сервера.
	- В раскрывающемся списке **Метод репликации** выберите **Over the network** (По сети) для начальной репликации, поскольку облака расположены на одном и том же сервере.

6. Дополнительно можно [настроить сетевое сопоставление](site-recovery-vmm-to-vmm.md/#step-5-configure-network-mapping).

	- В полях **Источник** и **Назначение** укажите имя сервера VMM.
	- В поле **Network on Source** (Сеть источника) выберите сеть виртуальной машины, которая настроена для защищаемого облака.
	- В поле **Network on Target** (Сеть назначения) выберите сеть виртуальной машины, настроенную для облака, которое необходимо использовать для защиты.
	- Можно настроить сетевое сопоставление между двумя сетями виртуальных машин на том же сервере VMM. Если одна и та же сеть VMM существует на двух разных сайтах, можно настроить сопоставление между такими сетями.
7. [Включите защиту](site-recovery-vmm-to-vmm.md/#step-7-enable-virtual-machine-protection) для виртуальных машин в облаке VMM, которое необходимо защитить. 
7. В консоли диспетчера Hyper-V настройте репликацию для виртуальной машины VMM с репликой Hyper-V. Виртуальную машину VMM не следует добавлять в любые облака VMM.


## Отработка отказа и восстановление

### Создание планов восстановления

Планы восстановления позволяют сгруппировать виртуальные машины, для которых совместно следует выполнять отработку отказа и восстановление.

1. При создании плана восстановления в полях **Источник** и **Назначение** укажите имя сервера VMM. В разделе **Выберите виртуальные машины** будут отображаться виртуальные машины, которые связаны с основным облаком.
2. Затем [создайте и настройте планы восстановления](https://msdn.microsoft.com/library/azure/dn337331.aspx).


### Восстановление

В случае аварии рабочие нагрузки можно восстановить, выполнив указанные ниже действия.

1. Вручную выполните отработку отказа реплики виртуальной машины VMM на сайт восстановления из консоли диспетчера Hyper-V.
2. После восстановления виртуальной машины VMM можно войти в консоли диспетчера восстановления Hyper-V на портале и выполнить незапланированную отработку отказа виртуальных машин с основного сайта на сайт восстановления.
3.  После завершения незапланированной отработки отказа пользователям будут доступны все ресурсы на основном сайте.
