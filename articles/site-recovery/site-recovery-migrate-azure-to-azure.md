#  Перенос виртуальных машин IaaS Azure между регионами Azure с помощью Azure Site Recovery

## Обзор

В этой статье описывается, как использовать Site Recovery для переноса виртуальных машин Azure из одного региона Azure в другой. Перед началом работы обратите внимание на следующее:

- Сейчас возможен только перенос. Это означает, что вы можете выполнить отработку отказа виртуальных машин из одного региона Azure в другой, но не сможете выполнить обратную отработку отказа этих виртуальных машин.
- В этой статье кратко описаны и использованы многие действия, полностью описанные в разделе [Репликация виртуальных машин VMware и физических серверов в Azure с помощью службы Azure Site Recovery](site-recovery-vmware-to-azure-classic.md), который предоставляет последние доработанные инструкции по настройке репликации. Мы рекомендуем при переносе следовать подробным инструкциям в этой статье.
- Для переноса **больше не следует использовать** инструкции в этой [устаревшей статье](site-recovery-vmware-to-azure-classic-legacy.md).

Комментарии или вопросы можно добавить в конце этой статьи или на [форуме по службам восстановления Azure](https://social.msdn.microsoft.com/forums/azure/home?forum=hypervrecovmgr).


## Предварительные требования

Вот что нужно для этого развертывания:

- **Сервер управления**: виртуальная машина под управлением Windows Server 2012 R2, которая выступает в качестве сервера управления. Установите компоненты Site Recovery (включая сервер конфигурации и сервер обработки) на этом сервере. Подробнее см. в [рекомендациях для сервера управления](site-recovery-vmware-to-azure-classic.md#management-server-considerations) и [предварительных требованиях к источнику](site-recovery-vmware-to-azure-classic.md#on-premises-prerequisites).
- **Виртуальные машины IaaS** — виртуальные машины, которые требуется перенести.

## Шаги по развертыванию

1. [Создайте хранилище](site-recovery-vmware-to-azure-classic.md#step-1-create-a-vault).
2. [Разверните сервер управления](site-recovery-vmware-to-azure-classic.md#Step-5-install-the-management-server).
3. После развертывания сервера управления убедитесь, что он может обмениваться данными с виртуальными машинами, которые требуется перенести.
4. [Создание группы защиты](site-recovery-vmware-to-azure-classic.md#step-8-create-a-protection-group). Группа защиты содержит защищенные компьютеры, которые имеют одинаковые настройки репликации. Укажите параметры репликации для группы, и они будут применяться ко всем компьютерам, добавляемым в эту группу.
5. [Установите Mobility Service](site-recovery-vmware-to-azure-classic.md#step-9-install-the-mobility-service). На каждую виртуальную машину, которую вы планируете защитить, требуется установить службу Mobility Service. Эта служба отправляет данные на сервер обработки. Службу Mobility Service можно установить вручную или передать и установить автоматически с помощью сервера обработки после включения защиты для виртуальной машины. Обратите внимание, что правила брандмауэра на виртуальных машинах IaaS, которые требуется перенести, необходимо настроить таким образом, чтобы разрешить принудительную установку этой службы.
6. [Включение защиты для виртуальных машин](site-recovery-vmware-to-azure-classic.md#step-10-enable-protection-for-a-machine). Добавьте виртуальные машины, которые вы хотите защитить, в группу репликации. 
7. Вы можете просмотреть виртуальные машины IaaS, которые требуется перенести в Azure, с помощью их частных IP-адресов. Этот адрес можно найти на панели мониторинга виртуальной машины в Azure.
8. На вкладке созданной вами группы защиты щелкните **Добавить компьютеры** > **Физические компьютеры**.

	![обнаружение EC2](./media/site-recovery-migrate-azure-to-azure/migrate-add-machines.png)

9. Укажите частный IP-адрес виртуальной машины.

	![обнаружение EC2](./media/site-recovery-migrate-azure-to-azure/migrate-machine-ip.png)
	
	После добавления виртуальной машины в группу защита будет включена, и запустится первоначальная репликация в соответствии с параметрами группы защиты.

10. [Запустите внеплановую отработку отказа](site-recovery-failover.md#run-an-unplanned-failover). После завершения начальной репликации вы можете запустить внеплановую отработку отказа из одного региона Azure в другой. При необходимости вы можете создать план восстановления и запустить внеплановую отработку отказа, чтобы перенести несколько виртуальных машин между регионами. [Узнайте подробнее](site-recovery-create-recovery-plans.md) о планах восстановления.
