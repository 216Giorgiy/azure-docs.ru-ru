# <a name="create-a-virtual-network-using-the-azure-cli"></a>Создание виртуальной сети с помощью интерфейса командной строки Azure

В этой статье вы узнаете, как создать виртуальную сеть. После создания виртуальной сети в ней развертываются две виртуальные машины для тестирования взаимодействия между ними в частной сети.

Если у вас еще нет подписки Azure, [создайте бесплатную учетную запись Azure](https://azure.microsoft.com/free/?WT.mc_id=A261C142F), прежде чем начинать работу.

[!INCLUDE [cloud-shell-try-it.md](../../includes/cloud-shell-try-it.md)]

Если вы решили установить и использовать интерфейс командной строки локально, для работы с этой статьей вам понадобится Azure CLI 2.0.4 или более поздней версии. Выполните командлет `az --version`, чтобы узнать установленную версию. Если вам необходимо выполнить установку или обновление, см. статью [Установка Azure CLI 2.0](/cli/azure/install-azure-cli). 

## <a name="create-a-resource-group"></a>Создание группы ресурсов

Создайте группу ресурсов с помощью команды [az group create](/cli/azure/group#az_group_create). Группа ресурсов — это логический контейнер, в котором происходит развертывание ресурсов Azure и управление ими. 

В следующем примере создается группа ресурсов с именем *myResourceGroup* в расположении *eastus*. Все ресурсы Azure создаются в расположении Azure (или регионе).

```azurecli-interactive 
az group create --name myResourceGroup --location eastus
```

## <a name="create-a-virtual-network"></a>Создать виртуальную сеть

Создайте виртуальную сеть с помощью команды [az network vnet create](/cli/azure/network/vnet#az_network_vnet_create). В следующем примере создается виртуальная сеть по умолчанию *myVirtualNetwork* с подсетью *default*. Так как расположение не указано, Azure создает виртуальную сеть в расположении, в котором находится группа ресурсов.

```azurecli-interactive 
az network vnet create \
  --name myVirtualNetwork \
  --resource-group myResourceGroup \
  --subnet-name default
```

После создания виртуальной сети отображается часть сведений, как показано ниже.

```azurecli
"newVNet": {
    "addressSpace": {
      "addressPrefixes": [
        "10.0.0.0/16"
```

Каждой виртуальной сети назначается один или несколько префиксов адреса. Так как при создании виртуальной сети не был указан префикс адреса, платформа Azure определила диапазон адресов по умолчанию — 10.0.0.0/16. Диапазон адресов указывается в нотации CIDR. Диапазон адресов 10.0.0.0/16 охватывает адреса 10.0.0.0–10.0.255.254.

Также в отображаемых сведениях указано, что для подсети *default*, указанной в команде, параметр **addressPrefix** имеет значение *10.0.0.0/24*. Виртуальная сеть может содержать подсети. Команда создала отдельную подсеть *default*, но для нее не был указан префикс адреса. Если для виртуальной сети или подсети не указан префикс адреса, по умолчанию Azure определяет для первой подсети префикс адреса 10.0.0.0/24. В результате подсеть охватывает адреса 10.0.0.0–10.0.0.254, однако из них доступны только 10.0.0.4–10.0.0.254, так как Azure резервирует первые четыре адреса (0–3) и последний адрес в каждой подсети.

## <a name="test-network-communication"></a>Тестирование взаимодействия по сети

Виртуальная сеть позволяет ресурсам Azure разных типов взаимодействовать друг с другом в частном порядке. Одним из таких ресурсов, который можно развернуть в виртуальной сети, является виртуальная машина. Создайте в виртуальной сети две виртуальные машины, чтобы в дальнейшем можно было проверить взаимодействие между ними.

### <a name="create-virtual-machines"></a>Создание виртуальных машин

Создайте виртуальную машину, выполнив команду [az vm create](/cli/azure/vm#az_vm_create). В следующем примере создается виртуальная машина *myVm1*. Также команда создает ключи SSH, если они не существуют в расположении ключей по умолчанию. Чтобы использовать определенный набор ключей, используйте параметр `--ssh-key-value`. Параметр `--no-wait` позволяет создать виртуальную машину в фоновом режиме, чтобы можно было перейти к следующему шагу.

```azurecli-interactive 
az vm create \
  --resource-group myResourceGroup \
  --name myVm1 \
  --image UbuntuLTS \
  --generate-ssh-keys \
  --no-wait
```

Azure автоматически создает виртуальную машину в подсети *default* виртуальной сети *myVirtualNetwork*, так как эта виртуальная сеть расположена в группе ресурсов и в команде не указана определенная виртуальная сеть или подсеть. Azure DHCP автоматически назначает адрес 10.0.0.4 виртуальной машине во время ее создания, так как это первый доступный адрес в подсети *default*. Расположение, в котором создается виртуальная машина, должно совпадать с расположением виртуальной сети. Виртуальная машина не обязательно должна быть в той же группе ресурсов, что и виртуальная сеть, хотя в данной статье это так.

Создайте вторую виртуальную машину. По умолчанию Azure также создает эту виртуальную машину в подсети *default*.

```azurecli-interactive 
az vm create \
  --resource-group myResourceGroup \
  --name myVm2 \
  --image UbuntuLTS \
  --generate-ssh-keys
```

Создание виртуальной машины занимает несколько минут. После создания виртуальной машины Azure CLI возвращает выходные данные следующего вида. 

```azurecli 
{
  "fqdns": "",
  "id": "/subscriptions/00000000-0000-0000-0000-000000000000/resourceGroups/myResourceGroup/providers/Microsoft.Compute/virtualMachines/myVm1",
  "location": "eastus",
  "macAddress": "00-0D-3A-23-9A-49",
  "powerState": "VM running",
  "privateIpAddress": "10.0.0.5",
  "publicIpAddress": "40.68.254.142",
  "resourceGroup": "myResourceGroup"
}
```

В приведенном примере можно видеть, что параметр **privateIpAddress** имеет значение *10.0.0.5*. Служба Azure DHCP автоматически назначила адрес *10.0.0.5* виртуальной машине, так как это первый доступный адрес в подсети *default*. Запишите значение **publicIpAddress**. Позже этот адрес будет использоваться для доступа к виртуальной машине из Интернета. Общедоступный IP-адрес не назначается в пределах префиксов адреса виртуальной сети или подсети. Общедоступные IP-адреса назначаются из [пула адресов, назначенных каждому региону Azure](https://www.microsoft.com/download/details.aspx?id=41653). Хотя Azure знает, какой общедоступный IP-адрес назначен виртуальной машине, операционная система на этой виртуальной машине не знает его.

### <a name="connect-to-a-virtual-machine"></a>Подключение к виртуальной машине

Используйте следующую команду для создания сеанса SSH с виртуальной машиной *myVm2*. Замените `<publicIpAddress>` общедоступным IP-адресом виртуальной машины. В примере выше используется IP-адрес *40.68.254.142*.

```bash 
ssh <publicIpAddress>
```

### <a name="validate-communication"></a>Проверка связи

Используйте приведенную ниже команду, чтобы проверить связь с *myVm1* из *myVm2*.

```bash
ping myVm1 -c 4
```

Вы получите четыре ответа с адреса *10.0.0.4*. Взаимодействие *myVm2* с *myVm1* возможно, так как обе виртуальные машины имеют частные IP-адреса, назначенные из подсети *default*. Вы можете проверить связь по имени узла, так как Azure автоматически обеспечивает разрешение DNS-имен для всех узлов в виртуальной сети.

Чтобы подтвердить возможность исходящих подключений к Интернету, введите следующую команду.

```bash
ping bing.com -c 4
```

Вы получите четыре ответа с сайта bing.com. По умолчанию любая виртуальная машина в виртуальной сети может устанавливать исходящие подключения к Интернету.

Выйдите из сеанса SSH и вернитесь к виртуальной машине.

## <a name="clean-up-resources"></a>Очистка ресурсов

Вы можете удалить ненужную группу ресурсов и все содержащиеся в ней ресурсы, выполнив команду [az group delete](/cli/azure/group#az_group_delete).

```azurecli-interactive 
az group delete --name myResourceGroup --yes
```
