# <a name="network-interfaces-in-azure"></a>Сетевые интерфейсы в Azure
Сетевой интерфейс обеспечивает взаимодействие между виртуальной машиной и сетью с базовым программным обеспечением. В этой статье описывается сетевой интерфейс и его применение в рамках модели развертывания Azure Resource Manager.

Хотя корпорация Майкрософт рекомендует развертывать новые ресурсы с использованием модели Resource Manager, виртуальные машины с сетевым подключением можно также развернуть с помощью [классической](virtual-network-ip-addresses-overview-classic.md) модели. Если вы знакомы с классической моделью развертывания, учтите, что в модели Resource Manager обеспечение обмена данными между виртуальными машинами отличается. Узнать об этих различиях можно в статье, посвященной [работе виртуальных машин в сети в рамках классической модели развертывания](virtual-network-ip-addresses-overview-classic.md#differences-between-resource-manager-and-classic-deployments) .

Сетевой интерфейс в Azure:

1. Это ресурс, который можно создать или удалить, а также настроить.
2. При создании должен быть подключен к одной подсети в одной виртуальной сети Azure. Если вы не работали с виртуальными сетями, см. этот [обзор виртуальных сетей](virtual-networks-overview.md). Сетевой интерфейс подключается к виртуальной сети, созданной в том же [расположении](https://azure.microsoft.com/regions) Azure и в той же [подписке](../azure-glossary-cloud-terminology.md#subscription), что и сам сетевой интерфейс. Для созданного сетевого интерфейса можно изменить подсеть, к которой он подключен, но нельзя изменить виртуальную сеть.
3. Имеет назначенное имя, которое нельзя изменить после создания. Имя должно быть уникальным в пределах [группы ресурсов](../azure-resource-manager/resource-group-overview.md#resource-groups)Azure, но не обязательно уникальным в пределах подписки, расположения Azure или виртуальной сети, к которой он подключен. Обычно в подписке Azure создается несколько сетевых интерфейсов. Мы рекомендуем разработать соглашение об именовании, которое упростит управление несколькими сетевыми адаптерами (в сравнении с использованием имен по умолчанию). Некоторые рекомендации представлены в статье [Recommended naming conventions for Azure resources](../guidance/guidance-naming-conventions.md) (Рекомендуемые соглашения об именовании для ресурсов Azure).
4. Может быть подключен к виртуальной машине, но только к одной и обязательно созданной в том же расположении, что и сам сетевой интерфейс.
5. Имеет MAC-адрес, который закрепляется за сетевым интерфейсом до тех пор, пока он подключен к виртуальной машине. MAC-адрес сохраняется при перезапуске виртуальной машины (с помощью функций операционной системы) или при остановке (освобождении) с последующим запуском через портал Azure, Azure PowerShell или интерфейс командной строки Azure. Если отключить сетевой интерфейс от виртуальной машины и подключить его к другой виртуальной машине, он получит новый MAC-адрес. После удаления сетевого интерфейса его MAC-адрес может быть назначен другому сетевому интерфейсу.
6. Должен иметь один основной **частный** *IPv4* , присваиваемый статически или динамически.
7. Может иметь один присвоенный общедоступный IP-адрес.
8. Поддерживает ускорение сети с технологией виртуализации операций ввода-вывода с единым корнем (SR-IOV) для некоторых размеров виртуальных машин под управлением определенных версий ОС Microsoft Windows Server. Дополнительные сведения об этой функции предварительной версии можно получить в статье [Accelerated networking for a virtual machine](virtual-network-accelerated-networking-powershell.md) (Ускорение сети для виртуальной машины).
9. Может получать трафик, не предназначенный для присвоенных ему частных IP-адресов, если для этого сетевого интерфейса включена IP-пересылка. Например, если на виртуальной машине запущен программный брандмауэр, интерфейс управляет маршрутизацией пакетов, не предназначенных для его IP-адресов. Для этого на виртуальной машине должна работать специальная программа для маршрутизации и пересылки трафика, но сначала необходимо включить IP-пересылку для сетевого интерфейса.
10. Часто создается в той же группе ресурсов, что и соответствующая виртуальная машина, или в той же виртуальной сети, к которой он подключен, хотя это не обязательно.

## <a name="vms-with-multiple-network-interfaces"></a>Виртуальные машины с несколькими сетевыми интерфейсами
К виртуальной машине можно подключить несколько сетевых интерфейсов, но при этом нужно учитывать следующее:  

* Размер виртуальной машины должен поддерживать использование нескольких сетевых интерфейсов. Чтобы узнать, какие размеры виртуальных машин поддерживают несколько сетевых интерфейсов, ознакомьтесь со статьями о размерах виртуальных машин [под управлением Windows Server](../virtual-machines/virtual-machines-windows-sizes.md) или [Linux](../virtual-machines/virtual-machines-linux-sizes.md).
* Виртуальная машина должна быть создана по крайней мере с двумя сетевыми интерфейсами. Если виртуальная машина создана только с одним сетевым интерфейсом, то, даже если ее размер поддерживает несколько сетевых интерфейсов, вам не удастся подключить к ней дополнительные сетевые интерфейсы. Если же виртуальная машина была создана по крайней мере с двумя сетевыми интерфейсами, то к ней можно будет подключить дополнительные сетевые интерфейсы, если ее размер поддерживает более двух сетевых интерфейсов.  
* Если к виртуальной машине подключено по крайней мере три сетевых интерфейса, то вы можете отключить от нее дополнительные сетевые интерфейсы (основной сетевой интерфейс отключить невозможно). Вы не сможете отключить какие-либо сетевые интерфейсы, если к виртуальной машине подключено не больше двух сетевых интерфейсов.  
* Порядок сетевых адаптеров внутри ВМ определяется случайным образом и может измениться по мере обновления инфраструктуры Azure. При этом IP-адреса и соответствующие MAC-адреса Ethernet останутся прежними. Например, предположим, что операционная система определяет Azure NIC1 как Eth1. У Eth1 IP-адрес 10.1.0.100 и MAC-адрес 00-0D-3A-B0-39-0D. После обновления инфраструктуры Azure и перезагрузки операционная система сможет идентифицировать Azure NIC1 как Eth2, но IP-адрес и MAC-адрес останутся такими же, как и при определении операционной системой Azure NIC1 как Eth1. Если перезагрузка инициируется пользователем, то порядок сетевых интерфейсов в операционной системе не меняется.  
* Если виртуальная машина входит в состав [группы доступности](../azure-glossary-cloud-terminology.md#availability-set), то у всех виртуальных машин в этой группе доступности должен быть либо один, либо несколько сетевых интерфейсов. Если у виртуальных машин по несколько сетевых интерфейсов, то их количество не обязательно должно совпадать, пока у каждой виртуальной машины имеется по крайней мере два сетевых интерфейса.
