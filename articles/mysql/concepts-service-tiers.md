# <a name="azure-database-for-mysql-options-and-performance-understand-whats-available-in-each-service-tier"></a>Параметры и производительность базы данных Azure для MySQL: возможности разных уровней служб
База данных Azure для MySQL предлагает два уровня служб: "Базовый" и "Стандартный". Уровень "Премиум" пока недоступен.

Каждый из них обеспечивает несколько уровней производительности для выполнения требований различных типов рабочих нагрузок. Чем выше уровень производительности, тем больше он предоставляет ресурсов, предназначенных для повышения пропускной способности. Вы можете изменять уровни производительности в пределах уровня служб динамически и без простоя приложений.

В будущем будет возможность повышения или понижения с одного уровня служб на другой. 

> [!IMPORTANT]
> В настоящее время данная служба доступна в виде общедоступной предварительной версии, и поэтому еще не обеспечивает выполнения соглашения об уровне обслуживания (SLA).

Можно создать отдельный сервер базы данных Azure для MySQL с выделенными ресурсами на уровне служб с определенным уровнем производительности. Затем можно создать одну или несколько баз данных на сервере, на котором ресурсы совместно используются несколькими базами данных. Ресурсы, доступные для отдельного сервера базы данных Azure для MySQL, выражаются в виде единиц вычислений и единиц хранения. Дополнительные сведения о единицах вычислений и хранения см. с статье [Общие сведения о единице вычислений и единице хранения](concepts-compute-unit-and-storage.md).

## <a name="choosing-a-service-tier"></a>Выбор уровня службы

В следующей таблице приведены примеры уровней службы, которые лучше всего подходят для рабочих нагрузок различных приложений.

| Уровень служб | Целевые рабочие нагрузки |
| :----------- | :----------------|
| Basic | Лучше всего подходит для небольших рабочих нагрузок, при которых требуется увеличение числа единиц вычислений и хранения, но не требуется гарантированное число операций ввода-вывода в секунду. В качестве примера можно привести серверы, используемые для разработки или тестирования, или небольшие редко используемые приложения. |
| Standard | Идеально подходит для облачных приложений, требующих гарантированного числа операций ввода-вывода в секунду с возможностью увеличивать число единиц вычислений и хранения независимо друг от друга, обеспечивая высокую пропускную способность. В качестве примера можно привести аналитические или веб-приложения. |
| Premium | Оптимально подходит для рабочих нагрузок, требующих минимальной задержки при обработке транзакций и операций ввода-вывода. При этом обеспечиваются высокие показатели числа операций ввода-вывода и пропускной способности. Поддерживает наибольшее число параллельных пользователей. Подходит для баз данных, поддерживающих критически важные приложения.<br />Уровень служб "Премиум" недоступен в предварительной версии. |


Чтобы выбрать уровень служб, сначала определите, требуется ли для ваших рабочих нагрузок гарантированное число операций ввода-вывода в секунду. Затем определите минимальный набор необходимых функций:

| **Возможности уровня служб** | **базовая;** | **Стандартный** | **Премиум** * |
| :------------------------ | :-------- | :----------- | :------------ |
| Максимальное число единиц вычислений | 100 | 2 000 | Недоступно в предварительной версии |
| Максимальный общий объем хранилища | 1050 ГБ | 10 000 ГБ | Недоступно в предварительной версии |
| Гарантированное число операций ввода-вывода в секунду, поддерживаемое хранилищем | Недоступно | Да | Недоступно в предварительной версии |
| Максимальное число операций ввода-вывода в секунду, поддерживаемое хранилищем | Недоступно | 3000 | Недоступно в предварительной версии |
| Срок хранения резервной копии | 7 дней | 35 дней | 35 дней |

> [!NOTE]
> В настоящее время уровень служб "Стандартный" в предварительной версии поддерживает до 800 единиц вычислений и объем хранилища не более 1000 ГБ.

Выбрав минимальный уровень служб, переходите к определению уровня производительности, который необходим для сервера базы данных Azure для MySQL (в единицах вычислений). Как правило, 200 и 400 единиц вычислений являются хорошей отправной точкой для приложений, в которых веб-нагрузки или аналитика требуют обработки большого числа параллельных пользователей. 

Тем не менее, вы можете масштабировать единицы вычислений независимо от единиц хранения, исходя из потребностей рабочей нагрузки. Если рабочая нагрузка требует корректировки вычислительных ресурсов, то можно динамически увеличить или уменьшить число единиц вычислений. Если для рабочей нагрузки требуется больше операций ввода-вывода или больший объем хранилища, то также можно масштабировать хранилище.

> [!NOTE]
> В предварительной версии уровни "Базовый" и "Стандартный" не поддерживают динамическое масштабирование хранилища. Мы планируем добавить эту функцию в будущем.

> [!NOTE]
> На уровне служб "Стандартный" число операций ввода-вывода в секунду масштабируется пропорционально подготовленному размеру хранилища в фиксированной пропорции 3:1. Хранилище объемом 125 ГБ гарантирует 375 подготовленных операций ввода-вывода в секунду с объемом ввода-вывода до 256 КБ. Если подготовить объем 1000 ГБ, то вы получите 3000 подготовленных операций ввода-вывода в секунду. Необходимо отслеживать потребление единиц вычислений на сервере и выполнять масштабирование, чтобы полностью использовать подготовленные операции ввода-вывода в секунду.

## <a name="service-tiers-and-performance-levels"></a>Уровни служб и уровни производительности

База данных Azure для MySQL предлагает несколько уровней производительности в пределах каждого уровня служб. У вас есть возможность выбирать уровень, который лучше всего соответствует требованиям вашей рабочей нагрузки. Используйте одно из следующих средств:
- [Портал Azure](quickstart-create-mysql-server-database-using-azure-portal.md)
- [Интерфейс командной строки Azure](quickstart-create-mysql-server-database-using-azure-cli.md)

Вне зависимости от количества баз данных, размещаемых на каждом сервере MySQL, вашей базе данных будет выделен гарантированный набор ресурсов. Кроме того, ожидаемые показатели производительности сервера останутся без изменений.

\* Максимальный размер хранилища сервера соответствует максимальному подготовленному размеру хранилища вашего сервера.

## <a name="scaling-up-or-down-a-single-server"></a>Увеличение или уменьшение масштаба отдельного сервера

Выбрав уровень служб и уровень производительности, вы сможете динамически увеличивать или уменьшать масштаб сервера в соответствии с требованиями рабочих нагрузок. Если вам нужно увеличить или уменьшить масштаб, вы легко можете изменить уровень базы данных с помощью портала Azure или Azure CLI.

При изменении уровня службы или уровня производительности базы данных создается реплика исходной базы данных с новым уровнем производительности, после чего устанавливаются подключения к реплике. Во время этого процесса данные не теряются, но в течение короткого периода, когда производится переподключение к реплике, соединения с базой данных становятся неактивны, поэтому некоторые текущие транзакции могут откатиться. Этот период может быть разным, но обычно он не превышает 4 секунд, а в более чем 99 % случаев его длительность составляет менее 30 секунд. Если в момент отключения подключений выполняется большое количество транзакций, этот период может быть длиннее.

Длительность всего процесса вертикального масштабирования зависит от размера и уровня служб сервера до и после изменения. Например, если изменяется число единиц вычислений и сервер переводится с уровня службы "Стандартный", на этот уровень или в его пределах, то процесс должен завершиться в течение нескольких минут. Новые свойства сервера не применяются до тех пор, пока изменение не завершится.

### <a name="details-about-scaling-up-or-down"></a>Дополнительные сведения об увеличении и уменьшении масштаба

- При снижении уровня сервера число единиц хранения этого сервера не должно превышать максимально допустимое число целевого уровня служб.
- Предложения службы восстановления отличаются для разных уровней служб. При переходе на более низкий уровень можно потерять возможность восстановления на момент времени или получить меньший срок хранения резервных копий. Дополнительные сведения см. в статье [Как восстановить сервер в базе данных Azure для MySQL](howto-restore-server-portal.md).
- Новые свойства сервера не применяются до тех пор, пока изменение не завершится.
