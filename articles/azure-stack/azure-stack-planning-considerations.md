# <a name="planning-considerations-for-azure-stack-integrated-systems"></a>Рекомендации относительно планирования интегрированных систем Azure Stack

*Область применения: интегрированные системы Azure Stack*

Если вы заинтересованы в интегрированной системе Azure Stack, следует рассмотреть некоторые рекомендации относительно планирования развертывания и внедрения системы в центр обработки данных. В этой статье представлен общий обзор таких рекомендаций.

Если вы решите приобрести интегрированную систему, ваш поставщик от изготовителя оборудования (OEM) проведет вас по большой части этапов процесса планирования. Он также выполнит фактическое развертывание.

## <a name="management-considerations"></a>Рекомендации по управлению

Azure Stack является закрытой системой, в которой инфраструктура заблокирована как с точки зрения разрешений, так и с точки зрения сети. Для блокировки всего несанкционированного входящего трафика и всей ненужной связи между компонентами инфраструктуры применяются списки управления доступом к сети (ACL). Это затрудняет несанкционированный доступ к системе.

Администратору не предоставляется неограниченный доступ к инфраструктуре для ежедневного управления и совершения операций. Операторы Azure Stack должны управлять системой через портал администратора или через Azure Resource Manager (с помощью PowerShell или REST API). К системе нельзя получить доступ с помощью других средств управления, таких как диспетчер Hyper-V или диспетчер отказоустойчивости кластеров. Чтобы защитить систему, сторонние программы (например, агенты) не могут быть установлены внутри компонентов инфраструктуры Azure Stack. Взаимодействие с внешним программным обеспечением управления и безопасности происходит с помощью PowerShell или REST API.

Когда требуется более высокий уровень доступа для устранения неполадок, которые не удается исправить, управляя оповещениями, обратитесь в службу поддержки. Благодаря службе поддержки существует способ временного предоставления полного доступа администратора к системе для выполнения более сложных операций. 

## <a name="deploy-connected-or-disconnected"></a>Развертывание с подключением и без подключения
 
Можно развернуть Azure Stack с подключением к Интернету (и к Azure) или без подключения. Для максимально эффективной работы с Azure Stack, включая гибридные сценарии между Azure Stack и Azure, следует выполнить развертывание с подключением к Azure. В следующей таблице приведены основные различия между режимами развертывания.

| Область | Режим развертывания с подключением | Режим развертывания без подключения |
| -------- | ------------- | ----------|
| Поставщик удостоверений | Azure Active Directory (Azure AD) и службы федерации Active Directory (AD FS) | Только AD FS |
| Синдикация Marketplace | Скачивание элементов непосредственно из Azure Marketplace в Marketplace в Azure Stack | Требуется ручной режим управления marketplace в Azure Stack |
| Модель лицензирования | Оплата по мере использования или на основе емкости | Только на основе емкости|
| Исправления и обновления  | Скачивание пакетов обновлений непосредственно в Azure Stack | Требуется съемный носитель и отдельное подключенное устройство |
| | | |

Нельзя изменить режим развертывания позднее без повторного развертывания всей системы.

## <a name="identity-considerations"></a>Вопросы относительно удостоверений

### <a name="choose-identity-provider"></a>Выбор поставщика удостоверений

Необходимо решить, какого поставщика удостоверений вы хотите использовать для развертывания Azure Stack: Azure AD или AD FS. Поставщики удостоверений нельзя изменить после развертывания без повторного развертывания всей системы.

Выбор поставщика удостоверений данных не имеет отношения к виртуальным машинам клиента, системе идентификации и учетным записям, которые они используют, возможности их подключения к домену Active Directory и т. д. Это отдельные аспекты.

**Причины использовать Azure AD**

- У вас уже есть ресурсы в Azure AD (например, Azure или Office 365).
- Вы хотите использовать то же удостоверение в облаках Azure и Azure Stack.
- Вам требуется поддержка мультитенантных сценариев, в которых можно размещать разные организации в одном экземпляре Azure Stack.
- Вы хотите использовать управление каталогами на базе REST через Azure AD Graph для подготовки пользователей, групп и т. д. с помощью API-интерфейсов.

> [!NOTE]
> Вы не можете одновременно подключать развертывание Azure Stack к экземпляру Azure AD и к существующему экземпляру AD FS. При развертывании с использованием Azure AD и необходимостью использовать существующий экземпляр AD FS вы можете объединить локальный экземпляр AD FS с Azure AD.

**Причины использовать AD FS**

- В случае, если отсутствует подключение или имеется только частичное подключение к Интернету.
- Существуют нормативные требования или требования к независимости данных.
- Вы хотите использовать собственную систему удостоверений (например, корпоративную службу Active Directory) для учетных записей операторов и пользователей. Чтобы сделать это, можно создать федерацию с существующим экземпляром AD FS (в Windows Server 2012, 2012 R2 или 2016) на базе Active Directory.
- У вас нет ресурсов в Azure, и вы не хотите использовать Azure AD.

> [!NOTE]
> Можно создать федерацию Azure Stack только с другим экземпляром AD FS на базе Active Directory. Другие поставщики удостоверений не поддерживаются. Если у вас есть другие поставщики удостоверений, которые вы хотите использовать с Azure Stack, рекомендуется выбрать развертывание на базе Azure AD.

### <a name="ad-fs-and-graph-integration"></a>Интеграция AD FS и Graph

Если требуется выполнить развертывание Azure Stack с использованием AD FS в качестве поставщика удостоверений, вы должны интегрировать экземпляр AD FS в Azure Stack с существующим экземпляром AD FS с помощью доверия федерации. Это позволяет удостоверениям в существующем лесу Active Directory проходить проверку в ресурсах Azure Stack.

Можно также интегрировать службу Graph в Azure Stack с существующей службой Active Directory. Это позволяет управлять доступом на основе ролей (RBAC) в Azure Stack. При делегировании доступа к ресурсу компонент Graph ищет учетную запись пользователя в существующем лесу Active Directory, используя протокол LDAP.

## <a name="licensing-model"></a>Модель лицензирования

Необходимо решить, какую модель лицензирования использовать. Для развертываний с подключением можно выбрать лицензирование с оплатой по мере использования или на основе емкости. Модель оплаты по мере использования требует соединения с Azure для уведомления об использовании, которое затем оплачивается через коммерческое предложение Azure. При развертывании без подключения к Интернету поддерживается только лицензирование на основе емкости. Дополнительные сведения о модели лицензирования см. [на странице с информацией о пакетах и ценах Microsoft Azure Stack](https://azure.microsoft.com/mediahandler/files/resourcefiles/5bc3f30c-cd57-4513-989e-056325eb95e1/Azure-Stack-packaging-and-pricing-datasheet.pdf).

## <a name="naming-decisions"></a>Решения именования

Вам нужно подумать о том, какое пространство имен вы будете использовать в Azure Stack, особенно имя региона и имя внешнего домена. Полное доменное имя (FQDN) развертывания Azure Stack для общедоступных конечных точек представляет собой комбинацию этих двух имен, &lt;*регион*&gt;&lt;*внешнее_FQDN*&gt;, например *east.cloud.fabrikam.com*. В этом примере порталы Azure Stack будут находиться по следующим URL-адресам:

- https://portal.east.cloud.fabrikam.com
- https://adminportal.east.cloud.fabrikam.com

| Имя | Описание | 
| -------- | ------------- |

## <a name="time-synchronization"></a>Синхронизация времени

Вы должны синхронизировать сервер времени Azure Stack с внешним сервером времени, разрешаемым по IP-адресу. Для развертывания без подключения к Интернету требуется сервер времени в корпоративной сети.

## <a name="network-connectivity"></a>Сетевое подключение

### <a name="dns-integration"></a>Интеграция DNS

Чтобы разрешить внешние имена DNS из Azure Stack (например, www.bing.com), вам необходимо предоставить DNS-серверы, которые Azure Stack может использовать для перенаправления DNS-запросов, для которых Azure Stack не является полномочным. В качестве входных данных для развертывания нужно предоставить как минимум два сервера, которые будут использоваться в качестве DNS-серверов пересылки для обеспечения отказоустойчивости.

Чтобы разрешить DNS-имена конечных точек Azure Stack извне Azure Stack (например, из корпоративного леса), необходимо интегрировать DNS-серверы, где размещена внешняя зона DNS для Azure Stack, с DNS-серверами, где размещена родительская зона, которую требуется использовать. Для этого требуется разрешение DNS-имен между Azure Stack и существующими зонами DNS в центре обработки данных. Для этого потребуются такие методы, как условное перенаправление или делегирование зоны. Мы советуем использовать условное перенаправление, если у вас есть прямой контроль над DNS-серверами, на которых размещается родительская зона вашего внешнего пространства имен DNS Azure Stack. Если внешняя зона DNS Azure Stack отображается как дочерний домен корпоративного доменного имени (например, azurestack.contoso.com и contoso.com), вместо этого следует использовать делегирование зоны.

### <a name="uplink-to-border-switches"></a>Линия связи с пограничными коммутаторами

Вам понадобится линия связи, настроенная к пограничным коммутаторам в вашем центре обработки данных. Вы можете направлять этот трафик третьего уровня от коммутаторов верхнего уровня (TOR), которые являются частью интегрированной системы Azure Stack, используя один из следующих методов:

- протокол BGP; 
- статическая маршрутизация.

Мы советуем использовать протокол BGP, так как он позволяет автоматически обновлять маршруты, которые публикуются мультиплексором балансировки нагрузки программного обеспечения (SLB MUX) во внешние сети. Это важно, если вы добавляете диапазоны общедоступных IP-адресов после развертывания. При установке пиринга BGP от коммутаторов TOR к агрегированному коммутатору, к которому подключены коммутаторы TOR, диапазоны общедоступных IP-адресов, которые вы добавляете после развертывания, автоматически объявляются в агрегированном коммутаторе без ручного вмешательства. При использовании статической маршрутизации необходимо вручную обновлять маршруты для новых диапазонов каждый раз, когда вы добавляете блок общедоступных IP-адресов.

### <a name="proxy-server"></a>Прокси-сервер

Azure Stack поддерживает только прозрачные прокси-серверы. Прозрачный прокси-сервер перехватывает запросы на сетевом уровне, не требуя специальной настройки клиента.

### <a name="publish-azure-stack-services"></a>Публикация служб Azure Stack

Вам нужно будет сделать службы Azure Stack доступными пользователям извне Azure Stack. Azure Stack настраивает несколько разных конечных точек для ролей инфраструктуры. Этим конечным точкам назначаются виртуальные IP-адреса из пула общедоступных IP-адресов. Для каждой конечной точки создается DNS-запись во внешней зоне DNS, которая указана во время развертывания. Например, пользовательскому порталу назначается запись DNS-узла "portal.<*регион*>.<*внешнее_FQDN*>". 

#### <a name="ports-and-urls"></a>URL-адреса и порты

Чтобы службы Azure Stack (такие как порталы, Azure Resource Manager, DNS и т. д.) были доступны для внешних сетей, необходимо разрешить входящий трафик к этим конечным точкам для определенных URL-адресов, портов и протоколов.
 
Если в вашем развертывании прозрачный прокси-сервер перенаправляет соединения к традиционному прокси-серверу, необходимо разрешить определенные порты и URL-адреса для исходящей связи. К ним относятся порты и URL-адреса для идентификации, синдикации marketplace, исправления и обновления, регистрации и использования.

Дополнительные сведения можно найти в разделе 
- [Входящие порты и протоколы](https://docs.microsoft.com/azure/azure-stack/azure-stack-integrate-endpoints#ports-and-protocols-inbound)
- [Исходящие порты и URL-адреса](https://docs.microsoft.com/azure/azure-stack/azure-stack-integrate-endpoints#ports-and-urls-outbound)

### <a name="connect-azure-stack-to-azure"></a>Подключение Azure Stack к Azure

Для гибридных облачных сценариев вам нужно будет спланировать, как вы хотите подключить Azure Stack к Azure. Виртуальные сети в Azure Stack можно подключить к виртуальным сетям в Azure двумя способами: 
- **Подключение типа "сеть — сеть"**. VPN-подключение по протоколу IPsec (протоколы IKE v1 и IKE v2). Для этого типа подключения требуется VPN-устройство или служба маршрутизации и удаленного доступа (RRAS). Дополнительные сведения о VPN-шлюзах в Azure см. в [этой статье](https://docs.microsoft.com/azure/vpn-gateway/vpn-gateway-about-vpngateways). Связь через этот туннель шифруется и защищена. Тем не менее пропускная способность ограничена максимальной пропускной способностью туннеля (100–200 Мбит/с).
- **NAT для исходящего трафика**. По умолчанию все виртуальные машины в Azure Stack будут подключаться к внешним сетям через NAT для исходящего трафика. Каждой виртуальной сети, созданной в Azure Stack, назначается общедоступный IP-адрес. Независимо от того, назначен ли виртуальной машине общедоступный IP-адрес или находится ли она за балансировщиком нагрузки с общедоступным IP-адресом, она будет иметь исходящий доступ через NAT для исходящего трафика, используя виртуальный IP-адрес виртуальной сети. Это работает только для связи, которая инициируется виртуальной машиной и предназначена для внешних сетей (Интернет или интрасеть). Это нельзя использовать для взаимодействия с виртуальной машиной извне.

#### <a name="hybrid-connectivity-options"></a>Гибридные варианты подключений

Для гибридного подключения важно рассмотреть, какое развертывание вы хотите предложить и где оно будет развернуто. Необходимо определить, нужна ли изоляция сетевого трафика для каждого клиента и тип связи с развертывании (в интрасети или Интернете).

- **Azure Stack с одним клиентом**. Развертывание Azure Stack, которое с точки зрения сети выглядит, как будто это один клиент. Существует много подписок клиентов, но, как и любая служба интрасети, весь трафик проходит по тем же сетям. Сетевой трафик из одной подписки перемещается по тому же сетевому подключению, что и другая подписка, и его не нужно изолировать через зашифрованный туннель.

- **Azure Stack с несколькими клиентами**. Развертывание Azure Stack, в котором трафик подписки каждого клиента, привязанный к сетям, которые являются внешними для Azure Stack, должен быть изолирован от сетевого трафика других клиентов.
 
- **Развертывание в интрасети**. Развертывание Azure Stack, которое находится в корпоративной интрасети, обычно на частном IP-адресе и за одним или несколькими брандмауэрами. Общедоступные IP-адреса не являются общедоступными, так как они не могут быть перенаправлены непосредственно через Интернет.

- **Развертывание в Интернете**. Развертывание Azure Stack, подключенное к общедоступному Интернету и использующее общедоступные IP-адреса для диапазона общедоступных виртуальных IP-адресов. Развертывание по-прежнему может находиться за брандмауэром, но диапазон общедоступных виртуальных IP-адресов напрямую доступен из общедоступного Интернета и Azure.

## <a name="external-monitoring"></a>Внешний мониторинг
Чтобы получить единое представление обо всех оповещениях устройств и развертывания Azure Stack, а также интегрировать оповещения в существующие рабочие процессы управления ИТ-службами для отправки запросов, вы можете интегрировать Azure Stack с внешними решениями мониторинга центров обработки данных.

Вместе с решением Azure Stack поставляется узел жизненного цикла аппаратного обеспечения. Это компьютер за пределами Azure Stack, который использует предоставленные поставщиком OEM средства для управления оборудованием. Вы можете использовать эти средства или другие решения, которые непосредственно интегрируются с существующими решениями мониторинга в вашем центре обработки данных.

## <a name="backup-and-disaster-recovery"></a>Резервное копирование и аварийное восстановление

Планирование резервного копирования и аварийного восстановления включает в себя планирование как для базовой инфраструктуры Azure Stack, в которой размещаются виртуальные машины IaaS и службы PaaS, так и для приложений и данных клиента. Необходимо учесть их отдельно.

### <a name="protect-infrastructure-components"></a>Защита компонентов инфраструктуры

Azure Stack создает резервные копии компонентов инфраструктуры в указанной вами общей папке.

- Вам потребуется внешний общий файловый ресурс SMB на существующем файловом сервере Windows или стороннем устройстве.
- Следует использовать этот же общий ресурс для резервного копирования сетевых коммутаторов и узла жизненного цикла оборудования. Ваш поставщик оборудования OEM предоставит рекомендации по резервному копированию и восстановлению компонентов, так как они являются внешними для Azure Stack. Вы отвечаете за выполнение рабочих процессов резервного копирования в соответствии с рекомендациями поставщика OEM.

Если происходит неустранимая потеря данных, вы можете использовать резервную копию инфраструктуры для повторной установки данных развертывания, таких как входные данные развертывания и идентификаторы, учетные записи служб, корневой сертификат ЦС, федеративные ресурсы (в развертываниях без подключения), планы, предложения, подписки, квоты, политика RBAC, назначения ролей и секреты хранилища ключей.
 
### <a name="protect-tenant-applications-on-iaas-virtual-machines"></a>Защита приложений клиента на виртуальных машинах IaaS

Azure Stack не создает резервную копию приложений и данных клиента. Необходимо предусмотреть резервное копирование и аварийное восстановление в среду вне Azure Stack. Защита клиента — это действие на основе клиента. Для виртуальных машин IaaS клиенты могут использовать гостевые технологии для защиты папок файлов, данных приложений и состояния системы. Однако как организация или поставщик услуг вы можете предложить решение для резервного копирования и восстановления в том же центре обработки данных или извне в облаке.

Чтобы создать резервную копию виртуальных машин IaaS Linux или Windows, необходимо использовать решения резервного копирования с доступом к гостевой операционной системе для защиты файлов, папок, состояния операционной системы и данных приложений. Вы можете использовать Azure Backup, System Center Data Center Protection Manager или поддерживаемые продукты сторонних производителей.

Для репликации данных во вторичное расположение и оркестрации отработки отказа приложений в случае сбоя можно использовать Azure Site Recovery или поддерживаемые продукты сторонних производителей. (В первоначальном выпуске интегрированных систем Azure Site Recovery не будет поддерживать отказоустойчивость. Тем не менее можно выполнить восстановление размещения вручную.) Кроме того, приложения, поддерживающие встроенную репликацию (например, Microsoft SQL Server), могут реплицировать данные в другое расположение, где выполняется приложение.

> [!IMPORTANT]
> В первоначальном выпуске интегрированных систем будут поддерживаться технологии защиты, которые работают на гостевом уровне виртуальной машины IaaS. Нельзя установить агенты на серверах базовой инфраструктуры.
