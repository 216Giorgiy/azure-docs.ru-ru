# Сравнение веб-сайтов Windows Azure, облачных служб и виртуальных машин

Windows Azure предоставляет несколько способов размещения веб-приложений, таких как [веб-сайты Windows Azure][], [облачные службы][] и [виртуальные машины][]. После знакомства с этими различными вариантами вы можете не быть уверены, что именно наилучшим образом соответствует вашим потребностям, или у вас могут быть сомнения относительно концепций, таких как IaaS и PaaS. Данная статья поможет вам понять возможности и сделать правильный выбор веб-сценария. Хотя все три варианта позволяют выполнять веб-приложения в Windows Azure с высокой масштабируемостью, существуют различия, которые могут помочь принять решение.

Во многих случаях веб-сайты Windows Azure является наилучшим вариантом. Эта служба предоставляет простые и гибкие варианты для развертывания и управления и может разместить большие веб-сайты. Можно быстро создать новый веб-сайт с помощью популярных программ, таких как WordPress из галереи веб-приложений, или можно переместить существующий веб-сайт в веб-сайты Windows Azure. С помощью [Windows Azure SDK WebJobs][] (в настоящее время в режиме предварительного просмотра) можно также добавить задания фоновой обработки. 

Также имеется возможность размещения веб-приложений на виртуальных машинах Windows Azure и облачных службах Windows Azure. Эти варианты хороши, когда для веб-уровня требуется дополнительный уровень управления и настройки, которые они предоставляют. Тем не менее за этот повышенный контроль приходится платить увеличением уровня сложности создания приложений, управления и развертывания. На следующей схеме показаны компромиссы из трех вариантов.

![ChoicesDiagram][ChoicesDiagram]

Веб-сайты просты в настройке, управлении и контроле, но имеют меньше параметров конфигурации. То есть если нет особой причины для размещения веб-сайта переднего плана на облачных службах или виртуальных машинах, используйте веб-сайты Windows Azure. - [Сводки служб](#services)

##<a name="scenarios"></a>Сценарии

### Я владелец малого бизнеса и мне необходимо недорогое размещение сайта с возможностями расширения.

Веб-сайты Windows Azure (WAWS) – идеальное решение для этой ситуации, поскольку эту службу можно использовать бесплатно и при необходимости добавить дополнительные возможности. Например, каждый бесплатный веб-сайт поставляется с доменом, предоставляемым Windows Azure (*ваша_компания*. azurewebsites.net). Если вы готовы начать использование собственного домена, можно добавить его всего за 9,80 долл. США в месяц (по состоянию на январь 2014 г.). Существует множество других служб и параметров масштабирования, позволяющих соответствовать повышенным требованиям пользователей. С помощью **веб-сайтов Windows Azure** можно:

- Начать работу с бесплатного уровня и при необходимости увеличить масштаб.
- Использовать коллекцию приложений для быстрой установки популярных веб-приложений, таких как WordPress.
- Добавлять дополнительные службы Windows Azure и функции в приложение при необходимости.
- Защитить веб-сайт с помощью HTTPS, используя сертификат, предоставляемый с вашим именем домена *ваша_компания*. azurewebsites.net.

### Я веб-дизайнер или графический дизайнер и мне необходимо создать веб-сайты для клиентов

Веб-разработчикам веб-сайты Windows Azure предлагают все необходимое для создания сложных веб-приложений. Веб-сайты обеспечивают тесную интеграцию со средствами, такими как Visual Studio и базы данных SQL. С помощью службы **Веб-сайты** разработчики могут:

- Использовать программы командной строки для [автоматизированных задач][scripting].
- Работать с популярными языками, такими как [.Net][dotnet], [PHP][], [Node.js][nodejs] и [Python][].
- Выбирать три различных уровня масштабирования для масштабирования высокой емкости.
- Осуществлять интеграцию с другими службами Azure, такими как [база данных SQL][sqldatabase], [Service Bus][servicebus] и [Хранилище][], или партнерскими предложениями от [магазина Windows Azure][azurestore], такими как MySQL и MongoDB.
- Осуществлять интеграцию с такими средствами, как Visual Studio, Git, WebMatrix, WebDeploy, TFS и FTP.

### Я переношу многоуровневое приложение с веб-ролью в облако

Если вы используете многоуровневое приложение, например, веб-сервер, который связывается с сервером базы данных для хранения и извлечения данных веб-сайта, в Windows Azure есть несколько вариантов. Эти параметры архитектуры включают веб-сайты, облачные службы и виртуальные машины. Во-первых, **Веб-сайты** являются хорошим вариантом для веб-уровня вашего решения и могут использоваться для создания двухуровневой архитектуры с базой данных SQL Windows Azure. Веб-сайты также позволяют запускать фоновые или долговременные процессы с помощью предварительного просмотра Windows Azure SDK WebJobs. Если требуется более сложная архитектура или более гибкие возможности масштабирования, облачные службы и виртуальные машины являются лучшим выбором. 

**Облачные службы** позволяют:

- Размещать веб-службы, службы среднего уровня и серверные службы в масштабируемых веб- и рабочих ролях. 
- Размещать только веб-службы, службы среднего уровня и серверные службы в рабочих ролях, сохраняя интерфейсный сервер на веб-сайтах Windows Azure. 
- Масштабировать интерфейсный сервер и серверные службы независимо друг от друга.

**Виртуальные машины** позволяют: 

- Более легко переносить настраиваемые среды, такие как образы виртуальных машин.
- Запускать программное обеспечение или службы, которые не могут быть настроены на Веб-сайтах и Облачных службах.

### Мое приложение зависит от настраиваемых сред Windows или Linux

Если для приложения требуется сложная установка или настройка операционной системы и программного обеспечения, Виртуальные машины, возможно, являются наилучшим решением. С помощью **Виртуальных машин** можно:

- Использовать галерею виртуальных машин для запуска операционной системы, например, Windows или Linux и настроить ОС для требований приложения. 
- Создавать и загружать настраиваемый образ существующего локального сервера для выполнения на виртуальных машинах в Windows Azure. 

### Мой веб-сайт использует программное обеспечение с открытым исходным кодом и его необходимо размещать в Windows Azure.

Все три варианта позволяют размещать языки и платформы с открытым исходным кодом. **Облачные службы** требуют использования задач запуска для установки и настройки необходимого программного обеспечения с открытым исходным кодом, которое работает на Windows. С помощью **Виртуальных машин** можно выполнять установку и настройку программного обеспечения в образе машины на базе Windows или Linux. Если ваша платформа с открытым исходным кодом поддерживается на веб-сайтах, это обеспечивает простой способ для размещения таких приложений, поскольку веб-сайты можно автоматически настраивать с языками и платформами, необходимыми приложению. **Веб-сайты** позволяют:

- Использовать многочисленные популярные языки с открытым исходным кодом, такие как [.NET][dotnet], [PHP][], [Node.js][nodejs] и [Python][]. 
- Настраивать WordPress, Drupal, Umbraco, DNN и многие другие веб-приложения сторонних производителей. 
- Переносить существующие приложения или создавать новые из коллекции приложений. 

### У меня есть бизнес-приложение, которому необходимо подключиться к корпоративной сети

Если вы хотите создать бизнес-приложение, для веб-сайта может потребоваться прямой доступ к службам или данным в корпоративной сети. Это можно сделать на **Веб-сайтах**, **Облачных службах** и **Виртуальных машинах**. Существуют различия в подходах, которые включают следующее:

- Веб-сайты могут надежно подключаться к ресурсам на предприятии с помощью ретрансляции шины Service Bus. Это позволяет другим службам в корпоративной сети выполнят задачи от имени веб-приложения без переноса всех данных в облако или настройки виртуальной сети. 
- Облачные службы и виртуальные машины, использующие преимущества виртуальной сети. По сути виртуальная сеть позволяет машинам, работающим в Windows Azure, подключаться к сети на предприятии. Windows Azure в этом случае становится расширением корпоративного центра обработки данных.

### Требуется разместить интерфейс API REST или веб-службу для мобильных клиентов

Веб-службы на базе HTTP позволяют поддерживать широкий спектр клиентов, включая мобильных клиентов. Платформы, такие как ASP.NET Web API, интегрируются с Visual Studio для упрощения создания и использования служб REST.  Эти службы предоставляются из конечной веб-точки, поэтому возможно использование любого размещения на Windows Azure для поддержки этого сценария. Тем не менее **Веб-сайты** отлично подходят для размещения интерфейсов API REST. С помощью веб-сайтов вы сможете:

- Быстро создать веб-сайт для размещения веб-службы HTTP в одном из глобально распределенных центров обработки данных Windows Azure.
- Переносить существующие службы или создать новые, используя потенциальные преимущества ASP.NET Web API в Visual Studio.
- Обеспечить выполнение соглашения об уровне обслуживания касательно доступности с помощью одного экземпляра или увеличить масштаб до нескольких выделенных машин. 
- Использовать опубликованный сайт для предоставления API REST для любых клиентов HTTP, включая мобильных клиентов.

##<a name="services"></a>Сводки служб

[Веб-сайты Windows Azure][] позволяют быстро создавать высокомасштабируемые веб-сайты в Windows Azure. Портал Windows Azure или программы командной строки можно использовать для настройки веб-сайта с популярными языками, такими как .NET, PHP, Node.js и Python. Поддерживаемые платформы уже развернуты и дополнительные действия по установке не требуются. Коллекция веб-сайтов Windows Azure содержит многие приложения независимых производителей, таких как Drupal и WordPress и платформ разработки, таких как Django и CakePHP. После создания сайта можно выполнить перемещение существующего веб-сайта или создать новый веб-сайт. Веб-сайты избавляют от необходимости управлять физическим оборудованием и также предоставляют несколько вариантов масштабирования. Можно перейти из общей многопользовательской модели в стандартный режим, где выделенные машины обслуживают входящий трафик. Веб-сайты позволяют выполнять интеграцию с другими службами Windows Azure, такими как базы данных SQL, Service Bus и хранилище. С помощью предварительного просмотра [Windows Azure SDK WebJobs][] можно также добавить задания фоновой обработки. В целом веб-сайты Windows Azure позволяют сфокусировать внимание на разработке приложений, поддерживая широкий спектр языков, приложений с открытым исходным кодом и методов развертывания (FTP, Git, веб-развертывание и TFS). Если отсутствуют специальные требования касательно облачных служб или виртуальных машин, веб-сайты Windows Azure вероятно являются наилучшим вариантом.

[Облачные службы][] позволяют создавать высокодоступные, неограниченно масштабируемые веб-приложения с помощью функциональной среды PaaS (платформа как услуга). В отличие от веб-сайтов облачная служба создается сначала в среде разработки, например Visual Studio, перед развертыванием в Windows Azure. Платформы, такие как PHP, требуют от пользователя выполнения действий или задач по развертыванию платформы при запуске роли. Главным преимуществом облачных служб является возможность поддержки более сложных многоуровневых архитектур. Одна облачная служба может состоять из клиентской части веб-роли и одной или нескольких рабочих ролей. Каждый уровень можно масштабировать независимо друг от друга. Также обеспечивается повышенный уровень контроля над вашей инфраструктурой веб-приложений. Например поддерживается использование удаленного рабочего стола на машинах, выполняющих экземпляры ролей. Можно также написать сценарий для дополнительных служб IIS и изменений конфигурации компьютера, который будет выполняться при запуске роли, в том числе для задач, требующих вмешательства администратора.

[Виртуальные машины][] обеспечивают возможность запуска веб-приложений на виртуальных машинах в Windows Azure. Эта возможность также известна как "инфраструктура как служба (IaaS)". Создавайте новые машины Windows Server или Linux на портале или загружайте существующие образы виртуальной машины. Виртуальные машины предоставляют наиболее полный контроль над операционной системой, конфигурацией и установленным программным обеспечением и службами. Это хороший вариант для быстрого переноса комплексных локальных веб-приложений в облако, так как машины могут быть перемещены целиком. С помощью виртуальных сетей эти виртуальные машины можно также подключать к корпоративным сетям на предприятии. Как и облачным службам, к этим машинам предоставляется удаленный доступ и возможность изменения конфигурации на уровне администратора. Тем не менее в отличие от веб-сайтов и облачных служб, образами виртуальных машин и архитектурой приложений необходимо управлять полностью на уровне инфраструктуры. К примеру, необходимо применять собственные пакеты исправлений для операционной системы.

<sup>2</sup> Хотя виртуальные машины можно масштабировать до нескольких экземпляров, службы, работающие на этих машинах, должны быть записаны для реализации этого масштабирования. Необходимо настроить дополнительную балансировку нагрузки для маршрутизации запросов на компьютерах. И наконец для всех компьютеров, участвующих в той же роли, необходимо создать территориальную группу для защиты их от одновременного перезапуска при обслуживании и сбоях оборудования.

Таким образом веб-развертывание не является механизмом развертывания для облачных служб в производственной среде.
