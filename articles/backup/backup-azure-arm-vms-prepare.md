# <a name="prepare-your-environment-to-back-up-resource-manager-deployed-virtual-machines"></a>Подготовка среды к архивации виртуальных машин, развернутых с помощью Resource Manager

В этой статье приводятся действия, Подготовка среды для резервного копирования Azure Resource Manager развертывания виртуальной машины (VM). Шаги процедур выполняются на портале Azure.  

Службе архивации Azure имеет два типа хранилищ для защиты виртуальных машин: резервными хранилищами и хранилищами служб восстановления. Резервного хранилища помогает защитить виртуальные машины, развернутые с помощью классической модели развертывания. Хранилище служб восстановления помогает защитить *классического развертывания и развертывания диспетчера ресурсов виртуальных машин*. Если вы хотите защитить развертывания диспетчера ресурсов виртуальной Машины, необходимо использовать хранилище служб восстановления.

> [!NOTE]
> В Azure предусмотрены две модели развертывания, позволяющие создавать ресурсы и работать с ними: [модель Resource Manager и классическая модель](../azure-resource-manager/resource-manager-deployment-model.md).

Перед защиты и резервного копирования развертывания диспетчера ресурсов виртуальной машины, убедитесь, что существуют следующие необходимые компоненты:

* Создайте хранилище служб восстановления (или определите существующем хранилище служб восстановления) *там же, как ВМ*.
* Выберите сценарий, определите политику архивации, а также элементы, которые необходимо защитить.
* Проверка установки агента виртуальной Машины на виртуальной машине.
* Проверьте сетевое подключение.
* Для виртуальных машин Linux, если требуется настроить среду резервного копирования для резервных копий, согласованных на уровне приложения, выполните [действия, чтобы настроить скрипты до моментального снимка и после создания моментального снимка](https://docs.microsoft.com/azure/backup/backup-azure-linux-app-consistent).

Если эти условия уже существуют в вашей среде, перейдите к разделу [резервное копирование виртуальных машин](backup-azure-arm-vms.md) статьи. Если необходимо настроить или проверить любого из этих требований, в этой статье описан действия.

## <a name="supported-operating-systems-for-backup"></a>Поддерживаемые операционные системы для резервного копирования
 * **Linux**: резервное копирование Azure поддерживает [список дистрибутивы подтверждает Azure](../virtual-machines/linux/endorsed-distros.md?toc=%2fazure%2fvirtual-machines%2flinux%2ftoc.json), за исключением CoreOS Linux. 
 
    > [!NOTE] 
    > Другие перевести your владельцем-дистрибутивы Linux может работать при условии, что агент виртуальной Машины для виртуальной машины и поддержка существует Python. Тем не менее мы не одобряет этих распределений для резервного копирования.
 * **Windows Server**: версии до Windows Server 2008 R2 не поддерживаются.

## <a name="limitations-when-backing-up-and-restoring-a-vm"></a>Ограничения при резервном копировании и восстановлении виртуальной машины
Перед подготовкой среде следует учитывать эти ограничения:

* Архивация виртуальных машин, к которым подключено более 16 дисков данных, не поддерживается.
* Резервное копирование виртуальных машин с данными больше 1023 ГБ размеры дисков не поддерживается.

  > [!NOTE]
  > У нас есть личной предварительной версии для поддержки резервного копирования для виртуальных машин с неуправляемой дисками размером 1 ТБ (или более поздней). Дополнительные сведения см. в [личной предварительной версии для поддержки резервного копирования больших дисков виртуальной Машины](https://gallery.technet.microsoft.com/Instant-recovery-point-and-25fe398a).
  >

* Архивация виртуальных машин с зарезервированным IP-адресом и без заданной конечной точки не поддерживается.
* Резервное копирование виртуальных машин, зашифрованного с помощью только что ключа шифрования BitLocker (BEK) не поддерживается. Резервное копирование виртуальных машин Linux, зашифрованного с помощью шифрования Linux единой ключ установки (LUKS) не поддерживается.
* Резервное копирование виртуальных машин, которые содержат общие тома кластера (CSV) или масштабируемого файлового сервера конфигурации не рекомендуется. Они требуют, включающий все виртуальные машины, включенные в конфигурацию кластера, во время выполнения моментального снимка. Служба Azure Backup не поддерживает согласованность нескольких виртуальных машин. 
* Резервных копий данных не содержит подключенных сетевых дисках, подключенных к виртуальной Машины.
* Замена существующей виртуальной машины во время восстановления не поддерживается. При попытке восстановить имеющуюся виртуальную машину произойдет сбой операции.
* Архивация и восстановление между регионами не поддерживаются.
* Резервное копирование и восстановление виртуальных машин ACLed хранилища сейчас не поддерживается. Резервное копирование виртуальных машин не поддерживается при наличии хранилища на функции виртуальной сети, которая позволяет учетных записей хранилища доступны только из некоторых виртуальных сетей и подсетей и/или IP-адресов.
* Можно выполнять архивацию виртуальных машин в все открытые регионах Azure. (См. [контрольный список](https://azure.microsoft.com/regions/#services) поддерживаемых регионов.) Если сейчас не поддерживается региона, в который вы ищете, он не появятся в раскрывающемся списке во время создания хранилища.
* Восстановление контроллера домена виртуальной машины, которая входит в конфигурацию с несколькими контроллерами домена, поддерживается только с помощью PowerShell. Дополнительные сведения см. в разделе [восстановление контроллера домена multi-DC](backup-azure-arm-restore-vms.md#restore-domain-controller-vms).
* Восстановление виртуальных машин с описанными ниже особыми конфигурациями сети поддерживается только в PowerShell. Виртуальные машины, созданные с помощью рабочего процесса восстановления в пользовательском Интерфейсе не будут иметь эти конфигурации сети после завершения операции восстановления. Дополнительные сведения см. в разделе [Восстановление виртуальных машин с помощью специальных конфигураций сети](backup-azure-arm-restore-vms.md#restore-vms-with-special-network-configurations).
  * Виртуальные машины в конфигурации балансировщика нагрузки (внутренняя и внешняя)
  * Виртуальные машины с несколькими зарезервированными IP-адресами
  * Виртуальные машины с несколькими сетевыми адаптерами

## <a name="create-a-recovery-services-vault-for-a-vm"></a>Создание хранилища служб восстановления для виртуальной Машины
Хранилище служб восстановления — это сущность, в которой хранятся резервные копии и точки восстановления, которые были созданы со временем. Хранилище служб восстановления также содержит политики резервного копирования, которые связаны с защищенных виртуальных машин.

Чтобы создать хранилище служб восстановления, сделайте следующее:

1. Войдите на [портале Azure](https://portal.azure.com/).
2. На **концентратора** последовательно выберите пункты **Обзор**, а затем введите **службы восстановления**. Началом ввода входные данные фильтруются в список ресурсов. Выберите **хранилищ служб восстановления**.

    ![Введите в поле и выбрав пункт «Хранилищами служб восстановления» в результатах](./media/backup-azure-arm-vms-prepare/browse-to-rs-vaults-updated.png) <br/>

    После этого отобразится список хранилищ служб восстановления,
3. На **хранилищ служб восстановления** последовательно выберите пункты **добавить**.

    ![Создание хранилища служб восстановления — шаг 2](./media/backup-azure-arm-vms-prepare/rs-vault-menu.png)

    **Хранилищ служб восстановления** откроется область. Предлагается для предоставления сведений о **имя**, **подписки**, **группы ресурсов**, и **расположение**.

    ![Панель «Хранилищами служб восстановления»](./media/backup-azure-arm-vms-prepare/rs-vault-attributes.png)
4. В поле **Имя**введите понятное имя хранилища. Имя должно быть уникальным в пределах подписки Azure. Введите имя, которое содержит 2 до 50 символов. Оно должно начинаться с буквы, и он может содержать только буквы, цифры и дефисы.
5. Выберите **подписки** для просмотра списка доступных подписок. Если вы не знаете, какая подписка будет использоваться, используется значение по умолчанию (или рекомендуемые) подписки. Существует несколько вариантов, только если работу или рабочую учетную запись связана с несколькими подписками Azure.
6. Выберите **группы ресурсов** для просмотра списка доступных групп ресурсов, или выберите **New** для создания новой группы ресурсов. Полные сведения для групп ресурсов см. в разделе [Обзор диспетчера ресурсов Azure](../azure-resource-manager/resource-group-overview.md).
7. Выберите **расположение** выберите географическую область для хранилища. Хранилище *должно* находиться в том же регионе, что и виртуальные машины, которые необходимо защитить.

   > [!IMPORTANT]
   > Если вы не знаете, какое место, в котором существует ВМ, закройте диалоговое окно создания хранилища и перейдите к списку виртуальных машин на портале. Если виртуальные машины в нескольких регионах, необходимо создать хранилище служб восстановления в каждом регионе. Прежде чем переходить к следующему расположению, необходимо создать хранилище в первом расположении. Нет необходимости указывать учетные записи хранения для хранения резервных копий данных. Хранилище служб восстановления и службы архивации Azure обрабатывать, автоматически.
   >
   >

8. Нажмите кнопку **Создать**. Для создания хранилища служб восстановления может потребоваться некоторое время. Отслеживать состояние уведомления в правом верхнем области портала. После создания своего хранилища, он отображается в списке хранилищами служб восстановления. Если вы не видите своего хранилища, выберите **обновление**.

    ![Список хранилищ службы архивации](./media/backup-azure-arm-vms-prepare/rs-list-of-vaults.png)

Теперь, когда вы создали хранилище, можно приступить к настройке репликации.

## <a name="set-storage-replication"></a>Набор носителей репликации
Параметр репликации хранилища позволяет выбирать между географически избыточной и локально избыточного хранилища. По умолчанию это геоизбыточное хранилище. Если данная резервная копия является основной, оставьте установленный параметр. Если вам нужно более дешевое и не такое надежное решение, выберите локально избыточное хранилище.

Чтобы изменить параметр репликации хранилища:

1. На **хранилищ служб восстановления** панели выберите вашего хранилища.
    При выборе ваше хранилище **параметры** (которая имеет имя хранилища вверху) и области сведения о хранилище.

   ![Выберите хранилище в списке резервных хранилищ](./media/backup-azure-arm-vms-prepare/new-vault-settings-blade.png)

2. На **параметры** области, с помощью ползунка вертикальной прокрутите вниз до **управление** и выберите **инфраструктуры резервного копирования**. В **Общие** выберите **конфигурация резервного копирования**. На **конфигурация резервного копирования** области, выберите вариант репликации хранилища для вашего хранилища. По умолчанию это геоизбыточное хранилище.

   ![Список хранилищ службы архивации](./media/backup-azure-arm-vms-prepare/full-blade.png)

   Если вы используете Azure как конечная точка основного хранилища резервных копий, продолжать использовать геоизбыточное хранилище. Если вы используете Azure как конечную точку неосновной резервного хранилища, выберите локально избыточное хранилище. Дополнительные сведения о вариантах хранения в [Обзор хранилища Azure репликации](../storage/common/storage-redundancy.md).

3. Если вы изменили тип репликации хранилища, выберите **Сохранить**.
    
После выбора места хранения хранилища, вы будете готовы к связывание виртуальной Машины с хранилищем. Перед началом связывания нужно обнаружить и зарегистрировать виртуальные машины Azure.

## <a name="select-a-backup-goal-set-policy-and-define-items-to-protect"></a>Выберите цель резервного копирования, задать политику и определить элементы для защиты
Идентифицируются перед регистрацией виртуальной Машины с хранилищем, выполнять процесс обнаружения, чтобы убедиться, что все новые виртуальные машины, которые были добавлены к подписке. Процесс запрашивает Azure список виртуальных машин в подписке, а также сведения, такие как имя облачной службы и области. 

На портале Azure *сценарий* ссылается на то, что будет помещено в хранилище служб восстановления. *Политика* расписание, как часто и когда создаются точки восстановления. Политика также включает диапазон хранения для точек восстановления.

1. Если хранилище служб восстановления уже открыто, перейдите к шагу 2. Если у вас нет открыть хранилище служб восстановления, откройте [портал Azure](https://portal.azure.com/). На **концентратора** последовательно выберите пункты **дополнительные службы**.

   a. В списке ресурсов введите **Службы восстановления**. При вводе, список будет отфильтрован входные данные. Когда вы увидите пункт **Хранилища служб восстановления**, щелкните его.

      ![Введите в поле и выбрав пункт «Хранилищами служб восстановления» в результатах](./media/backup-azure-arm-vms-prepare/browse-to-rs-vaults-updated.png) <br/>

      После этого отобразится список хранилищ служб восстановления, Если в вашей подписке нет хранилищ, этот список пуст.

      ![Просмотр списка хранилищ служб восстановления](./media/backup-azure-arm-vms-prepare/rs-list-of-vaults.png)

   Б. в котором нужно выбрать хранилище.

      **Параметры** области и панели мониторинга хранилища для выбранной хранилища откройте.

      ![Параметры области и хранилище панели мониторинга](./media/backup-azure-arm-vms-prepare/new-vault-settings-blade.png)
2. Выберите в меню панели мониторинга хранилища, **резервного копирования**.

   ![Кнопка резервного копирования](./media/backup-azure-arm-vms-prepare/backup-button.png)

   **Резервного копирования** и **цель резервного копирования** откроются панели.

3. На **цель резервного копирования** задайте **где выполняется рабочая нагрузка?** для **Azure** и **что вам требуется архивировать?** для  **Виртуальные машины**. Нажмите кнопку **ОК**.

   ![Резервное копирование и цели резервного копирования панелей](./media/backup-azure-arm-vms-prepare/select-backup-goal-1.png)

   Этот шаг регистрирует расширение ВМ в хранилище. **Цель резервного копирования** области закрывается и **политика архивации** откроется область.

   ![«Резервное копирование» и «Политика архивации» областей](./media/backup-azure-arm-vms-prepare/select-backup-goal-2.png)
4. На **политика архивации** области, выберите политику архивации, которую требуется применить к хранилищу.

   ![Выбор политики резервного копирования](./media/backup-azure-arm-vms-prepare/setting-rs-backup-policy-new.png)

   Подробные сведения о политике по умолчанию указаны в раскрывающемся меню. Если вы хотите создать политику, в раскрывающемся меню щелкните **Создать**. Инструкции по определению политики резервного копирования см. в разделе [Определение политики резервного копирования](backup-azure-vms-first-look-arm.md#defining-a-backup-policy).
    Выберите **ОК** для связи политики резервного копирования в хранилище.

   **Политика архивации** области закрывается и **выберите виртуальные машины** откроется область.
5. На **выберите виртуальные машины** панели выберите виртуальные машины для сопоставления с указанной политики и выберите **ОК**.

   ![Панель «Виртуальных машин select»](./media/backup-azure-arm-vms-prepare/select-vms-to-backup.png)

   Выбранная виртуальная машина пройдет проверку. Если вы не видите виртуальные машины, которые вы хотите увидеть, убедитесь, что они существуют в хранилище служб восстановления Azure местоположения и не защищены в другом хранилище. Панели мониторинга хранилища показывает расположение хранилища служб восстановления.

6. Теперь, когда вы определили все параметры для хранилища, **резервного копирования** выберите **Enable backup**. Этот шаг выполняется развертывание политики в хранилище и виртуальные машины. Этот шаг не создает начальную точку восстановления для виртуальной машины.

   ![Кнопка «Включить резервное копирование»](./media/backup-azure-arm-vms-prepare/vm-validated-click-enable.png)

После успешного включения резервного копирования, политикой резервного копирования будет выполняться по расписанию. Если вы хотите создать задание резервного копирования по запросу для резервное копирование виртуальных машин, в разделе [запуска задания резервного копирования](./backup-azure-arm-vms.md#triggering-the-backup-job).

Если имеется проблема при регистрации виртуальной машины, см. следующие сведения об установке агента виртуальной Машины и подключение к сети. Если защита виртуальных машин, созданных в Azure, включена, следующие сведения можно пропустить. Но если вы перенесли виртуальные машины в Azure, должны быть правильно установлен агент виртуальной Машины и что к виртуальной машине может подключаться к виртуальной сети.

### <a name="backup-extension"></a>Расширение резервного копирования
После установки агента виртуальной Машины на виртуальной машине службы резервного копирования Azure устанавливает расширение резервного копирования агент виртуальной Машины. Службу архивации без проблем обновления и исправления расширение резервного копирования.

Служба архивации устанавливает расширение резервного копирования ли Виртуальная машина запущена. На запущенной виртуальной машине проще получить точку восстановления, согласованную с приложениями. Тем не менее службы резервного копирования продолжает выполнять резервное копирование виртуальной Машины, даже если он был отключен и не удалось установить расширение. Это называется *автономной виртуальной Машины*. В этом случае точка восстановления будет *отказоустойчивой*.

## <a name="establish-network-connectivity"></a>Установка сетевых подключений
Чтобы управлять моментальные снимки виртуальной Машины, расширение резервного копирования должен быть подключен к Azure общих IP-адресов. Без правой подключение к Интернету время ожидания запросов HTTP виртуальной машины и сбою операции резервного копирования. Если развертывание ограничения доступа на месте — через группу безопасности сети (NSG), например — выберите один из следующих параметров для предоставления очистить пути для резервного копирования трафика:

* [Белый список IP-адрес центра обработки данных Azure диапазоны](http://www.microsoft.com/en-us/download/details.aspx?id=41653).
* Разверните прокси-сервер HTTP для маршрутизации трафика.

Когда нужно решить, какой вариант выбрать, преимущества и недостатки находятся в диапазоне от управляемость, высокий уровень контроля и затрат.

| Параметр | Преимущества | Недостатки |
| --- | --- | --- |
| Разрешенные диапазоны IP-адресов |Нет дополнительных затрат.<br><br>Нет дополнительных затрат. Открыть доступ в NSG можно с помощью командлета **Set-AzureNetworkSecurityRule**. |Сложно управлять как соответствующие диапазоны IP-адресов со временем изменяются.<br><br>Предоставляет доступ ко всей Azure, а не только хранения данных. |
| Использовать прокси-сервер HTTP |Четко контролировать прокси хранилища допускается URL-адреса.<br><br>Точка Интернет-доступ к виртуальным машинам.<br><br>Не подвергается влиянию изменений IP-адресов Azure. |Дополнительные затраты для работы виртуальной машины с программным обеспечением прокси-сервера. |

### <a name="whitelist-the-azure-datacenter-ip-ranges"></a>Добавьте диапазоны IP-адресов центра обработки данных Azure в список разрешений.
Белый список IP-адресов центра обработки данных Azure, в разделе [веб-сайте Azure](http://www.microsoft.com/en-us/download/details.aspx?id=41653) подробные сведения о диапазоны IP-адресов и инструкции.

Можно разрешить подключения к хранилищу из определенного региона с помощью [службы теги](../virtual-network/security-overview.md#service-tags). Убедитесь, что правило, которое позволяет получить доступ к учетной записи хранения имеет более высокий приоритет, чем правило, которое блокирует доступ к Интернету.

### <a name="use-an-http-proxy-for-vm-backups"></a>Использовать прокси-сервер HTTP для резервных копий виртуальной Машины
Когда выполняется резервное копирование виртуальной Машины, расширение резервного копирования на виртуальной Машине отправляет команды управления моментального снимка хранилища Azure с помощью API-Интерфейс протокола HTTPS. Направлять трафик расширение резервного копирования через HTTP-прокси, так как он является единственным компонентом, который настроен для доступа к Интернету.

> [!NOTE]
> Программное обеспечение определенного прокси-сервера, следует использовать не рекомендуется. Убедитесь, выбрать прокси-сервера, совместимого с действия по настройке ниже.
>
>

На следующем рисунке примере показаны шаги три конфигурации необходимо использовать прокси-сервер HTTP.

* Маршруты ВМ приложения весь трафик HTTP граница общедоступный Интернет через прокси-сервер виртуальной Машины.
* Прокси-сервера виртуальной Машины позволяет входящий трафик из виртуальных машин в виртуальной сети.
* Сетевая группа безопасности с именем денежным Покрытием блокировки должен правило безопасности, которое разрешает исходящий Интернет-трафик от прокси-сервера виртуальной Машины.

Чтобы использовать прокси-сервер HTTP для взаимодействия с общедоступный Интернет, выполните следующие действия.

> [!NOTE]
> Эти шаги использовать определенные имена и значения для этого примера. Когда вы будете ввод (или вставки) данные в коде, используйте имена и значения для развертывания.

#### <a name="step-1-configure-outgoing-network-connections"></a>Шаг 1: Настройка исходящих сетевых соединений
###### <a name="for-windows-machines"></a>Для компьютеров Windows
Эта процедура устанавливает конфигурацию прокси-сервера для учетной записи локальной системы.

1. Загрузить [PsExec](https://technet.microsoft.com/sysinternals/bb897553).
2. Откройте Internet Explorer, выполнив следующую команду с повышенными привилегиями:

    ```
    psexec -i -s "c:\Program Files\Internet Explorer\iexplore.exe"
    ```

3. В Internet Explorer, перейдите к **средства** > **обозревателя** > **подключений** > **параметры локальной сети**.
4. Проверьте параметры прокси-сервера для учетной записи system. Задайте IP-адрес прокси-сервера и порт.
5. Закройте Internet Explorer.

Следующий сценарий настраивает конфигурацию прокси-сервера и использует его для любого исходящего трафика HTTP или HTTPS. Если настройки прокси-сервера для текущей учетной записи пользователя (не учетная запись локальной системы), используйте этот скрипт для их применения к SYSTEMACCOUNT.

```
   $obj = Get-ItemProperty -Path Registry::”HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Internet Settings\Connections"
   Set-ItemProperty -Path Registry::”HKEY_USERS\S-1-5-18\Software\Microsoft\Windows\CurrentVersion\Internet Settings\Connections" -Name DefaultConnectionSettings -Value $obj.DefaultConnectionSettings
   Set-ItemProperty -Path Registry::”HKEY_USERS\S-1-5-18\Software\Microsoft\Windows\CurrentVersion\Internet Settings\Connections" -Name SavedLegacySettings -Value $obj.SavedLegacySettings
   $obj = Get-ItemProperty -Path Registry::”HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Internet Settings"
   Set-ItemProperty -Path Registry::”HKEY_USERS\S-1-5-18\Software\Microsoft\Windows\CurrentVersion\Internet Settings" -Name ProxyEnable -Value $obj.ProxyEnable
   Set-ItemProperty -Path Registry::”HKEY_USERS\S-1-5-18\Software\Microsoft\Windows\CurrentVersion\Internet Settings" -Name Proxyserver -Value $obj.Proxyserver
```

> [!NOTE]
> Если вы заметили «(407) прокси-сервера требуется проверка подлинности» в журнале сервера прокси-сервера, проверьте, правильно настроить проверку подлинности.
>
>

###### <a name="for-linux-machines"></a>Для компьютеров Linux
Добавьте следующую строку в файл ```/etc/environment``` :

```
http_proxy=http://<proxy IP>:<proxy port>
```

Добавьте следующие строки в файл ```/etc/waagent.conf``` :

```
HttpProxy.Host=<proxy IP>
HttpProxy.Port=<proxy port>
```

#### <a name="step-2-allow-incoming-connections-on-the-proxy-server"></a>Шаг 2: Входящих подключений на прокси-сервера
1. Откройте брандмауэр Windows на прокси-сервере. Самый простой способ получить доступ к брандмауэр — для поиска **брандмауэр Windows в режиме повышенной безопасности**.
2. В **брандмауэр Windows в режиме повышенной безопасности** диалоговое окно, щелкните правой кнопкой мыши **правила для входящих подключений** и выберите **новое правило**.
3. В правило мастера на **тип правила** выберите **настраиваемый** и выберите **Далее**.
4. На **программы** выберите **все программы** и выберите **Далее**.
5. На **протокол и порты** введите следующие сведения и выбрать **Далее**:
   * Для **тип протокола**выберите **TCP**.
   * Для **локальный порт**выберите **определенные порты**. В поле ниже укажите номер порта прокси-сервера, который был настроен.
   * Для **удаленный порт**выберите **все порты**.

В оставшейся части мастера примите параметры по умолчанию, пока не дойдете до конца. Затем укажите имя правила. 

#### <a name="step-3-add-an-exception-rule-to-the-nsg"></a>Шаг 3: Добавление правила исключения для NSG
Она добавляет исключение к группе NSG. Это исключение позволяет TCP-трафика из любого порта на 10.0.0.5 любой адрес в Интернете на порт 80 (HTTP) или 443 (HTTPS). Если требуется определенный порт на общий доступ в Интернет, необходимо добавить этот порт для ```-DestinationPortRange```.

Введите следующую команду в командной строке Azure PowerShell:

```
Get-AzureNetworkSecurityGroup -Name "NSG-lockdown" |
Set-AzureNetworkSecurityRule -Name "allow-proxy " -Action Allow -Protocol TCP -Type Outbound -Priority 200 -SourceAddressPrefix "10.0.0.5/32" -SourcePortRange "*" -DestinationAddressPrefix Internet -DestinationPortRange "80-443"
```
