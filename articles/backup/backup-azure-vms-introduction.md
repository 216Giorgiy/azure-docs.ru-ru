# Резервное копирование виртуальной машины Azure: введение

В этом разделе содержатся общие сведения о защите виртуальных машин Azure с помощью службы архивации Microsoft Azure. Вы узнаете следующее:

- Как работает резервное копирование виртуальных машин Azure.
- Процедура резервного копирования виртуальной машины Azure.
- Необходимые условия для успешного резервного копирования.
- Типичные ошибки и способы их решения.
- Список неподдерживаемых сценариев и способы влияния на изменения в продукте.

Дополнительные сведения о виртуальных машинах Azure см. в разделе [Документация о виртуальных машинах](https://azure.microsoft.com/documentation/services/virtual-machines/).

## Цель резервного копирования виртуальной машины Azure
Облачные вычисления позволяют приложениям работать в масштабируемой среде с высокой надежностью, поэтому Майкрософт разработала виртуальные машины Azure. Виртуальные машины Azure создают важные данные, которые необходимо резервировать. Типичные сценарии, требующие восстановления данных из резервных копий, приведены ниже:

- случайное или злонамеренное удаление файлов;
- повреждение виртуальной машины во время обновления;
- случайное или злонамеренное удаление всей виртуальной машины.

Резервное копирование данных таких виртуальных машин можно выполнить двумя различными способами:

- создать резервные копии отдельных источников данных из виртуальной машины;
- создать резервную копию всей виртуальной машины.

Чаще используется второй способ, так как копией всей ВМ гораздо проще управлять. Кроме того, с ее помощью можно легко восстановить какое-либо приложение или операционную систему целиком. Службу архивации Azure можно использовать для резервного копирования отдельных данных на виртуальной машине или всей виртуальной машины.

Преимущества для бизнеса от использования службы архивации Azure для резервного копирования виртуальной машины:

- автоматизация резервного копирования и восстановления для виртуальных машин;
- согласованные резервные копии приложения, которые обеспечивают восстановление данных из согласованного состояния;
- отсутствие простоя во время резервного копирования виртуальной машины;
- резервное копирование виртуальных машин под управлением Windows или Linux;
- точки восстановления доступны для легкого восстановления в хранилище службы архивации Azure;
- автоматическая очистка и сборка мусора из старых точек восстановления.

## Как работает резервное копирование виртуальных машин Azure
Чтобы выполнить резервное копирование виртуальной машины, необходимо сначала создать снимок данных на определенный момент времени. Служба архивации Azure запускает задание резервного копирования по расписанию и модуль резервного копирования для создания снимка. Расширение резервного копирования работает совместно со службой VSS в операционных системах для достижения согласованности и вызывает API моментального снимка BLOB-объекта службы хранилища Azure при достижении согласованности. Это делается для получения согласованных моментальных снимков дисков виртуальной машины без необходимости завершать ее работы.

После создания моментального снимка данные передаются в хранилище службы архивации Azure. Служба определяет и передает только блоки, измененные с момента последнего резервного копирования, что способствует эффективной работе хранилища службы архивации. После завершения передачи данных моментальный снимок удаляется и создается точка восстановления. Эта точка восстановления отображается на портале управления Azure.

![Архитектура резервного копирования виртуальных машин Azure](./media/backup-azure-vms-introduction/vmbackup-architecture.png)

>[AZURE.NOTE]Для виртуальных машин Linux возможно только резервное копирование с согласованием файлов.

## Вычисление защищенных экземпляров
Виртуальные машины Azure, для которых создаются резервные копии с помощью службы архивации Azure, подлежат оплате услуг в соответствии с [расценками службы архивации Azure](http://azure.microsoft.com/pricing/details/backup/). Вычисление защищенных экземпляров основано на *фактическом* размере виртуальной машины, который представляет собой сумму всех данных на виртуальной машине за исключением "диска ресурсов". Оплата зависит *не* от максимальных размеров всех дисков с данными, которые подключены к виртуальной машине, а от фактических данных, хранящихся на диске с данными. Аналогичным образом счет за хранилище службы архивации основан на объеме данных, хранящихся в службе архивации Azure, который представляет собой сумму фактических данных в каждой точке восстановления.

Например, возьмем виртуальную машину стандартного размера A2 с двумя дополнительными дисками максимальным объемом 1 ТБ каждый. В таблице ниже приводятся фактические данные, хранящиеся на каждом из этих дисков.

|Тип диска|Максимальный размер|Фактические данные|
|---------|--------|------|
| Диск ОС | 1023 ГБ | 17 ГБ |
| Локальный диск или диск ресурсов | 135 ГБ | 5 ГБ (не включен для резервного копирования) |
| Диск данных 1 |	1023 ГБ | 30 ГБ |
| Диск данных 2 | 1023 ГБ | 0 ГБ |

*Фактический* размер виртуальной машины в данном случае равен 17 ГБ + 30 ГБ + 0 ГБ = 47 ГБ. Этот показатель равен размеру защищенного экземпляра, на котором основывается ежемесячный счет. По мере роста объема данных на виртуальной машине изменяется размер защищенного экземпляра, который используется для выставления счетов.

Выставление счетов не начинается до завершения первого успешного резервного копирования. На этом этапе начнется выставление счетов за хранение и защищенные экземпляры. Выставление счетов продолжается до тех пор, пока существуют *резервные копии данных, хранящиеся в службе архивации Azure* для виртуальной машины. Остановка защиты не останавливает выставление счетов, если резервная копия данных продолжает храниться. Выставление счетов для определенной виртуальной машины будет остановлено, только если остановить защиту *и* удалить все резервные копии данных. Когда нет активных заданий резервного копирования (при остановке защиты), размер виртуальной машины во время последнего успешного резервного копирования становится размером защищенного экземпляра, на основе чего выставляется ежемесячный счет.

## Предварительные требования
### 1. Хранилище службы архивации
Чтобы начать резервное копирование виртуальных машин Azure, необходимо сначала создать хранилище службы архивации. Хранилище — это объект, в котором хранятся созданные резервные копии и точки восстановления. Хранилище также содержит политики резервного копирования, которые будут применяться к виртуальным машинам, для которых настроено резервное копирование.

На рисунке ниже показаны связи между различными объектами службы архивации Azure. ![Объекты и связи службы архивации Azure](./media/backup-azure-vms-introduction/vault-policy-vm.png)

### Создание хранилища службы архивации

1. Выполните вход на [Портал управления](http://manage.windowsazure.com/).

2. Последовательно щелкните **Создать** > **Службы данных** > **Службы восстановления** > **Хранилище архивации** > **Быстрое создание**. Если с учетной записью вашей организации связано несколько подписок, выберите ту, которую следует связать с хранилищем службы архивации. В каждой подписке Azure может быть несколько хранилищ службы архивации для организации защищаемых виртуальных машин.

3. В поле **Имя** введите понятное имя для идентификации хранилища. Это имя должно быть уникальным для каждой подписки.

4. В поле **Область** выберите географический регион для хранилища архивации. Обратите внимание, что хранилище должно быть в том же регионе, что и виртуальные машины, которые необходимо защитить. При наличии виртуальных машин в разных регионах создайте хранилище в каждом из них. Нет необходимости указывать учетные записи хранения для хранения данных резервного копирования — хранилище и служба архивации Azure сделают это автоматически. ![Создание хранилища службы архивации](./media/backup-azure-vms-introduction/backup_vaultcreate.png)

    >[AZURE.NOTE]Резервное копирование виртуальных машин с помощью службы архивации Azure поддерживается только в определенных регионах. См. список [поддерживаемых регионов](http://azure.microsoft.com/regions/#services). Если искомый регион в настоящее время не поддерживается, он не будет отображаться в раскрывающемся списке при создании хранилища.

5. Выберите команду **Создать хранилище**. Для создания резервного хранилища может потребоваться некоторое время. Уведомления о состоянии процесса см. в нижней части портала. ![Всплывающее уведомление о создании хранилища](./media/backup-azure-vms-introduction/creating-vault.png)

6. Отобразится сообщение об успешном создании хранилища, а на странице служб восстановления хранилище будет иметь активное состояние. ![Список хранилищ службы архивации](./media/backup-azure-vms-introduction/backup_vaultslist.png)

7. Если щелкнуть хранилище службы архивации, откроется страница **Быстрый запуск** с инструкциями для резервного копирования виртуальных машин Azure. ![Инструкции для резервного копирования виртуальной машины на странице панели мониторинга](./media/backup-azure-vms-introduction/vmbackup-instructions.png)

    >[AZURE.NOTE]Сразу после создания хранилища убедитесь, что выбрано соответствующее значение избыточности хранилища. Подробнее о [настройке избыточности хранилища в хранилище службы архивации][vault-storage-redundancy].

### 2. Агент виртуальной машины
Перед началом резервного копирования виртуальной машины Azure убедитесь, что на ней правильно установлен агент виртуальной машины. Для резервного копирования виртуальной машины службы архивации Azure устанавливает расширение для агента виртуальной машины. Так как агент виртуальной машины является необязательным компонентом при создании виртуальной машины, убедитесь, что установлен флажок агента ВМ до ее подготовки.

См. дополнительные сведения об [агенте виртуальной машины](https://go.microsoft.com/fwLink/?LinkID=390493&clcid=0x409) и [ее установке](http://azure.microsoft.com/blog/2014/04/15/vm-agent-and-extensions-part-2/).

>[AZURE.NOTE]Если вы планируете перенести виртуальную машину из локального центра обработки данных в Azure, убедитесь, что перед началом переноса скачан и установлен MSI агента виртуальной машины. Это также касается виртуальных машин, защищенных в Azure с помощью Azure Site Recovery.

## Ограничения во время предварительного просмотра

- Резервное копирование виртуальных машин, к которым подключено более пяти дисков, не поддерживается.
- Резервное копирование виртуальных машин с использованием хранилища уровня Premium не поддерживается.
- Резервное копирование виртуальных машин с использованием нескольких сетевых адаптеров или конфигурации с балансировкой нагрузки не поддерживается.
- Замена существующей виртуальной машины во время восстановления не поддерживается. Сначала необходимо удалить существующую виртуальную машину и все связанные диски, а затем восстановить данные из резервной копии.
- Резервное копирование виртуальных машин, восстановленных с помощью Azure Site Recovery, не поддерживается.
- Резервное копирования и восстановление между регионами не поддерживается.
- Резервное копирование виртуальных машин с помощью службы архивации Azure поддерживается только в определенных регионах. См. список [поддерживаемых регионов](http://azure.microsoft.com/regions/#services). Если искомый регион в настоящее время не поддерживается, он не будет отображаться в раскрывающемся списке при создании хранилища.
- Резервное копирование виртуальных машин с помощью службы архивации Azure поддерживается только в определенных версиях операционных систем:
  - **Linux**: список дистрибутивов, одобренных Azure, доступен [здесь](../virtual-machines-linux-endorsed-distributions.md). Другие собственные дистрибутивы Linux должны также работать, если агент виртуальной машины доступен на виртуальной машине.
  - **Windows Server**: версии до Windows Server 2008 R2 не поддерживаются.
