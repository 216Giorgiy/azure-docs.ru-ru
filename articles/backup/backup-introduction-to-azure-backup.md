# Что такое служба архивации Azure?
Служба архивации Azure — служба, используемая для резервного копирования и восстановления данных в Microsoft Cloud. Она заменяет существующее локальное или автономное решение для резервного копирования на надежное, безопасное и конкурентоспособное облачное решение. Она также позволяет защитить ресурсы-контейнеры, которые используются в облаке. Служба архивации Azure предоставляет службы восстановления на основе масштабируемой, долговечной и высокодоступной инфраструктуры мирового уровня.

[Просмотрите видеообзор службы архивации Azure](https://azure.microsoft.com/documentation/videos/what-is-azure-backup/)

## Зачем использовать службу архивации Azure?
Традиционные решения для резервного копирования используют облако в качестве конечной точки подобно диску или магнитной ленте. Несмотря на простоту, у этого подхода есть ограничения. Он предоставляет не все преимущества базовой облачной платформы. Это ведет к неэффективному и дорогому решению. Служба архивации Azure, с другой стороны, предоставляет все преимущества мощного и доступного облачного решения для резервного копирования. Ниже приведены некоторые из основных преимуществ службы архивации Azure.

| Функция | Преимущество |
| ------- | ------- |
| Автоматическое управление хранилищем | Отсутствие существенных затрат на локальные устройства хранения данных. Служба архивации Azure обеспечивает автоматическое выделение хранилища для резервных копий и управление им, при этом оплата осуществляется по мере использования. |
| Неограниченное масштабирование | Вы можете использовать все преимущества гарантии высокой доступности без расходов на обслуживание и мониторинг. Служба архивации Azure использует базовые возможности и масштабирование облака Azure, а также возможности ненавязчивого автоматического масштабирования. |
| Несколько вариантов хранения | Выберите хранилище для резервных копий в соответствии со своими потребностями: <li>локально избыточное хранилище блочных BLOB-объектов идеально подходит для клиентов, задумывающихся о расходах. При этом решение обеспечивает защиту данных от сбоев локального оборудования. <li>Хранилище блочных BLOB-объектов с георепликацией предоставляет три дополнительные копии в сопряженном центре обработки данных. Это гарантирует высокую доступность данных резервных копий даже в случае аварии на уровне веб-сайта Azure. |
| Отсутствие ограничений на передачу данных | Исходящие данные, передаваемые во время восстановления из хранилища службы архивации, не тарифицируются. Входящие данные, передаваемые в хранилище Azure, также не тарифицируются. |
| Централизованное управление | Портал Azure предоставляет простое и привычное взаимодействие с пользователем. По мере развития службы такие функции, как централизованное управление, позволят управлять инфраструктурой резервного копирования из одного расположения. |
| Шифрование данных | Эта функция делает безопасными передачу и хранение данных клиента в общедоступном облаке. Парольная фраза для шифрования содержится в источнике и никогда не передается и не хранится в Azure. Для восстановления любых данных необходим ключ шифрования, и только у клиента есть полный доступ к данным в службе. |  
| Согласованные с приложениями резервные копии | Согласованность резервных копий с приложениями в Windows гарантируют, что во время восстановления не потребуется исправление. Это уменьшает целевое время восстановления и позволяет клиентам быстрее вернуть рабочее состояние. |
| Длительный период удержания | Вместо того, чтобы платить за автономные решения для резервного копирования на магнитную ленту, клиенты могут выполнять его в службу Azure. Это привлекательное решение с семантикой резервного копирования на магнитную ленту, которое отличается очень низкой стоимостью. |

Хранилище службы архивации — предпочтительное хранилище для всех компонентов. System Center DPM и сервер службы архивации позволяют также создавать копии на локальном диске, но только System Center DPM предоставляет возможность записи данных на ленточное устройство хранения.

#### Добавочное резервное копирование
Независимо от целевого хранилища (диск, магнитная лента, хранилище службы архивации), каждый компонент поддерживает добавочную архивацию. Это позволяет гарантировать, что резервные копии эффективно используют хранилище и время, принимая только добавочные изменения с момента последнего резервного копирования и передавая их в целевое хранилище. Кроме того, резервные копии сжимаются для уменьшения используемого объема хранилища.

Компонент, который не выполняет сжатие, является расширением виртуальной машины. Все данные резервных копий копируются без сжатия из пользовательской учетной записи хранения в хранилище службы архивации в том же регионе. Хотя это немного увеличивает используемый объем хранилища, хранение данных без сжатия позволяет сократить время восстановления.

#### Дедупликация
Для System Center DPM и сервера службы архивации дедупликация поддерживается при [развертывании в виртуальной машине Hyper-V](http://blogs.technet.com/b/dpm/archive/2015/01/06/deduplication-of-dpm-storage-reduce-dpm-storage-consumption.aspx). Дедупликация выполняется на уровне узла. Для этого используется функция дедупликации Windows Server на виртуальных жестких дисках, подключенных к виртуальной машине в качестве хранилища для резервных копий.

>[AZURE.WARNING] Дедупликация в Azure недоступна ни для одного из компонентов службы архивации. При развертывании System Center DPM и сервера службы архивации в Azure нельзя выполнить дедупликацию дисков, подключенных к виртуальной машине.

Все данные трафика резервного копирования с серверов в хранилище службы архивации подвергаются шифрованию с использованием AES 256. Данные передаются через защищенное соединение HTTPS. Данные резервных копий также хранятся в зашифрованном виде в хранилище службы архивации. Парольная фраза для расшифровки есть только у клиента. Мы не можем расшифровывать данные резервных копий.

Чтобы создать резервные копии виртуальных машин Azure, необходимо явно настроить шифрование *в* виртуальной машине. Используйте BitLocker на виртуальных машинах Windows и **dm-crypt** на виртуальных машинах Linux. Служба архивации Azure не предусматривает автоматическое шифрование резервных копий данных, поступающих по указанному пути.

## Различия службы архивации и Azure Site Recovery
Многие клиенты путают восстановление резервных копий и аварийное восстановление. В обоих случаях происходит сбор данных и предоставляется семантика восстановления, но основные предлагаемые преимущества процессов отличаются.

Служба архивации Azure позволяет создавать резервные копии данных локально или в облаке. Azure Site Recovery обеспечивает координирование репликации, отработки отказа и восстановления размещения виртуальных машин и физических серверов. Только их сочетание создает полное решение для аварийного восстановления, так как стратегия аварийного восстановления данных должна предусматривать надежное хранение *и* восстановление данных (за это отвечает служба архивации), а также поддержку доступности рабочих нагрузок при сбоях (Site Recovery).

Чтобы принимать решения, касающиеся резервного копирования и аварийного восстановления, необходимо сформировать представление о следующих важных понятиях.

| Концепция | Сведения | Архивация | Аварийное восстановление |
| ------- | ------- | ------ | ----------------- |
| Целевая точка восстановления (RPO) | Объем потери данных, который допустим в случае, если необходимо выполнить восстановление. | Допустимые целевые точки восстановления решений для резервного копирования сильно отличаются. Резервное копирование виртуальных машин обычно предусматривает одну RPO в день, тогда как резервное копирование баз данных — всего в 15 минут. | Решения аварийного восстановления отличаются крайне низкими целевыми точками восстановления. Копия аварийного восстановления может отставать на несколько секунд или несколько минут. |
| Целевое время восстановления (RTO) | Количество времени, необходимое для полного восстановления. | Из-за большего количества RPO объем данных, которые нужно обработать решению для резервного копирования, обычно намного больше. Это увеличивает RTO. Например, восстановление данных с магнитных лент может занимать несколько дней в зависимости от того, сколько времени требуется для транспортировки ленты из места ее размещения. | Для решений аварийного восстановления характерно меньшее RTO, так как они более синхронизированы с источником и им необходимо обрабатывать меньшее количество изменений. |
| Сохранение | Как долго должны храниться данные | <p>В сценариях, когда требуется оперативное восстановление (при повреждении данных, случайном удалении файла, сбое операционной системы), резервные копии данных обычно хранятся в течение 30 дней или меньше.</p> <p>С точки зрения соответствия может потребоваться хранение данных в течение нескольких месяцев или даже лет. Данные резервных копий идеально подходят для архивации в таких случаях.</p> | Для аварийного восстановления необходимы только данные оперативного восстановления. Обычно оно длится несколько часов и до одного дня. Так как в решениях для аварийного восстановления используется детальный сбор данных, длительное хранение данных аварийного восстановления не рекомендуется. |
