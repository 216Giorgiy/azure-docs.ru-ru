# Что такое служба архивации Azure?
Служба архивации Azure — служба, используемая для резервного копирования и восстановления данных в Microsoft Cloud. Она заменяет существующее локальное или автономное решение для резервного копирования на облачное решение, которое является надежным, безопасным и конкурентоспособным. Она также защищает ресурсы, запущенные в облаке. Служба архивации Azure предоставляет службы восстановления на основе масштабируемой, долговечной и высокодоступной инфраструктуры мирового уровня.

[Просмотрите видеообзор службы архивации Azure](https://azure.microsoft.com/documentation/videos/what-is-azure-backup/)

## Зачем использовать службу архивации Azure?
Традиционные решения для резервного копирования рассматривают облако в качестве конечной точки аналогично диску или ленте. Хотя этот подход прост, он ограничен и использует не все преимущества базовой платформы облака. Это ведет к неэффективному и дорогому решению. Служба архивации Azure, с другой стороны, предоставляет все преимущества мощного и доступного облачного решения для резервного копирования. Ниже перечислены некоторые из основных преимуществ, предоставляемых службой архивации Azure.

| ФУНКЦИЯ | ПРЕИМУЩЕСТВО |
| ------- | ------- |
| Автоматическое управление хранилищем | Отсутствие существенных затрат на локальные устройства хранения данных. Служба архивации Azure автоматически выделяет резервное хранилище и управляет им с оплатой по мере использования. |
| Неограниченное масштабирование | Гарантия высокой доступности без расходов на обслуживание и мониторинг. Служба архивации Azure использует базовые возможности и масштабирование облака Azure, а также возможности ненавязчивого автоматического масштабирования. |
| Несколько вариантов хранения | Выберите резервное хранилище в соответствии со своими потребностями: <li>локально избыточное хранилище блочных BLOB-объектов (LRS) идеально подходит для клиентов, которые уделяют внимание расходам. При этом сохраняется защита данных от сбоев локального оборудования. <li>Хранилище блочных BLOB-объектов с георепликацией (GRS) предоставляет 3 дополнительные копии в сопряженном центре обработки данных, что гарантирует, что данные резервных копий останутся высокодоступными даже в случае аварии на уровне Azure. |
| Отсутствие ограничений на передачу данных | Исходящие данные, передаваемые во время восстановления из хранилища службы архивации Azure, не тарифицируются. Входящие данные, передаваемые в хранилище Azure, также не тарифицируются. |
| Централизованное управление | Простота и привычность портала Azure. По мере развития службы такие функции, как централизованное управление, позволят управлять инфраструктурой резервного копирования из одного расположения. |
| Шифрование данных | Безопасная передача и хранение данных клиента в общедоступном облаке. Парольная фраза для шифрования хранится в источнике и никогда не передается и не хранится в Azure. Для восстановления любых данных необходим ключ шифрования, и только клиент имеет полный доступ к данным в службе. |  
| Интеграция VSS | Согласованные резервные копии приложения в Windows гарантируют, что исправление во время восстановления не потребуется. Это уменьшает целевое время восстановления и позволяет клиентам быстрее вернуться в рабочее состояние. |
| Долгосрочное хранение | Вместо оплаты за автономные решения резервного копирования на ленту клиенты могут выполнять резервное копирование в службу Azure, которая обеспечивает привлекательное решение с семантикой решений резервного копирования на ленту по очень низкой стоимости. |

Хранилище службы архивации Azure является предпочтительным хранилищем для всех компонентов. SCDPM и сервер архивации Azure позволяют также создавать копию на локальном диске, но только SCDPM предоставляет возможность записи данных на ленточное устройство хранения.

#### Добавочное резервное копирование
Независимо от целевого хранилища (диск, лента, резервное хранилище), каждый компонент поддерживает добавочное резервное копирование. Это гарантирует, что резервные копии эффективно используют хранилище и время, принимая только добавочные изменения с момента последнего резервного копирования и передавая их в целевое хранилище. Кроме того, резервные копии сжимаются для уменьшения используемого объема хранилища.

>[AZURE.NOTE]Компонент, который не выполняет сжатие, является расширением виртуальной машины. Все данные резервных копий копируются без сжатия из пользовательской учетной записи хранения в хранилище архивации в том же регионе. Хотя это немного увеличивает используемый объем хранилища, хранение данных без сжатия позволяет сократить время восстановления.

#### Дедупликация
Для SCDPM и сервера архивации Azure дедупликация поддерживается при [развертывании в виртуальной машине Hyper-V](http://blogs.technet.com/b/dpm/archive/2015/01/06/deduplication-of-dpm-storage-reduce-dpm-storage-consumption.aspx). Дедупликация выполняется на уровне узла, используя функцию дедупликации Windows Server — на виртуальных жестких дисках, присоединенных к виртуальной машине как резервное хранилище.

>[AZURE.WARNING]Дедупликация в Azure недоступна ни для одного из компонентов службы архивации Azure! При развертывании SCDPM и сервера архивации Azure в Azure дедупликацию дисков, подключенных к виртуальной машине, выполнить нельзя.

Служба архивации Azure не производит автоматического шифрования резервных копий данных, поступающих по указанному пути.

## Чем служба архивации Azure отличается от Azure Site Recovery?
Многие клиенты путают резервное копирование и аварийное восстановление. В обоих случаях происходит сбор данных и предоставляется семантика восстановления, но основные предлагаемые преимущества в каждом случае различаются.

Служба архивации Azure создает резервные копии данных локально или в облаке. Azure Site Recovery координирует репликацию, отработку отказа и восстановление виртуальных машин и физических серверов. Для полного решения аварийного восстановления необходимо и то, и другое, поскольку стратегия аварийного восстановления данных должна надежно хранить и восстанавливать данные (служба архивации Azure) и поддерживать доступность рабочих нагрузок при возникновении сбоев (Azure Site Recovery).

При принятии решений, касающихся резервного копирования и аварийного восстановления, необходимо иметь представление о нескольких важных понятиях:

| ПОНЯТИЕ | ПОДРОБНЫЕ СВЕДЕНИЯ | РЕЗЕРВНОЕ КОПИРОВАНИЕ | АВАРИЙНОЕ ВОССТАНОВЛЕНИЕ (DR) |
| ------- | ------- | ------ | ----------------- |
| Целевая точка восстановления (RPO) | Объем потери данных, который допустим в случае, если необходимо выполнить восстановление. | Решения резервного копирования имеют большие различия в допустимых RPO. Резервное копирование виртуальных машин обычно имеет RPO в 1 день, тогда как резервное копирование баз данных — всего в 15 минут. | Решения аварийного восстановления имеют крайне низкие целевые точки восстановления. Копия аварийного восстановления может отставать на несколько секунд или несколько минут. |
| Целевое время восстановления (RTO) | Количество времени, необходимое для полного восстановления. | Из-за большего RRO объем данных, которые нужно обработать решению для резервного копирования, обычно намного выше, и это приводит к большим RTO. Например, восстановление данных с лент может занять несколько дней в зависимости от того, какое время требуется для транспортировки ленты из места ее размещения. | Решения аварийного восстановления иметь меньший RTO, так как они более синхронизированы с источником, и поэтому необходимо обработать меньшее количество изменений. |
| Сохранение | Как долго должны храниться данные | <p>Для сценариев, требующих оперативного восстановления (повреждение данных, случайное удаление файла, сбои операционной системы), резервные копии данных обычно сохраняются в течение 30 дней или меньшего срока.</p> <p>С точки зрения соответствия может потребоваться хранение данных в течение нескольких месяцев или даже лет. Данные резервных копий идеально подходят для архивации в таких случаях.</p> | Для аварийного восстановления необходимы только данные оперативного восстановления — обычно в пределах нескольких часов или дня. Из-за детального сбора данных, используемых в решениях для аварийного восстановления, длительное хранение с использованием данных аварийного восстановления не рекомендуется. |
