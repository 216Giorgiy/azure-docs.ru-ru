# <a name="connect-offline-computers-to-operations-management-suite-by-using-oms-gateway"></a>Подключение компьютеров без доступа к Интернету к Operations Management Suite с помощью шлюза OMS

Компьютеры, которые управляются при помощи Operations Management Suite (OMS) и отслеживаются с использованием System Center Operations Manager (Operations Manager), могут отправлять данные в службу Operations Management Suite при отсутствии доступа к Интернету. Шлюз OMS — это прокси-сервер переадресации HTTP, в котором HTTP-туннелирование осуществляется с помощью команды HTTP CONNECT. Шлюз OMS может собирать данные и отправлять их в службу Operations Management Suite от имени компьютеров, не подключенных к сети.  

Шлюз OMS поддерживает следующее:

* гибридные рабочие роли Runbook службы автоматизации Azure;  
* компьютеры Windows с Microsoft Monitoring Agent, которые напрямую подключены к рабочей области Operations Management Suite.
* System Center Operations Manager 2012 с пакетом обновления 1 (SP1) и накопительным пакетом обновления 7 (UR7), Operations Manager 2012 R2 с накопительным пакетом обновления 3 (UR3) или группу управления Operations Manager 2016, интегрированную с Operations Management Suite.  

Некоторые политики ИТ-безопасности запрещают подключение компьютеров в сети к Интернету. Это может относиться к POS-устройствам или серверам, поддерживающим ИТ-службы. Тем не менее вам необходимо подключить компьютеры к Operations Management Suite для мониторинга и управления. Для компьютеров без доступа к Интернету можно настроить прямое взаимодействие со шлюзом OMS. Шлюз будет получать конфигурации и передавать данные от имени этих компьютеров.  

Если эти компьютеры с агентом Operations Management Suite настроены для подключения к рабочей области Operations Management Suite напрямую, вместо этого все они будут взаимодействовать через шлюз OMS. Шлюз передает данные от агентов непосредственно в Operations Management Suite. Шлюз не анализирует передаваемые данные.

Если группа управления Operations Manager интегрирована с Operations Management Suite, вы можете настроить серверы управления для подключения к шлюзу OMS. Серверы будут получать сведения о конфигурации и отправлять собранные данные в зависимости от настроенного решения. Агенты Operations Manager отправляют на сервер управления некоторые данные, такие как оповещения, оценки конфигурации, размер экземпляра и емкость данных Operations Manager. Другие данные большого объема, например журналы служб IIS, сведения о производительности и событиях безопасности, отправляются непосредственно в шлюз OMS. 

Сервер шлюза Operations Manager, который развертывается в сети периметра (также известной как ***экранированная подсеть***) или другой изолированной сети для отслеживания ненадежных систем, не сможет взаимодействовать со шлюзом OMS. Серверы Operations Manager шлюза могут сообщать данные только серверу управления. Если группа управления Operations Manager настроена для взаимодействия со шлюзом OMS, сведения о конфигурации прокси-сервера автоматически распространяются на каждый компьютер под управлением агента, настроенного для сбора данных для Azure Log Analytics. Сведения о конфигурации прокси-сервера автоматически распределяются, даже если это параметр не указан.    

Обеспечить высокую доступность для компьютеров, подключенных напрямую, или для групп Operations Management, взаимодействующих с OMS через шлюз, можно с использованием балансировки нагрузки сети, чтобы перенаправлять и распределять трафик между несколькими серверами шлюза. Если один сервер шлюза выходит из строя, трафик перенаправляется на другие доступные узлы.  

Мы советуем установить агент Operations Management Suite на компьютере, на котором выполняется программное обеспечение шлюза OMS. Таким образом можно отслеживать шлюз OMS и анализировать данные о производительности или событиях. Кроме того, агент помогает шлюзу OMS определить конечные точки службы, с которыми необходимо взаимодействовать.

Каждый агент должен иметь сетевое подключение к шлюзу. Так он сможет автоматически передавать данные в шлюз и извлекать их из него. Не рекомендуется устанавливать шлюз на контроллере домена.

## <a name="prerequisites"></a>Предварительные требования

Компьютер, который назначен для запуска шлюза OMS, должен обладать следующими характеристиками:

* операционная система Windows 10, Windows 8.1, Windows 7, Windows Server 2012, Windows Server 2012 R2, Windows Server 2008 или Windows Server 2008 R2;
* Microsoft .NET Framework 4.5;
* 4-ядерный процессор (как минимум);
* 8 ГБ памяти (как минимум).

## <a name="download-oms-gateway"></a>Скачивание шлюза OMS

Существует три способа получения файла установки шлюза OMS последней версии.

* Скачать из [Центра загрузки Майкрософт](https://www.microsoft.com/download/details.aspx?id=54443).

* Скачать с портала OMS. После входа в рабочую область OMS:

    1. выберите **Параметры** > **Подключенные источники** > **Серверы с Windows**; 
    2. выберите **Скачать шлюз OMS**.

* Скачать [на портале Azure](https://portal.azure.com). После входа сделайте следующее:  

   1. в списке служб выберите **Log Analytics**;  
   2. Выберите рабочую область.
   3. в колонке рабочей области в разделе **Общие** выберите **Быстрый запуск**;
   4. в разделе **Выбор источника данных для подключения к рабочей области** щелкните **Компьютеры**;
   5. в колонке **Direct Agent** выберите **Скачать шлюз OMS**.<br><br> ![скачивание шлюза OMS](./media/log-analytics-oms-gateway/download-gateway.png)


## <a name="install-oms-gateway"></a>Установка шлюза OMS

Чтобы установить шлюз, выполните следующие действия. 

> [!NOTE]
> Если установлена предыдущая версия шлюза, которая раньше называлась сервером пересылки Log Analytics, она будет обновлена до текущего выпуска шлюза OMS.
>

1. В папке назначения дважды щелкните файл **OMS Gateway.msi**.
2. На странице **приветствия** нажмите кнопку **Далее**.<br><br> ![Мастер установки шлюза](./media/log-analytics-oms-gateway/gateway-wizard01.png)<br> 
3. На странице **лицензионного соглашения** установите флажок **Я принимаю условия лицензионного соглашения**. 
4. На странице со сведениями об **адресе порта и прокси-сервера**:

   1. Введите номер TCP-порта, используемого для шлюза. Программа установки настроит правило входящего трафика с этим номером порта в брандмауэре Windows. По умолчанию используется значение 8080.
      Допустимый диапазон номеров порта: 1–65535. При вводе значения, которое не входит в этот диапазон, появится сообщение об ошибке.
   2. Если для сервера, где установлен шлюз, требуется взаимодействие с прокси-сервером, введите адрес прокси-сервера, к которому должен подключиться шлюз (при необходимости). Например, **http://myorgname.corp.contoso.com:80**. Если оставить это поле пустым, шлюз будет пытаться подключиться к Интернету напрямую. Если ваш прокси-сервер требует проверки подлинности, введите имя пользователя и пароль.<br><br> ![Настройка прокси-сервера в мастере установки шлюза](./media/log-analytics-oms-gateway/gateway-wizard02.png)<br>   
5. Если Центр обновления Майкрософт отключен, появится страница **Центр обновления Майкрософт**. Тут можно его включить. В противном случае перейдите к следующему шагу.
6. На странице **Конечная папка** оставьте папку по умолчанию (C:\ProgramFiles\OMS Gateway) или укажите расположение, куда необходимо установить шлюз.
7. На странице **Готово к установке** выберите **Установить**. Может появиться диалоговое окно с запросом разрешения на установку. Выберите **Да**.
8. Выберите **Готово**. Проверить, запущена ли служба, можно в привязке services.msc. Если это так, **шлюз OMS** появится в списке служб. Его состояние должно быть **Выполняется**.<br><br> ![Проверка служб шлюза OMS](./media/log-analytics-oms-gateway/gateway-service.png)  

## <a name="set-up-network-load-balancing"></a>Настройка балансировки сетевой нагрузки 
Вы можете настроить шлюз для обеспечения высокой доступности с применением балансировки сетевой нагрузки, используя функцию балансировки сетевой нагрузки или аппаратные подсистемы балансировки нагрузки. Подсистема балансировки нагрузки управляет трафиком, перенаправляя запрошенные подключения от агентов Operations Management Suite и серверов управления Operations Manager по узлам. Если один сервер шлюза выходит из строя, трафик перенаправляется на другие узлы.

Сведения о проектировании и развертывании кластера балансировки сетевой нагрузки Windows Server 2016 см. в статье о [балансировке сетевой нагрузки](https://technet.microsoft.com/windows-server-docs/networking/technologies/network-load-balancing).  

Чтобы настроить кластер балансировки сетевой нагрузки Windows Server, выполните следующие действия:  

1. Войдите на сервер Windows, который входит в кластер балансировки сетевой нагрузки, используя учетную запись администратора.  
2. В диспетчере серверов откройте диспетчер балансировки сетевой нагрузки.
3. Выберите **Инструменты**, а затем **Диспетчер балансировки сетевой нагрузки**.
4. Чтобы подключить сервер шлюза OMS с установленным компонентом Microsoft Monitoring Agent, щелкните правой кнопкой мыши IP-адрес кластера и выберите **Добавить узел в кластер**.<br><br> ![Диспетчер балансировки сетевой нагрузки — "Добавить узел в кластер"](./media/log-analytics-oms-gateway/nlb02.png)<br> 
5. Введите IP-адрес сервера шлюза, который нужно подключить.<br><br> ![Диспетчер балансировки сетевой нагрузки — "Добавить узел в кластер: подключить"](./media/log-analytics-oms-gateway/nlb03.png)

### <a name="set-up-the-operations-management-suite-agent-and-operations-manager-to-use-the-oms-gateway-as-a-proxy-server"></a>Настройка агента Operations Management Suite и Operations Manager для использования шлюза OMS в качестве прокси-сервера

### <a name="set-up-a-standalone-operations-management-suite-agent"></a>Настройка автономного агента Operations Management Suite
Сведения о настройке параметров прокси-сервера для агента см. в статье [Настройка параметров прокси-сервера и брандмауэра в службе Log Analytics](log-analytics-proxy-firewall.md). В этом случае шлюзом является прокси-сервер. При развертывании нескольких серверов шлюзов после балансировщика сетевой нагрузки конфигурацией прокси-сервера агента Operations Management Suite является виртуальный IP-адрес балансировщика сетевой нагрузки:<br><br> ![Свойства Microsoft Monitoring Agent — настройки прокси-сервера](./media/log-analytics-oms-gateway/nlb04.png)

### <a name="set-up-operations-manager-all-agents-use-the-same-proxy-server"></a>Настройка Operations Manager (все агенты используют один прокси-сервер)
Настройте Operations Manager, чтобы добавить сервер шлюза. Конфигурация прокси-сервера Operations Manager применяется ко всем агентам, передающим данные в Operations Manager, автоматически, даже если соответствующий параметр не указан.

Чтобы использовать шлюз OMS с Operations Manager, требуется следующее:

* На сервере шлюза необходимо установить компонент Microsoft Monitoring Agent (версии **8.0.10900.0** или более поздней). Его необходимо настроить для рабочих областей агента Operations Management Suite, с которыми нужно установить взаимодействие.
* Шлюз должен иметь подключение к Интернету или должен быть подключен к прокси-серверу с подключением к Интернету.

> [!NOTE]
> Если не указать значение для шлюза, всем агентам передаются пустые значения.

Чтобы подключить сервер Operations Manager к рабочей области агента Operations Management Suite:

1. В консоли Operations Manager перейдите к **Operations Management Suite** > **Подключение** > **Настройка прокси-сервера**.<br><br> ![Operations Manager — настройка прокси-сервера](./media/log-analytics-oms-gateway/scom01.png)<br> 
2. Выберите **Использовать прокси-сервер для доступа к набору Operations Management Suite**. Введите IP-адрес сервера шлюза OMS или виртуальный IP-адрес балансировщика сетевой нагрузки. Он должен начинаться с префикса **http://**.<br><br> ![Operations Manager — адрес прокси-сервера](./media/log-analytics-oms-gateway/scom02.png)<br> 
3. Выберите **Готово**. 

### <a name="set-up-operations-manager-specific-agents-use-the-proxy-server"></a>Настройка Operations Manager (определенные агенты используют прокси-сервер)
В больших или сложных средах может потребоваться, чтобы только определенные серверы (или группы) использовали сервер шлюза OMS. Для этих серверов невозможно обновить агент Operations Manager напрямую. Это значение перезаписывается глобальным значением группы управления. Вместо этого необходимо переопределить правило, используемое для передачи этих значений.

> [!NOTE] 
> Вы можете использовать этот же прием конфигурации, чтобы запустить несколько серверов шлюза OMS в среде. Например, для каждого региона может понадобиться указать определенные серверы шлюза OMS.

1. В консоли System Center Operations Manager выберите рабочую область **Создание**.  
2. Выберите **Правила** и на панели инструментов System Center Operations Manager нажмите кнопку **Область**. Если эта кнопка недоступна, убедитесь, что на панели **Мониторинг** выбран именно объект, а не папка. Появится диалоговое окно **Ориентация объектов пакета управления** со списком общих целевых классов, групп или объектов. 
3. В поле **Искать** введите **служба работоспособности** и выберите ее в списке. Нажмите кнопку **ОК**.  
4. На панели инструментов консоли Operations Manager найдите **правило настройки прокси-сервера помощника**. Щелкните **Переопределения** и выберите команду **Override the Rule\For a specific object of class: Health Service** (Переопределить правило для конкретного объекта класса: служба работоспособности). Выберите конкретный объект в списке. При необходимости можно создать пользовательскую группу, содержащую объект службы работоспособности серверов, которые нужно переопределить. Затем примените переопределения к этой группе.
5. В диалогом окне **Свойства переопределения** установите флажок возле параметра **WebProxyAddress** в столбце **Переопределение**. В поле **Значение переопределения** введите URL-адрес сервера шлюза OMS. Он должен начинаться с префикса **http://**.

   >[!NOTE]
   > Вам не нужно включать правило. Правилом автоматически управляет переопределение, содержащееся в пакете управления переопределением безопасной ссылки System Center Advisor, предназначенном для группы серверов мониторинга System Center Advisor.
   > 

6. Чтобы указать пакет управления, выполните одно из следующих действий.
    * Выберите пакет управления в списке **Выберите целевой пакет управления**.
    * Чтобы создать новый открытый пакет управления, щелкните **Создать**. 
7. Нажмите кнопку **ОК**. 

### <a name="set-up-automation-hybrid-workers"></a>Настройка поддержки гибридных рабочих ролей службы автоматизации
При наличии в среде гибридных рабочих ролей Runbook службы автоматизации выполните временные решения вручную, чтобы настроить их поддержку.

Для выполнения следующих действий нужно знать регион Azure, где находится учетная запись службы автоматизации. Чтобы найти расположение:

1. Войдите на [портал Azure](https://portal.azure.com/).
2. Выберите службу автоматизации Azure.
3. Выберите соответствующую учетную запись службы автоматизации Azure.

## <a name="useful-powershell-cmdlets"></a>Полезные командлеты PowerShell
С помощью командлетов можно обновить параметры шлюза OMS. Прежде чем использовать командлеты PowerShell, не забудьте сделать следующее:

1. Установите шлюз OMS (MSI).
2. Откройте окно консоли PowerShell.
3. Чтобы импортировать модуль, введите эту команду: `Import-Module OMSGateway`. Если не возникла ошибка, значит импорт модуля выполнен успешно и вы можете использовать командлеты. 
4. Укажите `Get-Module OMSGateway`.
5. После внесения изменений с помощью командлетов перезапустите службу шлюза OMS.

Если при выполнении шага 3 произошла ошибка, значит модуль не импортирован. Возможно, эта ошибка возникла, потому что PowerShell не удалось найди модуль. Его можно найти по указанному пути установки шлюза: C:\Program Files\Microsoft OMS Gateway\PowerShell.

| Командлет | Параметры | Описание | Примеры |
| --- | --- | --- | --- |
| `Set-OMSGatewayConfig` |Ключ (обязательно) <br> Значение |Изменяет конфигурацию службы |`Set-OMSGatewayConfig -Name ListenPort -Value 8080` |
| `Get-OMSGatewayConfig` |Ключ |Получает конфигурацию службы |`Get-OMSGatewayConfig` <br> <br> `Get-OMSGatewayConfig -Name ListenPort` |
| `Set-OMSGatewayRelayProxy` |Адрес <br> Имя пользователя <br> Пароль |Задает адрес (и учетные данные) прокси-сервера ретрансляции (вышестоящего) |1. Задайте прокси-сервер ответа и учетные данные: `Set-OMSGatewayRelayProxy -Address http://www.myproxy.com:8080 -Username user1 -Password 123` <br> <br> 2) Задайте прокси-сервер ответа, для которого не требуется проверка подлинности: `Set-OMSGatewayRelayProxy -Address http://www.myproxy.com:8080` <br> <br> 3. Очистите параметры прокси-сервера ответа (это означает, что прокси-сервер ответа не требуется): `Set-OMSGatewayRelayProxy -Address ""` |
| `Get-OMSGatewayRelayProxy` | |Получает адрес (и учетные данные) прокси-сервера ретрансляции (вышестоящего) |`Get-OMSGatewayRelayProxy` |
| `Add-OMSGatewayAllowedHost` |Узел (обязательно) |Добавляет узел в список разрешенных |`Add-OMSGatewayAllowedHost -Host www.test.com` |
| `Remove-OMSGatewayAllowedHost` |Узел (обязательно) |Удаляет узел из списка разрешенных |`Remove-OMSGatewayAllowedHost -Host www.test.com` |
| `Get-OMSGatewayAllowedHost` | |Получает разрешенный в настоящее время узел (только локально настроенные разрешенные узлы; разрешенные узлы, скачанные автоматически, сюда не относятся) |`Get-OMSGatewayAllowedHost` |
| `Add-OMSGatewayAllowedClientCertificate` |Субъект (обязательно) |Добавляет субъект сертификата клиента в список разрешенных |`Add-OMSGatewayAllowedClientCertificate -Subject mycert` |
| `Remove-OMSGatewayAllowedClientCertificate` |Субъект (обязательно) |Удаляет субъект сертификата клиента из списка разрешенных |`Remove- OMSGatewayAllowedClientCertificate -Subject mycert` |
| `Get-OMSGatewayAllowedClientCertificate` | |Получает разрешенные в настоящее время субъекты сертификата клиента (только локально настроенные разрешенные субъекты; разрешенные субъекты, скачанные автоматически, сюда не относятся) |`Get-OMSGatewayAllowedClientCertificate` |
