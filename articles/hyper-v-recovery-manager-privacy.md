# Сведения о конфиденциальности для Azure Site Recovery

Эта таблица содержит дополнительные сведения о конфиденциальности для службы Microsoft Azure Site Recovery ("Служба"). Чтобы просмотреть заявление о конфиденциальности для служб Microsoft Azure, см.
[Заявление о конфиденциальности Microsoft Azure](http://go.microsoft.com/fwlink/?LinkId=324899).

<thead>
<tr>
	<th>Функция</th>
	<th>Действие</th>
	<th>Собираемая информация</th>
	<th>Использование</th>
	<th>Выбор</th>
</tr>
</thead>
<tbody>
<tr>
	<td><p><b>Регистрация</b></p></td>
	<td><p>Регистрация сервера со службой, чтобы можно было защитить виртуальные машины</p></td>
	<td><p>После регистрации службы Служба обеспечивает сбор и передачу следующих сведений.</p>
		<ul>
			<li>Сведения о сертификате управления с сервера VMM, который предназначен для обеспечения аварийного восстановления с помощью имени Службы сервера VMM</li>
			<li>Имя облаков виртуальных машин на сервере VMM</li>
		</ul>
</td>
	<td><p>Служба использует указанные выше сведения следующим образом.</p>
		<ul>
			<li>Сертификат управления - позволяет определить и проверить подлинность зарегистрированного сервера VMM для доступа к Службе. Служба использует ту часть сертификата, которая относится к открытому ключу, для защиты токена, доступ к которому может получить только зарегистрированный сервер VMM. Сервер должен использовать этот токен для получения доступа к возможностям Службы.</li>
			<li>Имя сервера VMM - имя сервера VMM необходимо для идентификации соответствующего сервера VMM, на котором расположены облака, и взаимодействия с ним. 
			</li>
			<li>Имена облаков с сервера VMM - имя облака необходимо при использовании описанной ниже возможности связывания и отмены связывания. Если решено связать облако из первичного центра обработки данных с другим облаком в резервном центре обработки данных, отображаются имена всех облаков из резервного центра обработки данных.</li>
		</ul>
</td>
	<td><p>Эти сведения представляют собой важную часть процесса регистрации Службы, так как помогают вам и Службе определить тот сервер VMM, для которого требуется обеспечить защиту Azure Site Recovery, а также идентифицировать правильный зарегистрированный сервер VMM. Если вы не хотите отправлять эти сведения в Службу, не пользуйтесь данной Службой. Если вы зарегистрировали сервер и затем хотите отменить регистрацию, это можно сделать, удалив информацию о сервере VMM на портале Службы (представляющем собой портал Azure).</p></td>
</tr>

<tr>
	<td><p><b>Включение защиты Azure Site Recovery</b></p></td>
	<td><p>Установленный на сервере VMM поставщик Azure Site Recovery является каналом связи со Службой. Поставщик представляет собой библиотеку динамической компоновки (DLL), размещенную в процессе VMM. После установки поставщика в консоли администратора VMM включается возможность Datacenter Recovery. В любой новой или существующей виртуальной машине в облаке можно включить свойство с именем Datacenter Recovery в целях защиты данной виртуальной машины. После установки этого свойства поставщик отправляет имя и идентификатор виртуальной машины в Службу. Виртуальная защита обеспечивается технологией репликации Windows Server 2012 или Windows Server 2012 R2 Hyper-V. Данные виртуальной машины реплицируются с одного узла Hyper-V на другой (который обычно находится в другом резервном центре обработки данных).</p></td>
	<td><p>Служба собирает, обрабатывает и передает метаданные для виртуальной машины, включая имя, идентификатор, виртуальную сеть и имя облака, к которому она принадлежит.</p>
	</td>
	<td><p>Служба использует указанную выше информацию для заполнения сведений о виртуальной машине на портале Службы.</p>
	</td>
	<td><p>Это важная часть службы, которую нельзя отключить. Если вы не хотите отправлять эти сведения в Службу, не включайте защиту Azure Site Recovery для каких-либо виртуальных машин. Обратите внимание, что все данные, отправляемые поставщиком в Службу, передаются по протоколу HTTPS.</p></td>
</tr>
<tr>
	<td><p><b>План восстановления</b></p></td>
	<td><p>Эта возможность помогает создать план взаимодействия для резервного центра обработки данных. Можно задать порядок, в котором виртуальные машины или группа виртуальных машин должны быть запущены на резервном сайте. Можно также указать автоматизированные скрипты и ручные действия, запускаемые и выполняемые во время восстановления для каждой виртуальной машины. Отработка отказа (описана в следующем разделе) обычно запускается на уровне плана восстановления для обеспечения скоординированного восстановления.</p></td>
	<td><p>Служба собирает данные о процессе и передает эти сведения в рамках работы плана восстановления.</p>
		<ul>
			<li>План восстановления, включая метаданные виртуальной машины.</li>
			<li>Метаданные скрипта автоматизации или заметка о ручном действии.</li>
		</ul>
	</td>
	<td><p>Указанные выше метаданные используются для создания плана восстановления на портале Службы.</p>
	</td>
	<td><p>Это важная часть службы, которую нельзя отключить. Если вы не хотите отправлять эти сведения в Службу, не создавайте для нее планы восстановления.</p></td>
</tr>
<tr>
	<td><p><b>Сетевое сопоставление</b></p></td>
	<td><p>Эта возможность позволяет сопоставить сетевые сведения из первичного центра обработки данных с резервным центром обработки данных. Если виртуальные машины восстанавливаются на сайте восстановления, это сопоставление помогает установить сетевое подключение для них.</p></td>
	<td><p>В рамках процесса сетевого сопоставления Служба собирает, обрабатывает и передает метаданные логических сетей для каждого сайта (первичного и центра обработки данных).</p>
	</td>
	<td><p>Служба использует эти метаданные для заполнения портала Службы, где можно сопоставить эти сведения о сети.</p>
	</td>
	<td><p>Это важная часть Службы, которую нельзя отключить. Если вы не хотите отправлять эти сведения в Службу, не используйте возможность сетевого сопоставления.</p></td>
</tr>
<tr>
	<td><p><b>Сбой - запланированный, незапланированный, текст</b></p></td>
	<td><p>Эта функция позволяет выполнить отработку отказа виртуальной машины из одного центра обработки данных, управляемого VMM, в другой центр обработки данных, управляемый VMM. Действие по отработке отказа запускается пользователем на портале Службы. Возможные причины отработки отказа включают в себя незапланированное событие (например, стихийное бедствие), запланированное событие (например, балансировка нагрузки центра обработки данных), тестовую отработку отказа (например, проверка плана восстановления).</p>
	<p>Поставщик на сервере VMM получает уведомление о событии от Службы и выполняет действие отработки отказа на соответствующем узле Hyper-V посредством интерфейсов VMM. Фактическая отработка отказа виртуальной машины с одного узла Hyper-V на другой (который обычно находится в другом резервном центре обработки данных) обрабатывается технологией репликации Windows Server 2012 или Windows Server 2012 R2 Hyper-V. После отработки отказа поставщик, установленный на сервере VMM резервного центра обработки данных, отправляет в Службу сведения об успешном выполнении.</p>
	</td>
	<td><p>Служба использует указанную выше информацию для заполнения состояния действия отработки отказа на портале Службы.</p>
	</td>
	<td><p>Служба использует указанные выше сведения следующим образом:</p>
		<ul>
			<li>Сертификат управления - позволяет определить и проверить подлинность зарегистрированного сервера VMM для доступа к Службе. Служба использует ту часть сертификата, которая относится к открытому ключу, для защиты токена, доступ к которому может получить только зарегистрированный сервер VMM. Сервер должен использовать этот токен для получения доступа к возможностям Службы.</li>
			<li>Имя сервера VMM - имя сервера VMM необходимо для идентификации соответствующего сервера VMM, на котором расположены облака, и взаимодействия с ним. 
			</li>
			<li>Имена облаков с сервера VMM - имя облака необходимо при использовании описанной ниже возможности связывания и отмены связывания. Если решено связать облако из первичного центра обработки данных с другим облаком в резервном центре обработки данных, отображаются имена всех облаков из резервного центра обработки данных.</li>
		</ul>
	</td>
	<td><p>Это важная часть службы, которую нельзя отключить. Если вы не хотите отправлять такие сведения в Службу, не используйте эту Службу.</p></td>
</tr>
</table>
