# Сведения о миграции в хранилище класса Premium
В хранилище данных SQL недавно введена возможность использования [хранилища класса Premium, обеспечивающего более предсказуемую производительность][]. Теперь мы можем переместить хранилища данных, расположенные в стандартном хранилище, в хранилище класса Premium. В этой статье содержатся сведения о том, как и когда произойдут автоматические миграции и как выполнить миграцию самостоятельно, если вы предпочитаете управлять простоем.

Если у вас есть несколько хранилищ данных, используйте приведенное ниже [расписание автоматических миграций][], чтобы определить время их миграции.

## Сведения об автоматической миграции
По умолчанию база данных будет перенесена автоматически в период с 18:00 до 06:00 по местному времени соответствующего региона в рамках [расписания автоматической миграции][], приведенного ниже. Во время миграции имеющееся хранилище данных использовать нельзя. По нашим оценкам, для миграции 1 ТБ емкости хранилища данных требуется около 1 часа. Мы также гарантируем, что в процессе миграции независимо от этапа плата не взимается.

> [AZURE.NOTE] Вы не сможете использовать имеющееся хранилище данных во время миграции. По завершении миграции оно снова переходит в оперативный режим.

Ниже приведены шаги, которые корпорация Майкрософт выполняет от вашего имени, чтобы осуществить миграцию без вашего вмешательства. В данном примере имеющемуся хранилищу данных в стандартном хранилище присвоено имя MyDW.

1.	Корпорация Майкрософт переименует хранилище на MyDW\_DO\_NOT\_USE\_[метка\_времени].
2.	Корпорация Майкрософт приостановит работу MyDW\_DO\_NOT\_USE\_[метка\_времени]. В это время она будет выполнять архивацию. Этот процесс может быть несколько раз приостановлен и возобновлен, если во время него возникнут проблемы.
3.	Майкрософт создаст хранилище данных с именем MyDW в хранилище класса Premium, используя резервную копию, созданную на шаге 2 выше. Хранилище MyDW не будет отображаться до завершения восстановления.
4.	По завершении восстановления хранилище MyDW вернется к прежнему уровню единиц DWU и состоянию (приостановлено или активно), в котором оно пребывало до миграции.
5.	По завершении миграции Майкрософт удалит хранилище MyDW\_DO\_NOT\_USE\_[метка\_времени].
	
> [AZURE.NOTE] Эти параметры не переносятся в процессе миграции:
> 
>	-  Auditing at the Database level will need to be re-enabled
>	-  Firewall rules at the **Database** level will need to be re-added.  Firewall rules at the **Server** level will not be impacted.

## Самостоятельная миграция в хранилище класса Premium
Если вы хотите управлять временем простоя, можно выполнить приведенные ниже шаги, чтобы перенести существующее хранилище данных из стандартного хранилища в хранилище класса Premium. Если вы решили выполнить миграцию самостоятельно, то это необходимо сделать до начала автоматической миграции в соответствующем регионе. Так вы избежите риска конфликтов (см. [расписание автоматической миграции][]).

> [AZURE.NOTE] Хранилище данных SQL с хранилищем класса Premium в настоящее время не геоизбыточное. Это означает, что после миграции хранилища данных в хранилище класса Premium данные будут находиться только в текущем регионе. Как только служба геоархивации будет доступна, она будет копировать хранилище данных в [парный регион Azure][] каждые 24 часа. Это позволит вам выполнить восстановление с помощью резервной копии службы геоархивации в любом регионе Azure. О доступности функции геоархивации для самостоятельной миграции будет объявлено на [основном сайте документации][]. При автоматической миграции такого ограничения нет.

### Инструкции по самостоятельной миграции
Если вы хотите управлять простоем, то самостоятельно перенести хранилище данных можно с помощью резервного копирования и восстановления. Предполагается, что для этапа восстановления миграции требуется около 1 часа на 1 ТБ памяти хранилища данных. Если вы хотите сохранить то же имя по завершении миграции, следуйте описанным ниже указаниям, чтобы [обойти этап переименования][].

1.	[Приостановите работу][] хранилища данных для автоматической архивации.
2.	[Восстановите][] данные из последнего моментального снимка.
3.	Удалите имеющееся хранилище данных в стандартном хранилище. **Если не выполнить этот шаг, с вас будет взиматься плата за оба хранилища данных.**

> [AZURE.NOTE] Эти параметры не переносятся в процессе миграции:
> 
>	-  Auditing at the Database level will need to be re-enabled
>	-  Firewall rules at the **Database** level will need to be re-added.  Firewall rules at the **Server** level will not be impacted.

#### Обход этапа переименования (необязательно) 
У двух баз данных на одном логическом сервере не может быть одинаковое имя. В настоящее время в хранилище данных SQL не поддерживается возможность переименования хранилища данных. Следуя указаниям ниже, можно обойти этот этап во время самостоятельной миграции. (Примечание. При автоматической миграции такого ограничения нет.)

В данном примере имеющемуся хранилищу данных в стандартном хранилище присвоено имя MyDW.

1.	[Приостановите][] MyDW для автоматической архивации.
2.	[Восстановите][] новую базу данных с другим именем, например MyDWTemp, из последнего моментального снимка.
3.	Удалите MyDW. **Если не выполнить этот шаг, с вас будет взиматься плата за оба хранилища данных.**
4.	Так как MyDWTemp — это только что созданное хранилище данных, резервная копия не будет доступна для восстановления в течение некоторого времени. Рекомендуется продолжить операции с MyDWTemp в течение нескольких часов, а затем перейти к шагам 5 и 6.
5.	[Приостановите][] MyDWTemp для автоматической архивации.
6.	[Восстановите][] новую базу данных с именем MyDW из последнего моментального снимка MyDWTemp.
7.	Удалите MyDWTemp. **Если не выполнить этот шаг, с вас будет взиматься плата за оба хранилища данных.**

> [AZURE.NOTE] Эти параметры не переносятся в процессе миграции:
> 
>	-  Auditing at the Database level will need to be re-enabled
>	-  Firewall rules at the **Database** level will need to be re-added.  Firewall rules at the **Server** level will not be impacted.

## Дальнейшие действия
Если у вас возникнут проблемы с хранилищем данных, [создайте запрос в службу поддержки][] и укажите "Миграция в хранилище уровня "Премиум"" в качестве возможной причины.
