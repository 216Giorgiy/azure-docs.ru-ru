# Создание цифровой маркетинговой кампании на веб-сайтах Azure
Это руководство содержит технический обзор использования веб-сайтов Azure для создания цифровых маркетинговых кампаний. Цифровая маркетинговая кампания обычно является краткосрочным мероприятием, предназначенным для достижения кратковременной маркетинговой цели. Необходимо рассмотреть два основных сценария. В первом сценарии сторонняя маркетинговая фирма создает кампанию для своего клиента и управляет этой кампанией в течение рекламной акции. Второй сценарий задействует маркетинговую фирму, создающую ресурсы цифровой маркетинговой кампании и затем передающую своему клиенту право владения этими ресурсами. Далее клиент запускает цифровую маркетинговую кампанию и управляет этой кампанией на свое усмотрение. 

[Веб-сайты Azure][websitesoverview] хорошо подходят для обоих сценариев. Они обеспечивают быстрое создание, поддерживают несколько платформ и языков, масштабируются для удовлетворения пользовательского спроса и совместимы со многими системами развертывания и управления версиями. Применение Azure также предоставляет доступ к другим службам Azure, таким как службы мультимедиа, что позволяет улучшить маркетинговую кампанию.

> [WACOM.NOTE]
> Если вы хотите начать работу с веб-сайтами Azure до регистрации учетной записи, перейдите по ссылке <a href="https://trywebsites.azurewebsites.net/">https://trywebsites.azurewebsites.net</a>, где вы сможете бесплатно и быстро создать кратковременный начальный сайт ASP.NET на веб-сайтах Azure. Не требуется вводить данные кредитной карты или брать на себя какие-либо обязательства.

Хотя для размещения веб-сайтов можно использовать [облачные службы Azure][csoverview] или [виртуальные машины Azure][vmoverview], такое решение не является идеальным для этого сценария, если только не требуется функция, которая не предоставляется веб-сайтами Azure. Описание возможных вариантов см. в статье [Веб-сайты Azure, облачные службы и виртуальные машины: когда они используются?][chooseservice].

Но существуют и другие возможности веб-сайтов Azure, которые можно использовать в конкретной реализации. ##<a name="deployexisting"></a>Развертывание существующих веб-сайтов
В сценарии глобального веб-присутствия были рассмотрены различные варианты создания и развертывания нового веб-сайта. Тем, кто недавно работает с веб-сайтами Azure, рекомендуется [ознакомиться с этой информацией][scenarioglobalweb]. Если вы часто создаете цифровые маркетинговые кампании, возможно, что у вас уже есть существующие веб-ресурсы, настроенные для различных рекламных акций. В этом разделе будут внимательно рассмотрены варианты развертывания различных типов веб-сайтов из системы управления версиями.

При повторном использовании веб-ресурсов для нескольких целей настоятельно рекомендуется использовать систему управления версиями, если она еще не применяется. Это позволяет хранить шаблоны распространенных веб-решений, которые можно разветвлять и настраивать для конкретных клиентов. Веб-сайты предоставляют возможность синхронизации со многими различными репозиториями исходных кодов. На вкладке **Панель мониторинга** выберите ссылку **Настроить развертывания в системе управления версиями**.

![DigitalMarketingDeploy1][DigitalMarketingDeploy1]

Откроется диалоговое окно с несколькими вариантами управления версиями. Среди них полнофункциональные системы управления версиями, такие как TFS, а также простые решения для развертывания, такие как Dropbox.

![DigitalMarketingDeploy2][DigitalMarketingDeploy2]

Для управления новым проектом, основанным на существующем базовом решении, можно использовать различные методы управления версиями. Например, можно скопировать ранее сохраненный базовый репозиторий, чтобы начать работу над новым проектом, или создать новую ветвь, чтобы отслеживать изменения в текущем проекте. Хороший пример использования ветвей в управлении различными развертываниями в одном и том же репозитории управления версиями см. в статье [Несколько сред с веб-сайтами Azure][gitstaging]. В этом сообщении показывается, как использовать ветвление репозитория git для управления тестовыми и производственными средами.

Подключив веб-сайт к системе управления версиями, вы сможете настраивать и отслеживать развертывания с портала. Дополнительную информацию об использовании системы управления версиями с веб-сайтами см. в статье [Публикация из системы управления версиями на веб-сайты Azure][publishingwithgit].

При использовании существующих веб-ресурсов также важны гибкие возможности размещения веб-сайтов различных типов. На вкладке **Настройка** можно выбрать для веб-сайта поддержку как .NET, так и PHP. 

![DigitalMarketingFrameworkVersions][DigitalMarketingFrameworkVersions]

Помимо этих вариантов конфигурации, веб-сайты автоматически обеспечивают поддержку Python 2.7 и Node.js. По умолчанию используемой версией Node.js является 0.10.5.

Одно из дополнительных преимуществ веб-сайтов Azure - скорость развертывания промежуточных веб-сайтов в Интернете. В течение планирования, создания прототипов и ранних этапов разработки сайта агентство и его клиент могут просматривать реальные рабочие версии сайта кампании перед его запуском. Когда веб-сайт готов к переносу в производственную среду, агентство может либо управлять развертыванием в производственной среде для клиента, либо предоставить клиенту веб-ресурсы для развертывания и управления.

##<a name="socialmedia"></a>Интеграция с социальными сетями
В большинстве цифровых маркетинговых кампаний используются сайты социальных сетей, например Facebook или Twitter. Одной точкой интеграции является использование удостоверений социальных сетей для проверки подлинности. Пример использования такого подхода с приложением ASP.NET см. в статье [Развертывание безопасного приложения ASP.NET MVC с членством, OAuth и базой данных SQL на веб-сайте Azure][deploysecurewebsite].

Но многие цифровые маркетинговые кампании не ограничиваются проверкой подлинности и используют интеграцию с социальными сетями в качестве ключевой части своей стратегии. На сайтах социальных сетей обычно предусмотрен раздел для разработчиков, описывающий различные способы интеграции приложений со службами социальных сетей. Службы, предоставляющие интерфейс API REST, можно использовать практически с любой веб-платформы. Но часто существует информация, специфическая для выбранного языка. Например, Twitter предоставляет [список доступных библиотек, поддерживающих API Twitter][twitter], включая библиотеки для .NET, Node.js и PHP. Это только один из примеров, подобные руководства для разработчиков следует искать прямо на сайтах нужных социальных сетей.

Для разработчиков ASP.NET, ориентирующихся на Facebook, Visual Studio предоставляет шаблон для приложения MVC 4 Facebook.  

![DigitalMarketingFacebook][DigitalMarketingFacebook]

Краткое описание использования этого шаблона с веб-сайтами см. в статье [Создание приложения Facebook с помощью шаблонов ASP.NET MVC Facebook и их бесплатное размещение на веб-сайтах Azure][fbtutorial]. Более подробный учебник и пример приложения см. в статьях [Приложение ASP.NET MVC Facebook для дня рождения][fbbirthdayapp] и [Новый шаблон приложения Facebook и библиотека для ASP.NET MVC][fbvstemplate].

Разработчику ASP.NET важно понимать, что интеграция с социальными сетями не ограничивается шаблонами, предоставляемыми Visual Studio. Шаблон просто помогает упростить процесс. Но, как упоминалось ранее, каждый сайт социальной сети обычно содержит сведения о других способах подключения из .NET и из многих других языков и платформ.

##<a name="scale"></a>Масштабирование в соответствии с пользовательским спросом
Облачные вычисления полезны для непрогнозируемых рабочих нагрузок. В эту категорию попадают и цифровые маркетинговые кампании. Трудно предсказать популярность относительно кратковременного маркетингового сайта, так как многое зависит от привлечения интереса пользователей и взаимодействия с соответствующими социальными сетями, что увеличивает трафик для сайта. Azure предлагает несколько вариантов масштабирования веб-сайтов и облачных служб.

- Масштабирование вручную, используя [Портал управления Azure][managementportal].
- Программное масштабирование с помощью [API управления службами][servicemanagementapi] или [сценариев PowerShell][powershell].
- Автоматическое масштабирование с помощью функции автомасштабирования (предварительная версия).

На портале управления перейдите на вкладку **Масштаб** для своего веб-сайта. Существует несколько параметров для масштабирования. Первый параметр определяет режим веб-сайта с возможными значениями **Бесплатный**, **В общем доступе** или **Стандартный**. 

![DigitalMarketingScale][DigitalMarketingScale]

Функции и возможности масштабирования растут с каждым уровнем. Например, сайты в режиме **Бесплатный** нельзя масштабировать на несколько экземпляров, а веб-сайты в режиме **В общем доступе** можно масштабировать на несколько (до 6) экземпляров. Возможность масштабирования существует и в режиме **Стандартный**. Этот режим запускает сайт на выделенных виртуальных машинах и предоставляет возможности изменения размеров машины: малый, средний и большой. Определив размер виртуальной машины, также можно выполнить горизонтальное масштабирование на нескольких экземплярах. В режиме **Стандартный** число экземпляров для масштабирования может достигать 10. Полный список различий между режимами см. в [правилах ценообразования для веб-сайтов][pricing].

При выборе режима **Стандартный** появляется дополнительная возможность включения функции автомасштабирования (предварительная версия). Она позволяет выполнять масштабирование на основе ЦП. Процент **Целевой ЦП** - это доля использования процессора, предоставляемая Azure для экземпляров веб-сайта. Если среднее использование ЦП меньше этого целевого значения, Azure уменьшает число экземпляров, так как эта же нагрузка при меньшем числе экземпляров приведет к увеличению коэффициента использования оставшихся экземпляров. Но нельзя уменьшить число экземпляров ниже минимального значения **Число экземпляров**. Аналогично, если средняя загрузка ЦП превышает значение **Целевой ЦП**, Azure увеличивает число экземпляров. Распределение нагрузки между дополнительными компьютерами приводит к уменьшению использования ЦП в каждой машине. Число добавляемых экземпляров ограничивается максимальным значением **Число экземпляров**.

![DigitalMarketingAutoScale][DigitalMarketingAutoScale]

Автоматическое масштабирование позволяет использовать ресурсы намного эффективнее. Например, цифровая маркетинговая кампания может быть наиболее активной в ночное время и по выходным. В этом случае также эффективно отрабатывается сценарий неожиданного увеличения популярности кампании. Автоматическое масштабирование позволяет эффективно обработать повышенную нагрузку, а затем уменьшить число экземпляров (и расходы) при уменьшении нагрузки. 

Дополнительную информацию о масштабировании веб-сайтов см. в статье [Масштабирование веб-сайтов][scalewebsite]. Эта тема также тесно связана с мониторингом. Дополнительную информацию см. в разделе данного руководства, описывающем [мониторинг кампании](#monitor).

> [WACOM.NOTE]
> Для веб-приложений, для которых выбрано использование облачных служб и веб-ролей, существует дополнительная возможность масштабирования, основанная на длине элементов в очереди. В облачной службе типовым архитектурным шаблоном являются роли, обрабатывающие серверные очереди. Дополнительную информацию о масштабировании облачных служб см. в статье <a href="http://www.windowsazure.com/ru-ru/manage/services/cloud-services/how-to-scale-a-cloud-service/">Масштабирование облачной службы</a>.

##<a name="integrate"></a>Интеграция с другими службами
Сайт цифрового маркетинга часто будет содержать мультимедийные данные, например потоковое видео. Размещение сайтов в Azure обеспечивает тесную интеграцию со связанными службами Azure. Например, можно использовать службы мультимедиа Azure для кодирования и потоковой передачи видео для веб-сайта. Дополнительную информацию о службах мультимедиа см. в статье [Введение в основные понятия и сценарии служб мультимедиа Azure][mediaservices].

Другие службы Azure можно использовать для создания более надежных приложений. Например, веб-сайты могут использовать распределенное кэширование, предоставляемое новой [службой кэша Azure (предварительная версия)][caching]. Либо можно использовать службы хранилища Azure для хранения данных и ресурсов приложения. Например, графики, видео и другие файлы большого размера могут в течение длительного времени храниться в BLOB-объектах. Для поддержки реляционных требований к данным также доступны службы баз данных, такие как база данных SQL Azure и MySQL.

##<a name="monitor"></a>Монитор кампании
Вкладка **Монитор** содержит метрики, которые помогут оценить успешность и производительность сайта цифрового маркетинга.

![DigitalMarketingMonitor][DigitalMarketingMonitor]

Например, можно просмотреть закономерности и уровни использования с помощью таких метрик, как **Время ЦП**, **Запросы** и **Исходящие данные**. Все эти метрики будут расти с ростом популярности маркетинговой кампании. Сведения об использовании можно применять для принятия решения о направлении масштабирования. Дополнительную информацию см. в статье [Мониторинг веб-сайтов][monitoring].

Для режимов **Бесплатный** и **В общем доступе** также следует учитывать квоты ресурсов. На вкладке **Панель мониторинга** портала показывается текущая статистика использования нескольких квот, а также когда эти квоты будут сброшены. Эта статистика использования зависит от выбранного режима. На следующем снимке экрана показана статистика для режима **Бесплатный**. В режиме **В общем доступе** отсутствуют квоты для метрики **Исходящие данные**. В режиме **Стандартный** отображаются только параметры **Хранилище файловой системы** и **Размер**.

![DigitalMarketingUsageOverview][DigitalMarketingUsageOverview]

В случае приближения к исчерпанию этих квот рассмотрите возможность перехода из режима **Бесплатный** в режим **В общем доступе** или из режима **В общем доступе** в режим **Стандартный**. В режиме **Стандартный** предоставляются выделенные ресурсы на малых, средних и больших виртуальных машинах. 

Помимо использования портала управления для получения этих сведений, существует ряд средств независимых производителей, предоставляющих дополнительные данные мониторинга для веб-сайтов. Например, можно установить надстройку New Relic из хранилища Azure. Пошаговое руководство по использованию New Relic для мониторинга см. в статье [Инструмент управления производительностью приложений New Relic для Azure][newrelic]. 

Наконец, для веб-сайтов в стандартном режиме можно включить мониторинг конечных точек и соответствующее оповещение. Монитор конечной точки периодически пытается подключиться к веб-сайту, а затем сообщает время ответа. Если время ответа превышает указанный порог, оповещения обеспечивают отправку уведомлений по электронной почте. Дополнительную информацию см. в разделе о мониторинге сценария [Глобальное веб-присутствие][scenarioglobalweb] и разделе [Практическое руководство. Получение уведомлений об оповещениях и управление правилами оповещения в Azure][receivealerts].

##<a name="summary"></a>Сводка
Веб-сайты Azure - это хороший вариант для многократного использования веб-контента, настраиваемого для отдельных маркетинговых кампаний. Веб-сайты поддерживает множество популярных языков, платформ и систем управления версиями, упрощая перенос этих ресурсов и процессов в облако. Данный шаблон приложения ASP.NET Facebook облегчает создание приложений для Facebook, но можно использовать интеграцию практически с любыми сторонними социальными сетями, поддерживающими веб-интерфейсы. Службы мультимедиа Azure и другие связанные службы Azure предоставляют дополнительные средства для создания хорошо разработанного сайта кампании. А несколько вариантов ручного и автоматического масштабирования полезны для обработки пользовательского спроса, который может оказаться трудно предсказуемым.
