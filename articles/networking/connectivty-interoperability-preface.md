# <a name="interoperability-in-azure-back-end-connectivity-features-test-setup"></a>Взаимодействие в функциях при подключении к серверной части в Azure. Настройка теста

В этой статье описывается настройка теста, с помощью которой можно выполнять анализ взаимодействия сетевых служб Azure на уровне управления и уровне данных. Давайте вкратце рассмотрим сетевые компоненты Azure:

-   **Azure ExpressRoute**. Частный пиринг Azure ExpressRoute позволяет напрямую подключать пространство частных IP-адресов локальной сети к развертываниям виртуальной сети Azure. С помощью ExpressRoute вы можете повысить пропускную способность и сделать подключение частным. Многие партнеры, работающие в экосистеме ExpressRoute, предоставляют подключения ExpressRoute с поддержкой соглашений об уровне обслуживания. Дополнительные сведения об ExpressRoute и его настройке см. в статье [Обзор ExpressRoute][ExpressRoute].
-   **VPN типа "сеть — сеть"**. С помощью VPN-шлюза Azure типа "сеть — сеть" вы можете безопасно подключить локальную сеть к Azure через Интернет или с помощью ExpressRoute. Сведения о настройке VPN типа "сеть — сеть" для подключения к Azure см. в статье [Сведения о VPN-шлюзе][VPN].
-   **Пиринг виртуальных сетей**. С помощью пиринга виртуальной сети можно установить подключение между виртуальными сетями в Azure. Дополнительные сведения о пиринге виртуальных сетей см. в статье [Руководство. Подключение виртуальных сетей с помощью пиринга и портала Azure][VNet].

Центральный элемент настройки теста соответствует виртуальной сети концентратора в регионе Azure 1. Виртуальная сеть концентратора подключена к несколькими сетям следующим образом:

-   Виртуальная сеть концентратора подключена к периферийной виртуальной сети с помощью пиринга виртуальных сетей. Периферийная виртуальная сеть получает удаленный доступ к обоим шлюзам в виртуальной сети концентратора.
-   Виртуальная сеть концентратора подключается к виртуальной сети филиала с помощью VPN типа "сеть — сеть". Это подключение использует eBGP для обмена маршрутами.
-   Виртуальная сеть концентратора подключается к локальной сети расположения 1 с помощью частного пиринга ExpressRoute в качестве основного пути. Здесь в качестве резервного пути используется подключение VPN типа "сеть — сеть". Далее в этой сети мы будем обозначать это подключение ExpressRoute как ExpressRoute 1. По умолчанию каналы ExpressRoute предоставляют избыточные подключения для обеспечения высокого уровня доступности. В ExpressRoute 1 подчиненный интерфейс пограничного маршрутизатора клиента (CE), обращенный к вторичному пограничному маршрутизатору Microsoft Enterprise (MSEE), отключен. Красная линия над двойной стрелкой на предыдущем рисунке представляет собой подчиненный интерфейс отключенного маршрутизатора CE.
-   Виртуальная сеть концентратора подключена к локальной сети расположения 2 с помощью другого частного пиринга ExpressRoute. Далее в этом документе мы будем обозначать второе подключение ExpressRoute как ExpressRoute 2.
-   ExpressRoute 1 соединяет виртуальную сеть концентратора и сеть локального расположение 1 с удаленной виртуальной сетью в регионе Azure 2.

## <a name="expressroute-and-site-to-site-vpn-connectivity-in-tandem"></a>Параллельное использование подключения ExpressRoute и VPN типа "сеть — сеть"

###  <a name="site-to-site-vpn-over-expressroute"></a>VPN типа "сеть — сеть" через ExpressRoute

VPN типа "сеть — сеть" можно настроить с помощью пиринга Microsoft ExpressRoute, чтобы организовать закрытый обмен данными между локальной сетью и виртуальными сетями Azure. Такая конфигурация гарантирует конфиденциальность, подлинность и целостность при обмене данными. Обмен данными также защищается от повторов. Дополнительные сведения о настройке VPN IPSec типа "сеть — сеть" в туннельном режиме с помощью пиринга Microsoft ExpressRoute см. [в этой статье][S2S-Over-ExR]. 

Основным ограничением настройки VPN типа "сеть — сеть" с использованием пиринга Майкрософт является пропускная способность. Пропускная способность туннеля IPsec ограничена емкостью VPN-шлюза. Пропускная способность VPN-шлюза ниже пропускной способности ExpressRoute. В такой ситуации можно оптимизировать использование пропускной способности ExpressRoute, настроив туннель IPSec для трафика, требующего высокого уровня безопасности, и частный пиринг для всего остального трафика.

### <a name="site-to-site-vpn-as-a-secure-failover-path-for-expressroute"></a>VPN "сеть — сеть" в качестве пути отработки отказа для ExpressRoute

ExpressRoute предоставляется в виде пары каналов, что обеспечивает избыточность и высокий уровень доступности. Вы можете настроить географически избыточное подключение ExpressRoute в разных регионах Azure. Как и в нашей настройке теста, вы можете создать путь отработки отказа для подключения ExpressRoute в пределах определенного региона Azure с помощью VPN типа "сеть — сеть". Если одни и те же префиксы объявляются через ExpressRoute и VPN типа "сеть — сеть", Azure отдает предпочтение ExpressRoute. Чтобы избежать асимметричной маршрутизации между ExpressRoute и VPN типа "сеть — сеть", в локальной сети следует также настроить конфигурацию ExpressRoute, прежде чем использовать VPN типа "сеть — сеть".

Дополнительные сведения о настройке параллельных подключений для ExpressRoute и VPN типа "сеть — сеть" см. в [этой статье][ExR-S2S-CoEx].

## <a name="extend-back-end-connectivity-to-spoke-vnets-and-branch-locations"></a>Расширение подключения серверной части к периферийным виртуальным сетям и расположениям филиалов

### <a name="spoke-vnet-connectivity-by-using-vnet-peering"></a>Подключение к периферийной виртуальной сети с использованием пиринга

Звездообразная архитектура виртуальных сетей нашла широкое применение. Главной является виртуальная сеть в Azure, которая выступает в качестве центральной точки подключения периферийных виртуальных сетей к локальной сети. Периферийные зоны — это виртуальные сети, которые устанавливают пиринг с концентратором и могут использоваться для изоляции рабочих нагрузок. Трафик передается между локальным центром обработки данных и центральной сетью через подключение ExpressRoute и (или) VPN. Дополнительные сведения об архитектуре см. в статье [Реализация звездообразной топологии сети в Azure][Hub-n-Spoke].

Пиринг виртуальных сетей в пределах региона позволяет периферийным виртуальным сетям использовать шлюзы виртуальной сети концентратора (шлюз VPN или ExpressRoute) для подключения к удаленным сетям.

### <a name="branch-vnet-connectivity-by-using-site-to-site-vpn"></a>Подключение к виртуальной сети филиала с помощью VPN типа "сеть — сеть"

Возможно, вам понадобятся виртуальные сети филиала, которые находятся в разных регионах, и локальные сети для обмена данными друг с другом через виртуальную сеть концентратора. Обычно в Azure решением для этой конфигурации является VPN типа "сеть — сеть". Альтернативой может быть использование виртуального сетевого модуля (NVA) для маршрутизации в концентраторе.

Дополнительные сведения см. в статьях [Сведения о VPN-шлюзе][VPN] и [Развертывание высокодоступных виртуальных сетевых модулей][Deploy-NVA].
