# <a name="azure-security-policies-monitored-by-azure-security-center"></a>Политики безопасности Azure, отслеживаемые в центре безопасности Azure
В этой статье содержится список определений службы "Политика Azure", которые можно отслеживать в центре безопасности.

## <a name="available-security-policies"></a>Доступные политики безопасности

В следующей таблице представлены сведения о встроенных политиках, которые отслеживаются с помощью центра безопасности.

| Политика | Предназначение политики |
| --- | --- |
|Аудит включения журналов диагностики в Service Fabric и Масштабируемых наборах виртуальных машин|Мы рекомендуем включить журналы для воссоздания следов действий и проведения исследования в случае инцидента или компрометации.|
|Аудит правил авторизации в пространствах имен концентратора событий|Клиентам концентратора событий не следует использовать политику доступа на уровне пространства имен, которая предоставляет доступ ко всем очередям и разделам в пространстве имен. Чтобы обеспечить соответствие модели безопасности с наименьшими привилегиями, следует создавать политики доступа на уровне сущностей для очередей и разделов, чтобы предоставлять доступ только к определенной сущности.|
|Аудит наличия правил авторизации в сущностях концентратора событий|Аудит наличия правил авторизации для предоставления доступа с наименьшими правами в сущностях концентратора событий|
|Аудит неограниченного сетевого доступа к учетным записям хранения|Проведите аудит неограниченного сетевого доступа с применением параметров брандмауэра учетной записи хранения. Вместо этого можно настроить правила сети так, чтобы учетная запись хранения была доступна только приложениям из разрешенных сетей. Чтобы разрешить подключения от конкретных локальных и интернет-клиентов, вы можете открыть доступ для трафика из конкретных виртуальных сетей Azure или для определенных диапазонов общедоступных IP-адресов.|
|Аудит использования настраиваемых правил RBAC|Аудит встроенных ролей, таких как "Владелец", "Участник", "Читатель", вместо настраиваемых ролей RBAC, использование которых сопряжено с ошибками. Работа с пользовательскими ролями рассматривается как исключение и требует строгой проверки с моделированием угроз.|
|Аудит включения журналов диагностики в Azure Stream Analytics|Аудит включения журналов и их хранение на срок до одного года. Это позволит воссоздать следы действий для анализа инцидентов безопасности или при компрометации сети.|
|Аудит безопасного перемещения в учетные записи хранения|Настройка требования выполнять аудит для безопасного переноса в учетную запись хранения. Безопасная передача данных — это параметр, при включении которого ваша учетная запись хранения принимает запросы только с безопасных подключений (HTTPS). Использование протокола HTTPS обеспечивает проверку подлинности между сервером и службой, а также защищает перемещаемые данные от атак сетевого уровня, таких как "злоумышленник в середине", прослушивание трафика и перехват сеанса.|
|Аудит подготовки администратора Azure Active Directory для SQL Server|Аудит подготовки администратора AAD для включения в SQL Server поддержки проверки подлинности AAD. Проверка подлинности Azure AD упрощает контроль разрешений и обеспечивает централизованное управление удостоверениями для пользователей баз данных и других служб Майкрософт.|
|Аудит правил авторизации в пространствах имен служебной шины|Клиентам служебной шины не следует использовать политику доступа на уровне пространства имен, которая предоставляет доступ ко всем очередям и разделам в пространстве имен. Чтобы обеспечить соответствие модели безопасности с наименьшими привилегиями, следует создавать политики доступа на уровне сущностей для очередей и разделов, чтобы предоставлять доступ только к определенной сущности.|
|Аудит включения журналов диагностики в служебной шине|Аудит включения журналов и их хранение на срок до одного года. Это позволит воссоздать следы действий для анализа инцидентов безопасности или при компрометации сети.|
|Аудит задания свойству ClusterProtectionLevel значения EncryptAndSign в Service Fabric|Service Fabric поддерживает три уровня защиты (None (Нет), Sign (Подписывание) и EncryptAndSign (Шифрование и подписывание)) для взаимодействия между узлами с использованием основного сертификата кластера. Задайте уровень защиты, чтобы обеспечить шифрование и цифровое подписывание всех сообщений, передаваемых между узлами.| 
|Аудит использования Azure Active Directory для проверки подлинности клиента в Service Fabric|Аудит проверки подлинности клиента только через Azure Active Directory в Service Fabric| 
|Аудит включения журналов диагностики для службы поиска|Аудит включения журналов и их хранение на срок до одного года. Это позволит воссоздать следы действий для анализа инцидентов безопасности или при компрометации сети.| 
|Аудит активации только безопасных подключений к кэшу Azure для Redis|Аудит активации только подключений по протоколу SSL к кэшу Azure для Redis. Использование безопасных подключений обеспечивает проверку подлинности между сервером и службой, а также защищает переносимые данные от атак сетевого уровня, таких как "злоумышленник в середине", прослушивание трафика и перехват сеанса.| 
|Аудит включения журналов диагностики в Logic Apps|Аудит включения журналов и их хранение на срок до одного года. Это позволит воссоздать следы действий для анализа инцидентов безопасности или при компрометации сети.| 
|Аудит включения журналов диагностики в Key Vault|Аудит включения журналов и их хранение на срок до одного года. Это позволит воссоздать следы действий для анализа инцидентов безопасности или при компрометации сети.|
|Аудит включения журналов диагностики в концентраторе событий|Аудит включения журналов и их хранение на срок до одного года. Это позволит воссоздать журнал действий для расследования инцидентов безопасности или при компрометации сети.| 
|Аудит включения журналов диагностики в Azure Data Lake Storage|Аудит включения журналов и их хранение на срок до одного года. Это позволит воссоздать следы действий для анализа инцидентов безопасности или при компрометации сети.| 
|Аудит включения журналов диагностики в Data Lake Analytics|Аудит включения журналов и их хранение на срок до одного года. Это позволит воссоздать следы действий для анализа инцидентов безопасности или при компрометации сети.| 
|Аудит использования классических учетных записей хранения|Используйте для своих учетных записей хранения новый выпуск Azure Resource Manager версии 2 с усовершенствованными функциями безопасности, включая более строгий контроль доступа (RBAC), улучшенный аудит, развертывание и управление на основе Resource Manager, доступ к управляемым удостоверениям и хранилищам ключей для секретов, проверку подлинности на основе Azure AD, а также поддержку тегов и групп ресурсов, упрощающих управление защитой.| 
|Аудит использования классических виртуальных машин|Используйте для своих виртуальных машин новый выпуск Azure Resource Manager версии 2 с усовершенствованными функциями безопасности, включая более строгий контроль доступа (RBAC), улучшенный аудит, развертывание и управление на основе Resource Manager, доступ к управляемым удостоверениям и хранилищам ключей для секретов, проверку подлинности на основе Azure AD, а также поддержку тегов и групп ресурсов, упрощающих управление защитой.| 
|Аудит настройки правил оповещений метрик в учетных записях пакетной службы|Аудит настройки правил оповещений метрик в учетной записи пакетной службы для включения нужной метрики.|

Возможный JIT-доступ к сети будет отслеживаться центром безопасности Azure для предоставления рекомендаций.|

|[Предварительная версия.] Аудит минимального числа владельцев подписки|Мы рекомендуем назначить более одного владельца подписки, чтобы обеспечить избыточность для административного доступа.|

|[Предварительная версия.] Аудит учетных записей с разрешениями на чтение, у которых не включена в подписке многофакторная проверка подлинности|Следует включить многофакторную проверку подлинности (MFA) для всех учетных записей подписки с разрешениями на чтение, чтобы предотвратить нарушение безопасности учетных записей или ресурсов.|
