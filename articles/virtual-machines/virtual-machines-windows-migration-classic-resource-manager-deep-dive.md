# Техническое руководство по поддерживаемому платформой переносу из классической модели в модель Azure Resource Manager

Эта статья представляет собой техническое руководство по миграции на уровне ресурсов и возможностей, которое позволит понять, как осуществляется перенос ресурсов из классической модели в модель Resource Manager на платформе. Прочтите статью с объявлением о выпуске службы [Поддерживаемый платформой перенос ресурсов IaaS из классической модели в модель Azure Resource Manager](virtual-machines-windows-migration-classic-resource-manager.md).

## Подробное руководство по миграции на уровне ресурсов и возможностей

Ниже представлены ресурсы в классической модели и в модели Resource Manager. Возможности и ресурсы, не указанные в этой таблице, в настоящее время не поддерживаются.

| Представление классической модели | Представление Resource Manager | Подробные примечания | | |
|--------------------------------------------------------|-------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|---|---|
| Имя облачной службы | DNS-имя | Во время миграции будет создана группа ресурсов со всеми ресурсами для каждой облачной службы с использованием шаблона именования `<cloudservicename>-migrated`. Имя облачной службы преобразуется в DNS-имя, связанное с общедоступным IP-адресом. | | |
| Виртуальная машина | Виртуальная машина | Свойства конкретных виртуальных машин будут перенесены как есть. Обратите внимание, что определенные сведения osProfile, например имя компьютера, не сохранились в классической модели развертывания и поэтому после миграции они будут отсутствовать. | | |
| Ресурсы дисков, подключенные к виртуальной машине | Скрытые диски, подключенные к виртуальной машине | Диски не моделируются в виде ресурсов верхнего уровня в модели развертывания Resource Manager. Они переносятся в качестве скрытых дисков в виртуальной машине. В настоящее время поддерживаются только диски, подключенные к виртуальной машине. Скоро будет предоставляться поддержка переноса несвязанных дисков. Чтобы обеспечить возможность миграции, теперь виртуальные машины Resource Manager могут использовать классические учетные записи хранения. Таким образом диски можно с легкостью перенести в модель Resource Manager без обновления. | | |
| Расширения виртуальных машин | Расширения виртуальных машин | Все расширения ресурсов (кроме расширений XML) будут перенесены из классической модели развертывания. | | |
| Сертификаты виртуальных машин | Сертификаты в хранилище ключей Azure | Если облачная служба содержит сертификаты службы, платформа создаст хранилище ключей Azure для каждой облачной службы и переместит сертификаты в хранилище ключей. Виртуальные машины будут обновлены для ссылки на сертификаты из хранилища ключей. | | |
| Конфигурация WinRM | Конфигурация WinRM в osProfile | Конфигурация WinRM будет перенесена как есть во время миграции. | | |
| Свойство группы доступности | Ресурс группы доступности | Спецификация группы доступности являлась свойством на виртуальной машине в классической модели развертывания. Группы доступности станут ресурсами верхнего уровня в ходе миграции. Обратите внимание, что не поддерживаются следующие конфигурации: несколько групп доступности на одну облачную службу или несколько групп доступности с виртуальными машинами, которые не находятся в группе доступности, в одной облачной службе. | | |
| Конфигурация сети на виртуальной машине | Основной сетевой интерфейс | Теперь конфигурация сети на виртуальной машине будет представлена в качестве ресурса основного сетевого интерфейса после миграции. Для виртуальных машин, которые не находятся в виртуальной сети, внутренний IP-адрес изменится во время миграции. | | |
| Несколько сетевых интерфейсов на виртуальной машине | Сетевые интерфейсы | Если с виртуальной машиной связаны несколько сетевых интерфейсов, каждый сетевой интерфейс (и все соответствующие свойства) станет ресурсом верхнего уровня в ходе миграции в модели развертывания Resource Manager. | | |
| Набор конечных точек с балансировкой нагрузки | Подсистема балансировки нагрузки | В классической модели развертывания платформа назначает скрытый балансировщик нагрузки для каждой облачной службы. Во время миграции будет создан ресурс балансировщика нагрузки, а набор конечных точек с балансировкой нагрузки преобразуется в правила балансировки нагрузки. | | |
| Правила преобразования сетевых адресов для входящих подключений | Правила преобразования сетевых адресов для входящих подключений | Входные конечные точки на виртуальной машине преобразуются в правила преобразования сетевых адресов для входящих подключений в балансировщике нагрузки во время миграции. | | |
| Виртуальный IP-адрес | Общедоступный IP-адрес с DNS-именем | Виртуальный IP-адрес станет общедоступным IP-адресом, который будет связан с балансировщиком нагрузки. | | |
| Виртуальная сеть | Виртуальная сеть | Виртуальная сеть и все ее свойства будут перенесены в модель развертывания Resource Manager. Будет создана группа ресурсов с именем `-migrated`. Сведения о неподдерживаемых конфигурациях см. [здесь](virtual-machines-windows-migration-classic-resource-manager.md). | | |
| Зарезервированные IP-адреса | Общедоступный IP-адрес с методом статического выделения | Зарезервированный IP-адрес, связанный с балансировщиком нагрузки, переносится с облачной службой или виртуальной машиной. Миграция несвязанных зарезервированных IP-адресов в настоящее время не поддерживается. | | |
| Общедоступный IP-адрес для каждой виртуальной машины | Общедоступный IP-адрес с методом динамического выделения | Общедоступный IP-адрес, связанный с виртуальной машиной, преобразуется в ресурс общедоступного IP-адреса со статическим методом выделения. | | |
| группы сетевой безопасности; | группы сетевой безопасности; | Группы безопасности сети, связанные с подсетью, клонируются в модель развертывания Resource Manager в ходе миграции. Обратите внимание, что группа безопасности сети не удаляется в ходе миграции в классической модели развертывания. Однако во время миграции блокируются операции на плоскости управления для этой группы. | | |
| DNS-серверы | DNS-серверы | DNS-серверы, связанные с виртуальной сетью или виртуальной машиной, переносятся в ходе миграции соответствующего ресурса наряду со всеми свойствами. | | |
| Определяемые пользователем маршруты | Определяемые пользователем маршруты | Определяемые пользователем маршруты, связанные с подсетью, клонируются в модель развертывания Resource Manager в ходе миграции. Обратите внимание, что определяемый пользователем ресурс не удаляется в ходе миграции в классической модели развертывания. Однако во время миграции блокируются операции на плоскости управления для этого ресурса. | | |
| Свойство IP-пересылки в конфигурации сети виртуальной машины | Свойство IP-пересылки в сетевой карте | Свойство IP-пересылки на виртуальной машине преобразуется в свойство на сетевом интерфейсе во время миграции | | |
| Балансировщик нагрузки с несколькими IP-адресами | Ресурсы балансировщика нагрузки с несколькими IP-адресами | Каждый общедоступный IP-адрес, связанный с балансировщиком нагрузки, преобразуется в ресурс общедоступного IP-ресурса и будет связан с балансировщиком нагрузки после миграции. | | |
| Внутреннее DNS-имя на виртуальной машине | Внутреннее DNS-имя на сетевой карте | Во время миграции внутренние DNS-суффиксы обновятся в соответствии с суффиксами модели развертывания Resource Manager. Обратите внимание, что разрешение имен будет продолжать работать после миграции. Тем не менее суффиксы, связанные с виртуальными машинами, изменятся. | | |
