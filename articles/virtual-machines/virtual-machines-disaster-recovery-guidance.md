#Что делать, если прерывание работы службы Azure влияет на виртуальные машины Azure

Корпорация Майкрософт прилагает все усилия для того, чтобы наши службы были всегда доступны. Иногда по независящим от нас обстоятельствам происходят незапланированные простои служб.

Корпорация Майкрософт предоставляет соглашения об уровне обслуживания (SLA) для служб, в которых гарантируется время бесперебойной работы и доступность служб. Соглашения об уровне обслуживания для отдельных служб Azure можно найти в разделе [Соглашения об уровне обслуживания Azure](https://azure.microsoft.com/support/legal/sla/).

Платформа Azure уже имеет массу возможностей для поддержки высокодоступных приложений. Дополнительные сведения об этих службах см. в разделе [Аварийное восстановление и высокий уровень доступности для приложений Azure](https://aka.ms/drtechguide).

В этом документе описывается подлинное аварийное восстановление, при котором из-за масштабного стихийного бедствия или обширного прерывания работы службы сбою оказывается подвержен весь регион. Такие события происходят нечасто, но необходимо быть готовым к тому, что сбой всего региона возможен. Если весь регион испытывает перебои в работе службы, то локально избыточные копии ваших данных становятся временно недоступными. Если включена георепликация, то в другом регионе будут храниться три дополнительные копии BLOB-объектов и таблиц службы хранилища Azure. В случае сбоя всего региона или аварии, при которой основной регион не может быть восстановлен, Azure связывает все записи DNS с геореплицированным регионом.
 
>[AZURE.NOTE]Учтите, что этот процесс нельзя контролировать и он выполняется только при сбоях на уровне всего региона. Поэтому для обеспечения самого высокого уровня доступности необходимо также использовать и другие стратегии архивации приложения. Дополнительные сведения см. в разделе [Стратегии аварийного восстановления для данных](https://aka.ms/drtechguide#DSDR).

Чтобы помочь вам в таких ситуациях, мы предлагаем следующие рекомендации для виртуальных машин Azure. Эти рекомендации нужно применять в случае сбоя службы для всего региона, в котором развернуто приложение виртуальной машины Azure.

##Вариант 1. ожидание восстановления 
В этом случае вам не нужно предпринимать какие-либо действия. Знайте, что мы активно работаем над восстановлением доступности службы. Текущее состояние службы можно просмотреть на [панели мониторинга работоспособности служб Azure](https://azure.microsoft.com/status/).

>[AZURE.NOTE]Это наилучший вариант, если вы не настроили Azure Site Recovery (ASR), архивацию виртуальных машин, геоизбыточное хранилище с доступом для чтения (RA-GRS) или геоизбыточное хранилище (GRS) до прерывания работы службы. Если вы настроили хранилище GRS (или RA-GRS) для учетной записи хранения, в которой размещаются виртуальные жесткие диски вашей виртуальной машины, вы можете восстановить базовый образ виртуального жесткого диска и попробовать подготовить из него новую виртуальную машину. Этот вариант не является предпочтительным, так как не гарантирует синхронизацию данных, если не используются архивация Azure или ASR, и поэтому может не работать.

Для клиентов, которым требуется немедленный доступ к виртуальным машинам, доступны два следующих варианта.

>[AZURE.NOTE]Учтите, что при использовании обоих вариантов возможна потеря части данных.

##Вариант 2. Восстановление виртуальной машины из архива 
Клиенты, у которых настроена архивация виртуальных машин, могут восстановить виртуальную машину из точки архивации и восстановления.

Для восстановления виртуальной машины из архива Azure обратитесь к статье "Восстановление виртуальных машин в Azure".

Чтобы спланировать инфраструктуру архивации виртуальных машин Azure, обратитесь к статье [Планирование инфраструктуры архивации виртуальных машин в Azure](../backup/backup-azure-vms-introduction.md).

##Вариант 3. Запуск отработки отказа с помощью Azure Site Recovery (ASR) 
Если вы настроили Azure Site Recovery для работы с виртуальными машинами Azure, можно восстановить виртуальные машины из их реплик. Эти реплики могут размещаться в Azure или даже локально. В этом случае можно создать новую виртуальную машину из ее существующей реплики. Для восстановления виртуальных машин из реплики Azure Site Recovery обратитесь к статье [Перенос виртуальных машин IaaS Azure между регионами Azure с помощью Azure Site Recovery](../site-recovery/site-recovery-migrate-azure-to-azure.md).

>[AZURE.NOTE]Несмотря на то, что ОС и диски данных виртуальной машины Azure будут реплицированы на вторичный виртуальный жесткий диск (VHD), если они находятся в учетной записи хранилища GRS или RA-GRS, то каждый виртуальный жесткий диск реплицируется независимо друг от друга и этот уровень репликации не гарантирует согласованность между виртуальными жесткими дисками. Если приложение или базы данных, использующие эти диски данных, имеют зависимости друг от друга, то репликация всех виртуальных жестких дисков в один снимок не гарантируется. Также не гарантируется, что реплика виртуального жесткого диска из хранилища GRS или RA-GRS позволит получить согласованный на уровне приложения моментальный снимок для загрузки виртуальной машины.
