# Мониторинг работоспособности и пользовательская проверка работоспособности 


Шлюз приложений Azure отслеживает работоспособность всех ресурсов в своем пуле внутренних серверов и автоматически удаляет из пула все ресурсы, которые считаются неработоспособными.

Наблюдение за работоспособностью выполняется в двух режимах: проверка работоспособности по умолчанию и пользовательская проверка.

## Проверка работоспособности по умолчанию

Если ни одной из пользовательских проверок работоспособности не настроено, шлюз приложений автоматически настраивает проверку работоспособности по умолчанию. Мониторинг выполняется путем отправки HTTP-запросов к IP-адресам, настроенным для пула внутренних серверов, и к порту, настроенному в параметрах HTTP серверной части для шлюза приложений.

Например. Вы настроили шлюз приложений, так что внутренние серверы A, B и C используются для получения сетевого трафика HTTP на порт 80. По умолчанию мониторинг работоспособности осуществляет проверку трех серверов каждые 30 секунд и ожидает получения HTTP-ответа, подтверждающего работоспособность. HTTP-ответ, подтверждающий работоспособность, имеет [код состояния](https://msdn.microsoft.com/library/aa287675.aspx) от 200 до 399.

Если проверка работоспособности по умолчанию для сервера А завершается сбоем, шлюз приложений удаляет такой сервер из пула внутренних серверов и останавливает поступающий на него сетевой трафик. Проверка работоспособности по умолчанию по-прежнему продолжает проверять сервер A каждые 30 секунд. Когда сервер A успешно отвечает на запрос проверки работоспособности по умолчанию, он возвращается в пул внутренних серверов, и передача трафика на этот сервер возобновляется.

При проверке по умолчанию используются только IP-адреса для проверки состояния. Чтобы проверить работоспособность, тестируя возможность подключения с использованием URL-адреса, необходимо использовать пользовательскую проверку.


## Пользовательская проверка 

Пользовательская проверка работоспособности позволяет добиться более детального контроля над мониторингом работоспособности. Пользовательская проверка позволяет настроить интервал, URL-адрес и путь проверки, а также количество неудачных попыток проверки, после которых экземпляр пула внутренних серверов будет считаться неработоспособным.


Параметры пользовательской проверки

- **Интервал проверки** — интервал проверки работоспособности в секундах.
- **Время ожидания проверки** — время ожидания проверки для проверки запроса HTTP.
- **Порог состояния неработоспособности** — количество неудачных запросов, при достижении которого экземпляр считается неработоспособным.  
- **Имя узла и путь** — если веб-сайт или веб-ферма не возвращают HTTP-ответ только для IP-адреса, необходимо настроить проверку имени узла и пути, чтобы получить допустимый HTTP-ответ, указывающий на работоспособность экземпляра. Например: у вас есть веб-узел http://contoso.com/, но он не возвращает допустимый HTTP-ответ. Имя узла и путь нужно настроить для предоставления допустимых HTTP-ответов, указывающих на работоспособность экземпляра веб-сервера. Например, чтобы пользовательская проверка возвращала допустимый HTTP-ответ, можно выполнить такую настройку: http://contoso.com/path/custompath.htm. 



>[AZURE.NOTE] Пользовательские проверки можно настраивать только с помощью PowerShell
