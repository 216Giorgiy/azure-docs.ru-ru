# Часто задаваемые вопросы о службе Azure Multi-Factor Authentication


Данный раздел содержит ответы на часто задаваемые вопросы о службе Azure Multi-Factor Authentication. Сюда входят вопросы об использовании службы, включая модель выставления счетов и удобство использования.

**В. Каким образом сервер Azure Multi-Factor Authentication обрабатывает данные пользователя?**

При использовании локального сервера Multi-Factor Authentication (MFA) данные пользователя хранятся на локальных серверах. В облаке данные пользователя постоянно не хранятся. Когда пользователь выполняет двухфакторную проверку подлинности, сервер MFA отправляет данные в облачную службу Azure Multi-Factor Authentication для проверки подлинности. При отправке запросов проверки подлинности в облачную службу в запросы и журналы включаются указанные ниже поля, что позволяет использовать их в отчетах о проверке подлинности клиентов или об использовании. Некоторые поля являются необязательными и могут включаться и выключаться на сервере Multi-Factor Authentication. Взаимодействие сервера MFA сервера с облачной службой многофакторной проверки подлинности осуществляется по протоколу SSL/TLS через порт 443 для исходящих подключений. В число этих полей входят следующие.

- Уникальный идентификатор: имя пользователя или внутренний идентификатор сервера MFA
- Имя и фамилия: необязательно
- Адрес электронной почты: необязательно
- Номер телефона: при выполнении голосовых вызовов или проверки подлинности с помощью SMS
- Маркер устройства: при выполнении проверки подлинности мобильного приложения
- Режим проверки подлинности 
- Результат проверки подлинности 
- Имя сервера MFA 
- IP-адрес сервера MFA 
- IP-адрес клиента: если он доступен



Кроме указанных выше полей, результат проверки подлинности (пройдена или получен отказ) и причины отказов также хранятся с данными проверки подлинности и могут быть получены из отчетов о проверке подлинности или использовании.




## Выставление счетов

**В. Будет ли с моей организации взиматься плата за телефонные звонки или текстовые сообщения, используемые для проверки подлинности пользователей?**

Все издержки покрываются платой за использование службы в расчете на пользователя или на отдельные проверки подлинности. С организаций не взимается плата за отдельные телефонные звонки или текстовые сообщения пользователям в рамках функционирования службы Azure Multi-Factor Authentication. Операторы могут взимать с владельцев телефонов плату, связанную с роумингом или другими затратами на получение телефонных вызовов или текстовых сообщений.

**В. Каким образом организации выставляются счета за использование службы Azure Multi-Factor Authentication?**

Когда на портале управления Azure создается поставщик многофакторной проверки подлинности, то выбирается модель выставления счетов и использования в расчете на пользователя или на отдельные проверки подлинности. Организация платит за использование этого ресурса в рамках подписки Azure так же, как в рамках подписки выставляются счета за использование виртуальных машин, веб-сайтов и т. д.

**В. Как выставляются счета в расчете на пользователя: на основе количества пользователей, для которых активирована служба Multi-Factor Authentication, или на основе количества пользователей, выполняющих проверки?**

При выставлении счетов учитывается количество пользователей, для которых включена служба Multi-Factor Authentication.

## Удобство использования

**В. Что делать, если я не получил ответ на свой телефон или забыл свой телефон?**

Если ранее вы указали в настройках, что будете использовать запасной телефон, повторите попытку, выбрав этот телефон на странице входа в ответ на запрос. Если другой метод не настроен, обратитесь к администратору и попросите обновить номер, назначенный в качестве основного телефона (мобильного или рабочего).

**В. Я удалил пользователя из роли администратора, но забыл отключить многофакторную проверку подлинности, и теперь она не отображается в списке. Как удалить эту функцию?**

- В зависимости от используемого портала, на левой панели щелкните «Пользователи» или «Пользователи и группы».
- Установите флажок рядом с пользователем, которого нужно изменить, и нажмите кнопку редактирования или значок правки.
- Выберите «Параметры», в разделе назначения роли выберите «Да» и добавьте пользователя в предыдущую роль администратора.
- Перейдите на страницу многофакторной проверки подлинности. Учетная запись теперь должна отображаться в списке на странице. 
- Выполните шаги, описанные выше, по отключению многофакторной проверки подлинности для учетной записи. 
- На этом этапе можно удалить учетную запись из роли администратора.


**В. Что делать, если пользователь обращается к администратору с проблемой блокирования учетной записи?**

Вы можете сбросить настройки пользователя, заставив его снова пройти процесс регистрации. Сведения об этом см. в статье [Управление параметрами пользователей в облаке с помощью службы Azure Multi-Factor Authentication](multi-factor-authentication-manage-users-and-devices.md).

**В. Что делать, если устройство пользователя, использовавшее пароли приложений, утеряно или украдено?**

Вы можете удалить все пароли приложений для предотвращения несанкционированного доступа. После приобретения нового устройства пользователь может создать их заново. Сведения об этом см. в статье [Управление параметрами пользователей в облаке с помощью службы Azure Multi-Factor Authentication](multi-factor-authentication-manage-users-and-devices.md).

**В. Что делать, если пользователь не может войти в приложения, не использующие браузер?**

- Пользователям с активированной многофакторной проверкой подлинности требуется пароль для входа в некоторые приложения, не использующие браузер.
- Им придется очистить данные для входа (удалить их), перезапустить приложение и войти, используя свои имя пользователя и пароль для приложения. 

Дополнительные сведения о создании паролей приложений см. в статье [Справочная информация о паролях приложений](multi-factor-authentication-end-user-app-passwords.md).


>[AZURE.NOTE]Современная проверка подлинности для клиентов Office 2013
>
> Клиенты Office 2013 (включая Outlook) теперь поддерживают новые протоколы проверки подлинности, благодаря чему в них можно активировать поддержку многофакторной проверки подлинности. Это означает, что после активации отпадает потребность в использовании паролей приложений с клиентами Office 2013. Дополнительные сведения см. в статье [Анонсирована общедоступная предварительная версия современной проверки подлинности Office 2013](https://blogs.office.com/2015/03/23/office-2013-modern-authentication-public-preview-announced/).

**В. Что делать, если текстовое сообщение не приходит или при ответе на двухстороннее текстовое сообщение истекает время ожидания проверки?**

Служба Azure Multi-Factor Authentication отправляет текстовые сообщения через SMS-агрегаторы. На надежность доставки текстовых сообщений может повлиять множество факторов, например используемый агрегатор, страна назначения, оператор мобильной связи и сила сигнала. Таким образом мы не гарантируем доставку текстовых сообщений и получение SMS-ответов при использовании двусторонних SMS. Одностороннее SMS предпочтительней двустороннего, так как оно более надежное и не требует оплаты международной SMS-пересылки, когда вы отвечаете на текстовое сообщение, отправленное из другой страны.

В некоторых странах, например в США и Канаде, проверки с помощью текстовых сообщений более надежны. Пользователям, которые при использовании Azure Multi-Factor Authentication не всегда получают текстовые сообщения, рекомендуется выбрать метод, в рамках которого используются мобильные приложения или телефонные звонки. Использовать мобильное приложение очень удобно, так как уведомления от него можно получать по сотовому и по Wi-Fi-подключению, а его секретный код отображается даже при полном отсутствии сигнала. Приложение Azure Authenticator доступно для [Windows Phone](http://www.windowsphone.com/store/app/azure-authenticator/03a5b2bf-6066-418f-b569-e8aecbc06e50), [Android](https://play.google.com/store/apps/details?id=com.azure.authenticator) и [iOS](https://itunes.apple.com/us/app/azure-authenticator/id983156458).


## Ошибки

**В. Что делать, если при проверке подлинности с помощью уведомлений мобильного приложения возникает ошибка «Запрос на проверку подлинности не для активированной учетной записи»?**

- Перейдите на страницу [https://account.activedirectory.windowsazure.com/profile/](https://account.activedirectory.windowsazure.com/profile/) и войдите в учетную запись своей организации.
- При необходимости щелкните «Другие способы проверки» и выберите другой вариант для проверки учетной записи.
- Щелкните «Дополнительная проверка безопасности».
- Удалите существующую учетную запись из мобильного приложения.
- Нажмите кнопку «Настроить» и следуйте инструкциям по перенастройке мобильного приложения.




**В. Что делать, если при попытке войти с помощью приложения, не использующего браузер, возникает ошибка 0x800434D4L?**

В настоящее время дополнительная проверка безопасности может использоваться только с приложениями или службами, доступными через браузер. Приложения, которые не используют браузер (называемые также полнофункциональными клиентскими приложениями) и которые установлены на локальном компьютере (как, например, Windows Powershell), не работают с учетными записями, которые требуются для дополнительной проверки безопасности. В этом случае может появиться ошибка 0x800434D4L.

Для решения этой проблемы нужно иметь отдельные учетные записи пользователя для операций, связанных с администрированием, и для операций, и не связанных с администрированием. И уже потом эти почтовые ящики можно связать, чтобы входить в Outlook с помощью учетной записи без прав администратора. Дополнительные сведения об этом см. в статье [Предоставление администратору возможности открывать и просматривать содержимое почтового ящика пользователя] (http://help.outlook.com/141/gg709759(d=loband).aspx?sl=1).
