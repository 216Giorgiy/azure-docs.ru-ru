# <a name="how-to-require-two-step-verification-for-a-user-or-group"></a>Как настроить требование двухфакторной проверки подлинности пользователя или группы

Можно выполнить одно из двух подходов, когда требуется двухшаговую проверку. Первый вариант — включить каждого пользователя Azure Multi-factor Authentication (MFA). Когда пользователи включаются по отдельности, они выполняют двухшаговую проверку при каждом входе в (за некоторыми исключениями, например при входе из надежных IP-адресов или при _сохраненных устройствах_ включены). Второй способ — настроить политику условного доступа, которая требует прохождения двухфакторной проверки подлинности при определенных условиях.

>[!TIP] 
>Выберите один из этих методов, чтобы требовать прохождения двухфакторной проверки подлинности, но не оба. Предоставление пользователю для многофакторной проверки подлинности Azure переопределяет все политики условного доступа.

## <a name="which-option-is-right-for-you"></a>Какой вариант подходит для вас?

**Включение многофакторной проверки подлинности Azure путем изменения состояния пользовательской** является традиционный подход, когда требуется двухшаговую проверку. Он подходит и для Azure MFA в облаке, и для сервера Azure MFA. Все пользователи, которые можно включить выполнить двухшаговую проверку, каждый раз при входе в. Предоставление пользователю переопределяет все политики условного доступа, которые могут повлиять на этого пользователя. 

**Включение многофакторной проверки подлинности Azure с политикой условного доступа** является более гибкий подход, когда требуется двухшаговую проверку. Он работает только для Azure MFA в облаке, однако и _условного доступа_ — [платные функции Azure Active Directory](https://www.microsoft.com/cloud-platform/azure-active-directory-features). Можно создать политики условного доступа, применяемые как к группам, так и к отдельным пользователям. Для групп с высоким риском можно задать больше ограничений, чем для групп с низким риском. Или вы можете сделать двухфакторную проверку подлинности обязательной только для облачных приложений с высоким риском, пропуская ее для приложений с низким риском. 

Оба варианта предлагать пользователям зарегистрироваться для многофакторной проверки подлинности Azure при первом запуске в систему после включения требования. Оба варианта также работать с настраиваемые [параметры многофакторной проверки подлинности Azure](multi-factor-authentication-whats-next.md).

Состояние пользователя отражает, указал ли администратор его в Azure MFA и завершил ли пользователь процесс регистрации.

Все пользователи начинают *отключено*. При регистрации пользователей в Azure MFA, их состояние меняется на *включено*. При включенных пользователей войти и завершить процесс регистрации, их состояние меняется на *принудительно*.

### <a name="use-powershell"></a>Использование PowerShell
Чтобы изменить состояние пользователя с помощью [Azure AD PowerShell](/powershell/azure/overview), изменить `$st.State`. Существуют три возможных состояния:

* Включено
* Принудительно
* Отключено  

Не переводите пользователей непосредственно в состояние *Применено*. В противном случае не браузер приложения перестанут работать, так как у пользователя не проходили регистрацию многофакторной проверки Подлинности Azure и получить [пароль приложения](multi-factor-authentication-whats-next.md#app-passwords). 

PowerShell — удобный инструмент для массового включения пользователей. Создайте сценарий PowerShell, который обходит список пользователей и включает их.

        $st = New-Object -TypeName Microsoft.Online.Administration.StrongAuthenticationRequirement
        $st.RelyingParty = "*"
        $st.State = “Enabled”
        $sta = @($st)
        Set-MsolUser -UserPrincipalName bsimon@contoso.com -StrongAuthenticationRequirements $sta

Следующий сценарий — пример:

    $users = "bsimon@contoso.com","jsmith@contoso.com","ljacobson@contoso.com"
    foreach ($user in $users)
    {
        $st = New-Object -TypeName Microsoft.Online.Administration.StrongAuthenticationRequirement
        $st.RelyingParty = "*"
        $st.State = “Enabled”
        $sta = @($st)
        Set-MsolUser -UserPrincipalName $user -StrongAuthenticationRequirements $sta
    }

## <a name="enable-azure-mfa-with-a-conditional-access-policy"></a>Включение Azure MFA с политикой условного доступа

_Условный доступ_ является платные функции Azure Active Directory с множество параметров конфигурации. Ниже описан один из способов создания политики. Дополнительные сведения см. в разделе [Условный доступ в Azure Active Directory](../active-directory/active-directory-conditional-access-azure-portal.md).

1. Войдите на [портал Azure](https://portal.azure.com) с использованием учетной записи администратора.
2. Выберите **Azure Active Directory** > **Условный доступ**.
3. Выберите **Новая политика**.
4. В разделе **Назначения** выберите **Пользователи и группы**. Используйте **Include** и **исключить** вкладок, чтобы указать, какие пользователи и группы политики управляет.
5. В разделе **Назначения** выберите **Облачные приложения**. Выберите параметр **Все облачные приложения**, чтобы включить все облачные приложения.
6. В разделе **Элементы управления доступом** выберите **Предоставление**. Выберите **Требовать многофакторную проверку подлинности**.
7. Включить **включить политику** для **на**, а затем выберите **Сохранить**.

Другие параметры в политике условного доступа предоставляют возможность указывать только в том случае, когда требуется двухшаговую проверку. Например, можно создать политики подобные: когда подрядчиков попытается обратиться к нашего приложения закупок из ненадежных сетей на устройствах, не присоединенных к домену, требуется двухшаговую проверку.
