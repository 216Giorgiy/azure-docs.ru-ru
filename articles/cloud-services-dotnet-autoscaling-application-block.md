## <a id="WhatIs"> </a>Что такое блок приложения автомасштабирования?

Блок приложения автомасштабирования позволяет автоматически масштабировать приложение Microsoft
Azure на основе правил, которые вы определяете специально для своего
приложения. Эти правила можно использовать, чтобы приложение Azure
поддерживало свою пропускную способность в ответ на изменения своей
рабочей нагрузки, а также чтобы управлять расходами, связанными с
размещением приложения в Azure. Наряду с масштабированием путем
увеличения или уменьшения количества экземпляров ролей в вашем
в приложении этот блок также позволяет применять другие действия масштабирования,
например регулирование определенных функциональных возможностей приложения или
применение пользовательских действий.

Можно выбрать размещение блока в роли Azure или в
локальном приложении.

Блок приложения автомасштабирования является частью [Microsoft Enterprise Library 5.0 Integration Pack для Azure][].

## <a id="Concepts"> </a>Основные понятия

На следующей схеме зеленая линия является проекцией количества
работающих экземпляров роли Azure за последние два дня. Число
экземпляров меняется автоматически с течением времени в ответ на набор
правил автомасштабирования.

![схема примера автомасштабирования](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling01.png)

Блок использует два типа правил для определения поведения автомасштабирования
вашего приложения:

-   **Правила ограничения.** Чтобы задать верхнюю и нижнюю границы количества
    экземпляров. Например, если необходимо, чтобы каждое утро между 8:00 и 10:00
    работало не менее четырех и не более шести экземпляров,
    вы можете использовать **правило ограничения**. На схеме красная и синяя
    линии представляют правила ограничений. Например, в точке **A** на
    схеме минимальное число экземпляров ролей увеличивается с двух до
    четырех, чтобы удовлетворить ожидаемое увеличение
    рабочей нагрузки приложения в это время. В точке **B** на схеме
    количество экземпляров ролей не может превышать пять,
    что позволяет контролировать эксплуатационные расходы на приложение.

-   **Реактивные правила.** Чтобы количество экземпляров менялось
    в ответ на непрогнозируемые изменения спроса, следует использовать **реактивные
    правил**. В точке **C** на схеме блок автоматически
    уменьшает число экземпляров роли с четырех до трех в
    ответ на уменьшение рабочей нагрузки. В точке **D** блок
    обнаруживает увеличение рабочей нагрузки и автоматически повышает
    количество работающих экземпляров ролей с трех до четырех.

Блок сохраняет свои параметры конфигурации в двух хранилищах:

-   **Хранилище правил.** Хранилище правил содержит бизнес-конфигурацию,
    список всех правил автомасштабирования, определенных для
    приложения Azure. Как правило, это хранилище представляет собой XML-файл,
    который расположен в месте, доступном для чтения блоку приложения автомасштабирования,
    например в хранилище больших двоичных объектов Azure или в локальном файле.

-   **Хранилище информации о службе.** В хранилище информации о службе содержится
    рабочая конфигурация, которая является моделью службы
    приложения Azure. Она включает в себя всю
    информацию о приложении Azure (например, имена ролей
    и информацию об учетной записи хранения), которая необходима блоку для
    сбора точек данных из целевого приложения Azure и
    выполнения операций масштабирования.

## <a id="PerfCounter"> </a>Сбор данных счетчика производительности из целевого приложения Azure

Реактивные правила могут использовать данные счетчиков производительности ролей в рамках
определения правила. Например, реактивное правило может отслеживать использование ЦП
ролью Azure для определения, следует ли блоку
инициировать операцию масштабирования. Блок считывает данные счетчиков производительности
из таблицы диагностики Azure с именем
**WADPerformanceCountersTable**, которая размещена в службе хранилища Azure.

По умолчанию Azure не записывает данные счетчиков производительности
в таблицу диагностики Azure в службе хранилища Azure. Таким образом, вам
следует изменить роли, из которых требуется собрать данные счетчиков производительности
для сохранения этих данных. Дополнительную информацию о включении
счетчиков производительности в приложении см. в разделе [Использование счетчиков производительности в Azure][].

.

## <a id="DefineServiceModel"> </a>Практическое руководство. Определение модели службы

Как правило, модель службы (описание среды
Microsoft Azure, которое содержит данные о подписках, размещенных
службах, ролях и учетных записях хранения) сохраняется в XML-файле. Вы можете найти
копию схемы для этого XML-файла в файле
**AutoscalingServiceModel.xsd** своего проекта. В Visual Studio
эта схема предоставляет Intellisense и способ проверки редактируемого
XML-файла модели службы.

Создайте новый XML-файл с именем **services.xml** в своем проекте.

В Visual Studio необходимо убедиться, что файл модели службы скопирован
в папку выхода. Для этого:

1.  Щелкните правой кнопкой мыши файл и выберите пункт **Свойства**.

2.  На панели "Свойства" задайте для параметра **Копировать в выходной каталог**
    значение **Всегда копировать**.

    ![Set Copy to Output Directory value](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling04.png)


	В следующем примере кода показан пример модели службы в файле **services.xml**:

    <?xml version="1.0" encoding="utf-8" ?>
    <serviceModel xmlns="http://schemas.microsoft.com/practices/2011/entlib/autoscaling/serviceModel">
      <subscriptions>
        <subscription name="[subscriptionname]"
                      certificateThumbprint="[managementcertificatethumbprint]"
                      subscriptionId="[subscriptionid]"
                      certificateStoreLocation="CurrentUser"
                      certificateStoreName="My">
          <services>
            <service dnsPrefix="[hostedservicednsprefix]" slot="Staging">
              <roles>
                <role alias="AutoscalingApplicationRole"
                      roleName="[targetrolename]"
                      wadStorageAccountName="targetstorage"/>
              </roles>
            </service>
          </services>
          <storageAccounts>
            <storageAccount alias="targetstorage"
              connectionString="DefaultEndpointsProtocol=https;AccountName=[storageaccountname];AccountKey=[storageaccountkey]">
            </storageAccount>
          </storageAccounts>
        </subscription>
      </subscriptions>
    </serviceModel>

## <a id="DefineAutoscalingRules"> </a>Практическое руководство. Определение правил автомасштабирования

Обычно правила автомасштабирования управляют рядом
экземпляров ролей в целевом приложении в XML-файле. Вы можете найти
копию схемы для этого XML-файла в файле **AutoscalingRules.xsd**
в своем проекте. В Visual Studio эта схема предоставляет
IntelliSense и способ проверки редактируемого XML-файла.

Создайте новый XML-файл с именем **rules.xml** в своем проекте.

В Visual Studio необходимо убедиться, что файл правил скопирован в
в выходную папку. Для этого:

1.  Щелкните правой кнопкой мыши файл и выберите пункт **Свойства**.

2.  На панели "Свойства" задайте для параметра **Копировать в выходной каталог**
    значение **Всегда копировать**.

В следующем примере кода показан пример набора правил в файле **rules.xml**
:

    <?xml version="1.0" encoding="utf-8" ?>
    <rules xmlns="http://schemas.microsoft.com/practices/2011/entlib/autoscaling/rules">
      <constraintRules>
        <rule name="default" enabled="true" rank="1" description="The default constraint rule">
          <actions>
            <range min="2" max="6" target="AutoscalingApplicationRole"/>
          </actions>
        </rule>
      </constraintRules>
      <reactiveRules>
        <rule name="ScaleUpOnHighUtilization" rank="10" description="Scale up the web role" enabled="true" >
          <when>
            <any>
               <greaterOrEqual operand="WebRoleA_CPU_Avg_5m" than="60"/>
            </any>
          </when>
          <actions>
            <scale target="AutoscalingApplicationRole" by="1"/>
          </actions>
        </rule>
        <rule name="ScaleDownOnLowUtilization" rank="10" description="Scale up the web role" enabled="true" >
          <when>
            <all>
              <less operand="WebRoleA_CPU_Avg_5m" than="60"/>
            </all>
          </when>
          <actions>
            <scale target="AutoscalingApplicationRole" by="-1"/>
          </actions>
        </rule>
      </reactiveRules>
      <operands>
        <performanceCounter alias="WebRoleA_CPU_Avg_5m"
          performanceCounterName="\Processor(_Total)\% Processor Time"
          source ="AutoscalingApplicationRole"
          timespan="00:05:00" aggregate="Average"/>
      </operands>
    </rules>

В этом примере есть три правила автомасштабирования (одно **правило
ограничения** и два **реактивных правила**), которые применяются к целевому объекту
**AutoscalingApplicationRole**, который является псевдонимом роли, определенной в
**модели службы**.

-   Правило ограничения всегда активно и задает минимальное число
    экземпляров роли, равное двум, и максимальное число экземпляров роли,
    равное 6.

-   Оба реактивных правила используют **операнд**
    **WebRoleA\_CPU\_Avg\_5m**, который вычисляет средний показатель использование ЦП
    за последние пять минут для роли Azure
    **AutoscalingApplicationRole.** Эта роль определяется в
    **модели службы**.

-   Реактивное правило **ScaleUpOnHighUtilization** увеличивает
    число экземпляров целевой роли на один, если средний
    коэффициент использования ЦП за последние пять минут был
    не менее 60 %.

-   Реактивное правило **ScaleDownOnLowUtilization** увеличивает
    число экземпляров целевой роли на один, если средний
    коэффициент использования ЦП за последние пять минут составлял менее 60 %.

## <a id="Configure"> </a>Практическое руководство. Настройка блока приложения автомасштабирования

Определив правила модели службы и автомасштабирования, необходимо
настроить блок приложения автомасштабирования для их использования. В этом
информация об оперативной конфигурации сохраняется в файле
файл конфигурации службы.

По умолчанию блок приложения автомасштабирования ожидает сохранения
правил автомасштабирования и модели службы в больших двоичных объектах Azure. В этом
примере выполняется настройка блока для загрузки правил из локальной файловой
системы.

### Настройте блок приложения автомасштабирования в ведущем приложении

1.  Щелкните правой кнопкой мыши файл **App.config** в обозревателе решений, а затем
    щелкните **Изменить файл конфигурации**.

2.  В меню **Блоки** щелкните **Добавить параметры автомасштабирования**.  
	![image](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling10.png)
  
3.  Разверните **Параметры автомасштабирования** и щелкните многоточие (...)
    рядом с элементом **Учетная запись хранения хранилища точек данных**, введите **имя учетной
    записи** и **ключ учетной записи** для учетной записи хранения Azure,
    в которую блок будет сохранять собираемые данные (см.
    [Практическое руководство. Определение модели службы][], чтобы получить информацию о том,
    как найти эти значения), а затем нажмите кнопку **ОК**:  

	![image](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling11.png)

4.  Разверните раздел **Параметры автомасштабирования**, чтобы открыть разделы **Хранилище
    правил** и **Хранилище информации о службе**. По умолчанию они
    настроены для использования хранилища больших двоичных объектов Azure:  
	![image](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling12.png)


5.  Щелкните знак "плюс" (+) рядом с пунктом **Хранилище правил**, выберите пункт **Задать
    хранилище правил**, щелкните **Использовать локальное хранилище правил файлов** и
    Щелкните **ДА**.

6.  В поле **Имя файла** введите **rules.xml**. Это имя
    файла, который содержит правила автомасштабирования:  
	![image](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling13.png)


7.  Щелкните знак "плюс" (+) рядом с пунктом **Хранилище информации о службе**, выберите пункт
    **Задать хранилище информации о службе**, щелкните **Использовать локальное хранилище
    файлов информации о службе** и нажмите кнопку **Да**.

8.  В поле **Имя файла** введите **services.xml**. Это имя
    файла, который содержит правила автомасштабирования:  
	![image](./media/cloud-services-dotnet-autoscaling-application-block/autoscaling14.png)


9.  В окне конфигурации библиотеки предприятия в меню **Файл**
    щелкните **Сохранить**, чтобы сохранить изменения конфигурации. Затем в
    окне конфигурации библиотеки предприятия в меню **Файл**, щелкните
    **Выход**.

Для получения подробной информации о действиях, выполняемых блоком
приложения автомасштабирования, необходимо записать сообщения,
записываемые в журналы. Например, при размещении блока в консольном
приложении можно просматривать сообщение журнала в окне вывода
Visual Studio. В следующем разделе показано, как настроить такое
поведение.

### Настройте ведение журналов в ведущем приложении блока приложения автомасштабирования

1.  В Visual Studio дважды щелкните файл **App.config** в
    обозревателе решений, чтобы открыть его в редакторе. Затем добавьте
    раздел **system.diagnostics** как показано в следующем примере:

        <?xml version="1.0" encoding="utf-8" ?>
        <configuration>
          ...
          <system.diagnostics>
            <sources>
              <sourcename="Autoscaling General" switchName="SourceSwitch" switchType="System.Diagnostics.SourceSwitch" />
              <sourcename="Autoscaling Updates" switchName="SourceSwitch" switchType="System.Diagnostics.SourceSwitch" />
            </sources>
            <switches>
              <addname="SourceSwitch" value="Verbose, Information, Warning, Error, Critical" />
            </switches>
          </system.diagnostics>
        </configuration>

2.  Сохраните изменения.

Теперь можно запустить блок приложения автомасштабирования ведущего консольного
приложения и посмотреть, как работают правила автомасштабирования с целевым
приложением Azure. При запуске ведущего консольного приложения
в окне вывода Visual Studio должны появиться сообщения, аналогичные
приведенному. Эти сообщения журнала помогают понять поведение
