#Ограничения предварительной версии Azure Stream Analytics Preview и известные проблемы

В данном документе описываются ограничения и известные проблемы [Azure Stream Analytics][stream.analytics.documentation] в версии для предварительного ознакомления.  В большинстве случаев данные ограничения вводятся с намерением стимулировать отклик пользователей или из-за существующих ограничений версии.

##<a name="limitations"></a> Ограничения

###Доступ по регионам
Задания Stream Analytics предварительной версии могут задаваться в Центральном регионе США и Западной Европе.

###Масштаб 
**Квота потоковой единицы**

В связи с существующими ограничениями принудительно задается квота в 12 потоковых единиц на регион на подписку. Для получения дополнительных сведений см. [Масштабирование заданий Azure Stream Analytics][stream.analytics.scale.jobs]. Если у вас возникнет необходимость увеличить указанные пределы, позвоните в [службу поддержки Майкрософт][microsoft.support] и мы сделаем все от нас зависящее в рамках ограничений публичной оферты. 

**Применение потоковой единицы**

В данной предварительной версии число предоставляемых заданию потоковых единиц может быть выше, чем выбранное или оплаченное.  Кроме того, потоковые единицы не будут сокращаться. Это значит, что наблюдаемая производительность может быть выше, чем обещанная гарантиями. Это зависит от вычислительных ресурсов.

**Ключ секции**

При масштабировании запроса директивой PARTITION BY поле для раздела должно быть PartitionId.  Разделение на другие пользовательские поля будет включено в более поздних версиях.
Для получения дополнительных сведений о масштабировании задания см. [Масштабирование заданий Azure Stream Analytics][stream.analytics.scale.jobs].

###Входные данные



**Символьная кодировка**

Для вводных источников CSV и JSON единственной поддерживаемой символьной кодировкой является UTF-8.


###Сложные запросы
Максимальное число поддерживаемых агрегатных функций в определении запроса задания Stream Analytics равняется 7.

###Управление жизненным циклом

**Обновление задания**

В настоящее время Stream Analytics не поддерживает изменение определений и конфигураций текущего задания в реальном времени.  Для изменения ввода, вывода, запроса, масштаба или конфигурации текущего задания следует выполнение этого задания остановить.

**Остановка и перезапуск задания**

При остановке задания его состояние не сохраняется. Это значит, что возможности перезапустить задание с точки его остановки нет.  Проблема этого ограничения будет решена в следующей версии.  Для получения практических рекомендаций о запуске и остановке выполнения задания см. [Руководство разработчика Azure Stream Analytics][stream.analytics.developer.guide]. 

###Мониторинг
Некоторые показатели, связанные с выполнением и производительностью при выполнении задания, например задержки, в предварительной версии недоступны.  Предварительная версия также обрабатывает задание по числу событий, а не размеру.

##<a name="knownissues"></a>Примечания о версии / известные проблемы

Здесь содержится список известных проблем в Azure Stream Analytics. Этот раздел со временем будет изменяться по ходу удаления позиций из списка, обнаружения новых проблем или новых подробностей об уже известных.


###Требуется повышенный уровень доступа к концентратору событий
На данный момент Stream Analytics требует уровня доступа по правилу общего доступа с управляемым доступом к источнику ввода и вывода концентратора событий.

###Задержка в конфигурации учетной записи хранилища Azure
При первом запуске задания в Stream Analytics вам будет предложено создать новую учетную запись для хранилища или ввести данные уже существующих для мониторинга заданий в Stream Analytics в соответствующем регионе.  Из-за задержек в мониторинге конфигурации, запуск еще одного задания в Stream Analytics в том же регионе в пределах 30 минут будет требовать введения данных другой учетной записи хранилища, а не показывать уже сконфигурированную в выпадающем меню мониторинга учетных записей хранилища.  Во избежание потребности в создании ненужных учетных записей хранилища следует подождать 30 минут после первого запуска предыдущего задания в том или ином регионе перед тем, как запускать новое задание в том же регионе. 

###В концентраторе событий задания используют клиентскую группу по умолчанию
При указании концентратора событий в качестве ввода задания Stream Analytics будут использовать клиентскую группу по умолчанию для приема данных из этого концентратора.  Выполнение этого без дополнительной конфигурации значит то, что никакие другие получатели не смогут иметь доступ к данному концентратору событий.  Чтобы концентратор событий мог иметь больше одного получателя, следует настроить новые клиентские группы.  Для получения дополнительных сведений см. [Руководство разработчика по концентраторам событий][azure.event.hubs.developer.guide].

###Добавление ввода/вывода - концентратор событий 
На третьей странице серии диалоговых окон "Добавление ввода и добавление вывода" для источников концентратора событий имеется выпадающее меню "Концентратор событий", которое содержит список пространства имен служебной шины в текущей подписке и вариант подключения к концентратору событий другой подписки.  При желании подключиться к концентратору событий той же подписки следует выбрать ее пространство имен служебной шины здесь.  При желании подключится к концентратору событий вне данной подписки выберите вариант "Использовать концентратор событий из другой подписки".  


###Не может ссылаться на один и тот же этап выполнения запроса несколько раз
В данной предварительной версии на каждый этап выполнения вложенного запроса, определенного директивой WITH, можно ссылаться лишь один раз.  В общих сценариях это может вызвать самосоединение по псевдонимам того же этапа.  Чтобы обойти такой вариант, рекомендуется задавать два этапа в одном вложенном запросе и под разными именами.

###Если преобразование типа не поддерживается, его результат выведет значение NULL
Любые события по преобразованию типа не поддерживаются в разделе "Типы данных" [Справочника по языку запросов Azure Stream Analytics][stream.analytics.query.language.reference], и на выходе их значение будет NULL.  В данной предварительной версии журнал учета ошибок для исключений такого преобразования не ведется. 

###Проблема нехватки памяти
Задание Streaming Analytics с широким окном допусков для событий отключений и/или сложных запросов с большим числом состояний может привести к нехватке памяти для этого задания, из-за чего это задание перезапустится. Операции запуска и остановки будут записаны в операционном журнале задания.  Чтобы избежать такого варианта, следует масштабировать запрос по нескольким разделам.  В новой версии данное ограничение будет снято за счет снижения производительности при выполнении задания, а не путем его перезапуска.

###Пустой сегмент концентратора в разделенном запросе приведет к отсутствию вывода
При выполнении разделенного запроса с неразделенным вложенным запросом в качестве второго этапа и при условии, что один из разделов концентратора событий на вводе оказывается пустым, такой запрос не выдаст никакого результата. Запись об ошибке при таком развитии событий будет внесена в операционный журнал данного задания.  Проверьте, что все разделы концентратора событий имеют входящие события, чтобы избежать возникновения такой проблемы.

###Ограничение по объему событий базы данных SQL
При использовании базы данных SQL в качестве источника вывода большой объем выходных данных может привести к тому, что задание Stream Analytics превысит время ожидания. Для решения этой проблемы следует или снизить объем выходных данных с помощью директив агрегирования или соединения, или выбрать хранилище BLOB-объектов или концентратор событий в качестве вывода.

###Ввод больших BLOB-данных не поддерживается
Извлечение больших файлов из хранилища BLOB-объектов может привести к системному сбою в задании Stream Analytics.  Чтобы избежать этой проблемы, не превышайте размер в 10 МБ BLOB-объекта.

###Директива LEFT OUTER JOIN не поддерживается
Директива LEFT OUTER JOIN в языке запросов Stream Analytics имеется, но не поддерживается.  Выполнение запроса с LEFT OUTER JOIN на протяжении нескольких минут может привести к ошибке из-за нехватки памяти.  Мы не рекомендуем использовать эту директиву для всех сценариев, кроме экспериментальных коротких запросов.  Данное ограничение будет снято в следующей версии.
