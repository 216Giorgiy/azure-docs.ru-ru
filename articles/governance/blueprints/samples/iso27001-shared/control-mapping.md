# <a name="control-mapping-of-the-iso-27001-shared-services-blueprint-sample"></a>Сопоставление элементов управления образца схемы "ISO 27001: общие службы"

Ниже в статье подробно описано, как образец схемы Azure Blueprints "ISO 27001: общие службы" сопоставляется с элементами управления ISO 27001. Дополнительные сведения об элементах управления см. в разделе [ISO 27001](https://www.iso.org/isoiec-27001-information-security.html).

Далее представлены сопоставления для элементов управления **ISO 27001:2013**. Используйте панель навигации справа для перехода непосредственно к сопоставлению конкретных элементов управления. Многие сопоставленные элементы управления реализуются с помощью инициативы [Политика Azure](../../../policy/overview.md). Чтобы просмотреть полную инициативу, откройте раздел **Политика** на портале Azure и выберите страницу **Определения**. Затем найдите и выберите встроенную инициативу политики **[Предварительная версия.] Аудит элементов управления ISO 27001:2013 и развертывание определенных расширений виртуальной машины для обеспечения требований аудита**.

## <a name="a612-segregation-of-duties"></a>A.6.1.2: разделение обязанностей

Наличие только одного владельца подписки Azure не позволит организовать административную избыточность. И наоборот, слишком большое число владельцев подписки Azure может увеличить вероятность бреши в системе безопасности через скомпрометированную учетную запись владельца. Эта схема помогает сохранить требуемое число владельцев подписки Azure, назначая два определения [Политики Azure](../../../policy/overview.md) для аудита числа владельцев для подписок Azure. Управление разрешениями владельца подписки помогает реализовать соответствующее разделение обязанностей.

- [Предварительный просмотр]: Audit minimum number of owners for subscription
- [Предварительный просмотр]: Audit maximum number of owners for a subscription

## <a name="a912-access-to-networks-and-network-services"></a>A.9.1.2: доступ к сети и сетевым службам

Azure реализует [управление доступом на основе ролей](../../../../role-based-access-control/overview.md) (RBAC) для управления пользователями, которые обращаются к ресурсам Azure. В рамках этой схемы вы управляете доступом к ресурсам Azure, присвоив семь определений [Политики Azure](../../../policy/overview.md). Эти политики ведут аудит использования типов и конфигураций ресурсов, которые предоставляют повышенные права доступа к ресурсам.
Определив ресурсы, нарушающие эти политики, вы сможете выполнить корректирующие действия, чтобы ограничить доступ к ресурсам Azure авторизованными пользователями.

- [Предварительный просмотр]: Deploy VM extension to audit Linux VM accounts with no passwords
- [Предварительный просмотр]: Deploy VM extension to audit Linux VM allowing remote connections from accounts with no
  паролиs
- [Предварительный просмотр]: Audit Linux VM accounts with no passwords
- [Предварительный просмотр]: Audit Linux VM allowing remote connections from accounts with no passwords
- Аудит использования классических учетных записей хранения
- Аудит использования классических виртуальных машин
- Аудит виртуальных машин, которые не используют управляемые диски

## <a name="a922-user-access-provisioning"></a>A.9.2.2: подготовка доступа пользователей

Azure реализует [управление доступом на основе ролей](../../../../role-based-access-control/overview.md) (RBAC) для управления пользователями, которые обращаются к ресурсам Azure. В рамках этой схемы присваиваются три определения [Политики Azure](../../../policy/overview.md) для аудита использования проверки подлинности [Azure Active Directory](../../../../active-directory/fundamentals/active-directory-whatis.md) для серверов SQL Server и [Service Fabric](../../../../service-fabric/service-fabric-overview.md). Использование проверки подлинности Azure Active Directory упрощает контроль разрешений и обеспечивает централизованное управление удостоверениями для пользователей баз данных и других служб Майкрософт. В рамках этой схемы также присваивается определение Политики Azure для аудита использования настраиваемых правил RBAC. Поняв, где реализуются настраиваемые правила RBAC, вы сможете определить необходимость и проверить правильность реализации, так как настраиваемые правила RBAC подвержены ошибкам.

- Аудит подготовки администратора Azure Active Directory для SQL Server
- Аудит использования Azure Active Directory для проверки подлинности клиента в Service Fabric
- Аудит использования настраиваемых правил RBAC

## <a name="a923-management-of-privileged-access-rights"></a>A.9.2.3: управление привилегированными правами доступа

В рамках этой схемы вы ограничиваете и контролируете привилегированные права доступа, присвоив четыре определения [Политики Azure](../../../policy/overview.md) для аудита внешних учетных записей с владельцем и/или разрешениями на запись и учетных записей с владельцем и/или разрешениями на запись без включенной многофакторной проверки подлинности.

- [Предварительный просмотр]: Audit accounts with owner permissions who are not MFA enabled on a subscription
- [Предварительный просмотр]: Audit accounts with write permissions who are not MFA enabled on a subscription
- [Предварительный просмотр]: Audit external accounts with owner permissions on a subscription
- [Предварительный просмотр]: Audit external accounts with write permissions on a subscription

## <a name="a924-management-of-secret-authentication-information-of-users"></a>A.9.2.4: управление информацией проверки подлинности секрета пользователей

В рамках этой схемы присваиваются три определения [Политики Azure](../../../policy/overview.md) для аудита учетных записей без включенной многофакторной проверки подлинности. Многофакторная проверка подлинности помогает защитить учетные записи даже в случае компрометации одной части сведений о проверке подлинности. Наблюдая за учетными записями без включенной многофакторной проверки подлинности, вы можете определить учетные записи, которые с большей вероятностью могли быть скомпрометированы. В рамках этой схемы также присваиваются два определения Политики Azure, которые ведут аудит разрешений файла пароля виртуальной машины Linux для создания предупреждения в случае их неправильной установки. Эта настройка позволяет предпринять корректирующее действие, чтобы убедиться, что структуры проверки подлинности не скомпрометированы.

- [Предварительный просмотр]: Audit accounts with owner permissions who are not MFA enabled on a subscription
- [Предварительный просмотр]: Audit accounts with read permissions who are not MFA enabled on a subscription
- [Предварительный просмотр]: Audit accounts with write permissions who are not MFA enabled on a subscription
- [Предварительный просмотр]: Deploy VM extension to audit Linux VM passwd file permissions
- [Предварительный просмотр]: Audit Linux VM /etc/passwd file permissions are set to 0644

## <a name="a925-review-of-user-access-rights"></a>A.9.2.5: обзор прав доступа пользователя

Azure реализует [управление доступом на основе ролей](../../../../role-based-access-control/overview.md) (RBAC), что упрощает управление тем, кто получает доступ к ресурсам Azure. С помощью портала Azure вы можете просмотреть, у кого есть доступ к ресурсам Azure, а также их разрешения. В рамках этой схемы присваиваются четыре определения [Политики Azure](../../../policy/overview.md) для аудита учетных записей, у которых должен быть приоритет для проверки, включая нерекомендуемые учетные записи и внешние учетные записи с повышенными разрешениями.

- [Предварительный просмотр]: Audit deprecated accounts on a subscription
- [Предварительный просмотр]: Audit deprecated accounts with owner permissions on a subscription
- [Предварительный просмотр]: Audit external accounts with owner permissions on a subscription
- [Предварительный просмотр]: Audit external accounts with write permissions on a subscription

## <a name="a926-removal-or-adjustment-of-access-rights"></a>A.9.2.6: удаление или корректировка прав доступа

Azure реализует [управление доступом на основе ролей](../../../../role-based-access-control/overview.md) (RBAC), что упрощает управление тем, кто получает доступ к ресурсам Azure. С помощью [Azure Active Directory](../../../../active-directory/fundamentals/active-directory-whatis.md) и RBAC вы можете обновить роли пользователей в соответствии с изменениями в организации. При необходимости можно заблокировать вход в учетные записи (или удалить их). При этом вы немедленно лишаетесь прав доступа к ресурсам Azure. В рамках этой схемы присваиваются два определения [Политики Azure](../../../policy/overview.md) для аудита нерекомендуемой учетной записи, рассматриваемой в качестве кандидата на удаление.

- [Предварительный просмотр]: Audit deprecated accounts on a subscription
- [Предварительный просмотр]: Audit deprecated accounts with owner permissions on a subscription

## <a name="a943-password-management-system"></a>A.9.4.3: система управления паролями

Зная о виртуальных машинах с нарушением политики надежности паролей, вы сможете выполнить корректирующие действия, чтобы убедиться, что пароли для всех учетных записей пользователей виртуальных машин соответствуют требованиям политики.

- [Предварительный просмотр]: Deploy VM extension to audit Windows VM should not allow previous 24 passwords
- [Предварительный просмотр]: Audit Windows VM enforces password complexity requirements
- [Предварительный просмотр]: Audit Windows VM maximum password age 70 days
- [Предварительный просмотр]: Audit Windows VM minimum password age 1 day

## <a name="a1011-policy-on-the-use-of-cryptographic-controls"></a>A.10.1.1: политика использования элементов управления шифрования

В рамках этой схемы вам позволено принудительно применить политику по использованию элементов управления шифрования, присвоив 13 определений [Политики Azure](../../../policy/overview.md), которые принудительно применяют конкретные элементы управления шифрованием и ведут аудит использования параметров слабого шифрования.
Зная, где в ресурсах Azure шифрование может быть настроено неоптимально, вы сможете выполнить корректирующие действия, чтобы настроить ресурсы в соответствии со своей политикой информационной безопасности. В частности, в политиках, присвоенных в рамках этой схемы, требуется шифрование для учетных записей хранилища больших двоичных объектов и учетных записей хранения озера данных; требуется прозрачное шифрование данных для баз данных SQL; выполнение аудита на отсутствие шифрования в учетных записях хранения, базах данных SQL, дисках виртуальной машины и переменных для учетной записи службы автоматизации; выполнение аудита небезопасных подключений к учетным записям хранения, приложениям-функциям, веб-приложениям, Приложениям API и Кэшу Redis; выполнение аудита слабого шифрования пароля виртуальной машины и аудита незашифрованного обмена данными с Service Fabric.

- [Предварительный просмотр]: Audit HTTPS only access for a Web Application

- [Предварительная версия.] Аудит развертывания агента анализа журналов в VMSS — образ виртуальной машины (ОС) отсутствует в списке

Адаптивное управление приложениями — решение из Центра безопасности Azure, помогающее управлять приложениями, которые можно запускать на виртуальных машинах, размещенных в Azure. В рамках этой схемы присваивается определение Политики Azure, которое отслеживает изменения в наборе разрешенных приложений. Ограничения на установку программного обеспечения позволяют снизить вероятность появления уязвимостей в программном обеспечении.

- [Предварительный просмотр]: Monitor possible app Whitelisting in Azure Security Center
